# 12: Locks, Latches, Spinlocks: データベースエンジンにおける同時実行制御

## エグゼクティブサマリー (Executive Summary)

現代のデータベース管理システムを支える同期メカニズムは、大きく分けて三つある。**Locks**、**Latches**、そして**Spinlocks**だ。この記事ではこの三者を、高レベルのキューイング理論からCPUのマイクロアーキテクチャに近い層まで、一段ずつ掘り下げて整理する。

**読めばわかること:**
- 論理的な一貫性を守るLocksと、物理的な一貫性を守るLatches/Spinlocksの根本的な違い。
- False Sharing、Cache Line Bouncing、コンテキストスイッチのコストなど、実際にスループットを削っている原因。
- B+TreeのLatch Crabbingアルゴリズム、Spinlockに使うExponential Backoff、そしてハードウェアトランザクショナルメモリ(HTM)へと続く進化の流れ。
- ロックフリーなデータ構造とキャッシュ最適化を、実際の設計にどう落とし込むか。

## 中核的な問題 (The Core Problem)

トランザクション量の多いマルチコアのデータベースシステムで、最も厄介な問題の一つは**リソース競合**だ。何千ものスレッドがB+TreeやBuffer Poolといった共有データ構造に同時にアクセスしようとする以上、データ破損を防ぐための同期メカニズムは避けて通れない。

ここで効いてくるのがアムダールの法則だ。システム全体の高速化は、直列にしか実行できない部分の割合によって頭打ちになる。ロック設計がまずいと、スレッドはひたすら待たされ続ける。さらに、同期プリミティブの選択を誤ると――たとえば時間のかかる処理にSpinlockを使ったり、逆にごく短い操作に論理Lockを持ち出したりすると――MESIプロトコルによるキャッシュの無効化が絶え間なく発生し、メモリバスの帯域幅を食いつぶしてしまう。

Locks、Latches、Spinlocksをきちんと区別して正しく使い分けなければ、コアを何百個も積んだサーバーで動かしているデータベースエンジンでさえ、シングルコアの旧世代システムより遅くなることすらある。

## 深い技術分析 (Deep Technical Analysis)

### 境界の分割: Locks vs. Latches vs. Spinlocks

データベースの世界でLocksとLatchesを混同するのは、基本的だがよくある間違いだ。両者は時間軸も対象も全く異なる目的のために存在する。

**1. Locks(論理ロック):**
- **目的:** トランザクション分離レベルに従い、データベースの論理的な一貫性を守る。
- **保護対象:** タプル(レコード)、ページ、テーブル。
- **保持期間:** 長い(トランザクション全体にわたることが多い)。
- **管理方法:** Lock Manager(大規模なハッシュテーブル)が担い、Wait-For Graphを使ったデッドロック検出もサポートする。

**2. Latches(物理ロック):**
- **目的:** RAM上の内部データ構造(B+Treeのノードなど)の物理的な一貫性を守る。
- **保護対象:** メモリ上のデータ構造、配列、リンクリスト。
- **保持期間:** 非常に短い(マイクロ秒からナノ秒単位)。
- **管理方法:** データ構造のメモリに直接組み込まれる。デッドロックを防ぐ責任はプログラマ自身にあり、Latch取得順序を厳格に守る必要がある。

**3. Spinlocks(マイクロアーキテクチャレベルのロック):**
- **目的:** 最も原始的な形のLatchで、ハードウェアに最も近いところで動く。
- **仕組み:** OSにコンテキストスイッチを要求する代わりに、ビジーウェイトループでメモリ変数を監視し続ける。

### B+TreeにおけるLatch Crabbing(Latch Coupling)アルゴリズム

Latchesはメモリやディスク上のデータを守る最前線の仕組みだ。典型的なB+Treeでは、あるスレッドがルートからリーフへ向かって読み進めている最中に、別のスレッドが挿入を行いノード分割でツリーの形が変わってしまうリスクが常につきまとう。

そこで使われるのが**Latch Crabbing**(カニ歩き)というテクニックだ。
1. スレッドが親ノードのLatchを取得する。
2. スレッドが子ノードのLatchを取得する。
3. 子ノードが「安全」かどうか(満杯でも空でもないか)を確認する。
4. 安全であれば、**親ノードのLatchを解放する**。
5. そのまま下へ進んでいく。

ただしボトルネックはルートノードに残る。すべてのスレッドが最初にルートでLatchを取り合うことになるからだ。これを緩和するため、現代のデータベースは**Optimistic Lock Coupling**を採用している。読み取り側はLatchを取る代わりにバージョンカウンタだけを読み、ノードを読んだ後にバージョンが変わっていたら(つまり誰かが書き込んでいたら)自動的にリトライする、という発想だ。

### Spinlockの解剖とMESIプロトコルが引き起こす悪夢

最下層では、SpinlockはCompare-And-Swap(CAS)のようなCPUのアトミック命令をもとに実装される。CAS命令は比較と更新を1クロックサイクルの中で完結させることを保証する。

ただ、CASループを回し続けること自体が物理的な問題を引き起こす。**MESI(Modified, Exclusive, Shared, Invalid)**というキャッシュコヒーレンシプロトコルが、コア間のL1/L2キャッシュの整合性を保っているからだ。
- コアAがSpinlockに書き込むCASを実行すると、その変数を含むキャッシュラインはModified(M)状態になる。
- ビジーウェイト中のコアB、C、Dは即座にそのキャッシュラインを無効化(Invalid)される。
- それらのコアはL3キャッシュかメインメモリから変数を再フェッチせざるを得ず、プロセッサ間バス(Intel QPIやAMD Infinity Fabric)全体にボトルネックが波及する。

**対策: Test-and-Test-and-Set (TTAS) とExponential Backoffの組み合わせ**

```cpp
class ExponentialBackoffSpinlock {
private:
    std::atomic<bool> lock_flag{false};

public:
    void lock() {
        int backoff_time = 1;
        while (true) {
            // Test: L1 キャッシュ上でローカルにスピンし、メモリバスを妨害しない
            while (lock_flag.load(std::memory_order_relaxed)) {
                _mm_pause(); // ハイパースレッディング (Hyper-threading) の救世主
            }
            // Test-And-Set: ロックが解除されているのを見たときだけ、高価な命令で書き込みを試みる
            bool expected = false;
            if (lock_flag.compare_exchange_weak(expected, true, std::memory_order_acquire)) {
                return;
            }
            // Backoff: 競合がある場合、指数関数的に増加する時間だけスリープする
            for (int i = 0; i < backoff_time; ++i) _mm_pause();
            backoff_time = std::min(backoff_time * 2, 1024);
        }
    }
    void unlock() { lock_flag.store(false, std::memory_order_release); }
};
```

`_mm_pause()`が入っているのには理由がある。これはCPUに「ここはビジーウェイトループだ」と知らせ、投機的実行によるメモリ順序違反を防ぎつつ、兄弟スレッドにALUリソースを譲るためのものだ。

### False Sharingとメモリアライメント

もう一つ見落としやすい問題が**False Sharing**だ。異なるデータ領域を守っているはずのロックAとロックBが、たまたま同じキャッシュラインブロック(通常64バイト)に乗ってしまうケースで起きる。コア1がロックAを、コア2がロックBを操作しているだけなのに、ハードウェアは変数単位で区別できないため、互いのキャッシュライン全体を絶えず無効化し合う(Cache Line Bouncing)。

C/C++でSpinlockを並べて配置する場合は、`alignas(64)`を使って各Spinlockが独立したキャッシュラインに収まるよう離しておく必要がある。これを怠るとFalse Sharingの被害を直接受けることになる。

### FutexとHardware Transactional Memory (HTM)

スレッド数が処理能力を超えると、Spinlockは単にCPUを無駄に燃やすだけの存在になる。Linuxカーネルはここで**Futex(Fast Userspace Mutex)**というハイブリッドな解決策を用意している。競合がない間はユーザースペースでアトミックにロックを取得し、競合が長引いた場合だけ`futex_wait`を呼んでカーネルにスレッドをスリープさせ、CPUを他のタスクに譲る。起床時は`futex_wake`が使われる。

同期の次の段階を担うのが**HTM(Hardware Transactional Memory、Intel TSXなど)**だ。HTMを使うと、クリティカルセクションをLatchなしで実行できる。ハードウェアがL1キャッシュをマイクロレベルで監視し、干渉するスレッドがなければ変更をそのまま「コミット」する。競合が検出された場合はアボートし、レジスタの状態を即座に復元する。CASのオーバーヘッドをなくし、ペシミスティックなロックをハードウェアレベルのオプティミスティックロックへと置き換える発想だ。

## 学んだ教訓と実践 (Lessons Learned)

1. **コンテキストスイッチのコストを侮らないこと。** Spinlock/Latchの代わりに従来のMutex/Lockを使うと、1回のコンテキストスイッチだけで数マイクロ秒かかる。インメモリデータベースの世界では、その数マイクロ秒でCPUは何千という演算を済ませられる。ごく短い処理にはSpinlockを使うべきだ。
2. **キャッシュラインを意識すること。** Spinlockの配列を何も考えずに並べて初期化してはいけない。`alignas(64)`などのパディングでFalse Sharingを避ける。ハードウェアへの無理解は、そのままソフトウェアの性能低下として跳ね返ってくる。
3. **Optimisticを基本線にすること。** ツリーやリスト全体をロックするより、Optimistic Lock Coupling(バージョンカウンタのチェック)を使うほうが良い。何千ものスレッドをブロックするくらいなら、たまに読み違えてリトライするほうがずっとましだ。
4. **Latchにはデッドロックマネージャーが存在しないと心得ること。** Locksと違い、Latchの取得順序を誤って設計しても誰も救ってくれない。物理的なデッドロックが発生すれば、プロセス全体が永久にハングする。マルチスレッドのコードでは、厳格な階層的順序を最初から決めておくこと。

## 結論 (Conclusion)

Locks、Latches、Spinlocksをめぐる技術は、決して抽象的な学術用語ではない。世界中の巨大なトランザクションシステムを実際に動かしている、細かな歯車とクランクシャフトのようなものだ。データ基盤から金融プラットフォームまで、ソフトウェアのデータ構造とマイクロプロセッサの物理特性(NUMA、キャッシュコヒーレンシ)の交差点を理解しているかどうかが、普通のプログラマーと優れたシステムアーキテクトを分ける境界線になる。同期システムの設計とは、単に安全なコードを書く作業ではなく、シリコンの物理法則と向き合う仕事そのものだ。
