# 16: Redisのシングルスレッドアーキテクチャ: なぜ高速なのか、そしていつボトルネックになるのか？

## 概要

Redisのシングルスレッドアーキテクチャは、最近のデータベース設計の中でもかなり逆張りな選択だ。他のインメモリシステムの多くはコア数を増やし、スレッドを増やす方向に進んでいるのに、Redisはコマンド実行の経路をあえて1本のスレッドに絞り込み、それでいて毎秒数十万オペレーションをさばいてみせる。これは偶然ではなく、ハードウェアのメモリ階層とOSのネットワークスタックが負荷下でどう振る舞うかを深く理解した上での設計判断だ。

このドキュメントでは、その選択がなぜうまくいくのか、そしてどこで限界に突き当たるのかを順に見ていく。

- **コアとなる設計判断:** なぜマルチスレッドを避けたのか。I/Oマルチプレキシング（`epoll`）とReactorパターンの組み合わせが、1本のスレッドで数万接続をどう捌くのか。
- **データ構造:** `redisObject`によるポリモーフィックなエンコーディングの仕組みと、`listpack`がポインタのオーバーヘッドをどう削るか。
- **メモリ管理の細部:** Jemallocが外部断片化を実質的に問題にしないレベルまで抑え込む仕組み。
- **破綻するポイント:** $\mathcal{O}(N)$コマンド、Transparent Huge Pages（THP）が引き起こすTLBミスの停滞、`fork()`によるレイテンシスパイク。
- **Redis 6.0以降の回答:** 単一スレッドのネットワーク帯域の天井を越えるために作られたハイブリッドなThreaded I/Oモデル。

## コアな問題

コンピュータサイエンスには長年の反射神経がある。ソフトウェアを速くしたければスレッドを増やせ、というものだ。MySQLやPostgreSQLといった従来型のデータベースサーバーも大体この路線を踏襲していて、1接続1スレッド、あるいはスレッドプールで並列性を稼いでいる。

ところがRedisはデータをすべてRAM上に置き、アクセスレイテンシは1マイクロ秒を大きく下回る。この条件下では計算が変わってくる。マルチスレッドが持ち込む2つのコストは、データがすでにメモリ上にある場合にこそ効いてくる。

1. **コンテキストスイッチ。** OSはスレッド間の切り替えに何千ものCPUサイクルを費やし、そのたびにキャッシュが汚染され、L1/L2のヒット率が落ちる。
2. **ロック競合。** 共有メモリに触れるスレッドは安全のためにmutexやspinlockを必要とする。アムダールの法則はここでは容赦がない。ロック待ちのスレッドが増えるほど、並列化で得られるはずの効果は削られていく。

つまりRedisが向き合った設計上の問いは、「同期の複雑さを払わずに数万の同時リクエストをどう捌くか」だった。その答えは、問題そのものを回避すること――計算コアを単一スレッドで動かし、ロックを不要にしてしまうことだった。

## シングルスレッドアーキテクチャの実際の仕組み

### I/Oマルチプレキシングとイベントループ（Reactorパターン）

接続ごとにスレッドを立てる代わりに、Redisは`epoll`（Linux）や`kqueue`（FreeBSD）といったOSの低レベルAPIをベースにした非同期のI/Oマルチプレキシングでネットワーク処理を行う。

内部では、`epoll`はファイルディスクリプタを赤黒木で管理し、準備完了イベントを二重リンクリストに保持する。NICがTCPパケットを受信すると、ハードウェア割り込みがTCPスタックを起動し、カーネルは$\mathcal{O}(1)$の時間でソケットを準備完了リストにプッシュする。

Redisのイベントループ（`ae`ライブラリ）はほとんど拍子抜けするほどシンプルだ。
```c
void aeMain(aeEventLoop *eventLoop) {
    while (!eventLoop->stop) {
        // epollが準備完了ソケットの配列を返すのを待つ
        aeProcessEvents(eventLoop, AE_ALL_EVENTS | AE_CALL_AFTER_SLEEP);
    }
}
```
ループを1周するたびに、Redisは準備完了ソケットの配列を受け取り、ネットワークからバイトを読み込み、RESPプロトコルを解析し、RAM上でコマンドを実行し、結果を書き戻す――これがすべて1本のスレッド上で順番に行われる。このシングルスレッド実行こそが、ロックを一切必要とせずに原子性をタダで手に入れている理由だ。

### ポリモーフィックな構造とJemallocによるメモリ最適化

Cの素の`malloc`は時間とともに断片化を起こしやすいことで知られている。Redisは**Jemalloc**と組むことでこれを回避している。Jemallocは固定サイズクラスのバケットから割り当てを行い、外部断片化を最小限に抑える。実際、Redisの断片化率 $F = \frac{RSS}{Allocated}$ はおおむね1.05前後に収まり、理想値にかなり近い。

データ構造の面では、Redis内のすべての値は`redisObject`という共通の入れ物にラップされている。
```c
typedef struct redisObject {
    unsigned type:4;
    unsigned encoding:4; // <-- ポリモーフィズムの鍵
    unsigned lru:LRU_BITS;
    int refcount;
    void *ptr;
} robj;
```
本当に工夫が凝らされているのは`encoding`フィールドだ。HashやSorted Setの要素数が少ないうちは、Redisはわざわざフルのハッシュテーブルやスキップリストを使わず、**`listpack`**――ポインタのオーバーヘッドがない連続バイト配列――に詰め込む。CPUが`listpack`を線形にスキャンするため、空間局所性のおかげでハードウェアプリフェッチャがほぼタダでL1キャッシュに全体を読み込める。構造が一定の閾値を超えると、Redisはひそかに`encoding`をフルのハッシュテーブル表現に切り替える。

### インクリメンタルリハッシュ: コストを分散させる

ハッシュテーブルの負荷係数が$\alpha > 1.0$を超えると、より大きなバッキング配列が必要になる――リハッシュだ。これを愚直にやると、構造をロックして一度に$\mathcal{O}(N)$のコピーを行う、いわゆるストップ・ザ・ワールドになる。シングルスレッドではこの種の停止は許容できない。

Redisはこれを**インクリメンタルリハッシュ**で回避する。辞書構造（`dict`）は`ht[0]`と`ht[1]`という2つのハッシュテーブルを並行して保持する。拡張が必要になった時点で`ht[1]`はすぐに確保されるが、データは一度には移動しない。代わりに、以降`dict`に触れるコマンドのたびに、副作用としていくつかのバケットが`ht[0]`から`ht[1]`へ少しずつ移される。この償却$\mathcal{O}(1)$のアプローチは、本来ひとつの大きなコストになるはずのものを何千もの微小な増分に分散させ、レイテンシが跳ねることなくナノ秒レベルで安定する。

### 本当のボトルネック: ネットワーク帯域幅と`fork()`

コマンド実行がシングルスレッドである以上、Redisのデプロイでは通常CPUが制約になることはほとんどない。実際の天井はネットワークハードウェアの帯域幅で、Redisは遅延時間の80%以上をカーネルのシステムコールとTCPバイトのコピーに費やしており、データ検索自体に使っているわけではない。

とはいえ、レイテンシを本気で壊しかねないものが2つある。

1. **暴走する$\mathcal{O}(N)$コマンド。** `KEYS *`はキースペース全体を走査する。シングルスレッドの上では、これが数秒間CPUを独占し、後ろに並ぶ他のすべてのコマンドをブロックしてしまう。
2. **`fork()`とTHPの組み合わせ。** RedisがRDBやAOFへ永続化する際、`fork()`を呼んでコピーオンライトの子プロセスを立ち上げる。OSで**Transparent Huge Pages（THP）**が有効になっている場合――通常の4KBページの代わりに2MBページを使う設定――Redisが行う書き込みのたびにカーネルは小さなブロックではなく丸ごと2MBのブロックをコピーすることになり、RAM帯域を圧迫し、最大500msにも及ぶレイテンシスパイクを引き起こす。

### Redis 6.0とThreaded I/Oの登場

ネットワークインターフェースが25Gbps〜100Gbpsの領域に入ると、単一のI/Oスレッドではもはや追いつけなくなった。Redis 6.0はここで本物のアーキテクチャ転換を行う――**Threaded I/O Mode**だ。

```mermaid

```

考え方はシンプルで、ネットワーク処理の純粋に機械的な部分――バイトの読み取り、RESPの解析――を補助的なI/Oスレッドのプールに任せる。解析済みのコマンドはロックフリーキューに積まれる。メインスレッドはこれまで通り動き続ける――逐次実行、ロックフリー、RAM上でロジックを実行し、結果をI/Oスレッドに戻して書き出させる。これは書き直しではなくハイブリッドだ。シングルスレッドの安全性はそのままに、ネットワークスループットはおよそ2倍以上（Redisの報告では約2.5倍）になる。

## 教訓

1. **I/OバウンドかCPUバウンドかを見極める。** Redisのアーキテクチャ全体がこの好例だ。ワークロードがインメモリデータへの本当に軽い操作であるなら、スレッドを増やしてもmutexのオーバーヘッドとコンテキストスイッチのコストが増えるだけになりがちだ。I/Oマルチプレキシングのループを1本のスレッドで包んだ方が、素朴にマルチスレッド化したものより速いことがある。
2. **`KEYS *`を本番環境に近づけない。** システムアーキテクトなら誰でも、`KEYS`や`FLUSHALL`のような$\mathcal{O}(N)$コマンドにアラートを仕掛けるか、そもそもリネームしてしまうべきだ。代わりにカーソルベースの`SCAN`を使って作業を小さく中断可能な単位に分割する。
3. **OSレベルのチューニングはコードと同じくらい重要。** ソフトウェアアーキテクチャを完璧に仕上げても、カーネルのデフォルト設定――典型例がTHP――で足をすくわれることがある。OSの設定をシステム設計の一部として扱い、後回しにしないことが欠かせない。
4. **I/Oとロジックを分離する。** Redis 6.0のThreaded I/Oモデルは、高スループットなシステム全般に通用するきれいなテンプレートだ。エンコーディングやパースといった重くてステートレスな作業はワーカースレッドに逃がし、状態を持つロジックは単一のロックフリーなコアに閉じ込めておく。

## 結論

Redisは単に速いキーバリューキャッシュというだけではない。不要な複雑さを積み増すのではなく削ぎ落としたときに何が起きるかを、実際に動く形で示した例だ。ロック競合を拒み、I/Oマルチプレキシングのイベントループに徹底的に寄り添い、データ構造の設計を最初からキャッシュ局所性に合わせることで、Redisは1本のよくできたスレッドが、はるかに多くのハードウェアを投入したシステムを上回れることを示している。ミニマリズムを丁寧に適用すること自体が、ひとつの高度なエンジニアリングだという教訓だ。
