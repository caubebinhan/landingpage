# RedisのZSETはskip listの上にどう組み立てられているか

## Executive Summary (概要 / Overview)

Redisが評価されてきたのは、単純な速さだけでなく、データ構造が実際のワークロードにどれだけうまく合っているかという点も大きい。**Sorted Set(ZSET)**はその代表例だ。重複のない要素の集合を保持し、それぞれに浮動小数点のスコアを紐づけ、自動的に順序を保つ。おかげで、要素数が数百万に達しても、ランク検索・スコア範囲検索・位置指定の取得がどれも高速なままだ。

このドキュメントでは、RedisがZSETを実際にどう組み立てているかを見ていく。結論を先に言うと、ハッシュテーブルと調整済みの**skip list**の組み合わせだ。Redisがなぜ平衡二分木を選ばなかったのか、skip listが自分の形を決める際の数学、`jemalloc`ベースの割り当て戦略、CPUキャッシュやOSとのやり取り、そして実運用でどこに使われているかまで順に見ていく。

## Core Problem Statement (中核となる課題)

検索・挿入・削除・範囲クエリを$\mathcal{O}(\log N)$でサポートしながら、ソートされた動的な集合を維持するというのは計算機科学の古典的な問題であり、RedisのエンジニアたちもZSETを設計する際に定番の候補を検討している。

1. **AVL木や赤黒木**は最悪計算量$\mathcal{O}(\log N)$を保証するが、木のバランスを保つこと自体がタダではない。挿入や削除のたびに回転と色の変更が発生する。Redisはシングルスレッドなのでロック競合は問題にならないが、それでも回転はCPUサイクルを消費し、レイテンシのジッターを増やす。範囲クエリ(`ZRANGE`)を実行するには、コストの高い中順走査か、各ノードを肥大化させる親ポインタの維持のどちらかが必要になる。

2. **B-TreeやB+-Tree**はキャッシュラインやOSのページサイズに合わせて調整されているので、ディスクベースのストレージには明らかな選択肢だ。しかしインメモリ構造では、ノードの分割やマージのたびにメモリコピーが発生し、実装の複雑さもRedisが掲げる「シンプルなソースコード」という方針に反する。

そこで1990年にWilliam Pughが発表した**skip list**がより魅力的な選択肢として浮上する。厳密な構造的バランスを強制する代わりに、挿入時のコイントスによる**確率的なバランス**で妥協する。実装がシンプルで、実用上高速、最下層の連結リストによって範囲クエリを自然にサポートし、チューニングもしやすい——これがRedisがZSETの土台にskip listを選んだ理由だ。

## Deep Technical Knowledge / Internals (詳細な技術知識 / 内部構造)

### dictとskip listの二段構え

RedisのZSETは単なるskip listではなく、同期した2つの構造の組み合わせだ。

- **`dict`**(ハッシュテーブル)は「この要素のスコアは?」という問いに$\mathcal{O}(1)$で答える。
- **`zskiplist`**は線形順序を保持し、ランククエリと範囲スキャンを$\mathcal{O}(\log N)$で実行できるようにする。

これは典型的な空間と時間のトレードオフだ。Redisはハッシュテーブルの分だけ余分なメモリを使うことで、`ZSCORE`が何も走査せずに済むようにしている。

### zskiplistNodeの内部:spanフィールド

Pughのオリジナルのskip list設計はランクのカウント問題を解決していなかった。数百万件の中で要素$X$が何番目にあるかを安く知る方法がなかったのだ。Redisはこれを、各`zskiplistLevel`に`span`フィールドを追加することで解決した。

実際のC言語の構造体はこうなっている。

```c
/* Skip Listのノードを定義する構造体 */
typedef struct zskiplistNode {
    sds ele;                              // 要素のコンテンツを含むSDS動的文字列
    double score;                         // ソートスコア (IEEE 754 倍精度)
    struct zskiplistNode *backward;       // ZREVRANGEをサポートするバックワードポインタ (level 0)
    struct zskiplistLevel {
        struct zskiplistNode *forward;    // フォワードポインタ (同じレベルの次のノードを指す)
        unsigned long span;               // スパン (スキップされた物理ノードの数)
    } level[];                            // ノードのレベルを含むFlexible Array Member
} zskiplistNode;

/* リスト制御ブロック */
typedef struct zskiplist {
    struct zskiplistNode *header, *tail;
    unsigned long length;                 // 要素の総数
    int level;                            // リストの現在の最高レベル
} zskiplist;
```

`span`は、現在のノードと`forward`ポインタが指す先のノードとの間に、ベースレベルで何個のノードが挟まっているかを記録する。`ZRANK`が実行されるときは、`header`から目的のノードまでの経路上の`span`値を単純に合計するだけでよく、本来$\mathcal{O}(N)$かかる走査が$\mathcal{O}(\log N)$の累積計算に置き換わる。

```mermaid

    subgraph Level_3_Expressway
        H3[Header] -- "span: 3" --> N3[Node C (Rank 3)]
        N3 -- "span: 2" --> T3[Tail]
    end
    subgraph Level_2_Transit
        H2[Header] -- "span: 1" --> N1[Node A (Rank 1)]
        N1 -- "span: 2" --> N3
        N3 -- "span: 2" --> T2[Tail]
    end
    subgraph Level_1_Base
        H1[Header] -- "span: 1" --> N1
        N1 -- "span: 1" --> N2[Node B (Rank 2)]
        N2 -- "span: 1" --> N3
        N3 -- "span: 1" --> N4[Node D (Rank 4)]
        N4 -- "span: 1" --> T1[Tail]
    end
    N1 -.->|backward| H1

    N4 -.->|backward| N3
```

### ランダムなレベル決定の裏にある数学

ZSETに新しい要素を挿入するたびに、そのノードが何層になるかを決める必要があり、Redisはベルヌーイ試行をシミュレートすることでこれを行う。

- すべてのノードは最低でもレベル1を持つ。
- もう1段上に上がる確率は$p = 0.25$。
- このコイントスは、失敗するか`ZSKIPLIST_MAXLEVEL`(32)に達するまで繰り返される。

あるノードがちょうど$k$層になる確率は幾何分布に従う。

$$ P(L=k) = p^{k-1}(1-p) $$

期待されるレベル数$\mathbb{E}[L]$は次の通り。

$$ \mathbb{E}[L] = \sum_{k=1}^{32} k \cdot p^{k-1}(1-p) \approx \frac{1}{1-p} $$

$p=0.25$を代入すると$\mathbb{E}[L] = 1.33$になる。つまり平均すると、1ノードあたりのポインタオーバーヘッドはわずか1.33層分だ。二分木が常に2本のポインタを固定で持つのと比べると、skip listの非対称な設計はメモリの点でかなり有利になる。

### 割り当て:jemallocとFlexible Array Member

Redisはヒープの割り当てを`jemalloc`(または`tcmalloc`)に任せている。断片化したワークロードをうまく扱えることが選定理由だ。

`zskiplistNode`はC99の**Flexible Array Member**という手法——構造体の末尾にある`level[]`——を利用しており、メタデータと全レベル分を含めたノード全体を1回の呼び出しで確保できる。

$$ \text{Size} = \text{sizeof}(zskiplistNode) + (k) \times \text{sizeof}(zskiplistLevel) $$

これにより各ノードのメモリは完全に連続し、内部断片化は起きない。その代償として、ZSET全体のリサイズが繰り返されると外部断片化のリスクが徐々に高まる。`jemalloc`はこれをかなりうまく吸収してくれるが、負荷の高いインスタンスでは`INFO MEMORY`経由で`mem_fragmentation_ratio`を監視しておく価値はある。

### CPUキャッシュが逆襲してくる場面

アルゴリズムとしては洗練されているskip listだが、現代のハードウェア上では明確な弱点がある。ポインタチェイシングへの依存度が高く、それはキャッシュに優しいCPUが最も苦手とする操作だ。

`zskiplistNode`のオブジェクトはヒープ上に散らばっているため、リストを辿ることは事実上ランダムなアドレスへのジャンプの連続になる。ジャンプのたびに64バイトのキャッシュラインのほとんどが無駄になり、次のノードのアドレスは事前に予測できない。結果として、L1/L2/L3のキャッシュミスとTLBミスが絶え間なく発生し、パイプラインはDRAMからのデータを待って何百サイクルもストールする。

これに対するRedisの答えが、**listpack**という別の構造だ。

### Listpack:Big-Oをキャッシュフレンドリーさと引き換える

`zset-max-listpack-entries`(デフォルト128)と`zset-max-listpack-value`(デフォルト64バイト)という2つの設定値を下回っている間、Redisはハッシュテーブルもskip listも一切作らず、ZSET全体を1本の連続したバイト列——**listpack**(旧`ziplist`の後継)——として保持する。

`[Header] [要素_1] [スコア_1] [要素_2] [スコア_2] ... [End]`

listpackへのルックアップは$\mathcal{O}(\log N)$ではなくフラットな$\mathcal{O}(N)$スキャンに退化するが、それでも実際には小規模な集合において数十倍速いことが多い。数百バイト程度の連続領域はL1/L2キャッシュに楽に収まり、シーケンシャルスキャンはハードウェアプリフェッチャの得意分野そのものだし、リサイズは`realloc`/`memmove`で行われ、CPUのベクトル命令でうまく処理される。設定した閾値を超えて初めて、Redisはこれを不可逆的に完全なskip listへ変換する。

### OSとの戦い:fork()とCopy-on-Write

Redisの永続化(RDBスナップショットやAOFの書き換え)は`fork()`でバックグラウンドプロセスを立ち上げることに依存し、Linuxはその結果生じるメモリ共有をページ単位のCopy-on-Writeで処理する。

これはskip listベースのZSETへの書き込みが始まるまでは問題ない。1回の挿入は、無関係な数十のノードに散らばった`span`と`forward`フィールドに触れることになり、それらのノードは物理的に分散しているため、たった1回の挿入が数十個の別々の4KBページにまたがるページフォールトを引き起こしうる——それぞれが実際のコピーを必要とする。これが積み重なると、Linuxのoom killerを招くほどのメモリ使用量の急増につながる。だからこそ、本番環境では`overcommit_memory = 1`と十分なRAMの余裕を確保することが欠かせない。

## Practical Applications & Case Studies (実用的なアプリケーションとケーススタディ)

### リアルタイムランキング

ゲームやeコマースのプラットフォームは、数千万ユーザーにわたって絶えず更新されるランキングにZSETを使う。スコア更新の`ZINCRBY`、上位K件を取る`ZREVRANGE`、現在の順位を取る`ZRANK`はすべてミリ秒未満で動作し、特に`ZREVRANGE`の速さは`span`メタデータがランク計算を安く解決してくれることにほぼ全面的に支えられている。

### スライディングウィンドウのレートリミッター

L7レイヤーのレート制限でよく使われるパターンは、マイクロ秒単位のタイムスタンプをスコアに、UUIDをメンバーに使う方法だ。`ZREMRANGEBYSCORE`で現在のウィンドウ外(たとえば60秒より前)のデータを削除し、`ZCARD`で残っている件数を即座に取得できる。安価で効果的なスライディングウィンドウ制限の作り方だ。

### 時系列データのインデックス

数百万のセンサーからテレメトリを収集するIoTプラットフォームでは、タイムスタンプをスコアとして使える。あるデバイスの$T_1$から$T_2$までのデータを取得する処理は、$\mathcal{O}(\log N + M)$の`ZRANGEBYSCORE`クエリになる。skip listは$T_1$までまっすぐジャンプし、そこから該当する$M$件を線形にスキャンする。

### 遅延タスクキュー

スコアを将来の実行時刻として使い、ワーカープロセスは`ZRANGEBYSCORE -inf <current_timestamp> LIMIT 0 1`でポーリングする。skip listはキューのサイズに関係なくこのルックアップを$\Theta(\log N)$に保ち、ワーカープール全体の応答性を維持する。

## Lessons Learned (得られた教訓)

1. **確率的なバランスで十分なことが多く、しかも扱いやすい。** 厳密な数学的バランスをランダムな幾何分布に置き換えることで、Redisは保守しやすく、レイテンシが安定していて、平衡木が必要とする調整のオーバーヘッドを避けられる構造を手に入れた。
2. **キャッシュの局所性は、賢い漸近計算量よりも効くことが思いのほか多い。** Listpackはその何よりの証拠だ。データセットが十分小さいうちは、連続メモリ上の単純な$\mathcal{O}(N)$スキャンが、ヒープに散らばった「賢い」$\mathcal{O}(\log N)$の探索を安定して上回る——データが大きくなって数学が勝ち始めるまでは。
3. **ポインタだらけの構造には隠れたOSレベルのコストがある。** インメモリのアルゴリズムがどれだけ優れていても、forkの際のページフォールトとCopy-on-Writeが実際の上限になる。大規模なRedisクラスタにRAMを割り当てる前に理解しておく価値がある。
4. **2つの構造を組み合わせる価値は十分にある。** `dict`と`zskiplist`を組み合わせるとメモリは余分にかかるが、順序を保ったままスコアの定数時間ルックアップが手に入る。状態を二重に持つことが正解になる好例だ。
