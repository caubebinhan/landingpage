# 長時間稼働データベースプロセスにおけるメモリフラグメンテーション: マイクロアーキテクチャへの影響と緩和策

## エグゼクティブサマリーと中核課題

高性能コンピューティングやデータエンジニアリングの現場では、長時間稼働するデータベースプロセスに忍び寄る**メモリフラグメンテーション(メモリ断片化)**が、静かだが厄介なボトルネックになる。スループットとレイテンシは何週間もかけてじわじわ悪化し、ある日突然レイテンシスパイクや壊滅的なOut-Of-Memory(OOM)クラッシュとして表面化することが多い。

問題の本質はこうだ。高い同時実行性を求められるOLTPやOLAPワークロード向けのDBMSは、毎秒数百万回という頻度でメモリの確保と解放を繰り返す。数週間、数ヶ月と稼働し続けるうちに、OSの下層にあるメモリアロケータはヒープ内部で構造的なエントロピーを避けられなくなる。

このエントロピーが壊すのはメモリアドレス空間の連続性だ。断片化が進むと、論理的には連続しているはずのデータ構造――B+Treeのノードや列指向配列など――が、物理的にはバラバラのページに散らばってしまう。この空間的な分散は、現代のCPUキャッシュ設計が前提とする空間的局所性を根こそぎ崩し、TLB(Translation Lookaside Buffer)ミス率を急上昇させ、MMU(Memory Management Unit)に何段ものページテーブルウォークを強いる。結果として、本来クエリを処理しているはずのCPUが、ただRAMを待つだけの時間を過ごすことになる。

この記事では、フラグメンテーションが進んだときにマイクロアーキテクチャレベルで何が起きるか、標準Cライブラリアロケータ(`glibc`)がなぜ十分でないか、そして現代のデータベースが組み込んでいる緩和策――Slabアロケータ、バッファプール、リージョンベースのアリーナ、Huge Pagesなどカーネルとの明示的なやり取りまで――を順に見ていく。

## メモリフラグメンテーションの理論的基盤

長時間稼働プロセスにおけるフラグメンテーションは、内部断片化と外部断片化という2つの異なる問題に分けられる。

### 内部断片化と外部断片化

1. **内部断片化**は、アロケータが要求されたサイズより厳密に大きいブロックを用意してしまうことで起きる。多くの場合、8バイトや16バイトといったハードウェアのアライメント制約や、アロケータ内部の「ビン」が定める量子化されたサイズ区分が原因だ。この未使用だが確保済みの領域は、実質的に使える物理メモリの割合を目減りさせる。
2. **外部断片化**は、確保済みのブロックの間に小さく非連続な空き領域が増えていく現象だ。これらの空き領域を合計すれば大きな確保要求を満たせるはずなのに、分断されているせいで満たせない。アロケータは`mmap`や`sbrk`でOSにさらにメモリを要求せざるを得なくなり、実際には十分な「空き」バイトを抱えているにもかかわらず、プロセスのRSS(Resident Set Size)だけが膨らんでいく。

### フラグメンテーションを数式で見る

フラグメンテーションの度合いを厳密に測るために、フラグメンテーション比率$\Phi$を定義してみる。$M_{total}$をOSから確保した総メモリ量、$M_{used}$を生存中のデータ構造が実際に使っている総メモリ量とすると、比率は次のように書ける。
$$\Phi = 1 - \frac{M_{used}}{M_{total}}$$

$\Phi$が時間$t$とともに単調に増加していくと、システムはある臨界点に近づく。その先では、サイズ$s$の要求に対する確保失敗確率$P_{fail}(s)$が、たとえ$M_{total} - M_{used} \gg s$であっても無視できない値になってしまう。

この確率は、空きブロックサイズの分布$F=\{f_1, f_2, \dots, f_n\}$を考慮した修正ポアソン分布でモデル化できる。
$$P_{fail}(s) = \prod_{i=1}^{n} (1 - H(f_i - s))$$
ここで$H(x)$はヘヴィサイドの階段関数だ。$P_{fail}(s)$が1に近づくと、アロケータはコストの高い圧縮(compaction)を起動せざるを得なくなり、メモリが本当に枯渇していればLinuxのOOM killerが発動することになる。

## マイクロアーキテクチャへの影響: TLBとキャッシュ階層

フラグメンテーションが進んだときにハードウェア側で起きることは深刻だ。現代のプロセッサは仮想アドレスから物理アドレスへの変換をキャッシュするために**TLB(Translation Lookaside Buffer)**に大きく依存している。

### TLBスラッシングとページテーブルウォーク

データベースプロセスの外部断片化が進むと、論理的には連続している配列が物理的には離れたページへ頻繁にマッピングされるようになる。この空間的分散が空間的局所性を破壊し、TLBミス率を跳ね上げる。

$T_{hit}$をTLBヒット時のレイテンシ、$T_{miss}$をTLBミス時のレイテンシ(4段階や5段階の基数木を辿るコストの高いページテーブルウォークが必要)とすると、実効メモリアクセス時間(EMAT)は次のように書ける。
$$EMAT = P_{hit} \cdot T_{hit} + (1 - P_{hit}) \cdot T_{miss}$$

フラグメンテーションが増えるほどTLBヒット率$P_{hit}$は急落する。MMUは頻繁にページテーブルを辿ることになり、CPUパイプラインはストールし、EMATは桁違いに膨れ上がる――L1 TLBヒットの約1nsから、主記憶からのフェッチを伴うページウォークの約100nsまで。

### L1、L2、L3キャッシュへの波及

この劣化はTLBだけにとどまらず、L1、L2、L3(LLC)のキャッシュ階層全体にも波及する。極端に断片化したヒープでは、本来なら同じ64バイトのキャッシュラインに収まっているべきデータが、複数のキャッシュラインに散らばってしまう。これがキャッシュの実効容量を減らし、キャッシュスラッシングを悪化させる。

連続的あるいは一定間隔のアクセスパターンを前提とするハードウェアプリフェッチャは、断片化した連結リストや不連続な物理ページに配置された配列を走査するときには、ほぼ何の役にも立たなくなる。

## アロケータ戦争: `glibc`、`jemalloc`、`tcmalloc`

標準Cライブラリのデフォルトアロケータ(`glibc malloc`)はptmallocをベースにした汎用アロケータだ。デスクトップアプリケーション向けには十分だが、長時間稼働するマルチスレッドのデータベースサーバーでフラグメンテーションを抑える能力はお世辞にも高くない。グローバルロックと共有アリーナに依存しているため、スレッド競合とフラグメンテーションの両方を招きやすい。

そのため、Redis、PostgreSQL、CockroachDBのような高性能データベースは、`jemalloc`(FreeBSD/Facebook発)や`tcmalloc`(Google製)といった専用アロケータを明示的にリンクしている。

### `jemalloc`のアーキテクチャ

`jemalloc`は割り当て頻度の高い環境でのフラグメンテーションと競合を抑えるために設計されており、いくつかの工夫がある。
1. **サイズクラス:** メモリを8バイト、16バイト、32バイト……といった離散的なサイズクラスに厳密に分ける。これにより内部断片化は数学的に最大約20%程度に抑えられる。
2. **スレッドローカルキャッシュ(tcache):** 各スレッドがロックフリーの独自メモリキャッシュを持つ。一定サイズ以下の確保はロックを取らずtcacheから即座に返され、$O(1)$の高速パスとコア間のフォールスシェアリング回避が両立する。
3. **アクティブパージ:** バックグラウンドでのパージ機構により、断片化した空きブロックを能動的にまとめてOSへ返却し、RSSのフットプリントを実際の使用量に近づけ続ける。

## アルゴリズム的緩和策: データベース独自のメモリアーキテクチャ

`jemalloc`のような優れたユーザー空間アロケータを使っても、アプリケーションの論理的なメモリビューとOSの物理メモリ管理との間には常にギャップが残る。高性能データベースの多くは、クリティカルパスで標準アロケータを完全に迂回し、独自のメモリ管理を実装している。

### Slabアロケータ

代表的な戦略の一つが、自前のSlabアロケータだ。Slab方式では、まずOSから「スラブ」と呼ばれる大きな連続チャンクを事前に確保する。

各スラブは均一サイズのスロットに分割され、特定の内部データ構造(トランザクションコンテキストやロック記述子など)専用に割り当てられる。オブジェクトの種類とサイズごとに厳格に分離することで、すべての確保が同じ寸法のスロットを扱うことになり、プール内での外部断片化が構造的に起きなくなる。

スロットが解放されるとロックフリーの単方向連結リストに戻される。スラブ単位の内部断片化は$S_{slab} \mod S_{obj}$で厳密に抑えられ、$S_{slab} \gg S_{obj}$であればほぼ無視できる。

### バッファプール(固定サイズページ)

主要なデータページについては、ほぼすべてのデータベースが固定サイズの**バッファプール**を採用している。バッファプール用のメモリは、初期化時に単一の大きな連続領域として確保される。

この領域は論理的に同一サイズのフレーム(PostgreSQLなら8KB、InnoDBなら16KB)に分割される。フレームサイズを厳密に固定することで、バッファプールはデータキャッシュに関する内部・外部両方の断片化を回避する。

ディスクから新しいページを読む必要があり、プールが満杯であれば、LRUやCLOCK-Proといったアルゴリズムが古いページを追い出し、まったく同じ連続フレームを新しいページに再利用する。このキャッシュは、稼働期間がどれだけ長くなろうとフラグメンテーションと無縁のままだ。

### アリーナアロケータ(リージョンベースのメモリ管理)

クエリ実行中――たとえば`JOIN`や`ORDER BY`を伴う`SELECT`――には、ハッシュテーブルやソートバッファのための動的な確保が発生する。これらの一時的な確保がグローバルヒープを断片化させないよう、エンジンは**アリーナアロケータ**(PostgreSQLの`MemoryContext`など)を使う。

アリーナアロケータはクエリ開始時に大きな連続ブロックを用意する。以降の確保要求はアリーナ内のポインタをインクリメントするだけで済み、オブジェクトごとのメタデータオーバーヘッドがない真の$O(1)$確保が実現する。

決定的なのは解放の仕方だ。アリーナ内の個々のオブジェクトは明示的に解放されることはない。クエリの実行が終わった時点で、アリーナ全体が一括で瞬時に回収される。この設計により、一時的なクエリメモリが長期的な外部断片化の原因になることは構造上あり得ない。

```cpp
template <typename T, size_t BlockSize = 4096>
class FragmentFreeArena {
private:
    struct Block {
        char data[BlockSize];
        size_t current_offset;
        Block* next;
        Block() : current_offset(0), next(nullptr) {}
    };
    Block* head_block;
    Block* current_block;

public:
    FragmentFreeArena() {
        head_block = new Block();
        current_block = head_block;
    }
    
    ~FragmentFreeArena() {
        Block* curr = head_block;
        while (curr != nullptr) {
            Block* next = curr->next;
            delete curr;
            curr = next;
        }
    }

    void* allocate(size_t size, size_t alignment = alignof(std::max_align_t)) {
        size_t current_ptr = reinterpret_cast<size_t>(current_block->data) + current_block->current_offset;
        size_t offset = (alignment - (current_ptr % alignment)) % alignment;
        
        if (current_block->current_offset + offset + size <= BlockSize) {
            void* ptr = current_block->data + current_block->current_offset + offset;
            current_block->current_offset += offset + size;
            return ptr;
        } else {
            Block* new_block = new Block();
            current_block->next = new_block;
            current_block = new_block;
            return allocate(size, alignment); 
        }
    }
    
    // 領域全体をO(1)で即座に解放する
    void reset() {
        Block* curr = head_block->next;
        while (curr != nullptr) {
            Block* next = curr->next;
            delete curr;
            curr = next;
        }
        head_block->next = nullptr;
        head_block->current_offset = 0;
        current_block = head_block;
    }
};
```

## OSとの付き合い方: Huge Pagesと`madvise`

メモリフラグメンテーションの管理は、OSカーネルの挙動と切り離しては考えられない。

### Transparent Huge Pages(THP)と明示的なHuge Pages

標準的なx86-64アーキテクチャは4KBページを使う。128GBのRAMを使うデータベースなら、これは3,350万件のページテーブルエントリに相当する。OSを**Huge Pages**(2MBや1GB)を使うよう設定すれば、エントリ数は劇的に減り、TLBのカバレッジは指数的に広がる。

ただし、Linuxの**Transparent Huge Pages(THP)**でメモリの連続性を積極的に追い求めると、厄介な副作用が出る。THPはカーネルスレッド(`khugepaged`)を通じて4KBページを2MBのHuge Pageへ動的に統合しようとする。

物理メモリが断片化している状態でカーネルがHuge Pageを確保しようとすると、**ダイレクトコンパクション**が走る。これは同期的でブロッキングな操作で、カーネルが物理ページを強制的に移動させている間、データベースプロセスは完全に止まってしまう。この停止は数百ミリ秒に及ぶこともあり、致命的なレイテンシスパイクにつながる。

ベストプラクティスとしては、**THPを完全に無効化**し(`echo never > /sys/kernel/mm/transparent_hugepage/enabled`)、代わりに明示的かつ静的に確保するHuge Pages(GRUBの`hugepages=N`)に頼るのが望ましい。これにより、連続したメモリ要件を事前に満たしておけるので、レイテンシの予測可能性が保たれる。

### `madvise`システムコール

データベースがメモリを解放しても、ユーザー空間のアロケータはそれを空きとしてマークするだけで、OS側は物理ページのマッピングを保持し続ける。人為的にRSSが膨らむのを防ぐため、`jemalloc`のようなアロケータは定期的に空き領域へ`madvise(MADV_DONTNEED)`や`MADV_FREE`を発行する。

このシステムコールは、指定した仮想アドレス範囲のページテーブルエントリを解体し、その下にある物理ページを回収するようカーネルに明示的に伝える。その後データベースが同じ仮想アドレスへアクセスすれば、カーネルはシームレスにゼロ埋めされた新しい物理ページを割り当てる。この仕組みによって、データベースは使っていないリソースをOSへ返却し、システム全体の安定性を保つことができる。

## 教訓とベストプラクティス

1. **DB用途にデフォルトの`glibc`をそのまま使わない:** C/C++/Rustでデータ集約型のアプリケーションを書くなら、`jemalloc`か`tcmalloc`を必ずリンクしよう。高い並行性の下での差は驚くほど大きい。
2. **Transparent Huge Pages(THP)は無効化する:** MongoDB、PostgreSQL、Redisなど主要なデータベースのほとんどで、THPは同期的なメモリコンパクションによる大きなレイテンシスパイクの原因になる。OSレベルでオフにしておくこと。
3. **バッファプールには明示的なHuge Pagesを使う:** 主要なデータキャッシュを事前確保のHuge Pagesに固定すれば、TLBスラッシングとOSレベルのスワッピングをほぼ排除でき、安定した性能を確保できる。
4. **リクエストごとにアリーナアロケータを使う:** Webサーバーやクエリエンジンを作るなら、リクエスト単位でアリーナからメモリを確保し、終わったらポインタをリセットするだけにする。これでフラグメンテーションが起きず、複雑なガベージコレクションのロジックも避けられる。
5. **RSSと実使用量を監視する:** データベースプロセスが10GBのデータを扱っていると報告しているのに`htop`上のRSSが30GBもあるなら、深刻なメモリフラグメンテーションか`madvise`設定の問題を抱えている可能性が高い。アロケータのバックグラウンドパージスレッドの設定を見直すべきタイミングだ。
