# 43: SSDのWrite Amplification: データベースがハードディスクを消耗させる仕組み

## 概要と中核的な問題

現代のデータセンターでは、NANDフラッシュベースのSSDが機械式のHDDをほぼ完全に置き換え、数百万IOPSという処理能力を実現している。しかしこの半導体には見過ごせない弱点がある。**寿命がプログラム/消去(P/E)サイクル数によって物理的に制限されている**という点だ。

問題の核心はシンプルだ。DBMSがOSに1ギガバイトのデータ書き込みを要求したとき、実際にドライブが書き込む物理的な量はそれよりずっと多くなることがある。この差分こそが**write amplification(ライトアンプリフィケーション、WA)**と呼ばれる現象だ。

MySQLやPostgreSQLのようなB-Tree構造に根ざした従来型データベースは、本質的にランダムな書き込みパターンを生む。この小さくバラバラなI/Oは、NANDフラッシュの「書き込む前に消去する」という物理的な制約と正面からぶつかる。結果として、write amplification factor(WAF)は10倍、20倍、時には30倍にまで跳ね上がることがある。数万ドル規模のSSDクラスタが、メーカーが約束する数年ではなく数ヶ月で物理的に寿命を迎えるということだ。

この記事では、フラッシュチップの量子レベルの物理現象から、FTL(Flash Translation Layer)がどうWAFを生み出すか、B-TreeとLSM-Treeでなぜここまで結果が違うのか、そしてZNS NVMeのような新しいアプローチが何を変えるのかを見ていく。

## フラッシュメモリのマイクロアーキテクチャ

現代のNANDフラッシュは、フローティングゲートMOSFETトランジスタ、あるいは多層3D NAND設計におけるCharge Trap Flash(CTF)技術の上に成り立っている。

### ファウラー・ノルドハイム・トンネル効果

微細構造のレベルでは、1ビットはフローティングゲート内に閉じ込められた電荷量として表現される。セルにデータを書き込むには――3ビットを表現するのに8段階のしきい値電圧を必要とするTLC NANDを例にすると――コントローラはコントロールゲートに高い正の電圧パルス($V_{prog}$、通常20V前後)を印加する。

この電圧が薄い酸化膜を貫くほど強い電場を作り出し、**ファウラー・ノルドハイム(FN)トンネル効果**によって電子をフローティングゲートへ強制的に移動させる。

消去はこの逆で、シリコン基板に大きな負電圧をかけて電子をフローティングゲートから引き抜く。これはかなり激しい電気的プロセスで、結晶格子にストレスを与え、絶縁層であるトンネル酸化膜を少しずつ摩耗させる。この摩耗こそがP/Eサイクル数の上限を決めている――TLCでおよそ3000回、QLCでは1000回未満だ。酸化膜が破れると電子は自然に漏れ出し、そのブロックは恒久的な不良ブロックとなり、データは失われる。

### PageとBlockのミスマッチ

NANDフラッシュの最も厄介な特性は、物理的な劣化そのものよりも、操作単位の不一致にある。
- 読み書きは**ページ**単位で行われる。通常4KBから16KBだ。
- 消去は**ブロック**全体――数千ページ分、4MBから16MB――に対してしか行えない。

さらに、すでに電荷を持つページへの上書きは物理的に不可能だ。ビットを0から1に変えるには、そのページが属するブロックをまるごと消去して空の状態に戻す必要がある。

## Flash Translation Layer(FTL)とガベージコレクション

HDDのように512バイト単位のセクタを自由に上書きできると想定しているOSとの折り合いをつけるため、SSDは**Flash Translation Layer(FTL)**と呼ばれるソフトウェア層を内蔵している。

FTLはSSD内蔵のDRAM上に大きなLogical-to-Physical(L2P)マッピングテーブルを保持する。OSがLBA 100への上書きを要求すると、FTLはその書き込みを新しい空の物理ページへリダイレクトし、古いデータを持っていたページを無効(ゴミ)としてマークする。

### WAFが実際に生まれる場所: ガベージコレクション

時間が経つと、各ブロックは有効なページとゴミページが混在した状態になる。空き容量が少なくなると**ガベージコレクション(GC)**が動き出す。
1. 最もゴミの多い「犠牲ブロック」を選ぶ。
2. そのブロックに残る有効ページを内部SRAMへ読み出す。
3. それらの有効ページを新しい空きブロックへ書き直す。
4. 20Vの高電圧パルスで犠牲ブロック全体を消去し、空間を取り戻す。

この静かな内部シャッフルこそが**write amplification**の実際の原因だ。SSDに4KBの新規書き込みを頼んだだけなのに、空き容量を作るためだけに4MBの古いデータを裏で書き直していることがある。

数式で表すと次のようになる。
$$WAF = \frac{\text{フラッシュに物理的に書き込まれたバイト数 (GCとHostによる)}}{\text{ホストによって論理的に書き込まれたバイト数 (OSによる)}}$$

$$WAF_{random} \approx \frac{1 + \alpha}{\alpha}$$
$\alpha$はオーバープロビジョニング比率だ。コンシューマー向けSSDでOPが7%しかない場合、$WAF \approx 15.2$になる。1TB書き込んだつもりが、実際には15.2TB分ドライブを消耗させていることになる。

## 悪条件が重なる場所: B-TreeデータベースとDoublewrite Buffer

ハードウェアFTLによるwrite amplificationは氷山の一角にすぎない。実際の総アンプリフィケーション係数は、複数の層をまたいだ積になる。
$$WAF_{total} = WAF_{DB} \times WAF_{FS} \times WAF_{SSD}$$

MySQL(InnoDB)やPostgreSQLがSSDにとって特に厳しい存在なのは、両者ともB+Treeアーキテクチャに根ざしているからだ。

### Torn PagesとDoublewrite Bufferの役割

B-Treeデータベースはデータを固定サイズの論理ページにまとめる(InnoDBで16KB、Postgresで8KB)。ところがファイルシステムやハードウェアはより小さい単位(通常4KB)でI/Oを割り当てる。16KBのページをディスクへフラッシュ中に電源が落ちれば、実際には4KBしか書き込めておらず、残り12KBは古いデータのままということが起こり得る。これが「torn page」だ――ログがあっても修復できない構造的な破損である。

この問題への対処法は次のとおり。
- **MySQL InnoDB**は**Doublewrite Buffer(DWB)**を使う。16KBのページを`.ibd`データファイルへ書き込む前に、まずその16KB全体を安全なDWB領域へシーケンシャルに書き込み、それが完了してから改めて`.ibd`へ書き込む。
- **PostgreSQL**は**Full Page Writes(FPW)**を使う。チェックポイント後の最初の変更では、たとえ1文字の変更でもページ全体(8KB)をそのままWALへコピーする。

**このパターンの$WAF_{DB}$を計算してみる:** `UPDATE users SET age = 30 WHERE id = 1`のように、約100バイトだけを変更するケースを考える。MySQLは次のように動く。
1. WALに100バイトを書き込む。
2. Doublewrite Bufferに16KBをフラッシュする。
3. データファイルに16KBをフラッシュする。

$$WAF_{DB} = \frac{100 \text{ (Log)} + 16384 \text{ (DWB)} + 16384 \text{ (Data)}}{100 \text{ (元のペイロード)}} = 328.6 \text{ 倍}$$

この328.6倍の書き込み量(その大半はランダムI/O)がSSDに叩き込まれると、さらにハードウェアのGC($WAF_{SSD} = 3.0$)が上乗せされる。合計すると$WAF_{total} = 328.6 \times 3.0 = 985.8$。100バイトの更新のために、ドライブは1メガバイート近い寿命を消費してしまう。

## LSM-Treeという逃げ道: RocksDB、Cassandra

B-Treeが行き詰まる一方で、RocksDB、ScyllaDB、Cassandraのような新世代データベースは**LSM-Tree(Log-Structured Merge-Tree)**を採用し、設計そのものでこの問題の大部分を回避している。

### インプレース更新をやめてAppend-Onlyへ

LSM-Treeにはインプレース更新という概念がそもそも存在しない。すべての書き込み・更新・削除は次のように処理される。
1. RAM上のMemTableへシーケンシャルに追記される(同時にWALへも高速にバックアップされる)。
2. MemTableが満杯になると(例えば64MB)、凍結されて大容量のシーケンシャルI/Oでディスクへフラッシュされ、読み取り専用のSorted String Table(SSTable)になる。

この大量のシーケンシャルI/Oは、SSDのFTLにとってほぼ理想的だ。データがまとまった連続したブロックとして届くため、局所的なゴミの断片化が起きにくい。結果として$WAF_{SSD}$は$1.0$近くまで下がる。

### 引き換えに来るCompactionのコスト

LSM-Treeは現在の書き込み性能を前借りし、その代わりに**Compaction**という形で後から代償を払う。古いレコードや削除マーカー(Tombstone)を含むSSTableが増えすぎると、CompactionはそれらをRAMへ読み込み、マージソートしてゴミを取り除き、新しいSSTableとしてシーケンシャルに書き出す。

データがより深い階層($L_1, L_2, L_3...$)へ次々と移動していくため、$WAF_{DB}$はコンパクション戦略(LeveledかTieredか)によって10倍から30倍程度になる。数字だけ見ると高く感じるが、ディスクに到達するI/Oパターンは依然として100%シーケンシャルだ。おかげでSSDコントローラは、ランダム書き込みが引き起こすようなガベージコレクションの遅延スパイクをほぼ経験せずに済む。

## この先の姿: Zoned Namespaces(ZNS)NVMe

インフラがクラウドのハイパースケール領域に近づくにつれ、従来のFTLがフラッシュの物理的実態を隠蔽するやり方は、予測しづらいテールレイテンシという形で問題を表面化させ始めている。ここで注目されているのが**Zoned Namespaces(ZNS)NVMe**という規格だ。

ZNSはFTLのマッピング層を実質的に取り払い、ドライブの物理構造を「ゾーン」という単位でOSとデータベースに直接見せてしまう。

ルールはかなり厳格だ。
1. ゾーンへの書き込みはシーケンシャルなappend-onlyでなければならない。
2. インプレースの上書きは許されない。
3. ゾーン内の空間を取り戻したい場合、ホストは`Zone Reset`コマンドを発行する。これによりそのハードウェアブロックへ高電圧の消去が走り、空の領域が返却される。

結果として$WAF_{SSD}$は正確に**1.0**に固定される。バックグラウンドのGCも、L2Pテーブル維持のためのRAM消費も要らない。アンプリフィケーションの管理は完全にLSM-Tree側(たとえばZNS対応版のRocksDB)に委ねられ、バイト単位で制御できるようになる。

## システムエンジニアへの教訓

WAFをコントロールすることは、Database/DevOps/SREを担う人間にとってほぼ必須のスキルだ。インフラ予算とレイテンシの安定性、その両方に直結する。

1. **コンシューマーSSDとエンタープライズSSDの違いを理解する:** エンタープライズSSDはオーバープロビジョニング率が高く(約28%)、L2P用のRAMも多く、FTLアルゴリズムも洗練されている。コンシューマーSSDでMySQLを動かすと、WAFが跳ね上がりドライブがあっという間に消耗する。
2. **Doublewrite Bufferの無効化を検討する:** ZFSのようなatomic writesに対応したファイルシステムや、torn page対策を持つNVMeコントローラを使っているなら、`innodb_doublewrite=0`にすることで16KB分のWAFを根本から取り除ける。
3. **ブロックサイズを揃える:** ファイルシステムのブロックサイズとデータベースのページサイズを、SSDのハードウェアセクタサイズ(通常4KB)と一致させる。ずれがあると物理I/Oが不必要に倍増する。
4. **用途に合わせてファイルシステムを選ぶ:** 組み込みやモバイル向けにはF2FS(Flash-Friendly File System)が適している。log-structuredな設計でappend-onlyの書き込みを行うため、eMMCフラッシュとの相性が良い。サーバー用途ではXFSやext4でも十分だが、I/Oパターンは常に観察しておきたい。
5. **ワークロードに合わせてエンジンを選ぶ:** IoTのデータ収集やログサーバーのように書き込みが全体の8割を超えるようなら、B-Treeエンジンは避けたほうがいい。Cassandra、ScyllaDB、InfluxDBのようなLSM-Treeエンジンのほうが、シーケンシャル書き込みの恩恵を活かせてドライブにも優しい。
