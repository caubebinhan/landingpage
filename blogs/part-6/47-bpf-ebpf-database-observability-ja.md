# 47: データベースObservabilityにおけるBPFとeBPF: カーネルの内側からデータベースを診る

## 概要と中核的な問題

分散マイクロサービスや高負荷なトランザクション処理の世界では、ボトルネックの正体はたいていデータベースにある。それにもかかわらず、実際に何が起きているかを診断するのは今なお難しい。

問題の根っこは、従来のデータベースが基本的に「ブラックボックス」として振る舞うことにある。SQLクエリがなぜ遅いのかを知ろうとすると、エンジニアはスロークエリログを有効にしたり、インバンドのプロファイリングエージェントを入れたり、`strace`や`tcpdump`といった昔ながらのOSツールに頼ったりすることになる。
- ログやインバンドエージェントを有効にすると、たいてい30〜50%のオーバーヘッドが乗り、原因を特定する前にシステム自体が悲鳴を上げる。
- `ptrace`ベースの`strace`は、システムコールのたびにOSがデータベースプロセスを止めるため、レイテンシが何百倍にも膨らむ。
- `tcpdump`は大量の生パケットを吐き出し、カーネルからユーザー空間へのコピーだけでCPUを大きく消費する。

**eBPF(Extended Berkeley Packet Filter)**は、このデータベースobservabilityの前提を変えた。エンジニアはカーネルの内部で直接、小さく検証済みのプログラムを走らせられるようになり、NICを通過するパケット、PostgreSQL内のあらゆる関数呼び出し、SSDへ書き込まれるブロックまで、ほぼゼロ――多くの場合1%未満――のオーバーヘッドで追跡できる。

この記事では、eBPF仮想マシンのマイクロアーキテクチャ、uprobeとkprobeを使って稼働中のデータベースへ再起動なしにフックする方法、そしてデータセンター規模でeBPF observabilityを運用したときに実際に何が壊れるかを見ていく。

## eBPF仮想マシンのマイクロアーキテクチャ

eBPFはもともとネットワークパケットのフィルタリング(tcpdumpの中のcBPF)として生まれ、今ではLinuxカーネルの中枢で動く汎用の仮想マシンへと成長した。

### ハードウェアにほぼ1対1で対応する命令セット

現代のeBPFアーキテクチャは、x86_64やARM64といった64ビットプロセッサの構造にかなり近い形でマッピングされている。eBPFの「プロセッサ」は11個の仮想64ビットレジスタ($R0$から$R10$)で動く。
- $R0$: 戻り値を保持する。
- $R1$〜$R5$: 関数呼び出し時の引数を保持する。
- $R6$〜$R9$: 関数呼び出しをまたいで保存されるレジスタ。
- $R10$: スタックへアクセスするための読み取り専用フレームポインタ。

Just-In-Time(JIT)コンパイルがeBPFバイトコードをネイティブなマシンコードに変換する。eBPFのレジスタモデルがx86_64にかなり近いため、このJITは高速に完了し、生成されるコードは直接コンパイルしたCコードとほぼ同等の速度で動く。

### 門番としてのVerifier

カーネルはバグを許さない領域だ。ゼロ除算、NULLポインタ、無限ループがカーネルコードに紛れ込めば、システム全体をクラッシュ(Kernel Panic)させかねない。だからeBPFプログラムは実行前に必ず**Verifier**を通過しなければならない。

Verifierは静的な制御フローグラフ(CFG)解析を使って、次を保証する。
1. 無限ループがないこと――プログラムは必ず終了することが証明できなければならない(eBPFは意図的にTuring完全ではない)。
2. 配列外へのメモリアクセスがないこと。
3. 許可されていないカーネルメモリ領域へのアクセスがないこと。
4. スタック使用量が512バイトを超えないこと。

Verifierの計算量はおおよそ$\mathcal{O}(N \times E)$($N$は状態数、$E$はフローグラフの辺の数)に収まる。安全と判定されて初めて、プログラムはJITへ渡される。

## eBPF Maps: コピーなしでデータを共有する

カーネル内で動くeBPFプログラムがクエリのレイテンシを収集したとして、それをコピーのコストなしにユーザー空間(Prometheusエクスポーターなど)へ届けるにはどうすればいいか。その答えが**eBPF Maps**だ。

Mapはカーネルのページアウトされない物理RAM上に確保されるキーバリュー構造だ。eBPFプログラムとユーザー空間のプロセスは、ファイルディスクリプタを介して同じMapを読み書きできるため、データコピーのためのシステムコールがまるごと不要になる。

### Hash Map(RCU保護)

Hash Mapは状態を保持するために使う。典型的な使い方はこうだ。リクエストパケットが届いたとき、eBPFはTCPタプル(IP、ポート)をキーにして`Time_Start`をHash Mapへ書き込む。レスポンスが送出されるとき、そのキーを引いてレイテンシを計算し記録する。

内部のハッシュ機構はRead-Copy-Update(RCU)を使っており、ロックフリーの同時読み取りを実現し、CPU競合なしに数千万IOPS規模のアクセス速度を出せる。

### Ring Buffer

カーネルからユーザー空間へイベント(遅いクエリの一覧など)をストリーミングするには、eBPFはRing Bufferを使う。これはロックフリーのMPSC(Multi-Producer Single-Consumer)構造の循環キューだ。イベントを生成するCPUコアは直接Ring Bufferへ書き込み、ユーザー空間のエージェントは`epoll`によるポーリングで取り出していく。ユーザー側が詰まっても、カーネル側がブロックされることはない。

## どこにフックを仕込むか

eBPFはデータベースの動きをどうやって観測しているのか。答えは、システムの様々な地点に用意されたフックだ。

### ネットワークフック(TC / XDP / Socket Filter)

データベースのプロセス自体には触れたくない、あるいは触れられない場合、ネットワークレイヤーで観測する手がある。eBPFはTraffic Control(TC)層や、NICドライバの最下層にあたるeXpress Data Path(XDP)にアタッチできる。PostgreSQLやMySQLのワイヤープロトコルに沿ってTCPを解析すれば、データベース自身に気づかれることなくクエリのレイテンシ、件数、再送率を測定できる。オーバーヘッドは非常に低いが、複数のTCPパケットにまたがって断片化したクエリを再構成するのはなかなか厄介だ。

### User Probe(uprobe/uretprobe)

これが最も鋭い武器だ。UprobeはeBPFを、データベースのバイナリ内にあるC/C++関数へ直接アタッチできるようにする。PostgreSQLなら、`exec_simple_query()`の入口にuprobeを、その戻り値にuretprobeを仕掛けるといった具合だ。

**裏側の仕組み(`int3`):** eBPFが関数にuprobeをアタッチすると、その関数の先頭の機械語命令をこっそり`int3`(ブレークポイントトラップ)に書き換える。CPUがそこに到達するとハードウェア例外が発生し、OSが制御を奪ってカーネルへ飛び、eBPFのハンドラを呼び出す。ハンドラの処理が終わると、CPUは元の命令を復元し、実行を再開する。

**uprobeの弱点:** uprobeを1回通過するコスト(例外の発生とコンテキストスイッチ)は、最近のCPUでおよそ1.5〜3.0マイクロ秒かかる。
- `exec_simple_query`のように1秒に1万回程度しか呼ばれない関数なら、消費時間は約30ms――オーバーヘッドにして3%程度で十分許容できる。
- 一方`btr_search_leaf`のようにB-Treeのリーフを毎秒100万回走査するような関数にuprobeを付けると、割り込みストームが起きる。実時間1秒に対してCPUは3秒分の仕事を強いられ、データベースは事実上ハングする。

uprobeを仕込む前には、必ず対象関数の呼び出し頻度を確認しておくべきだ。

### Kernel Probe(kprobe)

kprobeはデータベースとディスク・RAMとのやり取りを観測するために使う。`vfs_read()`や`vfs_write()`にkprobeをアタッチし、簡単なパターン認識と組み合わせれば、page cacheのミス率を直接追跡できる。MySQLの`SELECT`が実際に何回NVMeまで読みに行ったか、ブロックレイヤーで何ナノ秒を費やしたかまで見えるようになる。

## ハードウェアに合わせたチューニング: NUMAとキャッシュ

数十億トランザクションを処理する128コアサーバーでeBPF observabilityを動かすなら、Mapの設計はロジックだけでなくプロセッサの物理的な制約にも従う必要がある。

### NUMA問題とPer-CPU Map

NUMA(Non-Uniform Memory Access)構成では、あるCPUが別のCPUのローカルRAMにアクセスすると大きなレイテンシを払うことになる。コア0とコア64のeBPFプログラムが同じグローバルなHash Mapに同時に書き込もうとすれば、メモリ競合が発生し、相互接続バス(UPIやInfinity Fabric)が飽和してしまう。

解決策は**Per-CPU Map**だ。OSは物理コアごとにMapのローカルコピーを別々に確保し、あるコアで動くeBPFプログラムはそのコア専用のRAM領域にだけ書き込む。これにより、すべての書き込みがロックフリーかつ競合フリーになり、スループットはL1キャッシュの帯域幅という物理的な上限まで達する。

### 命令キャッシュを守る

プロセッサのL1命令キャッシュ(L1i)はかなり小さく、通常32KBしかない。データベース自身のロジック(クエリプランナー、実行エンジン)がこの領域の大半を必要としている。eBPFプログラムがコンパイル後に大きくなりすぎると、kprobeやuprobe経由で呼ばれるたびにL1iへ溢れ込み、データベース自身の命令を追い出してしまう。データベースの実行が再開するとキャッシュミスが発生し、L2/L3から命令を読み直す必要が生じ、目に見えるパフォーマンス低下(ストールサイクル)につながる。

実践的には、eBPFのコードはできるだけ無駄なく書くこと。チェックはまとめられるところはまとめ、ループのアンロールは慎重に使い、プログラム全体を4KB未満に抑えておけば、監視対象のデータベースとL1iを平和に共存させられる。

## システムエンジニアへの教訓

1. **kprobeとネットワークフックから始め、uprobeは慎重に扱う:** 本番環境でuprobeを初めて試すのは避けたい。まずステージング環境で`bcc`や`bpftrace`を使い、`count()`で関数の呼び出し頻度を測ってから、レイテンシ収集用のuprobeを展開する。
2. **データ構造はフラットに保つ:** eBPFにはヒープも`malloc`もない。複数ステップにまたがるSQLトランザクションを追跡したいなら、有限状態機械(FSM)はユーザー空間側に構築する。eBPFの役割はRing Bufferへイベントを流すことに徹し、複雑な相関ロジックをカーネル内に詰め込まない。
3. **Mapの容量を管理する:** すべてのeBPF Mapには`max_entries`という上限がある。Hash Tableが埋まると、新しいトランザクションは単に観測されなくなる。ユーザー空間のエージェントが古いキーを積極的に削除し、空きを作り続ける必要がある。
4. **権限とセキュリティを締める:** eBPFはカーネルのRAM全体――通信中のパスワードやシークレットも含めて――を読み取れてしまう。本番環境では`CAP_BPF`や`CAP_SYS_ADMIN`を持つユーザーだけに操作を限定し、コード署名で出所不明のeBPFプログラムを排除する。
5. **自前でobservabilityスタックを一から書かない:** Cilium、Tetragon、Pixieといった既存のエコシステムを使うか、Goなら`cilium/ebpf`ライブラリを使う。生データはPrometheusのヒストグラムへ流し込めば、GrafanaでP99やP99.9のパーセンタイル表示がそのまま手に入る。
