# SQL Window Functionsの背後にあるマイクロアーキテクチャ実行メカニズム

## エグゼクティブサマリー

SQL Window Functions(ウィンドウ関数)は、データエンジニアやアナリストが複雑な分析計算をこなす方法をがらりと変えた機能だ。行を1つの結果に潰してしまう集約関数(Aggregate Functions)とは違い、Window Functionsは入力レコードをそのまま保ちながら、周囲の「データフレーム(Window Frame)」との関係で値を計算する。

`ROW_NUMBER() OVER(PARTITION BY... ORDER BY...)`のようなSQL構文自体は洗練されているが、その裏側では、リレーショナルデータベース(RDBMS)や分散システム(Spark、Trino)の中でもとりわけ複雑でコストの高い実行エンジンが動いている。

この記事では、Window Functionsの実行メカニズムをパーティショニングやソートといったアルゴリズムの土台から、Monotonic DequeやSegment Treeといったデータ構造、さらにはSIMD、キャッシュライン、NUMA、JITコンパイルといったハードウェアレベルの話まで一通り追っていく。読み終える頃には、Window Functionsが実際どう動いているか、大規模システムでどこがボトルネックになりやすいか、データ設計をどう最適化すべきかが見えてくるはずだ。

## Window Functionsの核心的な問題

**問題とは何か?**
数百万人の顧客に対する7日間の移動平均を計算するクエリを考えてみよう。

`AVG(sales) OVER(PARTITION BY customer_id ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)`

ある1行の結果を返すために、システムはその行を読むだけでは済まない。次のことをやる必要がある。
1. 数百万の顧客を正しくパーティションに分ける(Partitioning)。
2. 各顧客の中で日付順に並べ替える(Sorting)。
3. 計算のために7行分の「スライディングウィンドウフレーム」を維持し、ウィンドウが進むたびに一番古い行を捨てて新しい行を足す。

ただの線形スキャンとは違い、この処理はかなり手の込んだローカル状態の管理を要求する。データがRAMに収まらないほど大きくなると、システムはクラッシュするか、ページング/スワッピングを続けざるを得なくなり、クエリの遅延は数秒から数時間へと膨れ上がる。現代のRDBMSが向き合っている課題は、この複雑な処理をこなしながらも、CPUのスループットを高く保ち、L1/L2/L3キャッシュをきちんと使い切ることだ。

## フェーズ1:データのパーティショニングとルーティング

論理パーティション(`PARTITION BY`基準)へデータを細分化するのが最初のステップだ。

### HashingとSIMDの力
グループ化は単純な文字列比較では行わない。**SIMD(AVX-512)**命令セットに乗せたMurmurHash3やxxHashのような高速ハッシュアルゴリズムを使う。8〜16個のハッシュキーをベクトルレジスタに同時にロードすることで、CPUは1クロックサイクルで16レコード分のパーティション振り分けを計算できてしまう。

$$h(x) = \text{xxHash}(x_k) \pmod P$$

### Radix Bufferとキャッシュスラッシング
キャッシュミスを避けるため、データストリームは**Radix Buffer**を経由してルーティングされる。このバッファサイズはL2/L3キャッシュの容量(だいたい256KBから数MB)にぴったり収まるよう厳密に調整されている。数千のパーティションへ一気にデータを流し込むと、CPUは絶えずキャッシュラインを入れ替える羽目になり(キャッシュスラッシング)、メモリ帯域幅が目に見えて落ち込む。

### External Partitioning技術とOSのmmap
パーティションがRAMの容量を超えて膨らむと(Out-of-Core)、RDBMSはExternal Partitioningを起動する。一時ファイルを`madvise(MADV_SEQUENTIAL)`フラグ付きの`mmap()`でディスクにマッピングする。このカーネルレベルの指示によって、OSはWindowアルゴリズムが実際にデータを要求する前からSSDからRAMへ積極的に先読みするようになり、I/O待ちがほぼなくなる。

## フェーズ2:データの並べ替え

パーティショニングが終わると、次に待っているのはコストの高い内部`ORDER BY`だ。理論上の下限は$\Omega(N \log N)$を下回らない。

### IntroSortと敗者木構造
データが完全にRAMに収まるなら、**IntroSort**(QuicksortとHeapsortのハイブリッド)が主役になる。RAMが足りなくなると、**External Merge Sort**の出番だ。データは細かく分割され、あらかじめソートされた上で、ソート済みラン(sorted runs)としてNVMeディスクに書き出される。

マージフェーズでは**敗者木(Loser Tree / Tournament Tree)**を使う。この構造はマイクロアーキテクチャ的にかなり都合が良く、新しい要素が入るたびに必要な比較は$\log_2(K)$回だけで、木のノード更新はほぼ同じキャッシュライン上で完結する。

## フェーズ3:評価状態マシン

ここからがアルゴリズムの本領だ。レコード$i$のウィンドウフレームは次のように定式化できる。

$$ W_i = \{ r_j \in P \mid \max(1, i - L) \le j \le \min(N, i + U) \} $$

### 単純な分析関数(ROW_NUMBER, RANK)
これらの関数なら、現在のインデックスを追うレジスタを1つ持っておくだけで済む。計算量は$\mathcal{O}(N)$で済み、非常に軽い。

### スライディングウィンドウ問題(Sliding Window Maximum/Minimum)
過去30日間のような可変フレームで`MAX()`や`MIN()`を求めるのは、実はそう簡単ではない。愚直にやると計算量は$\mathcal{O}(N \times W)$($W$はウィンドウサイズ)になり、$W = 10,000$ともなればクエリは事実上止まる。

ここでの定番の解決策が**単調両端キュー(Monotonic Deque)**だ。Dequeの中にはMAXになりうる候補だけを残し、絶対にMAXになり得ない小さすぎる値はその場で捨てる。結果として計算量は純粋な線形時間$\mathcal{O}(N)$まで下がる。

```rust
// Window MaximumのためのMonotonic Dequeアルゴリズムのシミュレーション
pub fn evaluate_max(&self) -> Vec<T> {
    let mut deque: VecDeque<usize> = VecDeque::new();
    for i in 0..n {
        // 1. ウィンドウからスライドした行を破棄する
        if let Some(&front_idx) = deque.front() {
            if i >= self.window_size && front_idx <= i - self.window_size { deque.pop_front(); }
        }
        // 2. 劣った候補を破棄する
        while let Some(&back_idx) = deque.back() {
            if self.data[back_idx] <= self.data[i] { deque.pop_back(); } 
            else { break; }
        }
        deque.push_back(i); // 3. 新しい候補を登録する
    }
}
```

### 桁落ち対策
浮動小数点データで`SUM()`を計算する場合、大きな数を足した後に小さな数を引く(スライディングウィンドウではよく起きる)と、精度がじわじわ失われていく。しっかりしたRDBMSは**Kahan Summation**を使って誤差を補正するバッファを持ち、累積誤差をゼロに近い水準に抑え込んでいる。

### 難しい関数のためのセグメント木
`COUNT(DISTINCT)`や不規則な`RANGE`ベースのウィンドウフレームでは、Dequeはもう役に立たない。そこでデータベースはRAM上に**セグメント木(Segment Tree)**や**フェニック木(Fenwick Tree)**をその場で構築する。コストは$\mathcal{O}(N \log N)$まで上がるが、複雑なクエリにも対応できるようになる。

## 分散アーキテクチャとオペレーティングシステムの制約

### 分散システム(MPP)のためのプレフィックスサム
SnowflakeやBigQueryのような大規模データウェアハウスでは、Window Functionの計算を1台のノードだけに任せるわけにはいかない。1つの顧客パーティションのデータが偏って多すぎる場合(Data Skew)、そのノードは容易にメモリ不足で落ちてしまう。

そこで使われるのが**プレフィックスサム(Prefix Sum / Scan)**アルゴリズムだ。1人の顧客のデータを複数のセグメントに分割し、複数のノードに割り当てる。ノードAが最初の10日分の合計を計算し、ノードBが次の10日分を計算、その後お互いに「残り」の値をブロードキャストする。結合法則(Associativity)を利用することで、分散処理の遅延は$\mathcal{O}(N / P + \log P)$程度まで抑えられる。

### ヒュージページとダイレクトI/O
巨大なキューや木構造を格納するのに、普通の4KBページでは荷が重い。仮想ページテーブルのミス(TLB Miss)がCPUの足を引っ張るからだ。そのためデータベースは**トランスペアレントヒュージページ(2MB/1GB)**を設定するのがほぼ必須になる。

さらに、Window Functionのデータスキャンは基本的に「一度読んで捨てる」性質の操作だ。OSのページキャッシュを素直に使うと、他の有用なデータがRAMから追い出されてしまう。そこでデータベースは`O_DIRECT`フラグを`io_uring`(Linuxの非同期I/O)と組み合わせて使い、カーネルを丸ごと迂回してNVMe SSDコントローラから自前のメモリ領域へ直接データを転送する。

### NUMAアーキテクチャとフォールスシェアリング
2ソケットのサーバーでは、RAMは2つのNUMAノード(0と1)に分かれている。RDBMSは**CPU Pinning**を使って、CPU 0上のスレッドがCPU 0に紐づいたRAMだけを読むようにする必要がある。さらに、2つのスレッドが同じ64バイトの物理キャッシュラインを取り合わないよう、C++側でデータオブジェクトをパディングする(`alignas(64)`)ことも欠かせない。これを怠ると、フォールスシェアリングが起きてメモリ帯域幅が大きく落ち込む。

## 教訓

Window Functionsの複雑な内部構造を追ってみると、システムエンジニアやDBAにとって役立つ教訓がいくつか見えてくる。

1. **ソートのコストを見くびらない:** 単にグループ化したいだけなら、Window Function内で`ORDER BY`を安易に使わないこと。`ORDER BY`を書けば、計算量$\mathcal{O}(N \log N)$の大掛かりなソートエンジンが起動し、場合によってはSSDへの書き込みも発生する。
2. **ウィンドウフレームのサイズがアルゴリズムを左右する:** 範囲を絞った`ROWS BETWEEN`は、常に`RANGE BETWEEN`より速い。後者は境界処理が複雑になり、セグメント木の構築が必要になることさえある。
3. **データの偏りには注意:** `PARTITION BY`の選び方がデータの分散のされ方を決める。「国」でパーティション分けして顧客の99%がベトナムに集中しているなら、データはほぼ1つのCPUスレッド(あるいは1ノード)に集中し、クラスタの並列性は事実上意味をなさなくなる。均等に分散するパーティションキーを選ぶこと。
4. **RAMの制限とシステム構成:** Sort/Hash用のメモリ割り当てパラメータ(PostgreSQLの`work_mem`など)は慎重に設定する必要がある。少なすぎるとハードディスク上での遅いExternal Merge Sortが走り、多すぎるとLinuxのOOM Killerがデータベースプロセスを終了させかねない。

## 結論

Window Functionの実行メカニズムは、組み合わせ的なグラフ構造、細かく作り込まれた最適化アルゴリズム、そしてハードウェアレベルでの緻密なチューニングが組み合わさった仕組みだ。これを理解しておくと、より賢いSQLを書けるようになるだけでなく、汎用ツールの限界を超えるビッグデータ分析システムを設計するための土台にもなる。
