# Google Spanner & TrueTime: 分散システムにおける時間の軸を曲げる

## エグゼクティブサマリー

CAP定理のせいで、ネットワーク障害が起きた瞬間、アーキテクトは一貫性 (Consistency) か可用性 (Availability) かの選択を迫られる。CassandraやDynamoDBをはじめ、今日の主要なデータベースの多くは可用性を選び、結果整合性 (Eventual Consistency) で手を打っている。それはアルゴリズムが未熟だからではない。地球上のどの2台のサーバーも、伝送遅延と水晶発振器のドリフトのせいで、完全に同期した時計を持てないという物理的な制約があるからだ。

Googleはこのトレードオフを受け入れないことを選んだ。**Google Spanner**は、真の**外部一貫性 (External Consistency)** を達成した、公に知られる限りでは唯一のグローバル分散リレーショナルデータベースだ。その核にあるのが**TrueTime API**——GPSと原子時計を組み合わせ、「今何時か」という曖昧な問いを、数学的に保証された誤差範囲を持つ値に変えてしまう、専用の時刻インフラである。

本稿ではTrueTimeのアーキテクチャを追っていく。誤差関数 $\epsilon(t)$、Paxosによる書き込みのルーティング、そして何より重要な**Commit Wait**——Spannerが時計の不確実性を吸収するためにトランザクションをわざと止める仕組みだ。

**問題の核心:**
グローバルなシステムで、ニューヨークのトランザクション $T_1$ が東京のトランザクション $T_2$ の開始前に完了したなら、因果律によりタイムスタンプ $S_1$ は $S_2$ より小さくなければならない。しかし東京の時計がニューヨークより5ミリ秒遅れているだけで、システムは平気で $S_2 < S_1$ と割り当ててしまう——これはグローバルなトランザクション順序付けを根本から壊すパラドックスだ。素直な解決策である中央タイムスタンプサーバーは、単にボトルネックを別の場所に移すだけで、大陸間の往復に数百ミリ秒かかる以上、Googleの規模では現実的ではない。

**教訓:**
1. **時間は決して正確ではないと受け入れる。** TrueTimeは1つの数値ではなく、区間 $[t_{earliest}, t_{latest}]$ を返す。真実はこの中のどこかに必ず存在する。
2. **Commit Waitは遅延と引き換えに一貫性を得る。** 書き込みを確認応答する前に、コーディネーターは不確実性の幅 $2\epsilon$ を丸ごと待ち、因果律を守る。
3. **異なる障害モードが互いを補い合う。** GPSは妨害されうるがドリフトしない。原子時計はドリフトするが妨害されない。両者を組み合わせれば、互いの死角をカバーできる。

## 分散コンピューティングにおける時間問題

すべてのサーバーは水晶発振器で時を刻んでいるが、2つとして全く同じ周波数で発振するものはない。チップの温度、電源の変動、ハードウェアの経年劣化が、わずかに周波数をずらす。これが**クロックドリフト**であり、一般的なハードウェアでは避けられない現象だ。

従来の解決策であるNTPは、LAN/WAN越しに基準サーバーへ時刻を問い合わせる。しかしルーターのキューバッファは常に変動するため、往復時間 (RTT) が対称になることはほとんどなく、NTPにはこの非対称性を正確に補正する手段がない。広域ネットワークでは、NTP同期された2台のサーバー間の時計誤差が100〜200msに達することも珍しくない。

これはかなり大きな幅だ。この間に何万ものトランザクションが順序を入れ替えられかねず、それがまさに外部一貫性と線形化可能性を壊す原因になる。

## TrueTimeアーキテクチャ:原子時計と衛星のネットワーク

NTPの限界を超えるため、Googleはソフトウェアだけに頼らず、専用のハードウェアを構築した。

### デュアルマスターネットワーク

TrueTimeのハードウェア層は、Googleの全データセンターに設置された2種類のTime Masterで構成される。

1. **GPS Masters** — 屋上のアンテナで衛星信号を読み取る。非常に精度が高いが、太陽嵐や電波干渉、大気の影響で信号が完全に途絶えることもある。
2. **Atomic Masters**(ルビジウム/セシウム原子時計) — サーバーホールの奥深くに設置され、電子遷移を測定する。信号が途絶えることはないが、時間の経過とともにわずかにドリフトする(1日あたり数マイクロ秒程度)。

この組み合わせがうまく機能するのは、2つの障害モードがほぼ直交しているからだ。GPS信号が途切れても原子時計が安定を保ち、原子時計がドリフトしすぎればGPSが引き戻す。両方を備えたトップクラスのマシンは、非公式に「Armageddon Masters」と呼ばれている。

### Marzulloのアルゴリズムとタイムタイムデーモン

各Spannerマシンはバックグラウンドで動作するTrueTimeデーモンを実行し、複数のMaster——ローカル/リモート、GPS/原子時計を問わず——に定期的にポーリングをかける。応答を集めた後、デーモンは**修正版Marzulloアルゴリズム**を実行し、報告された区間同士を交差させ、他と整合しない応答を返すMaster(ハードウェア障害や光ファイバーの遅延による「嘘つき」)を自動的に排除する。

## TrueTimeを支える数学:不確実性を枠にはめる

TrueTimeの本当の価値は、正確な数値を返すことではなく、証明可能に安全な**不確実性区間**を返すことにある。

APIは `TT.now()` を公開し、次を返す。
$$ [t_{earliest}, t_{latest}] $$
システムは、呼び出し時点の真の絶対時刻 $t_{abs}$ が次を満たすことを保証する。
$$ t_{earliest} \le t_{abs} \le t_{latest} $$

### 誤差関数 $\epsilon(t)$

不確実性を $\epsilon$ とすると、区間の幅は常に $2\epsilon$ になる。サーバーがMasterへの同期に成功した瞬間($t_{sync}$)、誤差は最小になる($\epsilon_{sync} \approx 0$)。そこから、ローカルの水晶がドリフトするにつれて誤差は増大していく。最悪ケースのドリフト率を $\rho$ とすると(Googleは通常 $\rho \approx 200\mu s/\text{秒}$ と仮定する)、

$$ \epsilon = \epsilon_{sync} + \rho \cdot (t - t_{sync}) $$

`TT.now()` が呼ばれると、サーバーはローカルのハードウェアクロック $C(t)$ を確認する。
- $t_{earliest} = C(t) - \epsilon$
- $t_{latest} = C(t) + \epsilon$

専用のGPS/原子時計ハードウェアのおかげで、平均 $\epsilon$ は**1〜7ミリ秒**程度に収まる——一般的なNTPインフラの約150msと比べればかなり小さい。

## Commit Wait:因果律を守るために時間を止める

Spanner内の各タブレットは**Paxos**によって複製される。書き込みが発生すると、Paxosリーダーはwrite-ahead logに記録する前にトランザクションへタイムスタンプを割り当てる。

### 単調タイムスタンプのルール

トランザクション $T_1$ が書き込みを要求すると、リーダーは `TT.now()` を呼び出し、$s = TT.now().latest$——区間内で取りうる最大値——を割り当てる。上限値を取ることで、このタイムスタンプが過去に起きた何かと衝突することは決してないと保証される。

### Commit Waitのルール

Commit Waitは次のように定める:**実時間 $t_{abs}$ が確実に $s$ を過ぎるまで、システムはトランザクションの完了を確認応答してはならない。**

それがいつ起きたと分かるのか。コーディネーターは $t_{earliest} > s$ になるまで `TT.now()` を呼び続ける。その時点で、定義上 $t_{abs} \ge t_{earliest}$ であるから、$t_{abs} > s$ が確実だと結論できる。

この待機時間は通常 $2\epsilon$ 程度——最悪でも約7ms程度になる。

**なぜ待つ必要があるのか。** 地球規模の時計に潜む相対論的なずれを吸収するためだ。$T_1$ に $s = 100$ が割り当てられたとする。リーダーは $t_{abs} > 100$ になるまで待ってからクライアントに成功を伝える。Client AがClient Bに伝え、Client Bがトランザクション $T_2$ を開始する。$T_2$ は $t_{abs} > 100$ の後にしか始まらないので、`TT.now()` を呼んだときその区間には必ず $t_{earliest} > 100$ が成り立つ——つまり $T_2$ のタイムスタンプ $s_2$ は必ず100より大きくなる。因果順序 $s_1 < s_2$ は、偶然ではなく設計によってグローバルに保たれる。

```cpp
// C++でのCommit Waitのマイクロアーキテクチャの疑似コード
struct TrueTimeInterval {
    int64_t earliest_us;
    int64_t latest_us;
};

class TrueTimeAPI {
public:
    TrueTimeInterval now();
};

class PaxosLeaderEngine {
private:
    TrueTimeAPI tt_api;
    int64_t last_assigned_timestamp = 0;
    
public:
    int64_t PrepareTransaction() {
        // 最大の未来時点を取得
        TrueTimeInterval current_time = tt_api.now();
        int64_t s = current_time.latest_us;
        
        // 厳密な単調増加を強制
        if (s <= last_assigned_timestamp) {
            s = last_assigned_timestamp + 1;
        }
        last_assigned_timestamp = s;
        return s;
    }

    void CommitTransaction(Transaction tx, int64_t s) {
        // 光ファイバーネットワークを介して多数派のクォーラム (Quorum) にデータを同期
        // このネットワーク転送時間 (約2〜5ms) は、待機時間の一部を吸収します。
        ReplicateToPaxosQuorum(tx, s);
        
        // 因果律を保護するスレッドロック - Commit Waitの開始
        while (true) {
            TrueTimeInterval wait_time = tt_api.now();
            if (wait_time.earliest_us > s) {
                // 不確実性が過ぎ去りました。現在の絶対時間は確実に > s
                break; 
            }
            // CPUを節約し、残りのマイクロ秒間スレッドを一時停止するようOSカーネルに要求
            HardwareInterruptNanosleep(wait_time.earliest_us - s);
        }
        
        // 安全。広域ネットワークに200 OK信号を送信
        RespondToClient(tx.client_id, SUCCESS);
    }
};
```

## 大陸をまたぐロックフリー読み取り

Commit Waitは書き込みに約7msの遅延という代償を課す。その代わりに、読み取りには本物の恩恵をもたらす:**大陸をまたぐロックフリー読み取り**だ。

ベトナムのアプリケーションがUserテーブルを読み取りたい場合、分散読み取りロックを取るためにわざわざ米国のデータセンターまで問い合わせる必要はない。単に `TT.now().latest` を呼んでタイムスタンプ $s_{read}$ を取得し、それをベトナムのデータセンターにあるローカルレプリカに渡すだけでいい。MVCCで保存されているそのレプリカは、$s_{read}$ に最も近いタイムスタンプを持つ行のバージョンを探して返すだけだ。世界のどこか他の場所で進行中の書き込みが、この読み取りと衝突したりブロックしたりすることは決してない。

読み取りのスループットは実質無制限にスケールし、レイテンシはほぼゼロに近づく。これがGoogle Adsを支える読み取り中心のインフラの核となる仕組みだ。

## マイクロアーキテクチャとシステムコールの視点

マイクロ秒単位の精度を実現するために、TrueTimeデーモンは `recvfrom()` のような通常のネットワークシステムコールに頼るわけにはいかない——OSのコンテキストスイッチが1回入るだけで $\epsilon$ の値が狂ってしまう。

- **ゼロコンテキストスイッチ設計:** TrueTimeデーモンのデータは、ユーザースペースに直接マッピングされた共有メモリセグメントに置かれる。
- **CPUレジスタの直接読み取り:** `TT.now()` の呼び出しは、水晶の発振信号をハードウェアレジスタから直接読み取る(例えばx86_64の `RDTSC` 命令)。投機的実行によるノイズを防ぐため、メモリバリア(`MFENCE`/`LFENCE`)と組み合わせる。
- **水晶温度の監視:** デーモンはCPUの温度センサーを継続的にポーリングする。温度が上がると水晶のドリフトは速くなるため、アルゴリズムは自動的に $\rho$ を最も保守的な値へと広げ、気づかないうちに同期がずれるリスクを避ける。

こうした設計判断が組み合わさることで、Google SpannerとTrueTime APIは、中央集権的なボトルネックを持たずにグローバル規模で外部一貫性を提供できる。時計同期という物理的な制約は、システムが推論できる一つの数値に変換され、静かにデータを壊す原因ではなくなる。
