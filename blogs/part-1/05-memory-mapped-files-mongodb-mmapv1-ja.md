# 05: Memory-Mapped Files (mmap)の探求: マイクロアーキテクチャの基盤とMongoDB MMAPv1の終焉

## エグゼクティブサマリー & 問題提起

データベースエンジンの歴史を振り返ると、あるひとつの選択がシステムの運命を大きく左右してきたことに気づく。キャッシュ管理と物理ストレージをどう扱うか、という選択だ。第一世代のNoSQL、代表格であるMongoDBの初期エンジンMMAPv1は、一見魅力的だが後になって重い代償を払うことになる近道を選んだ。I/Oとメモリ管理を、memory-mapped files(`mmap`)というメカニズムを通じてOSカーネルにまるごと委ねてしまったのだ。

この選択のおかげで、初期の開発スピードは圧倒的だった。しかし扱うデータがテラバイト級に膨らみRAM容量を超え、同時書き込みが何十万にも達するようになると、`mmap`に依存したモデルは軋み始める。データベース全体を止めるロック、スレッドを凍りつかせるページフォルトの連鎖、ストレージの断片化。これらが一斉に表面化した。

**根本的な問題はこうだ。** カーネルは汎用のメモリマネージャーであり、B-Treeのルートノードが何かも、どのページがクエリにとって重要かも知らない。LRUのような汎用アルゴリズムにページ破棄を任せていれば、いずれ重要なインデックスノードがディスクに追い出され、処理スレッド全体を止めるページフォルトを引き起こすのは時間の問題だった。

この記事では`mmap`の仕組みを追いながら、アムダールの法則を使ってマルチスレッド性能がなぜ崩壊したのかを数式で示し、MMAPv1が実際にどう息絶えたのかを解剖する。そのうえで、MVCCとユーザースペースのバッファプールを備えた**WiredTiger**への移行が、単なるアップグレードではなく生き残りをかけた選択だった理由を見ていく。

## Memory-Mapped Files (mmap)のマイクロアーキテクチャ動作メカニズム

Memory-mapped files、POSIXでいう`mmap`システムコールは、ディスク上のファイル内容をプロセスの仮想アドレス空間に直接マッピングし、`read()`や`write()`を完全に迂回する。一度マッピングされれば、プロセスはポインタ経由でRAM上の普通のバイト配列を扱うのと同じ感覚でファイルを読み書きできる。

### 仮想マッピングの確立プロセス

プロセスが`mmap()`を呼び出しても、カーネルはその場でディスクI/Oを**実行しない**。行うのは新しい仮想メモリ領域(VMA)の設定だけだ。このVMAは、仮想アドレス$V_{start}$から$V_{end}$までの範囲がディスク上のどのファイルに対応するかを記録している。

このマッピングは、CPUのメモリ管理ユニット(MMU)内にあるマルチレベルページテーブルと、Translation Lookaside Buffer(TLB)に深く依存している。

数式で表すなら、これは仮想空間$\mathcal{V}$からブロックデバイス$\mathcal{D}$を介して物理空間$\mathcal{P}$への変換だ。サイズ$S_{page}$(通常4KB)のページが仮想アドレス$V_{addr}$にあるとき、物理アドレス$P_{addr}$は実際にページフォルトが起きるまで解決されない。アドレス解決関数を$\mathcal{F}$とすると:

$$ P_{addr} = \mathcal{F}(V_{addr}) = P_{base} + (V_{addr} \pmod{S_{page}}) $$

### ページフォルトの解剖と遅延ペナルティ

`mmap`の中核にあるのは、**メジャーページフォルト**によって駆動される**デマンドページング**だ。

コード側がポインタ経由で$V_{addr}$範囲にアクセスし、対応する物理ページがまだロードされていないとき、MMUはページテーブルの中に有効なエントリを見つけられず、ハードウェア例外を発生させてカーネルに制御を渡す。

メジャーページフォルトの処理順序:
1. **Context Switch:** プロセスが一時停止する。
2. **Kernel VMA Lookup:** カーネルがフォルトの発生した仮想アドレスに対応するファイルとオフセットを特定する。
3. **I/O Fetch:** カーネルが空きのRAMページフレームを確保し、DMA転送でディスクからデータを引き込む。
4. **PTE Update:** ページテーブルエントリを更新し、$V_{addr}$を新しくロードされたRAMページに紐付ける。
5. **Resume:** 制御が戻り、プロセスがメモリアクセス命令を再実行する。

```mermaid
flowchart TD
    A[ユーザースペースプロセス] -->|V_addrのアクセスと変換| B(Memory Management Unit - MMU)
    B -->|バッファのルックアップ| C{Translation Lookaside Buffer - TLB}
    C -- TLB Hit --> D[物理アドレス P_addr]
    C -- TLB Miss --> E{Page Table Walk (ページテーブルウォーク)}
    E -- PTE 有効 --> F[TLBの更新とP_addrの返却]
    E -- PTE 無効 (Invalid) --> G((Major Page Fault))
    G --> H[カーネルモードへのコンテキストスイッチ]
    H --> I[VMAを通じたディスク上のブロックの特定]
    I --> J[物理RAMへのページ読み込みI/Oの調整]
    J --> K[ページテーブルエントリ - PTE の更新]
    K --> L[ユーザースペースプロセスの復元]
    F --> D
    L --> B
    D --> M[RAMデータの読み書き]
```

I/Oの数字だけを見ても、この経路がどれだけ高くつくかがわかる。$T_{L1}$をL1キャッシュのアクセス時間(1ns)、$T_{RAM}$をRAMアクセス時間(100ns)、$T_{disk}$をディスクレイテンシ(NVMeで100µs、HDDで5ms)とし、ページフォルトが発生する確率を$P_{fault}$とする。

期待アクセス時間$E[T_{access}]$は次のようになる。

$$ E[T_{access}] = (1 - P_{fault}) \cdot T_{RAM} + P_{fault} \cdot T_{disk} $$

$T_{disk} \gg T_{RAM}$ ($10^3$から$10^5$倍の開き)であるため、$P_{fault}$がわずか0.01でもシステム全体の性能は桁違いに落ち込む。一度のスキャンクエリだけで、データベースプロセスはコンテキストスイッチを繰り返すだけの存在になりかねない。

### mmapが見せる「ゼロコピー」の幻想

ページフォルトという代償を抱えながらも、`mmap`は初期のシステムに魅力的な幻想を与えた。ゼロコピーの読み書きだ。
通常の`read()`では、データはディスク→カーネルページキャッシュ→ユーザーバッファという経路を辿る。
`mmap`ではカーネルページキャッシュそのものがユーザーバッファになり、アプリケーションはそこへ直接読み書きする。

これによってMongoDBの初期エンジンは驚くほど速く書き上げることができた。バッファプールを構築する必要も、LRUエビクションを管理する必要も、バックグラウンドのフラッシュアルゴリズムを実装する必要もない。全部Linuxカーネルがやってくれるからだ。スタートアップにとって、これは市場投入までの時間で圧倒的な優位性になった。

ただし、この技術的負債はいずれ回収されることになる。

```c
// MongoDB MMAPv1のmmapの幻想を示す疑似コード
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>

void* init_database_mmap(const char* db_file_path, size_t* out_size) {
    int fd = open(db_file_path, O_RDWR);
    struct stat sb;
    fstat(fd, &sb);
    
    // カーネルにデータベース全体を仮想空間にマッピングするよう要求
    void* db_memory = mmap(NULL, sb.st_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
    
    *out_size = sb.st_size;
    close(fd); // OSはfdが閉じられてもマッピングを維持する
    
    return db_memory;
    // これ以降、MongoDBはC++のポインタを使用してdb_memory内のB-TreeとDocumentsに直接アクセスする
}
```

## MongoDB MMAPv1の解剖: アーキテクチャ、定量的な限界と死角

MMAPv1(バージョン3.0まで既定のエンジンだった)では、MongoDBのすべての物理ファイルが前述のメカニズムを通じてそのまま仮想空間にリンクされていた。

### 指数関数的なファイル割り当てのルール

MMAPv1は番号付きのデータファイルセグメント(`database.0`, `database.1`...)を作成する。新しいファイルはそれぞれ直前のファイルの2倍の容量を持ち、2GBの上限まで2の累乗で増えていく。

$$ S_i = \min(2^i \cdot 64 \text{ MB}, 2048 \text{ MB}) $$

狙いは、機械式HDDのシーク時間を減らすため、ドキュメントを連続したディスク領域にまとめて置くことだった。ファイル内部では、エクステントがBSONドキュメントの双方向連結リストを保持する。

### 動的割り当ての課題とドキュメント再配置

NoSQLのドキュメントは柔軟なスキーマを持つ。これは諸刃の剣で、今日1KBだったドキュメントが、アプリケーションが内部配列にデータを積み増すにつれ、明日には5KBに膨らむこともある。

ドキュメントの隣接領域がすでに埋まっていると、更新は**ドキュメント再配置**という厄介な処理を引き起こす。

1. MongoDBは新しく5KBの空き領域を要求する。
2. `memcpy`でドキュメント全体をそこへコピーする。
3. 古い1KB領域を解放する(ここで断片化が生まれる)。
4. すべてのインデックスの、すべてのリーフノードを新しい物理アドレスに向けて更新しなければならない。

余裕を持たせるために動的パディングが後から追加されたが、ストレージ消費が大きく増えるうえ、波及的なI/Oのリスクを完全には解消できなかった。

### アムダールの法則が示すボトルネック: データベースレベルのロック

RAMへのアクセスが生のC++ポインタで行われる以上、スレッドセーフの確保には高いコストがかかる。MMAPv1は当初プロセス全体を覆うグローバルなリーダー・ライターロックを使っており、後にデータベースレベルのロックまで緩和された。

とはいえ実態としては、`Users`コレクションへの書き込み一つで`Orders`や`Products`を含むデータベース全体がロックされてしまう。他の誰も読み書きできなくなる。

アムダールの法則がこの劣化を数式で説明してくれる。$P$を並列実行できる時間の割合、$S = 1 - P$をロックのクリティカルセクション内で消費される逐次実行時間とすると、$N$コアにスケールしたときの最大スピードアップは次の通りだ。

$$ \text{Speedup}(N) = \frac{1}{(1 - P) + \frac{P}{N}} $$

MMAPv1では一つのロックがデータベース全体を覆うため、$1-P$は非常に大きい。$N \to \infty$としても、スピードアップの上限は:

$$ \lim_{N \to \infty} \text{Speedup}(N) = \frac{1}{1 - P} $$

MMAPv1構成で128コアのサーバーを買うのは、実質的にお金を捨てるようなものだった。127個のコアはロック待ちでひたすらスピンし続けることになる。

### ページフォルトスラッシングという悪夢と、Yieldによる妥協

最悪のケースは、**データベースロック**と**メジャーページフォルト**が同時に起きるときだ。

1. スレッド$T_1$がデータベースの書き込みロックを取得する。
2. $T_1$が仮想アドレス$V_x$にあるドキュメントに書き込もうとする。
3. ところが$V_x$に対応する物理ページは、カーネルのLRUが「あまり使われていない」と判断し、ひそかにディスクへスワップアウトされていた。
4. $T_1$はページフォルトでブロックされ、カーネルがディスクからデータを引き戻すまで、10ミリ秒ほど待たされる。
5. その10ミリ秒の間、$T_1$は書き込みロックを保持したままだ。
6. 他の何千もの接続が完全に止まる。原因不明のまま、システムが「固まった」ように見える。

MongoDBが取った対策はYieldメカニズムだった。メモリにアクセスする前にLinuxの`mincore()`を呼び、そのページが本当にRAM上にあるかをOSに確認する。なければロックを自発的に手放し(Yield lock)、バックグラウンドでページインを走らせてから待つ。この場当たり的な対策は効果があったものの、並行処理のステートマシンをかなり複雑で予測しづらいものにしてしまった。

## MMAPv1の終焉とWiredTigerへの移行

データ量の爆発的な増加とともに、MMAPv1の弱点は致命的になった。メモリ管理を無条件にカーネルへ委ねるということは、データベースが自分自身のデータに対して盲目であることを意味する。OSにはB-Treeのルートノード(RAMに固定しておくべきもの)と、めったに使われない古いドキュメント(ディスクに追い出しても構わないもの)の違いなど分かりようがない。

### MVCCとユーザースペースという選択

2014年にMongoDBがWiredTigerを買収したことは、設計思想における大きな転換点になった。WiredTigerは`mmap`の自動処理を丸ごと捨て、ユーザースペースに独自の**バッファプール**を構築する。

これによって完全な制御権を取り戻せる。WiredTigerはデータの意味的な階層を意識したLFU/LRU系のエビクションポリシーを使い、B-Tree構造が常にRAM上に留まるようにする。

粗いデータベースレベルのロックの代わりに、WiredTigerは**MVCC(多版同時実行制御)**を採用した。書き込みは既存の行をその場で書き換えず、ポインタチェーンでつながった新しいバージョンをRAM上に作る。その結果:
- 読み取りが書き込みをブロックすることはない。
- ロックはドキュメント単位で行われる。
- 並列化率$P \approx 1$となり、アムダールの法則の下でも、WiredTigerは128コアサーバーの性能をほぼそのまま使い切れる。

### ブロック圧縮という武器

`mmap`を捨てたことで、圧縮という選択肢も開けた。`mmap`はディスクとRAMを1対1でマッピングするため、ディスク上のデータを圧縮することはできなかった。圧縮してしまうと、仮想バイトオフセットによるアドレッシングが成り立たなくなるからだ。

独立したバッファプールがあれば、処理中はRAM上で解凍されたデータを扱いつつ、フラッシャースレッドがディスクへ書き戻す段階でzstdやSnappyを通してブロックを圧縮できる。
圧縮率$\mathcal{C}_{ratio} = \frac{S_{uncompressed}}{S_{compressed}}$は3倍から5倍に達することも珍しくない。ストレージコストが下がるのはもちろん、同じ情報量を読むのに必要なバイト数が減るぶん、実効的なI/Oスループットも底上げされる。

```rust
// WiredTigerのようなユーザースペースバッファプールの優位性を示す抽象的な疑似コード
struct BufferPoolManager {
    page_table: HashMap<LogicalPageId, FrameId>, // DBMS独自の仮想マッピング
    frames: Vec<PageFrame>,
    eviction_policy: Box<dyn EvictionAlgorithm>,
    wal_manager: WriteAheadLog,
}

impl BufferPoolManager {
    fn fetch_page(&mut self, page_id: LogicalPageId) -> Result<&mut PageData, DiskError> {
        if let Some(&frame_id) = self.page_table.get(&page_id) {
            // Cache Hit: セマンティックLFUの更新
            self.eviction_policy.record_access(frame_id);
            Ok(self.frames[frame_id].get_data_mut())
        } else {
            // Cache Miss: DBMSがI/Oを自己調整し、スレッドを停止させるカーネル割り込み/ページフォルトを引き起こさない！
            let frame_id = self.evict_oldest_document_page()?; 
            
            // O_DIRECT + Async I/O (io_uring) を使用してデータを引き出す
            let compressed_data = disk_manager.async_read_page(page_id)?;
            let raw_data = snappy::decompress(compressed_data);
            
            self.frames[frame_id].fill(raw_data);
            self.page_table.insert(page_id, frame_id);
            Ok(self.frames[frame_id].get_data_mut())
        }
    }
}
```

### 結論

MMAPv1の終焉は、設計の失敗を示すものではなく、システムソフトウェアが必ずたどる進化のサイクルを証明しているにすぎない。`mmap`は自分の時代の役割を立派に果たした。MongoDBを驚くべき速さで立ち上げ、開発者を惹きつけ、Web 2.0時代のデータベースの顔になった。

しかし、本格的なマルチスレッド・ビッグデータとマイクロサービスの世界に足を踏み入れたとき、帯域幅とロック競合をめぐる数学的な制約は誤魔化しようがなかった。WiredTigerのMVCC、独立したバッファプール、ブロック圧縮によってユーザースペースを取り戻したことで、MongoDBは初期のNoSQL実験から、トランザクションを扱えるエンタープライズ級のデータストアへと生まれ変わった。

エンジニアにとって残る教訓はシンプルだ。アプリケーションが持つ本質的な意味論的知識を、それを理解しようもない汎用OSカーネルに丸投げしてはいけない。
