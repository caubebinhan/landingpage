Technical Whitepaper #50: Time-Series Databases (TSDB) – Dissecting the Gorilla Compression Algorithm and Memory Chunking Architecture

July 22, 2026 · admin · 8 min read

Executive Summary

This whitepaper provides a deep, comprehensive look at how the world’s leading time-series databases (TSDB) — such as Prometheus, InfluxDB, and Gorilla — handle the massive data volumes generated by IoT and Microservices systems. By analyzing the mathematical structure of the Gorilla compression algorithm (Delta-of-Delta and IEEE-754 XOR) as well as the Chunking memory management strategy at the operating-system kernel level, readers will understand the principle behind achieving a 64:1 compression ratio using just a handful of CPU clock cycles, and will draw invaluable design lessons for real-time distributed systems.

Introduction: The Immutable Nature of Time-Series Data

In the age of cloud computing and the Internet of Things (IoT), machines never stop talking. Billions of sensors, servers, and containers continuously emit telemetry signals. This kind of data is called Time-Series Data.

Unlike the tables found in typical RDBMS or NoSQL systems, a TSDB carries extremely demanding and distinctive physical characteristics:

  1. Extreme write throughput (Write-Heavy): Can reach millions of write operations per second.
  2. Time-Monotonic ordering: Data points always arrive with a strictly increasing timestamp. There is no such thing as an UPDATE to the past — almost everything is an APPEND.
  3. Simple Data Structure: A data point usually consists of only: a Timestamp (a 64-bit Int) and a Metric value (a 64-bit Float), along with a handful of identifying Tags.

This very monotony is both a colossal storage challenge and a golden opportunity for systems engineers to work microarchitectural magic.

The Core Problem: The IoT Data Tsunami and the Storage Bottleneck

1 Bandwidth, Disk I/O, and the Bankruptcy of RDBMS

Imagine a datacenter monitoring system with 100,000 servers. Each server emits 100 metrics (CPU, RAM, Disk I/O…) every second. Total: $100,000 times 100 = 10,000,000$ data points per second. Each data point is 16 bytes (8 bytes Timestamp + 8 bytes Float Value). Raw Ingestion Rate: $160 text{ MB/second} approx 13.8 text{ TB/day}$.

If this volume of data were pushed into PostgreSQL or MySQL via INSERT statements, the B-Tree Index system would immediately collapse under lock contention. Even using a system like Cassandra, the cost of purchasing enough NVMe storage to hold 13.8 TB per day over a 1-year retention lifecycle (roughly $sim 5$ Petabytes) would be a financial disaster.

2 The Failure of Dictionary-Based Compression Algorithms

Facing this pressure, an engineer’s natural reflex is to reach for data compression. However, industry-standard compression algorithms such as LZ77, Snappy, Gzip, and Zstandard operate on the principle of “Dictionary-based Compression.” They try to find repeating byte sequences. Unfortunately, timestamps are continuously increasing numbers (never repeating), and Floating-Point decimal values continuously churn the bits in their fractional part. Snappy or Zstd are essentially useless here, even burning substantial CPU while barely improving the compression ratio.

The problem to solve is: We need a dictionary-free compression algorithm that operates directly at the CPU microcode level without branching, fully optimized for sequences of integers and floating-point numbers. The Gorilla algorithm (introduced by Meta/Facebook in 2015) is the most perfect answer to this problem.

The Algorithmic Solution: The Gorilla Compression Algorithm (Math & Bit Logic)

Gorilla splits the problem into two streams: compressing Timestamps and compressing Metrics. The key insight lies in information theory: although absolute values change continuously, the difference between them (Delta) is extremely stable.

1 Timestamp Compression via Delta-of-Delta

Metric-collection systems usually operate on a fixed cycle (e.g., once every 10 seconds). Instead of storing 64 bits for every timestamp:

$T = [1000, 1010, 1020, 1030]$

We compute the first-order difference (Delta – $D_n = T_n – T_{n-1}$):

$D = [10, 10, 10]$

Due to network jitter, the actual Delta may look like:

$D = [10, 12, 9, 11]$

The algorithm then computes the second-order difference (Delta-of-Delta – $D^{(2)}_n = D_n – D_{n-1}$):

$D^{(2)} = [2, -3, 2]$

The mathematics behind Delta-of-Delta (DoD): in 96% of cases in a stable system, DoD = 0. Gorilla encodes this using variable-length encoding (similar in spirit to Huffman coding):

  • If DoD = 0: write exactly 1 bit 0. (Compressing 64 bits down to 1 bit!).
  • If DoD $in [-63, 64]$: write 10 + 7 data bits (9 bits total).
  • If DoD $in [-255, 256]$: write 110 + 9 data bits (12 bits total).
  • And so on for wider ranges.

2 Metric Value Compression via IEEE 754 XOR Logic

The 64-bit Floating-Point number (IEEE 754) is the nightmare of compression algorithms. It consists of 1 sign bit, 11 exponent bits, and 52 mantissa bits. A tiny change (say, from 2.0 to 2.01) completely scrambles the bit pattern. But Gorilla observed: a temperature reading from a sensor (e.g., 37.5°C) rarely jumps drastically. At the bit level, two adjacent values usually share the same Sign, the same Exponent, and a large chunk of the leading mantissa bits.

Instead of subtracting, Gorilla takes two adjacent values and applies a direct 64-bit logical XOR ($oplus$):

$$X_n = V_n oplus V_{n-1}$$

The result $X_n$ will be a 64-bit sequence containing many leading zero bits (Leading Zeros – LZ) and trailing zero bits (Trailing Zeros – TZ). In the middle sits the “Meaningful Bits” block.

  1. If $X_n = 0$ (value unchanged): write only 1 bit 0.
  2. If $X_n neq 0$, the system compares the LZ and TZ counts against the previous value. If the Meaningful Bits block falls within the previous block’s range, the system writes a 10 control bit and records only the differing bits, reusing the previous LZ/TZ structure.
  3. If the structure has changed, write 11 + 5 bits for the new LZ length + 6 bits for the length of the differing block + the differing block itself.

3 Hardware-Level Microarchitecture

Counting Leading Zeros doesn’t require expensive while loops. Thanks to the C++/Rust/Go compiler, this algorithm triggers the x86_64 CPU’s native LZCNT and TZCNT instructions directly. Counting the zeros completes in 1 clock cycle. Branching is kept to an absolute minimum, letting the CPU’s Hardware Branch Predictor run smoothly.

System Microarchitecture: Memory Paging Management and Chunking

Compressed data forms a continuous bit-stream. But if you need to query “yesterday’s” data, the system cannot decompress sequentially from the start of the year. This is where Chunking Architecture comes in.

1 What Is Memory Chunking?

The endless time series is chopped into “Chunks.” A Chunk typically represents either a time span (e.g., 2 hours) or a volume of bytes (e.g., 4 KB). At any given moment, each Time-Series has exactly one Active Chunk (resident in RAM, continuously receiving data in append-only fashion).

2 The CPU Core Trade-off: False Sharing vs. TLB Misses

If the Chunk size is designed too small (< 4KB): A single 4KB physical memory page will contain Chunks from many different Series. When different CPU cores update different Series that happen to sit adjacent to each other on the same Page, it triggers "False Sharing." The CPU's MESI protocol will continuously broadcast L1/L2 cache invalidation messages to each other, causing speed to plummet.

If the Chunk size is designed too large (e.g., 20MB): RAM becomes fragmented. The Translation Lookaside Buffer (TLB) gets exhausted from tracking too many physical addresses, leading to constant TLB Misses. The common solution is to consolidate Chunks into large “Memory Arenas” using Huge Pages (2MB), managed by RAM allocation libraries such as jemalloc, keeping the Cache Hit ratio consistently at its maximum.

3 The Lifecycle of a Data Block

Convergence with the Operating System: OS Page Cache and Mmap()

Once an Active Chunk has been sealed and becomes an Immutable Chunk, the TSDB begins a divine “delegation” trick to the Linux operating system’s Kernel.

1 Blurring the Boundary Between RAM and SSD

The TSDB flushes the Immutable Chunk down to NVMe SSD as Block Storage. Instead of managing file reads itself (which would cost read() calls and Kernel Space / User Space context switching), the TSDB uses the mmap() system call. mmap() flatly maps the blocks on SSD into the process’s Virtual Address Space.

  • When a user queries a chart of last month’s data, the TSDB simply performs a direct read on a memory pointer.
  • The hardware (Memory Management Unit) detects the data isn’t yet in RAM and fires a “Page Fault.”
  • The Linux Kernel immediately goes down to NVMe, fetches that 4KB block up into RAM, and even automatically guesses (Read-ahead heuristics) and pulls subsequent blocks into the L3 Cache.

2 Zero-Copy Deserialization

Because the byte structure of a Chunk on disk is 100% identical to the structure of the Gorilla bit-stream while it was still in RAM, the TSDB expends zero effort on “Deserialization.” The CPU reads the compressed bits directly from RAM (freshly pulled from disk via mmap), pushes them through the XOR/LZCNT function, and renders the chart immediately.

Lessons on Distributed Systems Design Philosophy

From the marvel of Gorilla and Chunking, we can distill the following core lessons:

  1. A deep understanding of the data domain (Domain-specific Optimization): There is no one-size-fits-all algorithm. Zstandard may reign supreme for text compression, but it loses badly against numerical time-series data. Gorilla proves that: when you precisely understand the physical nature, probability distribution of your data (the monotonicity of time, the correlation of sensor readings), you can design a simple algorithm (Delta-of-Delta, XOR) that beats every traditional information-theory approach.
  2. Immutability is the key to scalability: Designing a system around Immutable Chunks is the foundation for eliminating locking mechanisms, avoiding deadlocks, and letting the system fully exploit the OS Page Cache. The cache is never dirtied (no Dirty Pages) once a Chunk has been sealed, allowing the Kernel to evict data from RAM at any time under memory pressure.
  3. Hardware and Software must co-evolve: Designing a modern database is not just about writing logical lines of code. It requires the architect to speak the language of microarchitecture: Cache Lines (64 bytes), L1/L2 Cache Hit/Miss, TLB, SIMD, Hardware Instructions (LZCNT). Writing good code is not enough — writing code that respects the physical laws of Silicon is the pinnacle.

Conclusion

Building a high-performance Time-Series Database is a process of skillfully balancing network bandwidth, finite CPU resources, and physical storage limits. The Gorilla compression algorithm together with Chunking architecture combines minimalist Boolean mathematics, deep CPU microarchitecture understanding, and OS paging mechanisms to define the standard for observability databases. This understanding is the launchpad for backend engineers who want to design and master the data infrastructure of the Web 3.0 era and the global IoT device mesh.
