Exploring Memory-Mapped Files (mmap): The Micro-Architectural Foundation and the Fatal Decision Behind MongoDB’s MMAPv1

July 22, 2026 · admin · 11 min read

Executive Summary & Problem Statement

Throughout the history of database management systems (DBMS), the choice of how to manage caching and physical storage has determined the fate of the system. The first generation of NoSQL databases, most notably MongoDB (with its MMAPv1 engine), chose a shortcut that was highly appealing yet carried a fatal risk: fully delegating I/O and memory management to the operating-system kernel through the Memory-Mapped Files (mmap) mechanism.

This abstraction delivered an extremely fast development pace (time-to-market), but when the system had to face terabyte-scale data (exceeding available RAM) and hundreds of thousands of concurrent write transactions (high concurrency), the virtualization model of mmap collapsed. It exposed catastrophic problems: database-level locks, Page Fault Thrashing that froze threads, and storage fragmentation.

The core problem: The operating system’s kernel is a general-purpose manager. It has no “semantic knowledge” of a database’s B-Tree structure or the graph relationships between its files. Relying on the kernel’s automatic page-eviction mechanism, such as LRU (Least Recently Used), leads to critical index root nodes being evicted to disk, causing major page faults that paralyze the entire processing thread.

This technical report analyzes in depth the micro-architectural mechanism of mmap, mathematically demonstrates the collapse of multi-threaded performance via Amdahl’s Law, dissects the causes of MMAPv1’s death, and explains why the shift toward a self-managing engine like WiredTiger (with MVCC and a User-Space Buffer Pool) was an existential evolution for the database industry.

The Micro-Architectural Mechanism of Memory-Mapped Files (mmap)

Memory-Mapped Files (the mmap system call in POSIX) is a low-level system technique that allows the content of a file on physical disk to be mapped directly into a process’s Virtual Address Space. This method bypasses traditional I/O operations such as read() and write(). Once a file has been mapped, a process can read and write it exactly as if accessing an ordinary byte array in RAM, through pointers.

The Virtual Mapping Setup Process

When a process calls mmap(), the operating system kernel does not perform disk I/O immediately. Instead, it establishes a new Virtual Memory Area (VMA). This VMA structure records that the virtual address range from $V_{start}$ to $V_{end}$ represents the target file on disk.

This mapping event depends intimately on the multi-level page table and the Translation Lookaside Buffer (TLB) mechanism inside the CPU’s Memory Management Unit (MMU).

The mathematical equation representing this mapping is described through a transformation from virtual space $mathcal{V}$ to physical space $mathcal{P}$ combined with the block device $mathcal{D}$. If a memory page of size $S_{page}$ (typically 4KB) resides at virtual address $V_{addr}$, the physical address $P_{addr}$ is only determined when a page fault occurs. Let $mathcal{F}$ be the address resolution function:

$$ P_{addr} = mathcal{F}(V_{addr}) = P_{base} + (V_{addr} pmod{S_{page}}) $$

Anatomy of the Page Fault and Its Latency Penalty

At the heart of the mmap mechanism lies Demand Paging, triggered by Major Page Faults.

When an application tries to access, via a pointer, a region of the $V_{addr}$ array whose corresponding physical page has not yet been loaded into RAM, the MMU (Memory Management Unit) fails to find a valid mapping in the Page Table. It immediately triggers a hardware exception, handing control back to the kernel.

The Major Page Fault handling process:

  1. Context Switch: The application process is suspended (blocked).
  2. Kernel VMA Lookup: The kernel determines the physical file and offset corresponding to the faulting virtual address.
  3. I/O Fetch: The kernel allocates a free RAM page (Page Frame), then issues a DMA (Direct Memory Access) command to pull data from physical disk (HDD/NVMe) into this RAM page.
  4. PTE Update: The Page Table Entry is updated to link $V_{addr}$ to the newly loaded physical RAM page.
  5. Resume: Control is returned, and the application re-executes the memory-access instruction.
flowchart TD
    A[User Space Process] -->|Accesses via pointer V_addr| B(Memory Management Unit - MMU)
    B -->|Cache lookup| C{Translation Lookaside Buffer - TLB}
    C -- TLB Hit --> D[Physical Address P_addr]
    C -- TLB Miss --> E{Page Table Walk}
    E -- Valid PTE --> F[Update TLB and Return P_addr]
    E -- Empty PTE (Invalid) --> G((Major Page Fault))
    G --> H[Context Switch to Kernel Mode]
    H --> I[Determine Disk Block via VMA]
    I --> J[Dispatch I/O to Read Page into Physical RAM]
    J --> K[Update Page Table Entry - PTE]
    K --> L[Resume User Space Process]
    F --> D
    L --> B
    D --> M[Read/Write RAM Data]

From an I/O mathematics standpoint, this process incurs an extremely high latency. Let $T_{L1}$ be the L1 cache access time (1ns), $T_{RAM}$ be the RAM access time (100ns), and $T_{disk}$ be the disk latency (100µs for NVMe, 5ms for HDD). Let $P_{fault}$ be the probability of a page fault occurring.

The expected access time $E[T_{access}]$ is:

$$ E[T_{access}] = (1 – P_{fault}) cdot T_{RAM} + P_{fault} cdot T_{disk} $$

Because $T_{disk} gg T_{RAM}$ (a difference of $10^3$ to $10^5$ times), even a $P_{fault} = 0.01$ (just 1% of accesses faulting) drops system performance by thousands of times. A single scan query would turn a database process into a victim of continuous context switching.

The “Zero-Copy” Illusion of mmap

Despite the page-fault cost, mmap gave early systems an appealing illusion: zero-copy read/write. In traditional read(), data travels from Disk $rightarrow$ Kernel Page Cache $rightarrow$ User Buffer. With mmap, the Kernel Page Cache is the User Buffer. The application reads and writes directly into it.

This let MongoDB write engine code extremely fast. Engineers didn’t need to build a Buffer Pool, didn’t need to manage LRU eviction, didn’t need a background page-flushing algorithm. The Linux kernel did all of it for them. For a startup, this was an absolute competitive advantage in time-to-market.

But this technical debt would eventually come due.

// Pseudocode illustrating MongoDB MMAPv1's mmap illusion
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>

void* init_database_mmap(const char* db_file_path, size_t* out_size) {
    int fd = open(db_file_path, O_RDWR);
    struct stat sb;
    fstat(fd, &sb);
    
    // Ask the kernel to map the entire database into virtual space
    void* db_memory = mmap(NULL, sb.st_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
    
    *out_size = sb.st_size;
    close(fd); // The OS keeps the mapping alive even after fd is closed
    
    return db_memory;
    // From here, MongoDB accesses the B-Tree and Documents via raw C++ pointers straight into db_memory
}

Anatomy of MongoDB MMAPv1: Architecture, Quantitative Limits, and Blind Spots

In MMAPv1 (the default engine up through version 3.0), every physical MongoDB file was directly linked into virtual space through the mechanism described above.

The Linear (Exponential) File Allocation Rule

MMAPv1 establishes numbered data-file segments (database.0, database.1, …). Each new file has double the capacity of the previous one, following a base-2 exponential function up to a cap of 2GB:

$$ S_i = min(2^i cdot 64 text{ MB}, 2048 text{ MB}) $$

The purpose of this scheme is to keep documents stored on contiguous disk extents, reducing seek time for mechanical HDDs. Inside a file, Extents hold a doubly linked list of BSON documents.

The Dynamic Allocation Problem and Document Relocation

In NoSQL, documents have a flexible schema. A document that is 1KB today might grow to 5KB tomorrow as the application pushes more data into internal arrays.

If the space adjacent to the document is already occupied, an update triggers the disaster known as Document Relocation.

  1. MongoDB must request a new free 5KB region.
  2. It uses memcpy to copy the entire document there.
  3. It frees the old 1KB region (creating fragmentation).
  4. It must update EVERY LEAF NODE in EVERY INDEX to point to the new physical address.

A dynamic padding mechanism was added to reserve extra room in advance, but it consumes enormous storage overhead and does not fully solve the ripple I/O effect risk.

The Amdahl’s Law Bottleneck: The Database-Level Lock

Because RAM is accessed through raw C++ pointers, ensuring thread-safety becomes extraordinarily expensive. Initially, MMAPv1 used a single global process-level lock (a Global Reader-Writer Lock). It was later relaxed down to a Database-level Lock.

This means that if one thread is writing to the Users collection, the entire database — including Orders and Products — gets locked! No one can read or write anything.

Amdahl’s Law provides the mathematical proof of this multi-threaded performance degradation: Let $P$ be the fraction of execution time the system runs in parallel, and $S = 1 – P$ be the sequential execution time (spent inside the lock’s critical section). The maximum achievable speedup when scaling out to $N$ CPU cores is:

$$ text{Speedup}(N) = frac{1}{(1 – P) + frac{P}{N}} $$

In MMAPv1, since a single lock spans the entire database, the sequential fraction $1 – P$ is enormous. As the number of connections $N to infty$, performance is capped at:

$$ lim_{N to infty} text{Speedup}(N) = frac{1}{1 – P} $$

Buying a 128-core server for MMAPv1 is money thrown out the window, since 127 cores would sit continuously in a spin-waiting state waiting for the lock.

The Page Fault Thrashing Nightmare and the “Yield” Surrender

The worst outcome happens when Database Lock combines with Major Page Fault.

  1. Thread $T_1$ acquires the Write Lock on the database.
  2. $T_1$ tries to write into a document located at virtual address $V_x$.
  3. Unfortunately, the physical page corresponding to $V_x$ was just “silently” swapped out to disk by the OS’s LRU algorithm, which deemed it rarely used.
  4. $T_1$ is blocked by the kernel, generating a page fault, and must wait 10 milliseconds for the kernel to pull the data back up from disk.
  5. Throughout those 10 milliseconds, $T_1$ IS STILL HOLDING THE WRITE LOCK.
  6. Thousands of other connections freeze completely. The system “hangs” for no apparent reason.

To save itself, MongoDB integrated a Yield mechanism. Before accessing memory, it calls the Linux mincore() API to ask the OS whether that page is currently resident in RAM. If not, the thread voluntarily releases the lock (Yield lock), triggers a background process to push the I/O request (Background Page-in), and then waits. This patchwork solution made the concurrency state machine twisted and unpredictable.

The Demise of MMAPv1 and the Shift to WiredTiger

MMAPv1’s inadequacy became a fatal barrier as Big Data exploded. Unconditional dependence on the OS kernel for memory handling meant the DBMS was “blind” to its own data. The operating system does not understand the difference between a B-Tree Root Node (which must be pinned tightly in RAM) and a historical Document (which can be freely pushed to disk).

The MVCC and User-Space Compromise

In 2014, MongoDB acquired WiredTiger, marking a shift in micro-architectural engineering philosophy. WiredTiger rejects mmap‘s automatic mechanism. It builds its own Buffer Pool entirely within the application’s User-Space. This grants absolute control: WiredTiger uses an LFU/LRU-variant replacement algorithm combined with Semantic Hierarchy Awareness, ensuring the B-Tree structure always stays resident in RAM.

Instead of a crude Database-level lock (coarse-grained locks), WiredTiger applies Multi-Version Concurrency Control (MVCC). Every write operation does not directly modify the old data row, but instead creates a new copy in RAM, linked via pointer chains. As a result:

  • Readers never block Writers.
  • Locking happens at the Document level.
  • The parallelization ratio $P approx 1$. Amdahl’s Law now allows WiredTiger to make full use of the power of a 128-core server.

The Power of Block Compression

Abandoning mmap also unlocked the ability to compress data. Because mmap maps disk $leftrightarrow$ RAM 1:1, the on-disk data could never be compressed (compressing it would break the virtual byte-offset addressing).

With an independent Buffer Pool structure, data in RAM is decompressed for processing, but when the flusher thread writes it down to disk, WiredTiger passes the block through the zstd or Snappy algorithm. The compression ratio $mathcal{C}_{ratio} = frac{S_{uncompressed}}{S_{compressed}}$ easily reaches $3x$ to $5x$. This not only reduces cloud storage costs, but also multiplies I/O throughput (since the disk reads far fewer bytes for the same amount of information).

// Abstract pseudocode showing the superiority of a User-Space Buffer Pool (like WiredTiger)
struct BufferPoolManager {
    page_table: HashMap<LogicalPageId, FrameId>, // The DBMS's own private virtual mapping
    frames: Vec<PageFrame>,
    eviction_policy: Box<dyn EvictionAlgorithm>,
    wal_manager: WriteAheadLog,
}

impl BufferPoolManager {
    fn fetch_page(&mut self, page_id: LogicalPageId) -> Result<&mut PageData, DiskError> {
        if let Some(&frame_id) = self.page_table.get(&page_id) {
            // Cache Hit: update Semantic LFU
            self.eviction_policy.record_access(frame_id);
            Ok(self.frames[frame_id].get_data_mut())
        } else {
            // Cache Miss: the DBMS coordinates I/O itself — no Kernel Interrupt / stalled Page Fault!
            let frame_id = self.evict_oldest_document_page()?; 
            
            // Use O_DIRECT + Async I/O (io_uring) to pull the data up
            let compressed_data = disk_manager.async_read_page(page_id)?;
            let raw_data = snappy::decompress(compressed_data);
            
            self.frames[frame_id].fill(raw_data);
            self.page_table.insert(page_id, frame_id);
            Ok(self.frames[frame_id].get_data_mut())
        }
    }
}

Conclusion

The end of MMAPv1 was not a betrayal of a poorly designed system, but proof of the inevitable evolutionary lifecycle of systems software. mmap fulfilled its historical role brilliantly: it helped MongoDB launch at lightning speed, attract developers, and dominate the Web 2.0 era.

But upon entering the world of multi-threaded Big Data and Microservices architecture, the mathematical laws governing bandwidth and lock contention could not be evaded. By reclaiming user-space with WiredTiger’s MVCC, an independent Buffer Pool, and Block Compression, MongoDB reshaped itself from a primitive NoSQL solution into a mature, enterprise-grade transactional datastore.

The deepest lesson left for engineers: Never delegate an application’s core semantic domain knowledge into the hands of a general-purpose operating-system kernel.
