`io_uring` 対 `epoll`：データベースアーキテクチャにおける非同期I/Oの新時代

July 22, 2026 · admin · 4 分で読めます

エグゼクティブサマリーと中核課題（Core Problem Statement）

過去20年以上にわたり、Linuxサーバーのエコシステムは、C10K問題（1万の同時接続を処理する課題）を克服するためにepoll機構（kernel 2.5で導入）に依存してきた。しかし、ストレージハードウェアが機械式ディスク（HDD、ミリ秒単位のレイテンシ）からPCIe NVMe Gen 4/5規格のソリッドステートドライブ（マイクロ秒単位のレイテンシ）へと進化するにつれ、epollと従来の非同期I/Oモデルはマイクロアーキテクチャレベルで致命的な弱点を露呈することとなった。

中核課題：ScyllaDB、Aerospike、PostgreSQLのような現代的なデータベースエンジンは、ストレージの物理的限界によってではなく、Linuxオペレーティングシステムそのもののソフトウェアオーバーヘッドによってパフォーマンスを「絞め殺されて」いる。 epollや旧来のPOSIX AIO標準を用いて非同期I/O操作を行うには、CPUがユーザー空間（User Space）とカーネル空間（Kernel Space）の境界を絶えず越えなければならない。read()、write()、epoll_wait()といった各システムコール（syscall）は、コンテキストスイッチ、ページテーブル保護（KPTI）、データコピーのために数千CPUクロックサイクルを要求する。高速なNVMe環境では、CPUはデータを実際に移動させる時間よりも、OSとの「事務手続き」に多くの時間を費やしてしまう。

io_uring（Linux kernel 5.1から導入）の登場は、Linuxのi/Oアーキテクチャを完全に解体し再構築する革命である。ユーザー空間とカーネル空間の間で直接共有される2つのリングバッファ（Ring Buffer）をメモリマップ（mmap）することで、io_uringはアプリケーションが単一のシステムコールも発行することなく、数百万件のI/O要求を送信し、その結果を受け取ることを可能にする。

本レポートでは、io_uringのマイクロアーキテクチャにおける量子的原理を詳細に分析し、epollおよびlinux-aioの物理的限界と対比させ、データベース管理システム（DBMS）がこの技術的驚異を活用するために自らのコアエンジン全体をどのように再設計しているかを解説する。

epollの危機と非同期I/Oの起源

epollの本質（イベント駆動型のレディネス通知）

epollはネットワークI/O（ネットワークソケット）のために生まれた。その実行モデルは、完了通知（completion notification）ではなく、準備完了通知（readiness notification）という性質に縛られている。

動作の流れ：

  1. epoll_wait()を呼び出す（スレッドをブロックし、待機する）。
  2. ネットワークカードがパケットを受信すると、カーネルがあなたを起こす：「ソケット5にデータが届いています、読み取ってください」。
  3. アプリケーションはさらにread(5)というシステムコールを呼び出し、カーネルバッファからユーザーバッファへ実際にデータをコピーしなければならない。

総レイテンシ$T_{epoll}$は次のように数学的にモデル化できる。

$$T_{epoll} = t_{syscall(epoll_wait)} + t_{ctx_switch} + t_{syscall(read)} + t_{vfs_lookup} + t_{hardware_io} + t_{interrupt}$$

ディスクストレージI/Oにおけるepollの惨事

epollはネットワークに対しては見事に機能するが、ローカルディスク（ファイルシステムI/O）に対しては完全に、そして悲惨に失敗する。 Linuxでは、通常のストレージファイル（ext4、xfsなど）は常にepollに対して「準備完了（Ready）」状態を報告する。ディスク上のファイルに対してepoll_wait()を呼び出すと、それは即座に返る。しかしread()を呼び出したとき、もしデータがRAM上に存在しなければ（ページキャッシュミス）、Linuxカーネルはアプリケーションのスレッドを黙って（silently）ブロックし、ディスクからデータが届くのを待つ。

この「沈黙のブロッキング（silent blocking）」という挙動は、イベントループ（Event-Loop）の設計原理を完全に破壊する。たった一つのread()呼び出しがスタール（stall）するだけで、同じNodeJSやNGINXスレッド上で動作している何千もの他のクライアント接続がフリーズしてしまう可能性がある。

Linux AIO（io_submit）の崩壊

非同期のディスクI/Oを処理するため、Linuxはかつてlinux-aioインターフェース（io_submitとio_geteventsを介したもの）を提供していた。しかし、Linus Torvalds自身がこのアーキテクチャを公然と批判している。そこには数々の欠陥があった。

  1. O_DIRECTしかサポートしない： バッファリングを自前で管理し、ページキャッシュを完全にバイパスしなければならない。バッファードI/Oでは機能しない。
  2. メタデータによって依然としてブロックされる： O_DIRECTを使っていても、ファイルシステムがブロック割り当てやiノードロックの取得を必要とする場合、io_submitはブロックされる可能性がある。
  3. システムコールのコスト： 1バッチの要求を送信するだけでも、少なくとも1回のio_submitシステムコールが必要となる。

これらの拙い設計を回避するため、PostgreSQL、MySQL、MongoDBのようなデータベースは巨大なスレッドプール（Thread Pool）を作らざるを得なかった。彼らはブロッキングするread/write呼び出しをバックグラウンドスレッドに投げ込み、メインスレッドへの影響を避けている。これら何百ものI/Oスレッド間で絶え間なく発生するコンテキストスイッチは、CPUのL1/L2キャッシュに対する大虐殺（thrashing）を引き起こす。

io_uringのマイクロアーキテクチャ：共有メモリの驚異

io_uringは、Jens Axboe（Linuxカーネルのブロックレイヤーのメンテナ）によって、これまでの世代のあらゆる欠点を一掃するために設計された。それは「準備完了（Readiness）」モデルから「完了（Completion）」モデルへと移行する。あなたはカーネルに作業の記述を渡し、カーネルはAからZまでそれを処理し、完了すると結果をメモリに書き戻してくれる。

ロックフリーのリングバッファデータ構造

io_uringは一方向の2つの循環リングバッファを構築する。

  1. サブミッションキュー（Submission Queue、SQ）： タスク送信用のキュー（ユーザー -> カーネル）。SQE（Submission Queue Entry）スロットを含む。各SQEは「ファイルXから4KBを読み取り、バッファYへ入れる」といったコマンドを記述する。
  2. コンプリーションキュー（Completion Queue、CQ）： 結果受信用のキュー（カーネル -> ユーザー）。CQE（Completion Queue Entry）スロットを含む。各CQEは「先ほどのコマンドが読み取りを完了し、エラーコード0（成功）を返した」ことを報告する。

これら2つの配列構造は、アプリケーションプロセスの仮想メモリに直接メモリマップ（mmap）される。つまり、アプリケーションとカーネルは文字通り同じRAM領域を見ているのである。

メモリバリアによる同期

ここでは単一のロック（ミューテックスやスピンロック）すら使われていない！同期は完全に、アトミックなHeadポインタとTailポインタ、そしてハードウェアISAレベルのメモリバリア（Memory Barrier）の組み合わせに依存している。

データベースエンジンが100件の書き込みタスクをディスクへ送信したい場合、単にRAM上の100個のSQEスロットにデータを書き込み、SQ_Tailポインタをインクリメントするだけでよい。次に、カーネルを「呼び鈴」で起こして処理させるために、io_uring_enter()という単一のシステムコールを1回だけ呼び出す。 かつて100回のwrite()システムコールが必要だったところが、今ではわずか1回で済む。

究極の境地：SQPOLLによるシステムコールの完全排除

超低レイテンシ（ultra-low latency）の頂点に到達するため、io_uringはIORING_SETUP_SQPOLLという設定フラグを提供する。 この機能を有効にすると、カーネルは特定のCPUコアに固定（pin）された専用のカーネルスレッドを生成する。このスレッドは、アプリケーションのSQ_Tailポインタを継続的にポーリング（polling）するループを実行し続ける。

アプリケーションがSQに要求をプッシュすると、カーネルスレッドは共有メモリ領域を通じてそれを即座に検知し、自動的にその要求を拾い上げて処理する。この瞬間、システムコールの数はちょうどゼロである。 アプリケーションはI/Oを送信し、結果を受け取るが、コンテキストスイッチは一切発生しない。このときのI/Oレイテンシ（$T_{iouring_sqpoll}$）は、PCIeバス上を電気信号が伝わる時間だけとなる。

$$T_{iouring_sqpoll} = t_{mem_barrier} + t_{pcie_dma_transfer} + t_{nvme_flash_prog}$$

データベースエンジンのための高度な武器

io_uringは基本的なI/Oを解決するだけでなく、データベースエンジンに一連のロケットランチャーのような武器を装備させる。

固定バッファの登録（Fixed Buffers）

通常、read()やwrite()を呼び出すとき、Linuxカーネルはiovec構造体を設定し、ハードウェアが直接DMAできるようユーザーのRAMページをIOMMU（Input-Output Memory Management Unit）にマッピングし、その後再びアンマップしなければならない。この手順は非常にコストが高い。 io_uringを使えば、データベースは巨大なRAMブロックを事前に登録（pre-register）できる。カーネルはそのRAM領域を物理的にピン留め（pin）し、IOMMUの設定をあらかじめ構成する。以降、すべてのI/O操作（IORING_OP_WRITE_FIXED）は、いかなるアドレス変換手続きも経ることなく、ディスクコントローラからそのRAM領域へ直接飛んでいく。

連結されたリクエスト（Linked SQEs）

データベースにおいて、I/Oの順序は生死を分ける問題である。例えば、まずデータをディスクへ書き込み（コマンド1）、それがフラッシュされたことを保証し（コマンド2 — fsync）、それから初めてメタデータを更新する（コマンド3）必要がある。 Linux AIOでは、これを逐次的に行わなければならなかった。コマンド1の完了を待つ -> コマンド2を送信 -> コマンド2の完了を待つ -> コマンド3を送信、という具合である。 io_uringではIOSQE_IO_LINKフラグを使用する。3つのコマンドすべてを一度にキューへ送信すればよい。カーネルは、コマンド1が成功した場合にのみコマンド2が実行されることを保証する。これにより、ユーザーとカーネル間のやり取りの回数が大幅に削減される。

ネットワークとストレージの統合

io_uringの最大のブレークスルーは、あらゆる種類の操作（ネットワーク、ファイル、タイムアウト、fsync、fallocateなど）をサポートしている点である。 データベースはもはや2つのアーキテクチャ（ネットワークソケット用のepollとディスクI/O用のスレッドプール）を維持する必要がない。 アーキテクトは、1つのCPUコアに対して単一のイベントループ（Single Event-Loop）を1つ構築するだけでよい。そのコアはTCP接続を受け入れ（IORING_OP_ACCEPT）、HTTPリクエストを読み取り（IORING_OP_RECV）、データをディスクへ書き込む（IORING_OP_WRITEV）——これらすべてが、たった一つのio_uringリングへとまとめて詰め込まれる。

C++によるノンブロッキング実装

以下は、liburingを用いた現代的なストレージエンジンの疑似コードによる解剖モデルである。アプリケーションがC++のクラスオブジェクト（コンテキスト）をuser_dataに添付し、結果が返ってきた際にコマンドと照合する様子に注目してほしい。

#include <liburing.h>
#include <memory>
#include <cstdint>
#include <iostream>
#include <stdexcept>

// アプリケーション内部のコンテキストを運ぶカスタムリクエスト構造体
struct IOTransactionContext {
    int file_descriptor;
    uint64_t disk_offset;
    std::unique_ptr<char[]> memory_buffer;
    size_t length;
    uint32_t transaction_id;
};

class UltraFastStorageEngine {
private:
    struct io_uring ring;
    const unsigned int RING_DEPTH = 4096;

public:
    UltraFastStorageEngine() {
        struct io_uring_params params = {};
        // 最大限の最適化：SQPOLLを使ってシステムコールを完全に排除する
        params.flags |= IORING_SETUP_SQPOLL;
        params.sq_thread_idle = 2000; // 仕事がない場合、2ms後にカーネルスレッドをスリープさせる
        
        if (io_uring_queue_init_params(RING_DEPTH, &ring, &params) < 0) {
            throw std::runtime_error("カーネルが非対応、またはulimitによる制限があります！");
        }
    }

    void submit_async_write(IOTransactionContext* ctx) {
        // 共有リングバッファから空のSQEスロットを取得
        struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
        if (!sqe) {
            // SQが満杯なので、カーネルに消費させるため能動的に送信する
            io_uring_submit(&ring);
            sqe = io_uring_get_sqe(&ring);
        }
        
        // 低レベルの非同期書き込みオペコードを設定
        io_uring_prep_write(sqe, ctx->file_descriptor, 
                            ctx->memory_buffer.get(), ctx->length, ctx->disk_offset);
                           
        // 重要：CQE受信時にコンテキストを復元できるよう、SQEのメタデータ（64ビット整数）にC++オブジェクトのポインタを添付する
        io_uring_sqe_set_data(sqe, ctx);
    }

    void reap_completions_lockfree() {
        struct io_uring_cqe *cqe;
        unsigned head;
        unsigned count = 0;

        // ローカルRAM上だけでCQリングバッファを走査する（システムコールなし）
        io_uring_for_each_cqe(&ring, head, cqe) {
            // user_dataポインタをキャストし直し、コンテキストを復元する
            IOTransactionContext* ctx = static_cast<IOTransactionContext*>(io_uring_cqe_get_data(cqe));
            
            if (cqe->res < 0) {
                std::cerr << "トランザクション " << ctx->transaction_id 
                          << " でI/Oエラー（エラーコード：" << cqe->res << "）n";
            } else {
                // 成功時のビジネスロジックを処理する
                finalize_transaction(ctx, cqe->res);
            }
            count++;
        }
        
        if (count > 0) {
            // アトミックなCQ Headを更新し、収穫が完了したことをカーネルへ伝える
            io_uring_cq_advance(&ring, count);
        }
    }

private:
    void finalize_transaction(IOTransactionContext* ctx, int bytes_written) {
        // ネットワーク経由でクライアントへACKを送るか、WALへ記録する
        delete ctx; // メモリを片付ける
    }
    
    ~UltraFastStorageEngine() {
        io_uring_queue_exit(&ring);
    }
};

システムアーキテクトへの教訓とベストプラクティス

epollからio_uringへの世代交代は着実に進行している（Redis 7.0、PostgreSQL 15、NodeJS 20+も徐々に試験導入を進めている）。しかし、この宝刀を扱うには深い理解が必要である。

  1. OSキャッシュの限界を理解する： アプリケーションがio_uringでファイルを読み取る際、そのデータがすでにOSページキャッシュ（RAM）に存在する場合、カーネルのバックグラウンドワーカーio_wqが介入する必要が生じ、かえって遅くなる。io_uringが（他のあらゆる機構を大きく引き離す）絶頂のパフォーマンスに達するのは、O_DIRECT（ダイレクトI/O）フラグと組み合わせたときである。データベースは自前のバッファプールを構築し、オペレーティングシステムをバイパスしなければならない。
  2. セキュリティ上の脆弱性に注意する： カーネル構造体をmmapで直接ユーザー空間に露出させることは、歴史的に権限昇格のリスク（CVE）と数多く結びついてきた。コンテナシステム（Docker、Kubernetes）やSELinuxは、デフォルト設定で（seccompフィルタを通じて）io_uringを無効化していることが多い。DevOpsエンジニアは明示的にそれを解除（io_uringのシステムコールをホワイトリスト化）しなければ、データベースを起動させることすらできない。
  3. メモリのライフサイクルを慎重に管理する： C/C++/Rustでio_uringをプログラミングする際、サブミッションキュー（SQE）に渡すバッファは、完了通知（CQE）を受け取るまで有効でなければならない。もし不注意なスレッドがディスクの書き込み完了前にそのバッファを解放（delete/free）してしまうと、カーネルはゴミとなったRAM領域へDMAでデータをコピーしてしまい、システム全体を破壊するメモリ破損（Memory Corruption）を引き起こす。
  4. 超高速NVMe向けにポーリングI/O（IOPOLL）を使う： Intel OptaneやPCIe Gen 5ドライブのように応答レイテンシが10マイクロ秒未満の場合、完了を知らせるハードウェア割り込み（IRQ）機構そのものがボトルネックとなる（コンテキストスイッチに3〜4us要する）。IORING_SETUP_IOPOLLフラグを有効にし、カーネルがスリープする代わりにSSDの状態をビジーウェイトでポーリングするようにするとよい。これは1コアのCPUを100%燃焼させるが、物理的な量子限界に近いレイテンシをもたらす。
  5. Webサーバー向けにepollを早計に捨ててはならない： io_uringはネットワーク（ネットワークソケット）も扱えるとはいえ、ベンチマークによれば、典型的なWebサーバー（短命なHTTP/TCP接続）においてio_uringがepollに対して持つパフォーマンス上の優位性は、アーキテクチャ全体を書き直す複雑さを正当化するほど大きくはない。io_uringはストレージ領域における絶対の王者だが、ネットワーク領域ではepollは依然として歴戦の猛者であり続けている。
