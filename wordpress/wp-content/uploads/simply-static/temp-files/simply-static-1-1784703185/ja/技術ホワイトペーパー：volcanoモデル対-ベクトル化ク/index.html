技術ホワイトペーパー：Volcanoモデル対 ベクトル化クエリ実行 ― 現代データベースエンジンにおける比較

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー

リレーショナルデータベースのクエリ実行エンジンの設計と実装は、過去30年間で根本的かつ構造的な変革を遂げてきました。半導体ハードウェアアーキテクチャの絶え間ない幾何級数的な進化に牽引され、複雑な分析クエリを評価するためのパラダイムは劇的に変化しています。

データベース黎明期には、ハードディスクドライブ（HDD）のI/O帯域幅が絶対的なボトルネックでした。このディスク律速の時代において、Volcanoイテレータモデルが事実上の標準として登場し、クエリ評価のための非常にエレガントで、普遍的に組み合わせ可能な抽象化を提供しました。しかし、半導体の微細化がCPU処理能力、コア数、揮発性メインメモリ容量を急激に押し上げるにつれ、古典的なVolcanoモデルは現代のハードウェア上で深刻な構造的非効率性を示すようになり、壊滅的に低いクロックあたり命令数（IPC）スループットにつながりました。

このアーキテクチャ上の乖離は、ベクトル化クエリ実行（Vectorized Query Execution）の開発という、根本的なパラダイムシフトを必要としました。本ホワイトペーパーは、両方の実行モデルについて、その基本的なアルゴリズム、メモリアクセストポロジー、オペレーティングシステムとの相互作用、そして絶対的なハードウェア限界を解剖する、厳密かつ微視的で数学的根拠に基づいた分析を提示します。読者は、現代のデータベースがシリコンレベルでどのようにデータを処理するか、レガシーアーキテクチャに内在する問題、そしてデータベースエンジンを最適化するための重要な教訓について、深い理解を得ることができるでしょう。

はじめに：中核となる問題提起

問題: なぜ一部のデータベースエンジンは、全く同じハードウェア上で、分析クエリ（OLAP）を桁違いに高速に実行できるのでしょうか？

その答えは、アルゴリズムの高レベルな理論的複雑性（例えばビッグO記法）にあることはめったにありません。むしろ、性能差は、それらの高レベルなアルゴリズムを、正確なCPU命令と階層的なメモリアクセスへと機械的に変換する過程に深く根ざしています。レガシーなデータベースエンジンは、ディスクがボトルネックであり、CPUが常に待機している時代に設計されました。今日では、データはしばしば完全にRAM上にキャッシュされており（インメモリデータベース）、ボトルネックはCPUがメモリからデータを取得する能力（メモリ帯域幅）、命令をデコードする能力、そして実行パイプラインを飽和させ続ける能力へと、急激にシフトしています。

現代のスーパースカラCPU上で従来のクエリ実行モデルを実行すると、CPU使用率の壊滅的な低下が観察されます。CPUは、意味のあるリレーショナル演算を実行するのではなく、その大部分のクロックサイクルを、メモリ待ちのストールや投機的な予測ミスからの回復に費やしています。このソフトウェアアーキテクチャとハードウェアの現実との間の断絶こそが、ベクトル化実行が解決する中核的な問題なのです。

Volcanoモデルとタプル単位のパイプライン処理

Volcanoイテレータモデルは、1990年代初頭にGoetz Graefeによって正式に導入され、物理的なクエリ実行を、物理演算子の有向非巡回グラフ（典型的には木構造）として実装します。各演算子ノードは、特定のリレーショナル代数演算（選択、射影、結合、集約）に対応します。

アーキテクチャとデータフロー

Volcanoモデルを特徴づけるのは、open()、next()、close()という3つの仮想メソッドによって古典的に定義される、非常に標準化されたタプル単位の手続き型インターフェースです。データは、葉ノードからルートノードへと一方向に流れます。

親演算子がnext()呼び出しを実行すると、子演算子は自身の内部状態マシンを処理し、後続のnext()呼び出しを通じて自身の子からデータを取得し、ロジックを適用して、完全にマテリアライズされた単一のタプルを生成します。

効率という幻想

このパイプライン化された需要駆動型のアプローチは、中間結果セットがめったにマテリアライズされないため、従来のディスクベースのシステムでは非常にメモリ効率が良いものでした。しかし、現代のスーパースカラCPU上では、壊滅的かつ体系的なマイクロアーキテクチャ上のオーバーヘッドをもたらします。

解釈オーバーヘッドと仮想呼び出し

非効率性の主な原因は、仮想関数呼び出し（動的ディスパッチ）の広範な使用にあります。物理演算子は汎用インターフェースを継承する多態的なサブクラスとして実装されるため、すべてのnext()呼び出しは構造的に動的ディスパッチを必要とします。

現代のマイクロアーキテクチャでは、CPUの分岐ターゲットバッファ（BTB）が宛先を正確に予測できない場合、間接分岐が実行パイプラインをストールさせます。数十億のタプルを処理する深いVolcano実行木では、BTBは激しいキャパシティミスを経験し、持続的な予測ミスにつながります。

$$T_{volcano} approx sum_{i=1}^{N} sum_{j=1}^{D} left( C_{vcall}^{(i,j)} + C_{logic}^{(i,j)} + P_{miss}^{(i,j)} times C_{penalty} right)$$

この式において、$C_{vcall}$は単純な演算の実行時間を頻繁に支配します。手続き的な制御フローを調整するために明示的に必要とされる膨大な量のアセンブリ命令は、実際のリレーショナルロジックを実行する命令数を大幅に上回ります。

キャッシュスラッシングと行指向ストレージ

さらに、Volcanoモデルはネイティブに、N分ストレージモデル（NSM）、すなわち根本的に行指向ストレージとして知られる形式に従うデータを処理します。行指向タプルの特定のフィールドを読み取ると、メモリアクセスが分散します。現代のプロセッサは、厳密に64バイトのアラインされたチャンク（キャッシュライン）でDRAMからメモリを取得します。射影演算子が256バイトのタプルからわずか4バイトの整数のみを必要とする場合でも、ハードウェアは64バイトのキャッシュライン全体を取得し、貴重なメモリ帯域幅を浪費します。

// Volcano実行をモデル化するC++疑似コード
class PhysicalOperator {
public:
    virtual Tuple* next() = 0; 
};

class FilterOperator : public PhysicalOperator {
private:
    PhysicalOperator* child_;
    Predicate* pred_;
public:
    Tuple* next() override {
        // 木を下る繰り返しの仮想呼び出しがBTBの予測性を破壊する
        while (Tuple* t = child_->next()) {
            if (pred_->evaluate(t)) return t;
        }
        return nullptr;
    }
};

ベクトル化クエリ実行とハードウェアとの共生

ベクトル化クエリ実行は、スーパースカラシリコンとのメカニカル・シンパシー（機械的親和性）を達成し、ハードウェア利用率を最大化するために明示的に設計された、根本的でゼロベースのアーキテクチャ再設計を表しています。MonetDB/X100（VectorWise）によって先駆的に開発されたこのパラダイムは、タプル単位のアプローチを完全に放棄します。

データフロー優位への転換

個別のタプルを処理する代わりに、演算子は列指向形式で厳密に整理された、密で固定サイズのタプルのバッチ（VectorBatch）を渡すことで通信します。典型的な最適ベクトルサイズは数学的に1024から4096要素の範囲であり、ワーキングセットがCPUのL1またはL2データキャッシュに完全に収まるように慎重に調整されています。

膨大な数のタプルのバッチに手続き的な制御フローのオーバーヘッドを分散させることで、ベクトル化実行は仮想関数呼び出しの統計的影響を指数関数的に削減します。next()メソッドは単一のTupleポインタではなくVectorBatch参照を返し、計算プロファイルを制御フロー優位のパラダイムから厳密なデータフロー優位のパラダイムへと変換します。

密なインナーループとCPUプリフェッチ

演算子がベクトルバッチを処理する際、リレーショナルロジックは、密で連続的な配列を順次反復する、非常に予測可能でシンプルなforループとして物理的に実装されます。

この列指向レイアウトは、CPUキャッシュ階層に組み込まれた空間局所性の前提と完全に一致します。シーケンシャルなメモリアクセスは、ハードウェアのストライドプリフェッチャーおよびネクストラインプリフェッチャーを確実にトリガーし、実行ユニットが要求する前にDRAMからL1/L2キャッシュへキャッシュラインを積極的にロードします。これにより、メモリレイテンシが効果的に隠蔽され、メモリ帯域幅が完全に飽和します。

// ベクトル化実行を示すRust疑似コード
struct VectorBatch {
    columns: Vec<Vec<u8>>, 
    selection_vector: Vec<u16>, 
    count: usize,
}

impl FilterOperator {
    fn next_batch(&mut self) -> Option<VectorBatch> {
        let mut batch = self.child.next_batch()?;
        let data = get_typed_column::<i32>(&batch, self.filter_col_idx);
        let mut new_sel = Vec::with_capacity(batch.count);
        
        // 自動ベクトル化に非常に適した密なインナーループ
        for i in 0..batch.count {
            let row_idx = batch.selection_vector[i] as usize;
            // 分岐のない評価
            if data[row_idx] > self.threshold {
                new_sel.push(row_idx as u16);
            }
        }
        
        batch.selection_vector = new_sel;
        batch.count = batch.selection_vector.len();
        Some(batch)
    }
}

ベクトル化された時間（$T_{vectorized}$）の数式は、構造的な制御オーバーヘッドの極端な償却を示しています。

$$T_{vectorized} approx sum_{j=1}^{D} left( frac{N}{B} times C_{vcall}^{(j)} + sum_{k=1}^{N/B} C_{vector_logic}^{(j,k)} right)$$

ベクトルバッチサイズ$B$が増加するにつれ、解釈の償却コストはゼロに近づきます。実行は、ハードウェアのメモリ帯域幅（$BW_{dram}$）とALUスループットによって厳密に制約されるようになります。

SIMD（単一命令・複数データ）の力

ベクトル化の中で最も技術的に革新的な側面は、SIMDアーキテクチャ（Intel AVX-512、ARM SVE）との深い相乗効果です。これらの命令セットは、単一のクロックサイクル命令が複数のスカラーデータ要素に対して同時に動作することを可能にします。

ベクトル化エンジンはプリミティブ型の連続配列にデータを格納するため、現代のコンパイラ（LLVM/GCC）は密なインナーループを自動ベクトル化できます。あるいは、データベースエンジニアは明示的なSIMDイントリンシクスを利用します。単一のAVX-512命令は、16個の32ビット整数比較を物理的に同時に実行できます。

$$T_{vectorized_simd} approx frac{N}{B} times C_{vcall} + N times frac{C_{vector_logic}}{W_{simd}}$$

分岐予測ミスを完全に排除するため、ベクトル化エンジンは分岐のないプログラミングを採用します。if文の代わりに、ビット演算またはSIMDマスク演算を利用して選択ベクトルや物理的なビットマップを計算します。制御依存関係をアルゴリズム的にデータ依存関係へと変換することで、CPUパイプラインは完全に飽和した状態を維持し、分岐予測器の限界を回避します。

マイクロアーキテクチャ的な比較分析

科学的に厳密な比較により、マイクロアーキテクチャの実行に根ざした大きな格差が明らかになります。

  • リオーダーバッファ（ROB）の飽和: Volcanoモデルの頻繁な仮想呼び出しは直列化バリアとして機能し、ROBを枯渇させます。ベクトル化ループは、ROBに独立した命令の膨大なプールを提供し、積極的な動的ループアンローリングと複数のALUにまたがる並行命令実行を可能にします。
  • TLBスラッシング: トランスレーション・ルックアサイド・バッファ（TLB）は仮想アドレスを物理アドレスに変換します。Volcanoの断片化されたヒープ割り当ては壊滅的なTLBスラッシングを引き起こし、高価なページテーブルウォークをトリガーします。ベクトル化エンジンは、巨大で事前割り当てされた列指向のチャンク上で動作し、完璧な空間局所性を示します。透過的巨大ページ（Transparent Huge Pages、1GBブロック）と組み合わせると、TLBミスは根絶されます。
  • ハッシュ結合におけるソフトウェアパイプライン化プリフェッチ: Volcanoでは、ハッシュテーブルをタプルごとにプローブすることで、ロジックがランダムメモリアクセス（CPUストール）と激しく絡み合います。ベクトル化実行では、ハッシュ結合はループフィッションによってこれらのフェーズを分離します。エンジンはSIMDを使用してベクトルのハッシュを計算し、バケットに対する明示的なソフトウェアプリフェッチ命令（__builtin_prefetch）を生成し、その後別のループで比較を行います。これにより、高レイテンシのメモリアクセスとCPU計算が物理的にオーバーラップします。

教訓とベストプラクティス

  1. メカニカル・シンパシーは譲れない: ソフトウェアアーキテクチャは、CPUを汎用チューリングマシンとして扱うことはできません。極限の性能を得るには、SIMD、キャッシュライン、TLB階層、OoO(アウト・オブ・オーダー)パイプラインといったシリコンの特徴を明示的に活用するようにソフトウェアを形作る必要があります。
  2. 制御フローよりデータフロー: 分析ワークロードにおいては、条件ロジック（分岐）からデータ操作（ビットマップ、選択ベクトル）へ移行することで、データ分布の偏りに関わらず決定論的で高スループットな実行が保証されます。
  3. OLAPにおいて列指向は王である: ストレージ形式と実行形式は一致していなければなりません。行指向のVolcanoエンジンに列指向ストレージを供給すると、絶え間ないタプルの再構築が必要になり、利点が相殺されてしまいます。列指向ストレージ（Parquet、ORC）と組み合わせたベクトル化実行は、分析性能の頂点を represents します。
  4. 解釈オーバーヘッドに注意: システムプログラミングにおける抽象化（仮想関数など）が「ゼロコスト」であることはめったにありません。密でデータ集約的なループでは、多態的なディスパッチが命令パイプラインを破壊します。
  5. メモリ帯域幅は最後のフロンティア: ベクトル化によってCPUサイクルを最適化した後、性能はRAMがデータを供給できる速度によって制約されるようになります。データ配置の管理とメモリフットプリントの最小化が依然として重要です。

結論

VolcanoモデルとVectorized実行モデルの間の構造的な乖離は、システムエンジニアリングにおけるより広範な原則を浮き彫りにします。データ量が指数関数的に拡大し、ムーアの法則が単一スレッドのクロック速度に物理的な限界を課す中で、ハードウェアを無視した(Volcano)エンジンとハードウェアと共生する(Vectorized)エンジンとの格差は広がる一方でしょう。ベクトル化クエリ実行は、超大規模分析データ処理の未来にとって揺るぎないアーキテクチャ上の基盤であることを確固たるものとし、データベースがシリコンレベルでどのように機能するかを根本的に再定義しました。
