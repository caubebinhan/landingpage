PostgreSQLにおけるTOASTアーキテクチャ: メモリ管理の芸術と特大データストレージ技術

July 22, 2026 · admin · 3 分で読めます

問題の所在: 巨大な値と小さなページ

固定ブロックストレージを採用するデータベースは、遅かれ早かれ同じ壁にぶつかる。数十メガバイトに達するTEXTフィールドやJSONBドキュメント、BYTEAバイナリを、わずか8キロバイトのディスクページにどう収めるか、という問題だ。

単純な解決策としてページサイズ自体を（たとえば1MBに）広げる手もあるが、それでは小さなOLTPクエリの効率が犠牲になる。ちょっとした読み取りのために不必要に大量のデータをディスクから引っ張ってくることになり、I/Oの増幅がすぐに手に負えなくなる。

PostgreSQLはページサイズには手をつけない。代わりに、規定サイズを超える値を別のサブシステムへ迂回させる仕組みを持っている。それが TOAST（The Oversized-Attribute Storage Technique） である。

TOASTは大きな属性値を裏で分割・圧縮し、テーブル本体の外（アウトオブライン）へ格納する。しかもSQLクエリ層からは完全に透過的で、アプリケーション側は何も意識する必要がない。JSONBカラムの裏側でTOASTが実際に何をしているのかを追うことは、CPUキャッシュ階層（TLB、L1/L2）の挙動や、PostgreSQLがOSのページキャッシュ汚染をどう避けているかを学ぶ良い教材にもなる。TOASTアーキテクチャを掘り下げる価値はまさにそこにある。

ページ境界の数学とTOAST閾値定数

なぜTOASTが必要になったのかを理解するには、まずPostgreSQLのディスクページの物理構造を見る必要がある。

デフォルトでは、ブロックサイズを表す定数 BLCKSZ は8192バイト（$2^{13}$バイト）である。この限られた領域にデータが無秩序に詰め込まれることはなく、すべてのページはスロット付きページ（Slotted Page）構造に従う。

  1. ページヘッダー（PageHeaderData）： 24バイトを占め、pd_lsn（WALによるクラッシュリカバリ用のログシーケンス番号）、pd_checksum、そして pd_lower・pd_upper のポインタといったメタデータを保持する。
  2. ラインポインタ配列（ItemIdData）： ページ末尾にある静的配列で、各エントリが実際のタプルを指す。

これらすべてを支配する設計原則がひとつある。ページをまたぐ断片化を許さないことだ。1つのタプルは必ず単一ページの原子的なI/O読み取りの中に完全に収まらなければならず、2ページにまたがることは決して許されない。

とはいえ、この制約だけでは不十分である。PostgreSQLはタプル密度、つまり1ページに巨大なレコードを1つだけ詰め込むのではなく、それなりの数のレコードを収められる状態も求めている。そこで設計者たちは TOAST_TUPLE_THRESHOLD というハードリミットを導入した。Cのソースコードでは次のように定義されている。

$$ TOAST_TUPLE_THRESHOLD = lfloor frac{BLCKSZ}{4} rfloor – 1 $$

$BLCKSZ = 8192$ を代入すると 2047バイト になる（実際にはメモリアライメントのために切り下げられ、PostgreSQLはおおよそ2040バイトとして扱う）。

挿入されたタプルがこの約2040バイトのラインを超えた瞬間、ストレージエンジンにはTOASTルーティングを起動する以外の選択肢がなくなる。タプルをこの限界の下まで削ぎ落としてヒープページに収まるようにし、収まりきらない分は別のテーブルへ押し出す。

TOASTポインタの内部構造: varatt_external

たとえばBYTEA列に格納された10MBのPDFファイルのような巨大な値は、その場で varatt_external と呼ばれる18バイトのポインタ構造に置き換えられる。

その基盤となる varlena（可変長配列）形式は、ビットマスキングによって状態をエンコードしている。ヘッダーバイトの先頭ビットを読むだけで、パーサーはそのデータが今どの状態にあるかを正確に把握できる。

  • PLAIN: 生データが非圧縮のままインラインに格納されている状態。
  • COMPRESSED: データは圧縮済みだが、依然としてインラインに格納されている状態。
  • EXTERNAL: データがアウトオブラインに押し出され、TOASTポインタで参照されている状態。

この18バイトの varatt_external ポインタは、次の4つのフィールドを持つ。

  1. va_rawsize — 圧縮前の元データのサイズ。
  2. va_extsize — 圧縮・分割後にディスク上で占める物理サイズ。
  3. va_valueid — このチャンク群に割り当てられたOID。
  4. va_toastrelid — チャンクを保持しているTOASTテーブルのOID。

この間接参照が機能し始めると、メインヒープ上の行は事実上、サイドテーブルへの検索キーとして振る舞うようになる。(id INT, created_at TIMESTAMP, payload JSONB) という列を持つテーブルで、payload が5MBだとしよう。メインヒープ上でこのタプルが占めるのは $4 text{ バイト (id)} + 8 text{ バイト (timestamp)} + 18 text{ バイト (TOAST pointer)} = 30 text{ バイト}$ のみであり、5MBのデータ自体はヒープページに一切触れない。

MVCCにおける意味

この18バイトのポインタは、単なる省スペーステクニックにとどまらない。MVCCの更新コストの構造そのものを変えてしまう。

PostgreSQLは既存の行を決して上書きしない。payload 列には触れずに UPDATE table SET status='done' WHERE id=1 を実行しても、エンジンは新しい物理タプルバージョンを丸ごと作り直す必要がある。

もしアウトオブライン格納がなければ、この更新のたびに5MBのJSONB全体を新しいタプルへコピーすることになり、これは書き込み増幅（Write Amplification）の典型例だ。TOASTポインタがあれば、PostgreSQLがコピーするのは18バイトの va_valueid 参照だけで済む。新旧どちらのタプルバージョンも、TOASTテーブルにある同じ5MBのペイロードを指し示す。データの重複はなく、書き込み量も増えず、SSDの摩耗も大幅に抑えられる。

TOASTシャドウテーブルとバイナリチャンキング

では実際にデータはどこへ行くのか。PostgreSQLは pg_toast.pg_toast_XXX（XXXは親テーブルのOID）という名前の付随テーブルを裏側で自動的に作成する。通常の d コマンドからは見えない。

このTOASTテーブルは固定の3列構成を持つ。

  1. chunk_id（OID型） — 18バイトポインタの va_valueid と一致する。
  2. chunk_seq（INT型） — 0から始まるシーケンス番号で、チャンクを順序どおりに再組み立てするために使う。
  3. chunk_data（BYTEA型） — そのチャンクの生バイト列。

チャンキングの数式

このシャドウテーブル内の行も TOAST_TUPLE_THRESHOLD を守らなければならない。そのため、PostgreSQLは各チャンクのサイズ TOAST_MAX_CHUNK_SIZE をデフォルトで 1996バイト に制限している。

必要なチャンク数は単純な天井関数（ceiling function）に従う。

$$ N_{chunks} = lceil frac{S_{compressed}}{1996} rceil $$

10MBのファイル（10,485,760バイト）であれば、$10,485,760 / 1996 approx 5254$ 行が pg_toast_XXX に生成される計算になる。

読み取りを高速化するため、PostgreSQLは (chunk_id, chunk_seq) の組み合わせに対して専用のB-Treeインデックス pg_toast_XXX_index を構築する。クエリがその列を参照すると、エグゼキュータは $mathcal{O}(log n + K)$（Kはチャンク数）の計算量でB-Treeを走査し、5254個の断片を1つの連続したバッファへつなぎ合わせ、解凍してから結果を返す。

圧縮アルゴリズムと情報理論的な限界

1996バイトのチャンクに切り分ける前に、PostgreSQLはまず圧縮を試みる。物理I/Oを減らすためだ。利用できるアルゴリズムは2つ。長らくデフォルトだった PGLZ (PostgreSQL Lempel-Ziv) と、速度重視でPG 14から追加された LZ4 である。どちらもリング辞書方式と反復文字列のマッチングに基づいて動作する。

ただし、PostgreSQLは闇雲に圧縮を試すわけではない。情報理論、すなわちShannonエントロピーの考え方を使ってリスクを事前に評価する。バイナリ文字列 $X$ の情報量は次式で表される。

$$ H(X) = -sum_{i=1}^{n} P(x_i) log_2 P(x_i) $$

すでに圧縮済みのファイル——JPEG、MP4、gzipアーカイブなど——を挿入すると、そのバイト列はホワイトノイズに近い状態にあり、エントロピーは理論上の上限に近づいている。この状態でLempel-Ziv系のアルゴリズムをかけても得るものはなく、むしろサイズが増え、CPUサイクルだけを浪費することもある。

この無駄を避けるため、PostgreSQLはシンプルなヒューリスティックを採用している。データの先頭部分だけを対象に試験的な圧縮を実行し、そのトライアルでサイズを25%以上削減できなければ（つまり元のサイズの75%未満に収まらなければ）、その場で圧縮処理そのものを中止する。値はヘッダーで非圧縮とマークされ、そのまま生データとして保存される。この一つのチェックのおかげで、圧縮不可能なデータを延々と挿入するようなアプリケーションが相手でも、PostgreSQLのスループットは安定を保てる。

// Pseudocode: TOASTヒューリスティクス圧縮のマイクロアーキテクチャ
struct varlena* toast_insert_or_update(struct varlena* datum, StorageType strategy) {
    size_t raw_size = VARSIZE_ANY_EXHDR(datum);
    if (raw_size <= 2040) return datum; // 安全なデータブロック
    
    // テスト圧縮フェーズ (Heuristic Check)
    struct varlena* compressed_datum = perform_lz4_compression(datum);
    if (compressed_datum != nullptr && VARSIZE_ANY_EXHDR(compressed_datum) < raw_size * 0.75) {
        // エントロピーが低く、LZ4圧縮に成功
        datum = compressed_datum;
    } else {
        // エントロピーが高すぎるため、圧縮を放棄してCPUサイクルを節約
        free_memory_context(compressed_datum); 
    }
    
    // Shadow TableへのBinary Chunkingを進める...
    // Return 18-byte External Pointer
}

ハードウェアキャッシュとガベージコレクション（VACUUM）との連携

TOASTの設計が本当に効いてくるのは、CPUのキャッシュ階層（TLB、L1/L2）やLinuxのページキャッシュとの相互作用の場面だ。

OSページキャッシュ汚染を避ける

数百万行を対象にフルスキャンする分析クエリ、たとえば SELECT id, status FROM massive_table を思い浮かべてほしい。もし10MBのJSONB値が各レコードにインラインで埋め込まれていたら、このスキャンはNVMe上の無関係なJSONBデータを何ギガバイトもRAMへ引きずり出すことになり、その過程でB-Treeインデックスのルートノードのような本当に有用なページをLinuxのページキャッシュから追い出してしまう。これがOSキャッシュ汚染であり、本来なら軽いはずのスキャンを遅くする典型的な原因だ。

TOASTを使えば、メインタプルが持つのは18バイトのポインタだけになる。数百万行がわずか数メガバイトのRAMに収まり、実際のTOASTデータはディスク上に留まったまま、SELECT payload FROM ... のように明示的に要求されたときだけRAMにロードされる。コア列に対するシーケンシャルスキャンは、キャッシュの中に無駄なデータを引きずり込まない分、高速なままでいられる。

TLBミスを抑える

タプルを密に詰め込むことには、もう一つ副次的な効果がある。CPUが64バイトのキャッシュラインや4KBのメモリページを読み込むとき、1回のフェッチでより多くの有用なレコードを取得できるようになる。これは仮想アドレスから物理アドレスへの変換におけるTLB（Translation Lookaside Buffer）ミスを減らし、結果としてコストの高いページテーブルウォークによる処理停止を回避することにつながる。

孤立チャンクとVACUUM

TOASTテーブルは意図的にメインテーブルへの外部キーを持たない。もし持たせれば、あらゆる操作でロック競合のコストが発生するからだ。その代償として、メインテーブルの行を DELETE しても、対応するTOASTテーブル側のチャンクはすぐには削除されない。誰かが片付けるまで、孤立したチャンクとしてそこに残り続ける。

その片付け役がAutovacuumである。Autovacuumがメインテーブルをスキャンしてデッドタプルを見つけると、そのポインタから18バイトのOIDを取り出し、pg_toast_XXX テーブルへ移動して、B-Treeインデックスを使い該当するチャンクを物理的に削除し、解放した領域をFree Space Mapへ返却して次の挿入に備える。

こうして見ると、TOASTはストレージ層に後付けされた単なる圧縮トリックではない。タプルのエンコーディング、シャドウテーブル、インデックス、圧縮ヒューリスティック、そしてガベージコレクタが一体となって連携するシステムである。この連携があるからこそ、PostgreSQLのような古典的な行指向RDBMSが、かつてNoSQLの独壇場だったJSON中心のドキュメント指向ワークロードを、性能を犠牲にすることなく快適にホストできるのだ。
