長時間稼働データベースプロセスにおけるメモリフラグメンテーション：マイクロアーキテクチャ的影響とアルゴリズム的緩和策

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリーと中核課題（Core Problem Statement）

高性能コンピューティングとデータエンジニアリングの領域において、長時間稼働するデータベースプロセス内で発生するメモリフラグメンテーション（メモリ断片化、Memory Fragmentation）は、静かでありながら致命的なパフォーマンスボトルネックを構成する。これはスループットとレイテンシが長期の稼働期間にわたって徐々に劣化する形で現れ、しばしば突発的なレイテンシスパイクや壊滅的なメモリ不足（Out-Of-Memory、OOM）クラッシュに至る。

中核課題：高い同時実行性を要求されるトランザクション処理（OLTP）や分析ワークロード（OLAP）向けに設計されたデータベース管理システム（DBMS）は、かつてない頻度——しばしば毎秒数百万回——でメモリの割り当て（allocation）と解放（deallocation）を実行する。数週間、数か月にわたる継続稼働の中で、仮想メモリサブシステムと相互作用するオペレーティングシステムの基盤となるメモリアロケータ（allocator）は、ヒープ内部で深刻な構造的エントロピー（structural entropy）に必ず直面する。

このエントロピーはメモリアドレス空間の連続性を破壊する。メモリが高度に断片化すると、論理的には連続しているデータ構造（B+Treeのノードや列指向配列（columnar array）など）が物理的には異なるメモリページに散在するようになる。この空間的分散は、現代のCPUキャッシュ設計に内在する空間的局所性（spatial locality）の前提を完全に無効化し、TLB（Translation Lookaside Buffer）ミス率を急騰させ、MMU（Memory Management Unit）にコストの高い多段ページテーブルウォークを強いる。最終的に、本来クエリを実行しているべきCPUは、RAMを待つ時間に費やされてしまう。

本ホワイトペーパーでは、フラグメンテーションが高まった際のマイクロアーキテクチャ的帰結、標準Cライブラリアロケータ（glibc）の不十分さ、そして現代のデータベースシステムに組み込まれた高度に専門化されたアルゴリズム的緩和策——Slabアロケータ、バッファプール（Buffer Pool）、リージョンベースのアリーナ（Region-based Arena）、Huge Pagesのようなカーネルとの明示的な相互作用を含む——について厳密に検証する。

メモリフラグメンテーションの理論的基盤

長時間稼働プロセスにおけるフラグメンテーションは、内部断片化（internal）と外部断片化（external）という2つの異なる病理に大別される。

内部断片化 対 外部断片化

  1. 内部断片化（Internal Fragmentation）は、メモリアロケータが要求されたサイズよりも厳密に大きなメモリブロックを用意する際に発生する。これは通常、物理ハードウェアのアライメント制約（例：8バイトまたは16バイト境界）や、アロケータ内部の「ビン（bin）」によって定義されるブロックサイズの量子化規則を満たすためである。この未使用でありながら割り当て済みの空間は無駄な容量を表し、利用可能な物理メモリの実効利用率を低下させる。
  2. 外部断片化（External Fragmentation）は、割り当て済みブロックの間に散在する、小さく非連続な空きメモリセグメントの増殖として現れる。これらの空きセグメントの合計は大きな割り当て要求を満たすのに十分な場合があっても、その分断された性質ゆえにそうした要求を満たすことができない。アロケータはmmapやsbrkを通じてOSからさらにメモリを要求せざるを得なくなり、技術的には十分な「空き」バイトを保有しているにもかかわらず、プロセスのRSS（Resident Set Size）を膨張させてしまう。

フラグメンテーションの数学

メモリフラグメンテーションの程度を厳密に定量化するために、フラグメンテーション比率$Phi$を定義する。 $M_{total}$をオペレーティングシステムから要求された総メモリ量とし、$M_{used}$をデータベースプロセスが生存中のデータ構造を格納するために実際に利用している総メモリ量とする。フラグメンテーション比率は次のように表される。

$$Phi = 1 – frac{M_{used}}{M_{total}}$$

$Phi$が時間$t$とともに単調に増加するにつれ、システムは臨界閾値に近づく。この閾値では、サイズ$s$の要求に対する割り当て失敗確率$P_{fail}(s)$が、たとえ$M_{total} – M_{used} gg s$であっても無視できない大きさになる。

この確率は、空きブロックサイズの幾何分布$F={f_1, f_2, dots, f_n}$を考慮した修正ポアソン分布を用いてモデル化できる。具体的には次の通りである。

$$P_{fail}(s) = prod_{i=1}^{n} (1 – H(f_i – s))$$

ここで$H(x)$はヘヴィサイドの階段関数（Heaviside step function）である。$P_{fail}(s)$が1に達すると、アロケータはコストの高い圧縮（compaction）機構を起動せざるを得なくなり、あるいはメモリが完全に枯渇した場合はLinuxのOOM killerを発動させることになる。

マイクロアーキテクチャ的影響：TLBとキャッシュ階層

フラグメンテーションの高まりがもたらすハードウェア上の帰結は深刻である。現代のプロセッサは、仮想アドレスから物理アドレスへの変換をキャッシュするためにTLB（Translation Lookaside Buffer）に大きく依存している。

TLBスラッシングとページテーブルウォーク

データベースプロセスが深刻な外部断片化を示すと、論理的には連続している配列が、物理的には不連続な、離れたページに頻繁にマッピングされる。この空間的分散は空間的局所性を破壊する。結果として、TLBミス率が劇的に上昇する。

$T_{hit}$をTLBヒット時のレイテンシ、$T_{miss}$をTLBミス時のレイテンシ（4段階または5段階の基数木（radix tree）を辿るコストの高いページテーブルウォークを必要とする）とする。実効メモリアクセス時間（Effective Memory Access Time、EMAT）は次のように定義される。

$$EMAT = P_{hit} cdot T_{hit} + (1 – P_{hit}) cdot T_{miss}$$

フラグメンテーションが増大するにつれ、TLBヒット率$P_{hit}$は急激に低下する。MMUはページテーブルを頻繁に走査するようになり、CPUパイプラインをストールさせ、EMATを桁違いに膨張させる（L1 TLBヒットの約1nsから、主記憶からのフェッチを伴うページウォークの約100nsまで）。

L1、L2、L3キャッシュの劣化

さらに、この劣化はCPUのキャッシュ階層（L1、L2、L3/LLC）全体にも及ぶ。極度に断片化したヒープでは、本来同一の64バイトキャッシュライン内にコンパクトに収まっているべきデータ要素が、複数のキャッシュラインに散らばってしまう。これによりキャッシュの実効容量が減少し、キャッシュスラッシングが悪化する。

予測可能な連続または一定間隔（strided）のメモリアクセスパターンに依存するプロセッサのハードウェアプリフェッチャは、断片化した連結構造や、不連続な物理ページにマッピングされた配列を走査する際には、完全に無力になる。

アロケータ戦争：glibc 対 jemalloc 対 tcmalloc

標準Cライブラリのデフォルトアロケータ（glibc malloc）はptmallocをベースとした汎用アロケータである。デスクトップアプリケーション向けには適しているものの、長時間稼働するヘビーにマルチスレッド化されたデータベースサーバーにおいてフラグメンテーションを防ぐ能力は悪名高いほど低い。glibcはグローバルロックと共有アリーナを使用するため、スレッド競合とフラグメンテーションを甚大に引き起こす。

これに対抗するため、高性能データベース（Redis、PostgreSQL、CockroachDBなど）は、jemalloc（FreeBSD/Facebookによって開発）やtcmalloc（Google）といった専用アロケータを明示的にリンクする。

jemallocのアーキテクチャ

jemallocは、割り当て頻度が高い環境におけるフラグメンテーションと競合を緩和するために特別に設計されている。以下のような高度な技術を採用している。

  1. サイズクラス（Size Classes）： jemallocはメモリを離散的なサイズクラス（例：8バイト、16バイト、32バイト……大きなページサイズまで）に厳密に分離する。これにより内部断片化は数学的に最大約20%に制限される。
  2. スレッドローカルキャッシュ（tcache）： 各スレッドはロックフリーの独自メモリブロックキャッシュを持つ。一定サイズ以下の割り当ては、いかなるロックも取得することなくtcacheから即座に満たされ、$O(1)$の高速パスとCPUコア間でのフォールスシェアリングゼロが保証される。
  3. アクティブパージ（Active Purging）： jemallocはバックグラウンドでのメモリパージ機構を実装しており、断片化した空きブロックを能動的に統合してOSへ返却し、RSSのフットプリントを実際のアプリケーション使用量と厳密に相関させ続ける。

アルゴリズム的緩和策：データベース独自のメモリアーキテクチャ

jemallocのような高度なユーザー空間アロケータを用いても、アプリケーションの論理的メモリビューとオペレーティングシステムの物理メモリ管理との間の根本的な二分性は、継続的なアーキテクチャ上の精査を必要とする。高性能データベースは、クリティカルパスにおいて標準アロケータを完全に迂回し、独自設計のメモリ管理アーキテクチャを採用することが多い。

Slabアロケータ

最も重要な戦略の一つが、自律的なSlabアロケータの実装である。Slab割り当て方式では、メモリはOSから「スラブ（slab）」と呼ばれる大規模で連続したチャンクとして事前に割り当てられる。

各スラブはその後、均一なサイズのスロットに分割され、特定の内部データ構造（トランザクションコンテキストオブジェクト、ロック記述子など）を格納する用途に厳密に専用される。オブジェクトの種類とサイズによる厳格な分離を強制することで、Slabアロケータは構造的にプール内の外部断片化を排除する。なぜなら、すべての割り当てが同一寸法のスロットを操作するからである。

スロットが解放されると、それはロックフリーの単方向連結リストに追加される。スラブごとの内部断片化は$S_{slab} mod S_{obj}$によって厳密に制限され、$S_{slab} gg S_{obj}$の場合はまったく無視できる大きさとなる。

バッファプール（固定サイズページ）

主要なデータページに関しては、データベースは普遍的に固定サイズのバッファプール（Buffer Pool）を採用する。バッファプール用のメモリは、初期化時に単一の巨大かつ連続した仮想メモリ領域として割り当てられる。

この領域は論理的に同一サイズのフレーム（例：PostgreSQLでは8KB、InnoDBでは16KB）に分割される。均一なフレームサイズを厳密に維持することで、バッファプールはデータキャッシュに関連する内部断片化と外部断片化の両方を完全に回避する。

ディスクから新しいページを読み込む必要があり、プールが満杯の場合、LRUやCLOCK-Proのようなアルゴリズムが古いページを追い出し（evict）、新しいページのために正確に同じ連続メモリフレームを再利用する。このキャッシュは、無限の時間軸にわたって完全にフラグメンテーションから免疫を持つ。

アリーナアロケータ（リージョンベースのメモリ管理）

クエリ実行中（例：JOINとORDER BYを伴うSELECT）、データベースはハッシュテーブルやソートバッファのために動的で可変サイズの割り当てを行う。これらの一時的な割り当てがグローバルヒープを断片化させることを防ぐため、エンジンはアリーナアロケータ（Arena Allocator）（例：PostgreSQLのMemoryContext）を採用する。

アリーナアロケータは、クエリの開始時に大きく連続したブロックを用意する。以降のメモリ要求は、単にアリーナ内のポインタをインクリメントするだけであり、オブジェクトごとのメタデータオーバーヘッドがゼロの、真の$O(1)$割り当てを実現する。

決定的な特徴は解放処理にある。アリーナ内の個々のオブジェクトは決して明示的に解放されない。その代わり、クエリの実行が完了した際に、アリーナ全体が単一の瞬時的な操作で回収される。このアーキテクチャ設計は、一時的なクエリメモリが長期的な外部断片化に寄与し得ないことを根本的に保証する。

template <typename T, size_t BlockSize = 4096>
class FragmentFreeArena {
private:
    struct Block {
        char data[BlockSize];
        size_t current_offset;
        Block* next;
        Block() : current_offset(0), next(nullptr) {}
    };
    Block* head_block;
    Block* current_block;

public:
    FragmentFreeArena() {
        head_block = new Block();
        current_block = head_block;
    }
    
    ~FragmentFreeArena() {
        Block* curr = head_block;
        while (curr != nullptr) {
            Block* next = curr->next;
            delete curr;
            curr = next;
        }
    }

    void* allocate(size_t size, size_t alignment = alignof(std::max_align_t)) {
        size_t current_ptr = reinterpret_cast<size_t>(current_block->data) + current_block->current_offset;
        size_t offset = (alignment - (current_ptr % alignment)) % alignment;
        
        if (current_block->current_offset + offset + size <= BlockSize) {
            void* ptr = current_block->data + current_block->current_offset + offset;
            current_block->current_offset += offset + size;
            return ptr;
        } else {
            Block* new_block = new Block();
            current_block->next = new_block;
            current_block = new_block;
            return allocate(size, alignment); 
        }
    }
    
    // 領域全体をO(1)で即座に解放する
    void reset() {
        Block* curr = head_block->next;
        while (curr != nullptr) {
            Block* next = curr->next;
            delete curr;
            curr = next;
        }
        head_block->next = nullptr;
        head_block->current_offset = 0;
        current_block = head_block;
    }
};

OSとの相互作用：Huge Pagesとmadvise

メモリフラグメンテーションの管理は、OSカーネルと切り離して分析することはできない。

Transparent Huge Pages（THP）対 明示的Huge Pages

標準的なx86-64アーキテクチャは4KBのページサイズを使用する。128GBのRAMを消費するデータベースの場合、これは3,350万件のページテーブルエントリを必要とする。OSをHuge Pages（2MBまたは1GB）を使用するよう構成することで、エントリ数は劇的に削減され、TLBのカバレッジが指数関数的に増加する。

しかし、LinuxのTransparent Huge Pages（THP）を通じてメモリの連続性を積極的に追求すると、深刻な病理的挙動が引き起こされる。THPはカーネルスレッド（khugepaged）を通じて、4KBページを2MBのHuge Pageへと動的に統合する。

カーネルがHuge Pageの割り当てを試みるものの物理メモリが断片化している場合、ダイレクトコンパクション（direct compaction）が発動される。ダイレクトコンパクションは同期的でブロッキングな操作であり、カーネルが物理ページを強制的に移動させている間、データベースプロセスを停止させてしまう。この突発的な停止——数百ミリ秒に及ぶこともある——は壊滅的なレイテンシスパイクを引き起こす。

ベストプラクティスとしては、THPを完全に無効化すること（echo never > /sys/kernel/mm/transparent_hugepage/enabled）、そして代わりに明示的かつ静的に割り当てられたHuge Pages（GRUBにおけるhugepages=N）に依存することが推奨される。これにより、連続したメモリ要件を事前に満たすことで決定論的なレイテンシが保証される。

madviseシステムコール

データベースがメモリを解放すると、ユーザー空間のアロケータはそれを空きとしてマークするが、OSは物理ページマッピングを維持し続ける。人為的に膨張したRSSを防ぐため、jemallocのようなアロケータは定期的に空きメモリ領域に対してmadvise(MADV_DONTNEED)またはMADV_FREEを発行する。

この呼び出しは、指定された仮想アドレス範囲のページテーブルエントリを解体し、その下にある物理ページを回収するようカーネルに明示的に指示する。その後データベースがその仮想アドレスにアクセスした場合、カーネルはシームレスにゼロ埋めされた新しい物理ページを割り当てる。この機構により、データベースは未使用のリソースをOSに返却し、システム全体の安定性を維持する。

教訓とベストプラクティス

  1. DB向けにデフォルトのglibcを決して使わない： C/C++/Rustでデータ集約型アプリケーションを構築する場合、常にjemallocまたはtcmallocをリンクすること。高い並行性下でのパフォーマンスの差は驚異的である。
  2. Transparent Huge Pages（THP）を無効化する： ほぼすべての主要なデータベース（MongoDB、PostgreSQL、Redis）において、THPは同期的なメモリコンパクションにより大規模なレイテンシスパイクを引き起こす。OSレベルでオフにすること。
  3. バッファプールには明示的なHuge Pagesを使用する： 主要なデータキャッシュを事前割り当てされたHuge Pagesに固定することで、TLBスラッシングとOSレベルのスワッピングを完全に排除し、盤石なパフォーマンスを確保できる。
  4. リクエストごとにアリーナアロケータを使用する： Webサーバーやクエリエンジンを構築する際は、アリーナを使ってリクエストごとにメモリを割り当てること。リクエストが終わったらアリーナのポインタをリセットする。これによりフラグメンテーションが排除され、複雑なガベージコレクションアルゴリズムを回避できる。
  5. RSSと使用中メモリを監視する： データベースプロセスが10GBのデータを報告しているにもかかわらずhtopが30GBのRSSを示している場合、深刻なメモリフラグメンテーションまたはmadvise設定の問題を抱えている。アロケータのバックグラウンドパージスレッドをチューニングすること。
