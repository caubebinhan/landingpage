Google Spanner & TrueTime – 分散システムにおける時間の軸を曲げる (詳細な技術レポート)

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー (Executive Summary)

CAP定理により、アーキテクトはネットワーク障害時に一貫性 (Consistency) と可用性 (Availability) の間で選択を迫られてきました。最近のほとんどのデータベース (Cassandra、DynamoDB) は可用性を選択し、結果整合性 (Eventual Consistency) で妥協しています。この妥協の根本的な原因はアルゴリズムの未熟さではなく、宇宙の物理的障壁である「時間の相対性」にあります。伝送遅延 (network latency) と水晶発振器のドリフト (clock drift) のため、地球上で完全に同期して動作する時計を持つ2つのサーバーは存在しません。

しかし、Googleのエンジニアはその妥協を受け入れませんでした。彼らは絶対的な外部一貫性 (External Consistency) を達成した唯一のグローバル分散リレーショナルデータベースであるGoogle Spannerを作成しました。彼らの秘密兵器はTrueTime APIです。これは全地球測位システム (GPS) と原子時計 (Atomic Clocks) の組み合わせを使用し、時間を不確定な数値から枠に収められた数学的定数に変える時間インフラストラクチャネットワークです。

この技術レポートでは、TrueTimeのマイクロアーキテクチャ、時間誤差関数 $epsilon(t)$ の数学方程式、Paxosルーティングアルゴリズム、そして特にCommit Waitメカニズム（SpannerがCPUを強制的にスリープさせて相対性理論の時間の不確実性を吸収する方法）について深く解剖します。

問題の核心 (Problem Statement): グローバルネットワークにおいて、ニューヨークのトランザクション $T_1$ が東京のトランザクション $T_2$ が始まる前に完了した場合、因果律によりタイムスタンプ $S_1$ は $S_2$ より小さくなければなりません。しかし、東京の時計がニューヨークより5ミリ秒遅れている場合、システムは $S_2 < S_1$ と割り当てます。この時間のパラドックスは、データ同期 (Distributed Transactions) のあらゆる努力を破壊します。従来の解決策は中央のタイムスタンプサーバー (Central Timestamp Server) を使用することでしたが、これは大陸間の数百ミリ秒の伝送遅延を伴うボトルネックを生み出し、パフォーマンスを低下させます。

教訓と得られた知識 (Lessons Learned):

  1. 不確実性の受け入れ (Uncertainty): 時間を100%正確に測定することは不可能です。TrueTimeは1つの数値を返すのではなく、時間区間 $[t_{earliest}, t_{latest}]$ を返します。真実は常にこの区間内にあります。
  2. Commit Wait – 一貫性のための遅延のトレードオフ: 因果律を保護するため、書き込み完了を確認する前に、プロセッサは不確実性 $2epsilon$ に等しい時間待機しなければなりません。
  3. 障害ドメインの多様性 (Fault Domains): GPSは干渉を受けますがドリフトしません。原子時計は時間の経過とともにドリフトしますが干渉を受けません。両者を組み合わせることで、完璧なビザンチン障害耐性が生み出されます。

分散コンピューティングにおける時間危機の背景

世界中のすべてのサーバーは、継続的に発振する水晶振動子 (Quartz Oscillator) で時間を計測しています。チップの温度、電源、ハードウェアの経年劣化の違いにより、これらの水晶は不規則に振動します。この現象はClock Drift（時計のドリフト）と呼ばれます。

従来のネットワークタイムプロトコル (NTP) は、LAN/WANを使用して標準サーバーから時間を問い合わせます。しかし、ルーターのキューバッファ (Router Queue Buffer) が常に変化するため、RTT (Round Trip Time) は決して対称ではありません。NTPはこの遅延を正確に補正できず、2つのNTPサーバー間の時計の誤差は、広域ネットワーク上で100ms〜200msに達する可能性があります。

この200msの間に、何万もの金融トランザクション (Transactions) が順序通りにならず (Out-of-order)、外部一貫性/線形化可能性 (External Consistency/Linearizability) に違反する可能性があります。

TrueTimeアーキテクチャ: 量子時計と衛星のネットワーク

NTPの限界を打ち破るために、Googleは専用のハードウェアシステムを構築しました。

1 デュアルマスターネットワーク (Dual Master Network)

TrueTimeのハードウェアには、Googleのすべてのデータセンターに配置されたタイムマスター (Time Masters) が含まれます。

  1. GPS Masters: データセンターの屋根に直接アンテナが取り付けられています。GPSは衛星電波に基づいており、非常に正確ですが、太陽嵐、無線干渉、または大気の障害により通信が途絶える可能性があります。
  2. Atomic Masters (ルビジウム/セシウム原子時計): サーバーホールの地下深くに配置され、原子の電子遷移を測定します。信号が途絶えることはありませんが、極めて微小な線形ドリフト (Clock drift)（1日あたり数マイクロ秒）が発生します。

この組み合わせは直交障害ドメイン (Orthogonal Failure Domains) の芸術です。GPS信号が途切れた場合、原子時計が機能します。原子時計のドリフトが大きくなりすぎた場合、GPSが再調整します。両方を備えた一部の最高レベルのサーバーは、Armageddon Mastersと呼ばれます。

2 Marzulloフィルタリングアルゴリズムとマイクロオシレーション (TrueTime Daemon)

各Spannerサーバーはバックグラウンドプロセス (TrueTime Daemon) を実行します。このデーモンは、複数のマスター (Local DCおよびRemote DCのGPSとAtomicの両方) に対して定期的にポーリング (poll) を行います。 時間の応答のセットを受け取った後、デーモンは修正されたMarzulloアルゴリズム (Modified Marzullo’s Algorithm) を適用します。このアルゴリズムは時間区間を交差させ、ハードウェアの障害や光ファイバーの遅延による偽の応答 (Liars) を返すサーバーを自動的に隔離します。

TrueTime APIの数学的定理: 不確実性の枠組み

TrueTimeの偉大さは、正確な数値を返すことではなく、絶対に安全な不確実性区間 (Uncertainty Interval) を返すことにあります。

システムのAPIは TT.now() 関数を提供し、次を含むオブジェクトを返します。

$$ [t_{earliest}, t_{latest}] $$

システムは物理的な確実性をもって、コマンドが呼び出されたときの実際の時間 $t_{abs}$ が確実に以下を満たすことを保証します。

$$ t_{earliest} le t_{abs} le t_{latest} $$

1 誤差関数 $epsilon(t)$ (Epsilon)

不確実性を $epsilon$ とします。TrueTime区間の幅は常に $2epsilon$ です。 Spannerサーバーがマスターと正常に同期した瞬間（$t_{sync}$ と呼びます）、ネットワークの誤差は最小レベル ($epsilon_{sync} approx 0$) になります。 しかし、時間が経過するにつれて（$t_{sync}$ から遠ざかるにつれて）、サーバー上の水晶振動子がドリフトし始めます。$rho$ (rho) をハードウェアの最大ドリフト率 (Worst-case drift rate – 通常、Googleは $rho approx 200 mu s/text{秒}$ と仮定します) とします。 時間 $t$ における誤差の方程式は次のとおりです。

$$ epsilon = epsilon_{sync} + rho cdot (t – t_{sync}) $$

TT.now() 関数を呼び出すとき、サーバーはローカルのハードウェア時計 $C(t)$ をチェックします。

  • $t_{earliest} = C(t) – epsilon$
  • $t_{latest} = C(t) + epsilon$

独自のGPSおよびAtomicハードウェアネットワークのおかげで、Googleは平均 $epsilon$ 値をわずか1ミリ秒〜7ミリ秒に抑えています。（パブリックNTPネットワークでは、この数値は150msです）。

Spannerと「Commit Wait」プロトコル: 因果律を救うためにダイナミクスを一時停止する

Spanner内の各データフラグメント (Tablet) は、Paxosコンセンサスアルゴリズムによって保護されています。書き込み要求がある場合、PaxosのNode Leaderは、Write-Ahead Logに記録するために、そのトランザクションにタイムスタンプ (Timestamp) を割り当てる必要があります。

1 単調タイムスタンプのルール (Monotonic Timestamp Rule)

トランザクション $T_1$ が書き込みを要求したとき、Leaderは TT.now() を呼び出します。 タイムスタンプ $s = TT.now().latest$ を割り当てます。 最大のタイムスタンプ（可能な限り最も遠い未来）を取得することで、過去のトランザクションとタイムスタンプが重複しないことを保証します。

2 Commit Wait定理 (待機コミット)

Commit Wait定理は次のように規定しています：システムは、実際の時間 $t_{abs}$ が確実に $s$ の時点を過ぎるまで、トランザクションの完了 (ClientへのACK) を確認してはならない。 実際の時間が $s$ を過ぎたことをどうやって知るのでしょうか？ プロセッサ (Coordinator) は、$t_{earliest} > s$ の値を受け取るまで TT.now() 関数を継続的に呼び出します。このとき、$t_{abs} ge t_{earliest}$ であるため、システムは絶対的な数学的結論 $t_{abs} > s$ を導き出すことができます。

この条件が発生するのを待つためにシステムが「スリープ」(sleep) しなければならない時間は、通常 $2epsilon$ の距離（最大 $sim 7ms$）に等しいです。

なぜ待たなければならないのか？ この遅延は、地球上の相対論的な時間の乱れを完全に吸収します。 $T_1$ に $s = 100$ が割り当てられたと仮定します。Leaderは、実際の時間 $t_{abs} > 100$ になるまで待ってから、Clientに成功を報告しなければなりません。Client AがClient Bに通知します。Client Bはトランザクション $T_2$ を開始します。 $T_2$ は $t_{abs} > 100$ の後にのみ開始されるため、$T_2$ が TT.now() を呼び出すと、$T_2$ の時間区間は確実に $t_{earliest} > 100$ を持ちます。したがって、$T_2$ のタイムスタンプは $s_2 > 100$ に割り当てられます。 因果関係 (Causality) $s_1 < s_2$ は、世界規模で絶対に保護されます。

// C++でのCommit Waitのマイクロアーキテクチャの疑似コード
struct TrueTimeInterval {
    int64_t earliest_us;
    int64_t latest_us;
};

class TrueTimeAPI {
public:
    TrueTimeInterval now();
};

class PaxosLeaderEngine {
private:
    TrueTimeAPI tt_api;
    int64_t last_assigned_timestamp = 0;
    
public:
    int64_t PrepareTransaction() {
        // 最大の未来時点を取得
        TrueTimeInterval current_time = tt_api.now();
        int64_t s = current_time.latest_us;
        
        // 厳密な単調増加を強制
        if (s <= last_assigned_timestamp) {
            s = last_assigned_timestamp + 1;
        }
        last_assigned_timestamp = s;
        return s;
    }

    void CommitTransaction(Transaction tx, int64_t s) {
        // 光ファイバーネットワークを介して多数派のクォーラム (Quorum) にデータを同期
        // このネットワーク転送時間 (約2〜5ms) は、待機時間の一部を吸収します。
        ReplicateToPaxosQuorum(tx, s);
        
        // 因果律を保護するスレッドロック - Commit Waitの開始
        while (true) {
            TrueTimeInterval wait_time = tt_api.now();
            if (wait_time.earliest_us > s) {
                // 不確実性が過ぎ去りました。現在の絶対時間は確実に > s
                break; 
            }
            // CPUを節約し、残りのマイクロ秒間スレッドを一時停止するようOSカーネルに要求
            HardwareInterruptNanosleep(wait_time.earliest_us - s);
        }
        
        // 安全。広域ネットワークに200 OK信号を送信
        RespondToClient(tx.client_id, SUCCESS);
    }
};

大陸をまたぐロックフリー読み取り (Lock-Free Read-Only Transactions)

Commit Waitメカニズムは、書き込み (Write) タスクに対して $7ms$ の遅延という代償を払います。しかし、これは読み取り (Read) タスクに魔法をもたらします：大陸をまたぐLock-Free Readです。

ベトナムのアプリケーションがSpannerからUserデータテーブルを読み取りたい場合、分散読み取りロック (Read Lock) を取得するために米国に呼び出す必要はありません。単に TT.now().latest を呼び出してタイムスタンプ $s_{read}$ を取得します。その後、この $s_{read}$ をベトナムのデータセンターにあるローカルレプリカ (Local Replica) サーバーに渡します。 ベトナムのサーバー（Multi-Version Concurrency Control – MVCC 形式で保存されている）は、$s_{read}$ に最も近いタイムスタンプを持つUserのバージョン (Snapshot) を見つけて返すだけです。世界中で進行中の書き込み操作が、この読み取り操作と衝突したりブロック (Block) たりすることはありません。

読み取りパフォーマンスは無限にスケーラブル (Infinite Scalability) であり、遅延はほぼ $0$ です。これが、Google Adsネットワークの巨大な広告インフラの背後にあるコア技術です。

マイクロアーキテクチャ (Micro-Architecture) とシステムコールの視点

TrueTime APIがマイクロ秒 (micro-second) の精度を達成するためには、TrueTime Daemonは recvfrom() のような通常のネットワークシステムコールに依存することはできません。OSのコンテキストスイッチ (Context Switch) は $epsilon$ 値を歪めます。

  • ゼロコンテキストスイッチの分離: TrueTime Daemonのタイムデータは、ユーザースペース (User Space) の共有メモリセグメント (Shared Memory Segment) に直接マッピングされます。
  • CPUコアレジスタの読み取り: Spannerアプリケーションが TT.now() を呼び出すと、CPUハードウェアレジスタから直接水晶振動信号を読み取ります（例：x86_64チップの RDTSC アセンブリ命令を使用）。誤った投機的実行 (Speculative Execution) を防ぐために、メモリバリア (MFENCE / LFENCE) と組み合わせられます。
  • 水晶温度の監視: Daemonはまた、CPUの温度センサー (Temperature Sensors) を継続的にポーリングします。温度が上昇すると、水晶はより速くずれて振動します。アルゴリズムは自動的に補正し、アラーム範囲 $rho$ を最も保守的な閾値に広げ、同期が崩れるすべてのリスクを排除します。
