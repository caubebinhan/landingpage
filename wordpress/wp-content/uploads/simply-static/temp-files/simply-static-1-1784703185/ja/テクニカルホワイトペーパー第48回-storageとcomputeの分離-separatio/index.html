テクニカルホワイトペーパー第48回: StorageとComputeの分離 (Separation of Storage and Compute): クラウドネイティブアーキテクチャ、マイクロアーキテクチャのダイナミクス、および分散システムの未来

July 22, 2026 · admin · 4 分で読めます

エグゼクティブサマリー (Executive Summary)

この記事は、エンジニア、システムアーキテクト、研究者がストレージ (Storage) とコンピューティング (Compute) リソースの分離というトレンドの本質、動作原理、および深い教訓を理解できるように、詳細な技術ドキュメント (whitepaper) として設計されています。読み終えると、現代の分散アーキテクチャが解決すべき中心的な問題、システムが物理法則の限界 (ネットワーク遅延、メモリ帯域幅) をどのように克服するか、そしてSnowflake、BigQuery、Auroraなどの次世代クラウドネイティブデータベース (Cloud-native Databases) に適用される非常に価値のある設計の教訓をマスターすることができます。

はじめに：クラウドネイティブアーキテクチャを読み解く本質

データベース管理システム (DBMS) とビッグデータ処理プラットフォーム (Big Data platforms) の進化は、ランダムな変化ではありません。各パラダイムシフト (paradigm shift) は、物理ハードウェアとビジネスニーズの限界点に由来しています。過去10年間で、CPU、RAM、ハードディスクが同じ物理サーバーのシャーシにロックされているモノリシックアーキテクチャ (monolithic architecture) から、クラウド上での完全な分離アーキテクチャ (separation of compute and storage) への根本的なパラダイムシフトを目の当たりにしてきました。

しかし、ここで何を読んでいるのでしょうか？単なる流行のトレンドについて読んでいるわけではありません。マイクロ構造、ルーティングの数学的方程式、そして現代のソフトウェアがオペレーティングシステムを「騙して」光学的限界に近い伝送速度を達成する方法を深く掘り下げています。目標は「なぜ分離が良いのか」を知ることではなく、「ネットワーク遅延でシステムが崩壊しないようにどのように分離するのか」を知ることです。

中心的な問題：モノリシックアーキテクチャの崩壊 (The Core Problem)

1 ハードウェア結合の制約 (Hardware Coupling Constraints)

従来のモノリシックシステム (Shared-Nothing Hadoop, Teradata, オンプレミスRDBMS) では、リソース結合率はハードウェア構成の定数で定義できます：

$$R_c = frac{C_{capacity}}{S_{capacity}}$$

ここで、$C_{capacity}$ は計算能力 (CPUコア数、RAM容量) を表し、$S_{capacity}$ はローカルストレージ容量 (HDD/SSD) を表します。

中心的な問題 1：非対称な成長。 ビッグデータ時代において、データ量 (storage) は通常、指数関数的 ($O(e^x)$) に増加しますが、クエリ処理のニーズは通常、線形 ($O(x)$) にしか増加しません。モノリシックアーキテクチャでは、システムのハードディスク容量がなくなると、新しい物理ノード (nodes) を追加購入せざるを得ません。これらのノードには、CPU、RAM、およびディスクがすべて含まれています。その結果、データセンターはアイドル状態 (idle) のCPUコアで溢れかえりますが、システムはストレージ制限の100%に達したと報告し続けます。この $R_c$ のリソースの無駄は、設備投資 (CapEx) と運用コスト (OpEx) に多大な消費をもたらします。

2 データの重力 (Data Gravity)

中心的な問題 2：ワークロードのモビリティ。 データがローカル処理ノードに強く結合している場合、同じ生データセット (raw data) を異なる分析チーム (例：Data Science が Machine Learning を実行、Business Intelligence が Dashboard を実行、Data Engineering が ETL を実行) で共有することは、物理的なデータ複製を行わない限り不可能です。巨大なデータ複製 (ETL replication) 操作は時間がかかるだけでなく、一貫性 (Consistency) を破壊し、孤立したデータサイロ (Data Silos) を引き起こします。

解決策：リソース分離アーキテクチャの原理

根本的な解決策は、物理サーバー構造を打ち破り、これらの2つのリソース層を2つの独立したコンピューティングクラスターに分離することです。

1 Computeのステートレス化 (Statelessness)

計算層は現在、仮想マシン、コンテナ (containers)、またはServerless関数で構成されています。重要なポイント：これらはローカル状態を長期的に保持することは全くありません。カーネルパニック (kernel panic) によるエラーやクラウドインフラストラクチャによる回収 (Preemptible VM / Spot Instance) のために Compute ノードが突然シャットダウンされても、ユーザーデータのバイトは一切失われません。 これにより、システムオーケストレーターの弾力性のあるスケジューリングアルゴリズム (elastic scheduling algorithms) が、わずか数ミリ秒 (milliseconds) 以内に数千の CPU コアを割り当てまたは回収できるようになります。

2 大きなトレードオフ：ネットワークのボトルネック (The Network Bottleneck Trade-off)

絶対的な柔軟性をもたらす一方で、このアーキテクチャは遅延と帯域幅に関する深刻な物理的課題を提起します。CPUにとって、ローカルSSDからのデータ読み取りはわずか10〜100マイクロ秒 ($mu s$) しかかかりませんが、ネットワークを介して Object Storage からデータを呼び出すには数十ミリ秒 ($ms$) かかることが多く、これは約1,000倍遅いことを意味します。

ネットワーク同期を伴う分散システムのスケーリングにおけるアムダールの法則は次のように再定式化されます：

$$S(N) = frac{1}{(1 – p) + frac{p}{N} + C(N)}$$

ここで、$C(N)$ はデータ転送時のネットワーク遅延ペナルティです。基本的なクエリ実行時間は次のようにモデル化されます：

$$T_{total} = T_{init} + sum_{i=1}^{K} left( frac{D_i}{B_{net}} + L_{net} + T_{compute_i} right)$$

システムが窒息しないようにするためには、エンジニアは $D_i$ (ネットワーク経由で送信されるデータ量)、$L_{net}$ (遅延)、および $T_{compute}$ (CPU時間) を最小限に抑えることに集中する必要があります。

アルゴリズムの基盤と分散クエリ実行メカニズム

1 PACELC定理と一貫性

ネットワークが「システムバス」として機能する場合、パーティション (Partition) は避けられません。PACELC定理によれば、ネットワークシステムが安定している (Else) 場合でも、遅延 (Latency) と一貫性 (Consistency) の間には常にトレードオフがあります。 Compute層は大規模な分散バッファキャッシュ (Distributed Buffer Cache) を確立する必要があります。しかし、あるComputeノードがデータを書き込んだとき、他のComputeノードはどのようにして自身のバッファをクリアすることを知るのでしょうか？無効化ブロードキャスト (invalidation broadcasts) のコスト関数は $O(N)$ または $O(log N)$ に比例します。したがって、優れたデータウェアハウスシステムは、マルチバージョン同時実行制御 (MVCC – Multiversion Concurrency Control) を通じて 結果整合性 (Eventual Consistency) モデルにダウングレードすることを選択します。データテーブルは順次書き込まれ、LSM-Tree構造が最大限に適用されてランダムIO操作をシーケンシャルにまとめ、データのクリーンアップ (compaction) の負担をバックグラウンドノード (background nodes) に移すことで、インタラクティブな読み取りパフォーマンスに影響を与えないようにします。

2 Pushdown Analytics & JIT Compilation (クエリオプティマイゼーション)

$D_i$ の問題 (ネットワーク経由のデータの削減) に対処するための究極の技術は、Computation Pushdown (計算をストレージ層に押し下げること) です。 コストベースオプティマイザ (CBO) はもはやハードドライブの回転を測定するのではなく、次の最適化微分関数を使用します：

$$Cost(P) = alpha cdot W_{CPU} + beta cdot W_{Memory} + gamma cdot W_{Storage_IO} + delta cdot W_{Network_Transfer} + epsilon cdot W_{Serialization}$$

Compute層は、単一の行を SELECT するためだけに1テラバイトの Parquet テーブル全体をRAMに引き込むことはしません。代わりに：

  1. フィルタ条件 (例：WHERE user_id = 123) の抽象構文木 (AST) を抽出します。
  2. このASTをシリアライズ (Serialize) し、RPCを介してストレージノード (Storage Node / SmartNIC) に送信します。
  3. ストレージノードでは、小さなマイクロプロセッサ (ARM/RISC-V) が LLVM を使用してそのASTを機械語にJIT (Just-In-Time) コンパイルします。
  4. ネットワークに送信される前に、NAND Flash メモリチップから離れた瞬間にデータをフィルタリングします。

    5. 結果は？ フィルタ条件を本当に満たすわずか1キロバイトのデータだけがネットワークを通過して Compute ノードに返されます。これにより、ボトルネックとなるネットワークIOがストレージハードウェアでの超並列フィルタリング処理に変わります。

// Pseudocode: Pushdown Filter Serialization (C++ Paradigm)
struct FilterExpression {
    enum Operator { EQUALS, GREATER_THAN, IN_BLOOM_FILTER };
    Operator op;
    uint32_t column_id;
    std::vector<uint8_t> scalar_value_bytes;
};

class StorageNodeComputeEngine {
public:
    std::shared_ptr<ArrowRecordBatch> execute_pushdown(
        const std::string& parquet_path, const std::vector<FilterExpression>& filters) {
        auto file_metadata = ParquetReader::Open(parquet_path)->metadata();
        std::vector<int> matching_row_groups;

        // BƯỚC 1: Lọc bằng Siêu Dữ Liệu (Zero-Decompression Pruning)
        // (STEP 1: メタデータによるフィルタリング (Zero-Decompression Pruning))
        // Dùng Min/Max zone maps & Bloom Filters, không cần giải nén dữ liệu thật
        // (Min/Max zone maps & Bloom Filters を使用し、実データの展開は不要)
        for (int i = 0; i < file_metadata->num_row_groups(); ++i) {
            if (evaluate_zone_maps(file_metadata->RowGroup(i)->statistics(), filters)) {
                matching_row_groups.push_back(i);
            }
        }

        // BƯỚC 2: Thực thi bộ lọc vector hóa (Vectorized JIT execution bằng AVX-512)
        // (STEP 2: ベクター化されたフィルタの実行 (AVX-512 による Vectorized JIT execution))
        std::shared_ptr<ArrowRecordBatch> result_batch = allocate_result_buffer();
        for (int rg_idx : matching_row_groups) {
            auto chunk = file_reader->ReadRowGroup(rg_idx);
            result_batch->append(SIMD_Vectorized_Filter::apply(chunk, filters));
        }
        
        // BƯỚC 3: Trả về lượng dữ liệu đã được nén tối thiểu, giải phóng mạng
        // (STEP 3: 最小限に圧縮されたデータ量を返し、ネットワークを解放する)
        return result_batch;
    }
};

物理的限界とオペレーティングシステムマイクロアーキテクチャのダイナミクス

ここは、ソフトウェアエンジニアがアルゴリズムのコードを書くのが上手いだけでなく、シリコンについて真に理解していなければならない場所です。

1 Kernel Bypass と RDMA / RoCEv2 テクノロジー

POSIX (Linux) オペレーティングシステムでは、Storage層からネットワークパケットが到着するたびに、ネットワークカード (NIC) がハードウェア割り込み (hardware interrupt) を報告し、CPUがタスクを停止し、オペレーティングシステムがパケットを取得し、TCP/IPヘッダーを取り除き、チェックサムを検証し、ペイロードに組み立て、Kernel Space から User Space (アプリケーションのRAM) にコピーします。この Context Switching と Memory Copying のプロセスは、CPUの貴重なクロックサイクルを完全に「燃やし尽くし」、L3 Cache を溢れさせます。

解決策： Kernel Bypass テクノロジー。 RoCEv2 プラットフォーム上の RDMA (Remote Direct Memory Access) を通じて、ネットワークハードウェア (SmartNIC) は、ネットワークから受信したバイトストリームを User Space プロセスの仮想アドレス (Virtual Address) に直接書き込む (Direct Memory Access – DMA) 権限を持ちます。

結果：計算ノードのCPUは、データがメモリアレイ (Buffer) にきちんと収まるまで、ネットワークで何が起こっているかを全く知りません。ネットワーク遅延は、数十ミリ秒から数マイクロ秒のレベルまで圧縮されます。

2 NUMAアーキテクチャとハードウェア認識メモリアロケーションスケジューリング (NUMA-Awareness)

RDMAを介して数百ギガバイトのデータを引き出すには、オペレーティングシステムがそれらをディスクにスワップ (swap) しないように、ターゲットメモリ領域を「ロック」 (Memory Pinning/mlocked) する必要があります。そうしないと Kernel Panic エラーが発生します。 さらに、大規模な物理サーバーは、マルチCPUソケットの NUMA (Non-Uniform Memory Access) アーキテクチャを使用します。SmartNIC が CPU-Socket-1 に接続されたRAM領域にデータを書き込んだのに、オペレーティングシステムが CPU-Socket-2 の処理スレッドを割り当ててそのデータを分析しようとすると、巨大なデータストリームが内部バス (Intel UPI や AMD Infinity Fabric など) を横断しなければなりません。これは、極めて深刻なハードウェアのボトルネックを生み出します。 これを克服するために、データベース管理システムは NUMA-Aware スケジューリング機能を備えている必要があります：ネットワークデータを受け取るための大きなRAM (Huge-pages) の割り当てを常に確保し、そのRAM領域を保持している正確な物理コアにCPU処理スレッドを強制的に固定 (Core Affinity Binding) します。

得られた教訓 (Lessons Learned) と深い知識

Storage/Compute分離システムをマイクロレベルで分解した後、読者は以下の原則的で重要な技術的教訓を導き出すことができます：

  1. 教訓1：「ネットワークはもはや致命的な障壁ではなく、CPUリソースの浪費が障壁である」：最新の100Gbps/400Gbps ClosネットワークとRDMAテクノロジーにより、ネットワーク経由のIOはローカルNVMeのパフォーマンスに近づくことができます。問題はもはや光伝送の速度ではなく、割り込み処理とコピーにおけるLinuxオペレーティングシステムの間接コスト (overhead) です。カーネルバイパス (Kernel Bypass) は、バックエンド開発の未来における必然的なトレンドです。
  2. 教訓2：データをアルゴリズムに移動させるのではなく、アルゴリズムをデータに移動させる (Pushdown Principle)：物理的な分離 (Separation) は、ネットワーク経由でデータを盲目的に引き寄せることを意味しません。メタデータ (Metadata, Zone Maps, Bloom Filters) は、ハードウェアレベルからジャンクデータブロックをプルーニング (Pruning) するのに役立つ不可欠な鍵です。
  3. 教訓3：ハードウェアとソフトウェアの共生 (Hardware-Software Co-design)：今日の分散データベースは、抽象的なPOSIX仮想マシン上で実行される純粋なJava/C++コードのようには設計できません。プログラマーは、データがマザーボード自体のバス上で停滞するのを防ぐために、L1/L2 Cache Lines、NUMA Sockets、および SIMD ベクター化命令 (AVX-512) について深く理解している必要があります。
  4. 教訓4：ストレージ/メモリの境界が曖昧になっている：Compute Express Link (CXL) と NVMe-oF のおかげで、データセンター内の数百メートル離れた場所にあるSSDハードドライブアレイを、計算中のCPUのRAMアドレスマップに直接「マッピング」 (mapped) できるようになり、データセンター全体が単一の統合されたマルチコアコンピュータに変わります。

結論

モノリシックアーキテクチャから Compute/Storage 分離アーキテクチャへの移行は、クラウド時代の不可欠な技術革命です。これにより、リソース拡張における非柔軟性 (Scalability) が根本的に解決され、データの重力 (Data Gravity) が打ち砕かれます。物理的な遅延と一貫性に関して巨大な課題を生み出しますが、システムエンジニアは数学 (クエリオプティマイゼーション)、コンピューターサイエンス (pushdown, vectorization)、および電子物理学 (RDMA, NUMA, SmartNICs) を巧みに組み合わせて、最も経済的なコストで瞬く間に数ペタバイトのデータを飲み込むことができるSnowflakeやDatabricksなどのビッグデータ処理システムを作成しました。

この分離アーキテクチャを理解することは、弾力性があり、耐久性があり、次の10年のテクノロジーに対応できるマルチテナント (Multi-tenant) SaaSおよびPaaSソフトウェアを設計するための究極の足がかりとなります。
