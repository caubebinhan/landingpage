技術白書: SQL Window Functionsの背後にあるマイクロアーキテクチャ実行メカニズム

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー (Executive Summary)

SQL Window Functions（ウィンドウ関数）は、データエンジニアやアナリストが複雑な分析計算を実行する方法に革命をもたらしました。データの行を単一の結果に折りたたむ集約関数（Aggregate Functions）とは異なり、Window Functionsは入力レコードのそれぞれを保持しながら、周囲の「データフレーム（Window Frame）」と関連する値を計算します。

しかし、エレガントなSQL構文（ROW_NUMBER() OVER(PARTITION BY... ORDER BY...)など）の背後には、リレーショナルデータベース管理システム（RDBMS）や分散システム（Spark、Trino）において、最も複雑でコストが高く、高度な技術を要する実行エンジンの1つが存在しています。

本白書では、アルゴリズムの断片化（Partitioning、Sorting）の概念から、最新のデータ構造（Monotonic Deque、Segment Tree）、ハードウェアマイクロアーキテクチャレベルでの深い相互作用（SIMD、Cache Lines、NUMA、JIT Compilation）に至るまで、Window Functionsの実行メカニズムを詳細に解剖します。このドキュメントを読み終える頃には、Window Functionsの動作原理を完全に理解し、大規模システムにおけるパフォーマンスのボトルネックを特定し、データ最適化設計の教訓（Lessons Learned）を引き出すことができるようになります。

Window Functionsの核心的な問題 (The Core Problem)

問題とは何か？ 数百万人の顧客を対象とした7日間の移動平均（7-day moving average）を計算するクエリを考えてみましょう。 AVG(sales) OVER(PARTITION BY customer_id ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)

特定の行に結果を返すために、システムはその行を読み取るだけでなく、以下のことを行う必要があります。

  1. 数百万の顧客を適切にパーティションにグループ化する（Partitioning）。
  2. 各顧客内で日付順にデータを並べ替える（Sorting）。
  3. 計算のために7行の「スライディングウィンドウフレーム」を維持しつつ、ウィンドウが移動するにつれて最も古い行を破棄し、最も新しい行をフレームに追加する。

通常の線形スキャンとは異なり、このプロセスは非常に洗練されたローカル状態の保持を必要とします。データが大きすぎてRAMに収まらない場合、システムはクラッシュするか、ページング/スワッピング（Paging/Swapping）を継続的に実行しなければならず、クエリの遅延が数秒から数時間へと増大します。現代のRDBMSの課題は、CPUの処理スループットを最大レベルに保ち、L1/L2/L3 Cache全体を活用しながら、この複雑な問題をいかに解決するかにあります。

フェーズ1：データのパーティショニングとルーティング (Data Partitioning)

論理パーティション（PARTITION BYに基づく）へのデータの細分化が最初のステップです。

HashingとSIMDの力

システムは単純な文字列比較命令でグループ化を行いません。SIMD (AVX-512) 命令セットに実装された超高速ハッシュアルゴリズム（MurmurHash3やxxHashなど）を使用します。8〜16個のハッシュキーをベクトルレジスタ（Vector Registers）に同時にロードすることで、CPUは単一のクロックサイクル（clock cycle）で16レコードのパーティショニングを計算できます。

$$h(x) = text{xxHash}(x_k) pmod P$$

Radix BufferとCache Thrashing

キャッシュミス（Cache Miss）現象を防ぐために、データストリームはRadix Bufferを介してルーティングされます。このバッファサイズは、L2/L3 Cacheのスペース（通常256KBから数MB）にぴったり収まるよう厳密に調整されています。データを何千ものパーティションに同時に注ぎ込むと、CPUは継続的にCache Lineを置き換える必要が生じ（Cache Thrashing）、メモリ帯域幅が著しく低下します。

External Partitioning技術とOS mmap

パーティションがRAMを超えて膨張した場合（Out-of-Core）、RDBMSはExternal Partitioningをアクティブにします。一時メモリファイルをmadvise(MADV_SEQUENTIAL)フラグと組み合わせたmmap()関数を介してディスクにマッピングします。このカーネルレベルのディレクティブは、Windowアルゴリズムが実際にそれを要求する前にSSDからRAMへデータを継続的にプッシュするアグレッシブな先読み（Aggressive Read-Ahead）モードをオンにするようOSに「促し」、I/O遅延（I/O latency）を排除します。

フェーズ2：データの並べ替え (Sorting Algorithm Dynamics)

パーティショニングが完了すると、システムは非常にコストのかかる問題、内部のORDER BYに直面します。数学的な計算量は$Omega(N log N)$を下回ることはありません。

IntroSortと敗者木 (Loser Tree) 構造

データが完全にRAM内に収まる場合、IntroSort（QuicksortとHeapsortのハイブリッド）が支配的になります。しかしRAMが枯渇すると、External Merge Sortアルゴリズムが介入する必要があります。データは細分化され、予備的にソートされ、ソート済みラン（sorted runs）としてNVMeディスクに書き込まれます。

マージフェーズ（Merge Phase）において、システムは敗者木（Loser Tree / Tournament Tree）を使用します。この技術はマイクロアーキテクチャのメモリにとって非常に最適化されています。新しい要素が木に入るとき、システムは$log_2(K)$回の比較のみを必要とし、木ノード構造のすべての更新操作はCPU Cache Lineの同じ物理ストライプ上で実行されます。

フェーズ3：評価状態マシン (Window Frame Evaluation)

ここでアルゴリズムの魔法が真に輝きます。レコード$i$のウィンドウフレームは次のように数学化されます。

$$ W_i = { r_j in P mid max(1, i – L) le j le min(N, i + U) } $$

単純な分析関数 (ROW_NUMBER, RANK)

これらの関数の場合、システムは現在のインデックスの軌跡を保持するレジスタ（Registers）を維持するだけで済みます。計算量は超高速の$mathcal{O}(N)$です。

スライディングウィンドウ問題 (Sliding Window Maximum/Minimum)

スライディングフレーム（例：過去30日間）でのMAX()またはMIN()の計算は単純ではありません。素朴に計算すると計算量は$mathcal{O}(N times W)$（$W$はウィンドウサイズ）になります。$W = 10,000$の場合、クエリはフリーズします。 ここでの偉大な解決策は、単調両端キュー（Monotonic Deque）を使用することです。DequeはMAXになる可能性のある候補のリストを保持し、小さすぎる（MAXになることは決してない）行を即座に破棄します。結果として、コストは純粋な線形時間$mathcal{O}(N)$にまで減少します。

// Window MaximumのためのMonotonic Dequeアルゴリズムのシミュレーション
pub fn evaluate_max(&self) -> Vec<T> {
    let mut deque: VecDeque<usize> = VecDeque::new();
    for i in 0..n {
        // 1. ウィンドウからスライドした行を破棄する
        if let Some(&front_idx) = deque.front() {
            if i >= self.window_size && front_idx <= i - self.window_size { deque.pop_front(); }
        }
        // 2. 劣った候補を破棄する
        while let Some(&back_idx) = deque.back() {
            if self.data[back_idx] <= self.data[i] { deque.pop_back(); } 
            else { break; }
        }
        deque.push_back(i); // 3. 新しい候補を登録する
    }
}

桁落ち (Catastrophic Cancellation) 対策

浮動小数点データに対してSUM()を計算する場合、大きな数値を連続して加算した後に小さな数値を減算する（スライディングウィンドウフレームによる）と、バイナリの精度が失われます。高度なRDBMSはKahan Summation（カハンの加算アルゴリズム）を適用して補正エラーバッファを維持し、累積偏差をゼロ付近のイプシロン閾値に保ちます。

難しい関数のためのセグメント木 (Segment Tree)

COUNT(DISTINCT)のような関数や、不規則なRANGEに基づくウィンドウフレームの場合、Dequeは役に立ちません。データベースは直ちにRAM内にセグメント木（Segment Tree）またはフェニック木（Fenwick Tree）を構築します。このときコストは$mathcal{O}(N log N)$に上昇し、複雑なクエリに対応します。

分散アーキテクチャとオペレーティングシステム (Distributed & OS-Level Constraints)

分散システム (MPP) のためのプレフィックスサム (Prefix Sum)

SnowflakeやBigQueryなどの大規模なデータウェアハウス（Data Warehouse）システムでは、Window Functionの計算を1つの仮想マシン（Node）に配置することはできません。1つの顧客パーティションのデータが多すぎる（Data Skew）場合、そのNodeはハング（OOM）します。 システムはプレフィックスサム (Prefix Sum / Scan) アルゴリズムを使用してこれを解決します。1人の顧客のデータは複数のセグメントに分割され、多くのNodeに割り当てられます。Node Aは最初の10日間の合計を計算し、Node Bは次の10日間の合計を計算します。その後、それらはお互いに「残り」の部分をブロードキャスト（Broadcast）します。数学の結合法則（Associativity）を利用することで、システムは分散遅延を$mathcal{O}(N / P + log P)$の閾値まで減少させます。

ヒュージページ (Huge Pages) とダイレクト I/O (OSバイパス)

巨大なキューと木の構造を格納するため、データベースは通常の4KB RAMを使用できません。仮想ページテーブルのミス（TLB Miss）はCPUを破壊します。データベースはトランスペアレントヒュージページ (Transparent Huge Pages – 2MB/1GB) を設定することが義務付けられています。

さらに、Window Functionのデータスキャンは「一度読んだら捨てる」操作です。OS Page Cacheを使用すると、他の有用なデータがRAMから押し出されてしまいます。データベースはO_DIRECTフラグをio_uring（Linux Asynchronous I/O）と組み合わせて使用し、オペレーティングシステムのカーネルを完全にバイパスして、NVMe SSDコントローラから内部メモリ領域にデータを直接転送します。

NUMAアーキテクチャとフォールスシェアリング (False Sharing) 現象

2 CPUサーバーでは、RAM構造は2つのNode（NUMA Node 0および1）に分割されます。RDBMSはCPU Pinningを使用して、CPU 0の計算スレッドがCPU 0にアタッチされたRAMのみを読み取るようにする必要があります。さらに、2つのスレッドが同じ64バイトの物理Cache Lineを上書きする（False Sharing）のを避けるために、C++でデータオブジェクトをパディング（alignas(64)）する必要があります。これは半導体回路の帯域幅を崩壊させる可能性のある現象です。

教訓 (Lessons Learned & Best Practices)

Window Functionの複雑なアーキテクチャを紐解くことで、システムエンジニアやDBAは以下の貴重な教訓を得ることができます。

  1. ソートのコストを理解する (The Cost of Sorting): 単にグループ化が必要なだけなら、Window Function内でORDER BYを乱用しないでください。どのORDER BYコマンドでも、巨大なソートエンジン（計算量$mathcal{O}(N log N)$およびSSDへの書き込みリスクあり）が起動します。
  2. ウィンドウフレームのサイズがアルゴリズムを決定する: 小さな制限範囲を持つROWS BETWEENを使用することは、常にRANGE BETWEENよりも高速です（後者はシステムに複雑な境界ロジックの処理を強要し、セグメント木を構築することさえあります）。
  3. データスキューネス (Data Skewness) の危険性: PARTITION BY句はデータが分散される方法を形作ります。国でパーティション化し、顧客の99%がベトナムにいる場合、すべてのデータが1つのCPUスレッド（または1つのNode）に集中し、クラスタの並列性が無意味になります。均等な分散を持つパーティションキーを探してください。
  4. RAMの制限とシステム構成: Sort/Hash用の専用メモリ割り当てパラメータ（PostgreSQLのwork_memなど）を賢く設定してください。割り当てが少なすぎると、クエリはハードドライブで非常に遅いExternal Merge Sortをトリガーします。割り当てが多すぎると、LinuxのOOM Killerがデータベースプロセスを終了させます。

結論

Window Functionの実行メカニズムは、複雑な組み合わせグラフ構造、微分最適化アルゴリズム、およびハードウェアレベルでの物理的なオーバークロック操作の複雑な交響曲です。このメカニズムを理解することは、よりスマートなSQLコードを書くのに役立つだけでなく、データアーキテクトが一般的なツールの限界を超える優れたビッグデータ分析システムを設計するための足がかりにもなります。
