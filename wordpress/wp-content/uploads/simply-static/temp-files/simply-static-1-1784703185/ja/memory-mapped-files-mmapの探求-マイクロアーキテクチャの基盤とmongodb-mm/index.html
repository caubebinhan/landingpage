Memory-Mapped Files (mmap)の探求: マイクロアーキテクチャの基盤とMongoDB MMAPv1の終焉

July 22, 2026 · admin · 4 分で読めます

エグゼクティブサマリー & 問題提起

データベース管理システム (DBMS) の発展の歴史において、キャッシュ管理と物理ストレージの選択はシステムの運命を決定づけます。MongoDB (MMAPv1エンジン) に代表される第一世代のNoSQLは、非常に魅力的でありながら致命的なリスクを伴う近道を選びました。それは、Memory-Mapped Files (mmap) メカニズムを通じて、I/Oとメモリの管理をオペレーティングシステムのカーネルに完全に委ねることでした。

この抽象化は、ソフトウェア開発における圧倒的なスピード（市場投入までの時間の短縮）という利点をもたらしましたが、システムがテラバイト規模のデータ（RAM容量の超過）と何十万もの同時書き込みトランザクション（高並行性）に直面したとき、mmapの仮想化モデルは崩壊しました。データベースレベルのロック (Database-level Lock)、スレッドを停止させるページフォルトの連続発生 (Page Fault Thrashing)、そしてストレージの断片化 (Storage Fragmentation) という壊滅的な問題が露呈したのです。

根本的な問題: OSのカーネルは汎用的なマネージャーです。B-Tree構造やデータベースファイル間のグラフ関係に関する「セマンティックな知識 (semantic knowledge)」を持っていません。カーネルのLRU (Least Recently Used) のような自動的なページ破棄 (page eviction) メカニズムに依存した結果、インデックスの重要なルートノード (root nodes) がディスクに追い出され (evict)、処理スレッド全体を麻痺させるメジャーページフォルト (major page fault) を引き起こしました。

このテクニカルレポートでは、mmapのマイクロアーキテクチャのメカニズムを深く分析し、アムダールの法則 (Amdahl’s Law) を通じてマルチスレッドパフォーマンスの崩壊を数学的に証明します。さらに、MMAPv1の死因を解剖し、なぜ WiredTiger (MVCCとユーザースペースのバッファプールを備える) のような自己管理型エンジンへの移行がデータベース業界の生存に不可欠な進化であったのかを解説します。

Memory-Mapped Files (mmap)のマイクロアーキテクチャの動作メカニズム

Memory-Mapped Files (POSIXにおけるmmapシステムコール) は、物理ディスク上のファイルの内容をプロセスの仮想アドレス空間 (Virtual Address Space) に直接マッピング (mapping) することを可能にする低レベルのシステム技術です。この方法は、read()やwrite()のような従来のI/O操作をバイパスします。ファイルがマッピングされると、プロセスはポインタを介してRAM内の通常のバイト配列にアクセスするのと全く同じように、ファイルの読み書きを行うことができます。

仮想マッピング (Virtual Mapping) の確立プロセス

プロセスがmmap()を呼び出すとき、OSのカーネルは直ちにディスクI/Oを実行しません。代わりに、新しい仮想メモリ領域 (Virtual Memory Area – VMA) を設定します。このVMA構造は、仮想アドレス範囲 $V_{start}$ から $V_{end}$ がディスク上のターゲットファイルを表すことを記録します。

このマッピングイベントは、CPUのメモリ管理ユニット (MMU) 内のマルチレベルページテーブル (Multi-level Page Table) とトランスレーション・ルックアサイド・バッファ (Translation Lookaside Buffer – TLB) メカニズムに密接に依存しています。

このマッピングを表す数式は、ブロックデバイス $mathcal{D}$ と組み合わせた、仮想空間 $mathcal{V}$ から物理空間 $mathcal{P}$ への変換として記述されます。サイズ $S_{page}$ (通常4KB) のメモリページが仮想アドレス $V_{addr}$ にある場合、物理アドレス $P_{addr}$ はページフォルト (Page Fault) が発生したときにのみ決定されます。$mathcal{F}$ をアドレス解決関数とします：

$$ P_{addr} = mathcal{F}(V_{addr}) = P_{base} + (V_{addr} pmod{S_{page}}) $$

ページフォルト (Page Fault) の解剖と遅延ペナルティ (Latency Penalty)

mmapメカニズムの心臓部は、メジャーページフォルト (Major Page Faults) によってトリガーされるデマンドページング (Demand Paging) です。

アプリケーションがポインタを使用して $V_{addr}$ 配列の領域にアクセスしようとしたとき、対応する物理ページがRAMにロードされていない場合、MMU (Memory Management Unit) はページテーブル内に有効なマッピングを見つけることができません。MMUは直ちにハードウェア例外 (Hardware Exception) をトリガーし、カーネルに制御を移します。

メジャーページフォルトの処理プロセス：

  1. Context Switch: アプリケーションプロセスがブロック (blocked) されます。
  2. Kernel VMA Lookup: カーネルは、フォルトが発生した仮想アドレスに対応する物理ファイルとオフセットを特定します。
  3. I/O Fetch: カーネルは空のRAMページ (Page Frame) を割り当て、DMA (Direct Memory Access) コマンドを発行して物理ディスク (HDD/NVMe) からこのRAMページにデータをプルします。
  4. PTE Update: ページテーブルエントリ (Page Table Entry) を更新し、$V_{addr}$ を新しい物理RAMページにリンクします。
  5. Resume: コンテキストを復元し、アプリケーションはメモリアクセス命令を再実行します。
flowchart TD
    A[ユーザースペースプロセス] -->|V_addrのアクセスと変換| B(Memory Management Unit - MMU)
    B -->|バッファのルックアップ| C{Translation Lookaside Buffer - TLB}
    C -- TLB Hit --> D[物理アドレス P_addr]
    C -- TLB Miss --> E{Page Table Walk (ページテーブルウォーク)}
    E -- PTE 有効 --> F[TLBの更新とP_addrの返却]
    E -- PTE 無効 (Invalid) --> G((Major Page Fault))
    G --> H[カーネルモードへのコンテキストスイッチ]
    H --> I[VMAを通じたディスク上のブロックの特定]
    I --> J[物理RAMへのページ読み込みI/Oの調整]
    J --> K[ページテーブルエントリ - PTE の更新]
    K --> L[ユーザースペースプロセスの復元]
    F --> D
    L --> B
    D --> M[RAMデータの読み書き]

I/Oの数学的観点から見ると、このプロセスは極端に高いレイテンシを伴います。$T_{L1}$ をL1キャッシュアクセス時間 (1ns)、$T_{RAM}$ をRAMアクセス時間 (100ns)、$T_{disk}$ をディスクレイテンシ (NVMeで100µs、HDDで5ms) とします。ページフォルトが発生する確率を $P_{fault}$ とします。

期待されるアクセス時間 $E[T_{access}]$ は次のようになります：

$$ E[T_{access}] = (1 – P_{fault}) cdot T_{RAM} + P_{fault} cdot T_{disk} $$

$T_{disk} gg T_{RAM}$ ($10^3$ から $10^5$ 倍の差) であるため、$P_{fault} = 0.01$ (アクセスの1%でフォルトが発生) であっても、システムのパフォーマンスは数千分の一に低下します。データスキャンクエリは、データベースプロセスを絶え間ないコンテキストスイッチの犠牲者に変えてしまいます。

mmapの「ゼロコピー (Zero-Copy)」の幻想

ページフォルトのコストにもかかわらず、mmapは初期のシステムに魅力的な幻想をもたらしました。それはダブルメモリコピーが不要 (Zero-copy read/write) という点です。 従来の read() では、データは ディスク $rightarrow$ カーネルページキャッシュ $rightarrow$ ユーザーバッファ へと移動します。 mmap では、カーネルページキャッシュがそのままユーザーバッファになります。アプリケーションはそこに直接読み書きを行います。

これにより、MongoDBはエンジンコードを極めて高速に記述することができました。エンジニアはバッファプールを構築する必要も、LRUによるエビクションを管理する必要も、バックグラウンドのフラッシュ (background flush) アルゴリズムを実装する必要もありませんでした。Linuxカーネルがすべてを代行してくれたからです。スタートアップにとって、これは市場投入までの時間 (time-to-market) において絶対的な競争優位性でした。

しかし、この技術的負債には高い代償を払うことになります。

// MongoDB MMAPv1のmmapの幻想を示す疑似コード
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>

void* init_database_mmap(const char* db_file_path, size_t* out_size) {
    int fd = open(db_file_path, O_RDWR);
    struct stat sb;
    fstat(fd, &sb);
    
    // カーネルにデータベース全体を仮想空間にマッピングするよう要求
    void* db_memory = mmap(NULL, sb.st_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
    
    *out_size = sb.st_size;
    close(fd); // OSはfdが閉じられてもマッピングを維持する
    
    return db_memory;
    // これ以降、MongoDBはC++のポインタを使用してdb_memory内のB-TreeとDocumentsに直接アクセスする
}

MongoDB MMAPv1の解剖: アーキテクチャ、定量的な限界と死角

MMAPv1 (バージョン3.0までのデフォルトエンジン) では、MongoDBのすべての物理ファイルは前述のメカニズムを通じて仮想空間に直接リンクされています。

指数関数的なファイル割り当ての法則 (Exponential Allocation)

MMAPv1は番号付けされたセグメントデータファイル (database.0, database.1…) を作成します。各新しいファイルは前のファイルの2倍の容量を持ち、2GBの制限に達するまで2の累乗に従います：

$$ S_i = min(2^i cdot 64 text{ MB}, 2048 text{ MB}) $$

この目的は、機械的なHDDのシーク時間を短縮するために、レコード (documents) を連続したディスク領域 (contiguous extents) に保存することです。ファイル内には、BSONドキュメントの双方向連結リスト (Doubly Linked List) を含むエクステント (Extent) があります。

動的割り当てとドキュメント再配置 (Document Relocation) の課題

NoSQLでは、ドキュメントは柔軟なスキーマを持っています。今日1KBのサイズだったドキュメントが、アプリケーションが内部の配列 (arrays) にデータを追加したことで、明日には5KBに膨れ上がる可能性があります。

もしドキュメントに隣接するスペースがすでに占有されている場合、更新処理はドキュメント再配置 (Document Relocation) という惨劇を引き起こします。

  1. MongoDBは新しい5KBの空き領域の割り当てを要求しなければなりません。
  2. memcpy を使用してすべてのデータをそこにコピーします。
  3. 古い1KBの領域を解放します (断片化 – fragmentation – を生成します)。
  4. 新しい物理アドレスを指すように、すべてのインデックス (Indexes) のすべてのリーフノード (leaf nodes) を更新します。

余裕を持たせるために動的なパディングメカニズムが追加されましたが、それは莫大なストレージ容量を消費し、波及的なI/Oの影響 (ripple I/O effects) のリスクを完全に解決するものではありませんでした。

アムダールの法則のボトルネック: データベースレベルのロック (Database-Level Lock)

RAMへのアクセスは生のC++ポインタ (bare pointers) によって行われるため、スレッドセーフ (Thread-safety) の保護は非常に高くつきます。当初、MMAPv1はプロセスレベルのグローバルロック (Global Reader-Writer Lock) を使用していました。後にデータベースレベルのロック (Database-level Lock) に引き下げられました。

これは、1つのスレッドが Users コレクションに書き込んでいる場合、データベース全体 (Orders や Products を含む) がロックされることを意味します！誰も読み書きできなくなります。

アムダールの法則 (Amdahl’s Law) は、このマルチスレッドパフォーマンスの低下に対する数学的な証明を提供します： $P$ をシステムが並行して実行できる時間の割合とし、$S = 1 – P$ を順次実行の時間 (ロックのクリティカルセクション内) とします。 $N$ 個のCPUコアにスケールアウトした場合の最大スピードアップは：

$$ text{Speedup}(N) = frac{1}{(1 – P) + frac{P}{N}} $$

MMAPv1では、1つのロックがデータベース全体をカバーするため、順次処理の割合 $1 – P$ が非常に大きくなります。接続数 $N to infty$ のとき、パフォーマンスは次のレベルに制限されます：

$$ lim_{N to infty} text{Speedup}(N) = frac{1}{1 – P} $$

MMAPv1のために128コアのサーバーを購入することはお金をドブに捨てるようなものです。なぜなら127のコアはロックを待ってスピンウェイト状態 (Spin-waiting) になり続けるからです。

ページフォルトスラッシングの悪夢と降伏 (Yield)

最悪の事態は、データベースロックとメジャーページフォルトが組み合わさったときに発生します。

  1. スレッド $T_1$ がデータベースを書き込みロック (Write Lock) でロックします。
  2. $T_1$ が仮想アドレス $V_x$ にあるドキュメントに書き込もうとします。
  3. 不運なことに、$V_x$ に対応する物理ページは、カーネルのLRUアルゴリズムによって使用頻度が低いと見なされ、OSによって「密かに」ディスクにスワップアウト (swap out) されたばかりでした。
  4. $T_1$ はカーネルによってブロック (Blocked) され、ページフォルトが発生します。カーネルがディスクからデータを引き出すまで $10$ ミリ秒待たなければなりません。
  5. その $10$ ミリ秒間、$T_1$ は書き込みロックを保持し続けます。
  6. 他の何千もの接続が完全にフリーズします。システムは原因不明のまま「凍結」します。

自らを救うため、MongoDBはYield (譲歩) メカニズムを統合しました。アクセスする前に、Linuxの mincore() API関数を呼び出し、そのページがRAMにあるかどうかOSに尋ねます。もし無ければ、スレッドは自発的にロックを解放 (Yield lock) し、I/Oリクエストを発行するバックグラウンドプロセス (Background Page-in) を実行して待機します。この場当たり的な解決策により、並行状態マシン (concurrency state machine) は歪み、予測不可能なものになりました。

MMAPv1の終焉とWiredTigerへの移行

ビッグデータ (Big Data) の爆発的な増加に伴い、MMAPv1の不備は致命的な障壁となりました。メモリ処理をOSのカーネルに無条件に依存することは、DBMSが自身のデータに対して「盲目」であることを意味します。オペレーティングシステムは、B-Treeのルートノード (RAMに固定 – pin – する必要がある) と履歴ドキュメント (自由にディスクに追い出せる) の違いを理解していません。

MVCCの妥協とユーザースペース (User-Space)

2014年、MongoDBはWiredTigerを買収し、マイクロアーキテクチャの技術哲学における大きな転換をマークしました。WiredTigerは mmap の自動メカニズムを拒絶しました。代わりに、アプリケーション空間 (User-Space) に完全に位置する独自の バッファプール (Buffer Pool) を構築しました。 これにより絶対的な制御権がもたらされました。WiredTigerはセマンティックな階層認識 (Semantic Hierarchy Awareness) と組み合わせたLFU/LRU置換アルゴリズムを使用し、B-Tree構造が常にRAMに存在することを保証します。

粗いDBレベルのロック (Coarse-grained locks) を使用する代わりに、WiredTigerは 多版同時実行制御 (Multi-Version Concurrency Control – MVCC) を適用しました。すべての書き込み操作は、古いデータ行を直接変更するのではなく、ポインタチェーン (Pointer Chains) で接続された新しいコピーをRAM内に作成します。 これにより：

  • 読み取りが書き込みをブロックすることはありません (Readers don’t block Writers)。
  • ロック (Locking) はドキュメントレベル (Document-Level) で行われます。
  • 並行処理の割合 $P approx 1$。アムダールの法則によれば、WiredTigerは今や128コアサーバーのパワーを完全に活用できます。

ブロック圧縮 (Block Compression) の力

mmapを放棄したことで、データ圧縮の可能性も解放されました。mmapはディスクとRAMを1:1でマッピングするため、ディスク上のデータは圧縮できませんでした (圧縮すると、仮想バイトオフセットのアドレスが完全に間違ってしまいます)。

独立したバッファプール構造により、RAM内のデータは処理のために解凍されますが、フラッシャースレッド (Flusher) がディスクに書き込む際、WiredTigerはzstdまたはSnappyアルゴリズムを通じてブロックを圧縮します。 圧縮率 $mathcal{C}_{ratio} = frac{S_{uncompressed}}{S_{compressed}}$ は容易に $3x to 5x$ に達します。これはクラウドストレージのコストを削減するだけでなく、I/Oスループットを何倍にも向上させます (同じ情報量に対してハードディスクが読み取るバイト数が減るため)。

// WiredTigerのようなユーザースペースバッファプールの優位性を示す抽象的な疑似コード
struct BufferPoolManager {
    page_table: HashMap<LogicalPageId, FrameId>, // DBMS独自の仮想マッピング
    frames: Vec<PageFrame>,
    eviction_policy: Box<dyn EvictionAlgorithm>,
    wal_manager: WriteAheadLog,
}

impl BufferPoolManager {
    fn fetch_page(&mut self, page_id: LogicalPageId) -> Result<&mut PageData, DiskError> {
        if let Some(&frame_id) = self.page_table.get(&page_id) {
            // Cache Hit: セマンティックLFUの更新
            self.eviction_policy.record_access(frame_id);
            Ok(self.frames[frame_id].get_data_mut())
        } else {
            // Cache Miss: DBMSがI/Oを自己調整し、スレッドを停止させるカーネル割り込み/ページフォルトを引き起こさない！
            let frame_id = self.evict_oldest_document_page()?; 
            
            // O_DIRECT + Async I/O (io_uring) を使用してデータを引き出す
            let compressed_data = disk_manager.async_read_page(page_id)?;
            let raw_data = snappy::decompress(compressed_data);
            
            self.frames[frame_id].fill(raw_data);
            self.page_table.insert(page_id, frame_id);
            Ok(self.frames[frame_id].get_data_mut())
        }
    }
}

結論

MMAPv1の終焉は、お粗末な設計に対する裏切りではなく、システムソフトウェアの避けられない進化のライフサイクルの証です。mmapは歴史的な役割を見事に果たしました。MongoDBの迅速な立ち上げを支援し、開発者を惹きつけ、Web 2.0の時代を支配しました。

しかし、マルチスレッドのビッグデータとマイクロサービスアーキテクチャ (Microservices) の世界に足を踏み入れたとき、帯域幅とロックの競合に関する数学の法則は避けられませんでした。WiredTigerのMVCC、独立したバッファプール、ブロック圧縮を備えたユーザースペースを取り戻すことで、MongoDBは原始的なNoSQLソリューションから完全なエンタープライズ向けのトランザクショナルデータストア (Transactional Datastore) へと再構築されました。

エンジニアに残された最も深遠な教訓：アプリケーションの中核となるセマンティックなドメイン知識 (Semantic Domain Knowledge) を、汎用的なオペレーティングシステムのカーネルに決して委ねてはならない。
