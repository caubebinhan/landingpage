技術ホワイトペーパー #51: HTAPデータベース – ハイブリッドトランザクション/分析処理のアーキテクチャ

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー

この技術ホワイトペーパーは、ハイブリッドトランザクション/分析処理 (Hybrid Transactional/Analytical Processing – HTAP) データベースシステムの包括的かつマイクロアーキテクチャ的な分析を提供します。歴史的に、データベース業界はOLTP (オンライン・トランザクション処理) とOLAP (オンライン分析処理) の間の厳格な分岐を受け入れてきました。HTAPはこの妥協を打ち破ります。デュアルフォーマット・メモリトポロジー (Delta-Mainアーキテクチャ) の数学、異なるデータレイアウト間での多版同時実行制御 (Multi-Version Concurrency Control – MVCC) の複雑さ、そしてSIMDベクトル化とLLVM JITコンパイルの圧倒的な計算能力を探求することで、本書はシステムエンジニア、データベースアーキテクト、および研究者に、次世代の統合データエンジンを理解し構築するために必要な深い技術知識を提供します。

はじめに: データ処理の歴史的な分岐

40年以上にわたり、データベースエコシステムは根本的な二面性によって特徴付けられてきました。システムは高速に書き込むように設計されているか、または高速に読み取るように設計されているかのどちらかです。

  • OLTP (Online Transaction Processing): PostgreSQLやMySQLのようなシステムは、高頻度、低レイテンシ、ポイントクエリ、および更新用に最適化されています。これらは行指向 (N-ary Storage Model) レイアウトを使用し、1つのディスクI/Oまたはキャッシュラインのフェッチでレコード全体 (例：ユーザープロファイル) を取得または変更できるようにします。
  • OLAP (Online Analytical Processing): Snowflake、ClickHouse、BigQueryなどのシステムは、大量のデータを集計する複雑で読み取りが多く、長時間実行されるクエリに最適化されています。これらは列指向 (Decomposition Storage Model) レイアウトに依存しており、シーケンシャルメモリ帯域幅を最大化し、極端なデータ圧縮を可能にします。

問題は何でしょうか？今日のビジネスは、ライブトランザクションデータに対するリアルタイムの分析を必要としています。このギャップを埋めるための従来のアプローチでは、OLTPシステムからデータを抽出し、変換し、OLAPデータウェアハウスにロード (ETL) します。これにより許容できない遅延ウィンドウが発生し、「リアルタイム分析」が「昨日の分析」に変わってしまいます。

HTAPは、これらのワークロードを単一のエンジンに統合し、パフォーマンスの分離を厳密に維持しながらETLパイプラインを排除するように設計されたアーキテクチャのパラダイムシフトです。しかし、単一のエンジンが根本的に対立する2つの目的をどのように満たすことができるのでしょうか？

コアとなる問題: インピーダンスミスマッチ

1 行ストアの分析ペナルティ

OLAPクエリ (例：SELECT SUM(salary) FROM employees) が行ストアで実行されると、CPUは単一の salary 属性を読み取るためだけに、行全体をL1/L2キャッシュにロードする必要があります。これにより大規模なキャッシュ汚染 (address や biography などの不要な属性のロード) が発生し、メモリ帯域幅を浪費します。CPUはデータ不足に陥り、システム全体がボトルネックになります。

2 列ストアのトランザクションの非柔軟性

逆に、列ストアで INSERT または UPDATE を実行することは運用の悪夢です。単一の論理行の変更では、ディスク上に散在する数十または数百の異なる列方向ファイルへのシークと書き込みが必要になります。この散在するランダムI/Oはトランザクションスループットを完全に破壊し、マイクロ秒の操作をミリ秒の操作に変えてしまいます。

3 データ移動のコスト (ETL)

2つの別々のデータベースを維持するには、ETLパイプラインが必要です。ETLは壊れやすく、計算コストが高く、数分から数時間に及ぶことが多い古いデータ（鮮度の限界 $Delta T$）をもたらすことで悪名高いです。現代のアルゴリズム取引、不正検出、または動的価格設定において、1時間のデータの遅れは、数百万ドルの収益の損失に直結します。

統合エンジンソリューション: Delta-Mainアーキテクチャ

インピーダンスミスマッチを解決するために、HTAPデータベース (TiDB、SingleStore、SAP HANAなど) はマルチフォーマット、デュアルストレージ・アーキテクチャを採用しています。データは論理的に単一のスキーマ内に存在しますが、同じシステム内で物理的には行形式と列形式の両方でインスタンス化されます。

1 Delta-Mainメモリトポロジー

マイクロ秒レベルの変更ごとに2つの物理的表現を同期的に維持することは非常にコストがかかるため、HTAPエンジンは インメモリのDelta-Mainアーキテクチャ を利用します。

  • Delta Store (行指向): すべての受信書き込み操作 (INSERT、UPDATE、DELETE) は、RAM内の並行性が高く、ロックフリー (lock-free) の行指向バッファに取り込まれます。これにより高速なトランザクションの変更を吸収し、サブミリ秒のコミットを保証します。
  • Main Store (列指向): 履歴データの大部分は、高度に圧縮され、読み取りに最適化された列形式に格納されます。

2 非同期コンパクションとTuple Movers (タプルムーバー)

行指向のDelta Storeが利用可能なすべてのRAMを消費し、分析を遅らせるのを防ぐために、非同期バックグラウンドパイプライン (通常、Tuple Mover または Compactor と呼ばれます) が継続的に実行されます。Deltaストア内の不変 (immutable) な行のスナップショットを取得し、それらを列形式に転置し、強力な圧縮 (Run-Length Encoding、Dictionary Encoding、Bit-packing) を適用して、Main Storeにフラッシュします。

属性 $A$ に対する分析クエリ $C_{olap}$ の数学的コストには、両方のスキャンが含まれます:

$$C_{olap} = C_{column_scan}(N – Delta) + C_{row_scan}(Delta)$$

$C_{row_scan}$ は $C_{column_scan}$ よりもタプルあたりのコストが大幅に高いため、Tuple Moverのアルゴリズムは、$Delta$ を小さく保つために十分に積極的に動作しつつ、OLTPスレッドのCPUサイクルやメモリ帯域幅を枯渇させないように慎重に動作する必要があります。

3 ハードウェアへの影響: NUMA、TLB、およびHuge Pages

インメモリHTAPシステムを構築するには、OSとハードウェアの制限を深く理解する必要があります。標準の4KB OSメモリページを使用すると、数テラバイトの列データを急速にスキャンすることで Translation Lookaside Buffer (TLB) ミスが引き起こされます。HTAPエンジンは常にLinuxカーネルを構成して Huge Pages (大容量ページ) (2MBまたは1GB) を割り当て、TLBを常駐させ、ページウォーカーをアイドル状態に保ちます。

さらに、Non-Uniform Memory Access (NUMA) トポロジーでは、CPUソケット0のスレッドがCPUソケット1のRAM内にある列データをスキャンしようとすると、データがQPI/UPIインターコネクトを通過する必要があり、寄生レイテンシが追加されてスループットがボトルネックになります。HTAPアロケータは、実行中の分析スレッドのNUMAノードに特定の列チャンクを固定し、ローカルのメモリ帯域幅を最大化します。

HTAPにおける多版同時実行制御 (MVCC)

同じエンジンで高スループットの書き込みと長時間実行される読み取りを同時に処理するには、防弾のアイソレーションが必要です。従来の2相ロック (Two-Phase Locking – 2PL) はここでは壊滅的です。5分間のOLAP読み取りクエリは数百万行で共有ロックを獲得し、OLTP書き込みを5分間完全にブロックしてしまいます。

1 MVCCを介したスナップショットアイソレーション

HTAPは均一に Multi-Version Concurrency Control (MVCC) を活用します。ライターはデータを所定の位置で上書きしません。新しいバージョンのタプルを追加します。リーダーはロックを取得しません。特定の論理読み取りタイムスタンプ ($T_{read}$) に対応する、不変で時間的に一貫したデータベースのスナップショットから読み取ります。

2 デュアルフォーマット間での可視性の解決

複雑さは、デュアルフォーマット間で可視性を解決するときに生じます。

  1. 分析クエリは、高度に圧縮されたColumnar Main Storeをスキャンします。ただし、これらのタプルの一部はDelta Storeで更新または削除された可能性があります。エンジンは、Roaring Bitmapまたは無効化リストを使用して、これらの古いタプルを除外します。
  2. クエリは同時にDelta Storeをスキャンします。各行について、次の可視性述語 (visibility predicate) を評価します:

$$Visible(V) = (BeginTS le T_{read}) land (EndTS > T_{read})$$

この条件を満たす特定のバージョンのタプルのみが、最終結果セットにマージされます。

3 ガベージコレクションとウォーターマーク

HTAPエンジンが1秒間に10,000回の更新を処理する場合、1秒間に10,000個の古いバージョンが生成されます。システムは、最も古いアクティブなトランザクションを表すグローバルな $T_{watermark}$ を維持します。バックグラウンドのVacuum (バキューム) プロセスは継続的にDelta Storeをスキャンし、$EndTS < T_{watermark}$ となるタプルのバージョンを物理的に削除してメモリを再利用し、キャッシュ密度を維持します。

ハードウェアアクセラレーションによるクエリ実行

MVCCのオーバーヘッドとデュアルフォーマット調整のコストを補うために、HTAP分析エンジンは従来の1タプルずつの (Volcano) 処理をバイパスし、最新のCPUマイクロアーキテクチャの限界まで活用します。

1 ベクトル化実行とSIMD

ベクトル化はデータをバッチ (1024または4096個の値のチャンク) で処理します。これにより、仮想関数ディスパッチのオーバーヘッドが償却され、タイトなループがL1命令キャッシュ (Instruction Cache) に完全に常駐します。 重要なのは、ベクトル化により Single Instruction, Multiple Data (SIMD) 機能 (Intel AVX-512など) が解放されることです。CPUは16 x 32ビット整数を単一の512ビットレジスタにロードし、16個のすべての値に対して単一のクロックサイクルで算術演算または述語チェック (salary > 50000) を実行できます。これにより、スカラー実行と比較して理論的に $16times$ のスピードアップが得られます。

2 LLVMを介したJust-In-Time (JIT) コンパイル

代替または補完的なアプローチは、LLVMベースのJITコンパイルです。事前コンパイルされた一般的なC++関数を実行する代わりに、データベースは実行時にユーザーのSQLクエリ専用に調整された特注のマシンコードを動的に記述してコンパイルします。JITコンパイルは複数の演算子 (operators) を融合させ、データを完全にCPUレジスタ内に保持し、L1キャッシュへのメモリラウンドトリップを完全に排除します。

3 鮮度とパフォーマンスのトレードオフ

HTAPにおいて、データの鮮度は制御可能なスペクトルです。アナリストが5分前のデータ ($Delta T = 5 text{ min}$) を許容できる場合、Query Optimizer (クエリオプティマイザ) は行指向のDelta Storeを完全にバイパスし、不変のColumnar Main Storeのみをスキャンできます。CPUに負荷のかかる可視性述語チェックとデュアルフォーマットのマージを回避することで、クエリは数桁速く実行されます。これは美しい数学的トレードオフを明らかにしています。マイクロ秒レベルの鮮度を犠牲にすることで、ギガバイト/秒単位のスキップスループットの向上が得られます。

学んだ教訓と設計哲学

HTAPシステムの進化は、ソフトウェアおよびシステムエンジニアリングにいくつかの深い教訓を与えています:

  1. 統合システムはスマートなトレードオフについてである: HTAPは魔法のように物理法則を排除するわけではありません。RAMの容量 (デュアルフォーマットの保存) とバックグラウンドのCPUサイクル (Tuple Movers) を、運用のシンプルさ (ETLなし) およびリアルタイムの鮮度と交換することで、OLTP/OLAPのインピーダンスミスマッチを解決します。
  2. データの鮮度はブール値ではなくスペクトルである: 最新のシステムは、アプリケーションがそれぞれの鮮度要件を交渉できるようにする必要があります。鮮度の限界 $Delta T$ をクエリオプティマイザに公開することで、大規模な最適化が可能になり、分析的洞察にとって厳格なシリアライザビリティ (strict serializability) が多くの場合不必要なボトルネックであることを証明しています。
  3. ハードウェアとソフトウェアのコデザインは必須である: 一般的なPOSIXの抽象化を使用して、1秒間に10億行のエンジンを構築することはできません。真のHTAPパフォーマンスを実現するには、開発者はキャッシュライン、NUMA境界、TLBページサイズ、およびSIMDレジスタを明示的に尊重するコードを記述する必要があります。ソフトウェアはシリコンの物理的な現実に従わなければなりません。

結論

HTAPデータベースは、運用システムと分析システムの間の数十年にわたる壁を解体することに成功した、現代のデータエンジニアリングの頂点です。インメモリのDeltaストア、バックグラウンドの列圧縮、ロックフリーのMVCC、およびSIMDベクトル化の繊細なシンフォニーをオーケストレーションすることにより、HTAPエンジンは、ビジネスがライブデータ上で生成されたまさにそのミリ秒に複雑な機械学習モデルと集計を実行できるようにします。データの量と速度が爆発的に増加し続ける中、ハードウェア最適化の絶対的な限界から生まれたHTAPアーキテクチャは、将来のデータベースプラットフォームの紛れもない標準となるでしょう。
