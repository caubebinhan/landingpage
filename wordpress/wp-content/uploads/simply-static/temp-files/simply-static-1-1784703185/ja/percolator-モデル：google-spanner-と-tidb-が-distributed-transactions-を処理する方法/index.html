Percolator モデル：Google Spanner と TiDB が Distributed Transactions を処理する方法

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー (Executive Summary)

本ドキュメントは、分散データベース (Distributed Databases) のトランザクション処理の方法を永遠に変えたアーキテクチャの基盤である Percolator モデルに関する詳細な技術分析 (whitepaper) を提供します。2010年に Google によって導入された Percolator は、巨大な NoSQL ストレージ (Bigtable) 上で完全な ACID トランザクション機能を提供するという、不可能と思われていた問題を解決しました。

読者が得られる知識：

  • コアアーキテクチャの理解: Percolator が多版同時実行制御 (MVCC) とステートレスな2フェーズコミットプロトコル (Stateless Coordinator / 2PC) を組み合わせてどのように機能するかを把握します。
  • 物理メカニズムの分析: ハードディスク I/O (NVMe) を最適化するために LSM-Tree 構造上に直接データストレージを構築するマルチカラム ($A_{data}$, $A_{lock}$, $A_{write}$) 手法を探求します。
  • 分散ネットワークの最適化に関する知識: Timestamp Oracle (TSO) がボトルネックになる理由と、TiDB が Async Commit を使用してそれをどのように克服するかを理解します。
  • Google Spanner の量子進化: Google がルビジウム原子時計と GPS (TrueTime API) を応用してリアルタイムを操作し、CAP定理の限界を曲げる方法を目撃します。

コアな問題点 (The Core Problem)

クラウドコンピューティングとマイクロサービスのアーキテクチャでは、データは地理的に分散した複数のデータセンターに配置された数千の独立した物理サーバーにわたってハッシュ化 (hashed) およびシャーディング (sharded) されます。複数のレコードを同時に更新するトランザクションにおけるデータの一貫性 (Consistency) を維持することは困難な課題です。

  • 従来のリレーショナルデータベース (RDBMS) は集中型ロックマネージャ (Centralized Lock Manager) を使用します。分散環境では、これは単一障害点 (SPOF) を生み出し、ネットワーク遅延 (Network Latency) による致命的なパフォーマンスのボトルネックを引き起こします。
  • X/Open XA アーキテクチャを介した古典的な2フェーズコミット (2PC) プロトコルでは、コーディネーター (Coordinator) がトランザクションの状態をハードディスクに保存する必要があります。コーディネーターが途中でクラッシュした場合、参加しているすべてのサーバーのリソースが無期限にブロックされる問題 (Blocking Problem) が発生します。

Google は、ボトルネックがなく、書き込みトランザクションにブロックされることなく読み取りトランザクション (read-only) を超高速で実行できる、完全な分散メカニズムを必要としていました。これが Percolator モデルが誕生した理由です。

詳細な技術分析 (Deep Technical Analysis)

理論的基盤: MVCC と時間の座標

Percolator の理論的基盤は、多版同時実行制御 (Multi-Version Concurrency Control – MVCC) と、楽観的並行性制御 (Optimistic Concurrency Control – OCC) に基づく分散型の 2PC プロトコルの密接な組み合わせを中心に構成されています。

データの各バージョンは静的なスカラー値 (scalar value) ではなく、二次元の時間空間における座標点です。トランザクション $T_i$ が開始すると、システムは開始タイムスタンプ (start timestamp) を割り当て、これを $T_{s,i}$ と表します。このタイムスタンプは偏光レンズのように機能します。$T_i$ は、コミットされた (committed) トランザクション $T_j$ によって作成された $T_{c,j} < T_{s,i}$ のタイムスタンプを持つデータバージョンのみを観察する権限を持ちます。これにより Snapshot Isolation (SI) の分離レベルが保証され、ダーティリード (Dirty Read) とファントムリード (Phantom Read) の現象が完全に排除され、ライトスキュー (Write Skew) のわずかなリスクのみが残ります。

データのミクロ構造: 3カラム手法 (Three-Column Layout)

これらの抽象的な概念を具現化するために、Percolator は Bigtable/LSM-Tree のストレージレイヤー内に独自のデータシャーディングミクロ構造を設計しました。論理属性 $A$ は暗黙的に3つの別々の物理カラムにコンパイルされます。

  1. $A_{data}$ カラム (Payload): 生データが保存される場所です。トランザクションが更新されると、新しい値 $V_{new}$ はキー $T_s$ を用いてここに直接プッシュされます。このプロセスは、トランザクションが完了していなくても行われます。
  2. $A_{lock}$ カラム (Semaphore): 分散排他ロックメカニズムを管理します。書き込み時、トランザクションはこのカラムにフラグを設定する必要があります。このフラグを見た並行トランザクションは、デッドロック (Deadlock) を避けるために OCC の原則に従って能動的にバックオフ (backoff) しなければなりません。
  3. $A_{write}$ カラム (Source of Truth): 真実のレジスタ。トランザクションが正常にコミットされた場合にのみ更新されます。レコードには $T_c$ のキーが含まれ、値は $A_{data}$ カラム内の座標 $T_s$ を逆指すポインタ (pointer) です。

この間接ポインタを使用した手法は、NVMe ディスク帯域幅を非常に効果的に最適化し、書き込み増幅 (write amplification) の現象を排除します。重いデータ (large binaries) はディスクに一度だけフラッシュされます。

ステートレス 2PC メカニズム (Stateless Coordinator) とプライマリロック (Primary Lock)

Percolator は「ステートレスコーディネーター」手法を具現化しています。コーディネーター (通常はクライアントライブラリ) は状態を一切保存せず、トランザクションの進行はデータレイヤーに直接刻み込まれます。

トランザクションプロセス：

  1. プレライトフェーズ (Prewrite Phase): クライアントは変更するレコードのセット ($K$) を分析します。クライアントはランダムに1つのプライマリロック (Primary Lock) $k_p$ を選択します。残りのキーはセカンダリロック (Secondary Locks) となります。クライアントはすべてのストレージデバイスに Prewrite コマンドをブロードキャストします。ストレージノードは競合をチェックします (将来のトランザクションが上書きしたか、またはキーを保持しているか)。安全であれば、新しい値が $A_{data}$ に書き込まれ、フラグが $A_{lock}$ に書き込まれます (セカンダリロックは $k_p$ へのポインタを保持します)。
  2. コミットフェーズ (Commit Phase): クライアントはコミットタイムスタンプ $T_c$ を取得します。クライアントは Commit コマンドを プライマリロック $k_p$ を保持している特定のサーバーにのみ 発行します。$k_p$ のロックフラグがまだ安全であれば、ストレージノードは時点 $T_c$ において $A_{write}$ にポインタを書き込み、$A_{lock}$ のフラグを削除します。
  3. バックグラウンド非同期同期 (Background Async): $k_p$ がコミットされた瞬間、分散トランザクション全体がグローバルに成功したと宣言されます。残りの数千のセカンダリロックに対するコミット操作は、バックグラウンドスレッド (background threads) にプッシュされます。クライアントは待つ必要がありません。

TSO の問題と TiDB の Async Commit ソリューション

すべての Percolator トランザクションには、Timestamp Oracle (TSO) による時間の割り当てが必要です。TSO は数百万の RPC 割り込みを処理しなければならないため、ネットワーク遅延 (Network Latency) の致命的なボトルネック (Bottleneck) になります。RPC バッチ処理を適用しても、RTT (ラウンドトリップタイム) の遅延は依然として存在します。

Percolator を継承するオープンソースデータベースである PingCAP TiDB は、このレイヤー全体を再構築しました。Raft Consensus プロトコルを深く組み込むことで、TiDB は 非同期コミットトランザクション (Async Commit) / 1PC 構造を生み出しました。

TiDB は TSO にクエリを実行する代わりに、ローカルストレージノード (TiKV) が自らのローカルクロックに基づいて予想コミット時間を自己補間することを要求します。コーディネーターはこれらの数値を集計し、グローバルな Max $T_c$ を決定します。第2のコミットフェーズは完全に排除され、バックグラウンドで実行されます。RTT サイクルは 2 から 1 に削減され、コントロールネットワーク帯域幅の 50% を節約します。

Google Spanner の物理的挑戦: TrueTime API

Google Spanner はさらに一歩進んで、時間空間そのもののミクロな動作メカニズムに直接影響を与えます。脆弱な TSO を使用する代わりに、Google はデータセンターに GPS 信号受信ネットワークとルビジウム原子時計を装備し、TrueTime API システムを作成しました。

$TT.now()$ 関数は不確実なウィンドウ (uncertainty window) $[t_{earliest}, t_{latest}]$ を返します。この不確実性の境界の幅 $epsilon$ は常に 7 ミリ秒未満に制限されます。

Spanner はこの理論を コミット待機ルール (Commit Wait Rule) に適用します。2PC プロセスの最終フェーズで、Spanner は $T_c = t_{latest}$ を割り当てます。コーディネーターは直ちに成功を報告するのではなく、OS カーネルを $TT.now().earliest > T_c$ になるまで「スリープ」 (Sleep) させます。この 7 ミリ秒のバッファは因果関係の逆転リスクを永久に排除し、絶対的な 100% の信頼性で厳密なグローバル外部一貫性 (Strict Global External Consistency) を保証し、従来の TSO の RTT の障壁を完全に打ち破ります。

教訓と実践 (Lessons Learned)

  1. 2PC に対する見方の変化: 2PC は必ずしも遅く行き詰まるものではありません。「ステートレスコーディネーター」 (メタデータをデータに直接刻み込む) と「プライマリーロック」アンカーを使用することで、Percolator は従来の 2PC の SPOF 問題を完全に解決しました。マイクロサービスを構築する際、すべてのシステムアーキテクトはこの設計から学ぶべきです。
  2. LevelDB / LSM-Tree の力: Percolator の3カラム構造は、B+Tree を使用すると数百万のランダムアクセス (Random Access) が発生するため、ハードディスク (HDD) を破壊する可能性があります。LSM-Tree を使用することで、これらの操作はすべてシーケンシャルライト (Sequential Writes) に変換されます。ソフトウェアの最適化はデータ構造の特性と密接に関連していなければなりません。
  3. OS I/O 管理 (Group Commit): fsync() 関数はパフォーマンスの敵です。TiKV のような優れたデータベースエンジンは、グループコミット (Group Commit) 手法と io_uring メカニズムおよび Direct I/O を組み合わせて使用し、ディスクフラッシュの複雑さを $mathcal{O}(M)$ から $mathcal{O}(1)$ に変更します。独立した操作ごとに fsync を呼び出してはいけません。
  4. ハードウェアによる限界の打破: ソフトウェアが飽和点 (TSO のボトルネック) に達した場合、唯一の解決策はハードウェアの介入です。ルビジウム原子時計を展開するという Google の決定は、Spanner を地球上で No.1 のデータベースに変えました。

結論 (Conclusion)

2PC/OCC アーキテクチャを備えた Google の Percolator プロトタイプから、TiKV/TiDB の Raft および Async Commit ネットワーク最適化の改良を経て、Spanner 上の TrueTime API の誕生に至るまでの道のりは、コンピュータサイエンスの叙事詩です。直列化可能性グラフ理論、非同期 Linux カーネルアーキテクチャ、量子力学 (不確実な時間の処理)、および LSM-Tree ストレージ技術を融合させることで、システム設計者は弱点のないデータストアを作成し、今後数十年にわたるクラウドコンピューティングの未来を形作りました。
