Transaction Isolation Levels: Write Skew、Read Skew、および Phantom Reads の徹底解剖

July 22, 2026 · admin · 3 分で読めます

エグゼクティブ・サマリー (本記事の概要)

このテクニカルホワイトペーパーは、並行データベースシステムにおける Transaction Isolation Levels の極めて厳密な調査を提供するものであり、特に3つの悪名高い異常である Read Skew、Write Skew、および Phantom Reads のマイクロアーキテクチャ、アルゴリズム、数学的な複雑さを徹底的に解剖します。現象論的でしばしば不十分な ANSI SQL-92 規格の定義を超え、本ドキュメントはこれらの異常を検出し軽減するために必要な正式な Serialization Graph (SG) 手法を探求します。

読者のための重要なポイント:

  • 異常の全体像の理解: 従来の Dirty Reads や Non-repeatable Reads の定義が、最新の MVCC (Multi-Version Concurrency Control) や Snapshot Isolation 環境においてなぜ不十分であるか。
  • 深い技術的メカニズム: cache line bouncing (MESI protocol)、CPU memory reordering、および NUMA architecture などの現象が、ソフトウェアレベルのロックや分離アルゴリズムとどのように相互作用するか。
  • 高度な軽減戦略: Serializable Snapshot Isolation (SSI)、Epoch-Based Memory Reclamation、および Next-Key Locking を介した Predicate Locking の近似実装の設計詳細。
  • 実践的なエンジニアリングの教訓: データの不変性 (data invariants) を損なうことなく、高スループットのトランザクションシステムをどのようにアーキテクトしデバッグするか。

根本的な問題 (何が問題なのか？)

並行性の高い OLTP システムにおいて、データベースエンジンは ACID プロパティ (Atomicity, Consistency, Isolation, Durability) を維持しながら、毎秒数千の重複するトランザクションを実行しなければなりません。根本的な対立は、Strict Serializability (数学的な正確さを保証するが、順次実行を強制することでパフォーマンスを破壊する) と Concurrency (ハードウェアの利用率を最大化するが、静かなデータ破損のリスクを伴う) の間に存在します。

ANSI SQL-92 規格は、特定の異常の防止に基づいて isolation levels (Read Uncommitted, Read Committed, Repeatable Read, Serializable) を分類しようとしました。

  1. Dirty Reads: コミットされていないデータの読み取り。
  2. Non-repeatable Reads (Read Skew): トランザクションが同じタプルを2回読み取り、異なる結果を得ること。
  3. Phantom Reads: 並行する挿入/削除により、再実行時に範囲クエリが異なる行のセットを返すこと。

しかし、この古典的なフレームワークは、Snapshot Isolation (PostgreSQL や Oracle など) を使用するシステムに本質的に蔓延する破壊的な異常である Write Skew を完全に無視していました。根本的な問題は、最新のデータベースエンジンが基本的なロックに依存できないことです。破壊的な CPU cache invalidation やメモリのオーバーヘッドを避けながら、分離違反を示すトポロジー上のサイクルを捕らえるために、複雑でメモリ最適化された数学的構造 (依存関係追跡のための Directed Acyclic Graphs など) を維持しなければなりません。

深い技術分析 (詳細な知識)

トランザクション異常と Serialization Graphs の基礎

isolation levels を正式化するには、トランザクションの履歴を操作の正式なスケジュールに分解する必要があります。トランザクション $T_i$ が読み取り操作 $r_i(x)$ と書き込み操作 $w_i(x)$ から成り、コミット $c_i$ またはアボート $a_i$ で終了するとします。スケジュール $S$ は、その実行が何らかの直列スケジュールと同等である場合、strictly conflict-serializable (厳密な競合直列化可能) と呼ばれます。

私たちは Serialization Graph (SG) —ノードがコミットされたトランザクションであり、エッジが競合を表す Directed Acyclic Graph (DAG)— を構築することによって異常を検出します。

  • Write-Read (WR): コミットされていない/新しくコミットされたデータの読み取り。
  • Read-Write (RW): あるトランザクションが別のトランザクションが読み取ったデータを上書きする逆依存関係 (anti-dependency)。
  • Write-Write (WW): ブラインド書き込み (blind writes)。

スケジュール $S$ は、その Serialization Graph $SG(S)$ がサイクルを含まない場合に限り、strictly conflict-serializable となります。Generalized Isolation Level フレームワーク (Adya らによって提案された) は、すべての異常を有向サイクルとしてモデル化します。分離の適用は、CPU のボトルネックとなることなく、$O(1)$ または $O(V+E)$ の時間で特定の巡回構造を動的にプルーニングすることに直接マッピングされます。

Read Skew とマイクロアーキテクチャのコヒーレンシ

Read Skew は、トランザクションが構造的に一貫したスナップショットを認識すべきであるという不変性を侵害します。Read Committed では、$T_1$ が $x$ を読み取り、$T_2$ が $x$ を上書きしてコミットし、$T_1$ が再度 $x$ を読み取ると新しい値を取得します。

Snapshot Isolation は、単調増加するトランザクションタイムスタンプ $TS(T_i)$ を割り当てることでこれを軽減します。エンジンは可視性関数 $V(x, T_i)$ を評価し、$CTS(T_j) le TS(T_i)$ である $x$ の物理バージョンを見つけます。

内部的には、タプルのバージョンは原子的な Compare-And-Swap (CAS) を使用して、ロックフリーのリングバッファ内で管理されます。Core A がタプルを更新するとき、それは cache line の排他的所有権 (MESI protocol の ‘Modified’ 状態) を取得します。Core B がバージョンチェーンを読み取ると、cache miss が誘発され、Core A に共有 L3 キャッシュ (‘Shared’ 状態) に書き戻すよう強制します。この cache line bouncing がスループットを制限します。

さらに、古いバージョンを回収するには Epoch-Based Reclamation が必要です。munmap のようなシステムコールを使用する単純なメモリ解放は、すべてのコアを停止させる Inter-Processor Interrupts (IPIs) を介した Translation Lookaside Buffer (TLB) シュートダウンを引き起こします。高度なエンジンは、Huge Pages 上にカスタムの slab allocators を使用することでこれを回避します。

// Rustでのロックフリーな可視性評価
use std::sync::atomic::{AtomicPtr, Ordering};
use crossbeam_epoch::{self as epoch, Atomic};

struct TupleVersion {
    val: i64,
    commit_ts: u64,
    prev: Atomic<TupleVersion>,
}

fn evaluate_visibility<'g>(
    chain_head: &Atomic<TupleVersion>, txn_start_ts: u64, guard: &'g epoch::Guard
) -> Option<i64> {
    let mut current_ptr = chain_head.load(Ordering::Acquire, guard);
    while !current_ptr.is_null() {
        let current_version = unsafe { current_ptr.deref() };
        if current_version.commit_ts <= txn_start_ts {
            return Some(current_version.val);
        }
        current_ptr = current_version.prev.load(Ordering::Acquire, guard);
    }
    None
}

Write Skew の破壊力

Write Skew は特に Snapshot Isolation を悩ませます。これは、2つのトランザクションが一貫したスナップショットを評価し、交差するデータを読み取りながら、アプリケーションの不変性によってバインドされた分離したセットを変更する場合に発生します。

例: 不変条件 $A + B ge 0$。

  • $A=100, B=100$。
  • $T_1$ は $A, B$ を読み取り、$A$ を 150 減らす。(ローカル状態: $100+100-150 ge 0$)。
  • $T_2$ は $A, B$ を読み取り、$B$ を 150 減らす。(ローカル状態: $100+100-150 ge 0$)。
  • それらの書き込みセットが交差しないため、両方ともコミットされる ($W(T_1) cap W(T_2) = emptyset$)。
  • 結果: $A = -50, B = -50$。不変条件が粉砕された。

数学的には、Write Skew は Serialization Graph の「危険な構造」、つまり2つの隣接する逆依存関係 (RW) エッジを持つサイクルに対応します: $T_i xrightarrow{rw} T_j xrightarrow{rw} T_k$。

これを根絶するために、エンジンは Serializable Snapshot Isolation (SSI) を展開します。SSI は、トランザクションノードの inConflict および outConflict フラグを使用して、実行時に読み取り-書き込みの逆依存関係を動的に追跡します。トランザクションがピボットになった (入出力両方の RW エッジを示した) 場合、システムはサイクルを断ち切るために1つのトランザクションを強制的にアボートします。CPU ソケット間でこれを効率的に追跡するには、Thread-Local Storage (TLS) にある非常に密度の高い、分割可能なハッシュセットが必要です。

Phantom Reads と Predicate Locking

Phantom Reads は動的な範囲クエリ (SELECT * WHERE condition) に影響を与えます。$T_1$ が範囲をクエリしてセット $mathcal{S}_1$ を確認します。$T_2$ が条件に一致する新しい行を挿入します。$T_1$ が再クエリすると $mathcal{S}_2$ を確認します ($mathcal{S}_1 neq mathcal{S}_2$)。

まだ存在しない行を物理的にロックすることはできません。理論的な解決策は Predicate Locking、つまり数学的述語 $f(row) rightarrow boolean$ をロックすることです。しかし、すべての書き込み操作に対して任意のブール述語を評価することは、計算上困難です (NP困難)。

最新のデータベースは、B+ Trees 上の Next-Key Locking または Gap Locking を使用してこれを近似します。範囲クエリがリーフノードに到達すると、エンジンは特定のレコードだけでなく、連続するレコード間の物理的な「ギャップ」に対しても共有読み取りロックを取得します。

gap locks の管理には、物理的リソース ID を複雑なキューにマッピングする、futex によって保護されたハッシュバケットを使用する非常に洗練されたロックマネージャが必要であり、その背後では Tarjan の Strongly Connected Components アルゴリズムを実行する常時バックグラウンドのデッドロック検出 (Deadlock Detection) がサポートしています。

教訓とベストプラクティス (得られた教訓)

  1. SQL-92 の Isolation の定義を信用しない: 「Repeatable Read」または「Snapshot Isolation」の下で PostgreSQL や Oracle を使用する場合、明示的に SERIALIZABLE に引き上げるか、アプリケーションレベルの SELECT ... FOR UPDATE ロックを実装しない限り、根本的に Write Skew に対して脆弱です。
  2. ロックのハードウェアコストを理解する: ロックが取得されるたびに、アトミックな操作が cache line を変更し、MESI protocol を介して他のすべての CPU コアにわたってそれを無効化します。高スループットの挿入における gap locks は、必然的に大規模な CPU 相互接続の輻輳を引き起こします。
  3. Phantom ギャップに注意する: アプリケーションが集計制約 (例：「医師は1日に8シフト以上持つことはできない」) に依存している場合、標準の行ロック (row locks) では同時挿入から保護されません。Next-Key Locking (MySQL の Serializable では自動的に発生します) に依存するか、集計を個別のロックされた行にマテリアライズ (materialize) する必要があります。
  4. HTM は未来である: Intel TSX のような Hardware Transactional Memory (HTM) により、マイクロプロセッサはL1/L2キャッシュ内で完全に競合検出を処理でき、拡大するソフトウェア SSI グラフをバイパスして、マルチソケット NUMA ボード上でかつてないスケーリングを実現します。

結論

トランザクションの分離を保証することは、単なるデータベース意味論の演習ではありません。それは、シリコン物理学、メモリ遅延、およびグラフ理論に対する過酷な戦いなのです。現象論的な SQL-92 の定義から Serialization Graphs の正式な数学的厳密さへの移行により、最新のエンジンは毎秒数百万のトランザクションを処理できるようになりました。SSI、Epoch-Based Reclamation、および Gap Locking の低レベルの実装を理解することは、データ損失ゼロでハイパースケールのトランザクションシステムを構築しようとするアーキテクトにとって不可欠です。
