専門技術白書: Cascades Framework – 現代のQuery Optimizerシステムにおけるゴールドスタンダード・アーキテクチャ

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー (Executive Summary)

Query Optimizerは、全てのリレーショナルデータベース管理システム (RDBMS) の真の「頭脳」です。数ミリ秒で完了するクエリとサーバー全体をフリーズさせるクエリの違いは、抽象的なSQLコマンドを物理的な実行計画にコンパイルするシステムの仕組みにあります。過去30年間にわたり、Goetz Graefeによって設計されたCascades Frameworkはゴールドスタンダードとして台頭し、Microsoft SQL Server、Apache Calcite、CockroachDB、Greenplumなどの分散型および高性能スーパーデータベースの中核的な基盤となっています。

この白書では、Cascades Frameworkのアーキテクチャ構造を深く分析します。Memoグラフ (Memoization Graph) の構造、トップダウン評価 (Top-down evaluation) アルゴリズム、分枝限定刈り込み (Branch-and-bound pruning) メカニズム、そしてこのアーキテクチャがマイクロアーキテクチャ (Micro-architecture) レベルでハードウェアとどのように共生しているかを解剖します。対象読者は、大規模データシステムを習得、最適化、およびデバッグ (debug) するために深い運用原理を理解する必要があるデータベースエンジニアやソフトウェアアーキテクトです。

はじめに: クエリ最適化の問題 (The Optimization Problem)

問題とは何か？ プログラマーがSQLクエリを記述するとき、彼らは「どのようにデータを取得するか」 (How) ではなく、「どのような結果を求めているか」 (What) を指定しています。10個のデータテーブルを結合 (join) する単純なクエリは、176億通り以上の方法 (実行計画ツリー) で実行される可能性があります。Query Optimizerの課題は、この176億通りの探索空間をナビゲートし、それぞれのコスト (Cost) を計算し、CPU、I/O、ネットワーク帯域幅の消費が最も少ないものを選択することです。

この問題の複雑さはNP-Hardの閾値に達するため、ブルートフォース (Brute-force) やボトムアップの動的計画法 (古典的なSystem RのようなBottom-up Dynamic Programming) を使用すると、システムはSQLコマンドの実行方法を見つけるためだけにすべてのRAMとCPUを使い果たしてしまいます。

画期的なソリューション: Cascades Frameworkは、3つの直交するコンポーネントを分離するという設計哲学により、この組み合わせ爆発の問題を解決します。

  1. 論理探索空間 (Logical Search Space)。
  2. 物理コストモデル (Physical Cost Models)。
  3. ルール実行エンジン (Rule Execution Engine)。

その結果、Cascadesはデータベース開発者にとって非常に拡張性 (Extensible) の高いアーキテクチャを提供すると同時に、巨大な探索空間を最小限のRAM容量に圧縮します。

アーキテクチャの基盤: Memoデータ構造 (The Memo Structure)

Cascadesの心臓部はMemo構造であり、有向非巡回グラフ (Directed Acyclic Graph – DAG) です。何十億もの個別の解析ツリーを生成する代わりに、Memoは共通のノードを共有することでそれらを圧縮します。

Logical Equivalence Classes (論理的同値類)

初期の論理計画がロードされると、それは複数のGroupに分割されます。各Groupは論理的同値類を表します。つまり、背後にあるアルゴリズムに関係なく、同じデータ結果セットを生成するすべての代数式の集合です。 各Groupの内部には、複数のGroup Expression (グループ式) が含まれています。Group Expressionの入力オペランド (operands) は、他の式を直接指すのではなく、子Groupを指します。この抽象化により、代数ツリーは極めて高密度に共有されたネットワークに変わります。

$n=10$のテーブルの場合、生成される結合ツリーの数は $N(n) = frac{(2n-2)!}{(n-1)!}$ です。テラバイト級のRAMを消費する代わりに、Memoの遅延ポインタ (lazy pointer) 共有メカニズムは空間計算量を $mathcal{O}(n cdot 2^n)$ に削減します。実際、RAMの消費量は常に数メガバイトに抑えられます。

プロパティ管理 (Properties Management) と Enforcers

Memoエコシステムは、2つのプロパティドメインを定義します。

  • 論理プロパティ (Logical Properties): 列構造、フィルター条件、カーディナリティ (Cardinality) 推定などの静的メタデータ。計算の重複を避けるため、Group全体で1回だけ保存されます。
  • 物理プロパティ (Physical Properties): データのソート順 (Sort Order) やネットワークの分散パーティション (Distribution Partition) など。

要件と実際の状況を一致させるために、CascadesはEnforcersシステムを使用します。データが列 $A$ でソートされている必要があるとします。Cascadesは、非常に高速ですがソートされていないデータ読み取りアルゴリズムを見つけました。それを捨てる代わりに、その高速なアルゴリズムにEnforcer (Sort演算子) を追加する方が、(B+ Tree Scanのような) 既にソートされているアルゴリズムを使用するよりもコストが低いかどうかを確認します。 このプロセスのBellman方程式:

$$ C_{opt}(G, P) = min left( min_{e in G} left( C_{local}(e) + sum_{i=1}^{k} C_{opt}(G_i, P_i) right), C(E_P) + C_{opt}(G, emptyset) right) $$

アルゴリズムのメカニズム: トップダウン探索空間と刈り込み (Top-down Search Space and Pruning)

Memo内のすべての変換はRules (ルール)によって制御されます (例: A JOIN B を B JOIN A に交換)。CascadesとSystem Rの最大の違いは、トップダウン (Top-Down) のトラバーサル戦略と分枝限定法 (Branch-and-Bound) の組み合わせです。

ボトムアップアルゴリズムでは、すべてのツリーブランチのコストをボトムアップで集計する必要があります。対照的に、トップダウンはルートから開始し、物理的要件 (「ソートされたデータが必要」など) を子ノードに伝播させます。これにより、上位レベルで要求されるプロパティを満たさない場合、システムが無意味な子ブランチ (例: Hash Join) を生成するのを防ぎます。

分枝限定刈り込み (Branch-and-Bound Pruning)

再帰的なトラバーサルの過程で、システムが $Cost_{limit} = 1000$ の完全な計画を見つけると、この記録が保存されます。アルゴリズムが別のブランチにジャンプすると、コストが累積されます。途中で、すでにトラバーサルしたステップの総コストと、まだトラバーサルしていないステップの下限 (Lower Bound) の合計が $1000$ を超えた場合、Cascadesは即座にその下の巨大なブランチに入ることなくすべて刈り取ります。

$$ C_{accumulated} + C_{local}(e) + sum_{i in text{unoptimized_children}} LB_{cost}(G_i) geq Cost_{limit} $$

Promise Function (ヒューリスティックナビゲーション)

速度を最適化するために、CascadesはPromise Functionを適用してルールの優先順位を決定します。コストを最も削減する可能性が高いルールが最初に適用され、超低コストの $Cost_{limit}$ が迅速に生成されるため、Branch-and-Bound武器は他のブランチをより早く排除できます。

ハードウェアレベルの交差点 (Hardware Micro-architecture)

数学的理論やグラフ理論だけでは不十分です。真のOptimizerは、オペレーティングシステムやCPUのマイクロアーキテクチャ (Micro-architecture) と親和性を持って動作する必要があります。

メモリ管理: 断片化の回避 (Heap Fragmentation)

Cascadesのプロセスでは、わずか数ミリ秒の間に数千万個の Group と GroupExpr オブジェクトが生成および破棄されます。malloc() (C) や new (C++) を継続的に使用すると、膨大なガベージフラグメンテーション (Heap Fragmentation) とロック競合 (Lock Contention) が発生します。最新のOptimizerはBump-pointer Arena Allocatorsを使用します。これらは、mmap を通じてオペレーティングシステムから大きなRAMブロック (Linuxでは通常2MB/1GBのHuge Pagesを使用) を要求し、その内部で割り当てを行います。これにより、TLB Missesが完全に排除されます。

Cache-Line Packing と Hardware Prefetcher

CPUはRAMからバイト単位で読み取るのではなく、Cache Line (64 Bytes) のブロック単位で読み取ります。Cascadesのソースコード構造では、#pragma pack と alignas(64) を使用して、GroupExpr オブジェクトが64バイト内に完全に収まるように強制します。 この細心の設計により、CPUのHardware Prefetcher回路がアクティブになります。CPUがNode $A$を計算しているとき、ハードウェアはプロアクティブにNode $B$をL3 Cacheからコアレジスタにプッシュし、Pipeline Stall (データ待機レイテンシ) を完全に吹き飛ばします。

Branchless Programming と SIMD

何十億ものプロパティマトリックス計算を評価する際、Cascadesの if/else ループ構造はCPUのBranch Predictorを破壊します。現代のエンジンは、ビット単位のマスク演算子 (Bitwise mask) を使用して、これらの比較をBranchless Programming (分岐のないプログラミング) に変換します。 さらに、システムは順次比較するのではなく、SIMD (AVX-512) を使用して16個または32個の論理的同値性チェック操作を単一のクロックサイクルに圧縮し、リアルタイムクエリ処理 (Real-time OLAP) の限界を突破します。

学んだ教訓 (Lessons Learned & Best Practices)

データベースエンジニア (Database Engineers) とソフトウェアアーキテクト (Software Architects) 向け:

  1. システムの「刈り込み」 (Pruning) の仕組みを理解する: クロス列制約 (cross-column constraints) が不明確なSQLコマンドを記述すると、Cardinality Estimation システムが歪められます。推定が間違っていると、$LB_{cost}$ も歪められ、Branch-and-Boundが誤って真に最適な計画を刈り取り (prune)、最悪の計画を残す原因となります。常にStatistics (統計) を維持し、外部キー (Foreign Keys) 関係を宣言してください。
  2. Physical Properties の力: B-Treeインデックス (Index) が、Full Table Scanよりも遅いのに使用されたりされなかったりするのはなぜかと疑問に思ったことはありませんか？それは、B-Treeの「ソート済み」 (Sorted) 特性が、上位レベルでのSort操作を回避するのに役立つ非常に貴重な Physical Property だからです。Top-Down Enforcersを理解することは、Schemaの設計を最適化するのに役立ちます。
  3. ソフトウェア設計には Mechanical Sympathy が必要: アルゴリズムのBig-O計算量がどれほど最適であっても、数百万のGarbage Objectsを生成し、Cache Linesを破壊し、TLB Missesを引き起こす場合、ハードウェアに優しいブルートフォースアルゴリズムよりも遅くなります。Arena Allocatorの考え方を大規模な計算システムに適用してください。
  4. 関心事の分離 (Separation of Concerns): Cascadesの最大のアーキテクチャ上の教訓は、Rules、Cost Model、およびSearch Engineの分離です。複雑なビジネスソフトウェアシステムの設計において、論理ブロック (Business Rules vs Execution Engineなど) を分離することで、無限の拡張性 (Extensibility) がもたらされます。

結論

1995年の学術的なアイデアから、Cascades Frameworkは30年間にわたり進化を遂げ、データベース管理システムの領域を支配するようになりました。グラフと組合せ数学の傑作であるだけでなく、システムプログラミングの芸術における確固たる証拠でもあります。最も抽象的な代数的概念が、シリコン、Cache Lines、SIMDレジスタの最も無味乾燥で機械的な法則に頭を下げ、調和しなければならない場所なのです。
