Write-Ahead Logging (WAL)とARIESリカバリアルゴリズム：マイクロアーキテクチャと数学的基盤の徹底解説

July 22, 2026 · admin · 6 分で読めます

エグゼクティブサマリーと問題提起

データベース設計や分散ストレージシステムのエンジニアリングにおいて、高いパフォーマンスを維持しながらデータの永続性 (Durability) を保証することは常に究極の課題です。停電やソフトウェアエラー (kernel panic) が発生すると、保存されていないデータはRAMからすべて消失してしまいます。システムが1秒間に数十万のトランザクションを処理する速度を落とすことなく、データをバイト単位で正確な状態に回復するにはどうすればよいでしょうか？

核となる問題： 複雑なデータ構造 (B+Treeなど) を物理ディスクに直接書き込むと、莫大なランダムI/O (Random I/O) コストがかかり、原子性 (Atomicity) を保証できません。データページの書き込み途中でシステムがクラッシュすると、そのページは完全に破損してしまいます (Torn Page)。

この問題に対する究極の解決策が、Write-Ahead Logging (WAL) とARIESリカバリアルゴリズムの組み合わせです。この記事では、WALのマイクロアーキテクチャ、マルチスレッド同期プロセスの背後にある数学的モデル、Group Commitメカニズムの待ち行列理論、および混沌としたデータ復旧問題を破壊不可能な完璧な論理グラフに変換するARIESアルゴリズムの状態収束定理について深く掘り下げます。最後に、I/Oインフラストラクチャ設計の教訓 (Lessons Learned) を導き出します。

Write-Ahead Logging (WAL) の理論的基礎とマイクロアーキテクチャ

リレーショナルデータベースと最新の分散ストレージシステムは、ACID定理の2つの核心的な特性である原子性 (Atomicity) と永続性 (Durability) を保証するために、かけがえのないマイクロアーキテクチャ基盤としてWrite-Ahead Logging (WAL) に依存しています。データ状態の変更をディスク上のデータ構造 (B+TreeやHeap filesなど) に直接書き込む代わりに、システムはこれらの変更をログレコード (log records) にシリアライズし、WALストリームの末尾に追記 (append-only) します。

WALの究極の原則は数学的に厳密に述べられています。データページ $P_i$ に対する最後の変更を表すログレコードがディスク上での永続性を保証されるまで、バッファプール (Buffer Pool) から物理ストレージデバイスへのデータページ $P_i$ のフラッシュ (flush) は許可されません。

ページ $P_i$ を更新した最後のレコードのLog Sequence Numberを $LSN_{page}$ とし、ディスクに安全にフラッシュされた最大のLSNを $LSN_{flushed}$ とすると、WALの核心的な不等式は稼働中のすべての時点で常に成立しなければなりません：

$$ LSN_{page} le LSN_{flushed} $$

Log Sequence Number (LSN) の構造と数学的意味

Log Sequence Number (LSN) の物理構造は通常、単調増加する64ビット符号なし整数 (monotonically increasing 64-bit unsigned integer) です。これは、物理WALストリーム内のログレコードの絶対位置を指定する論理オフセットポインタ (logical offset pointer) として機能します。

各WALレコードには、次のような構造のメタデータがカプセル化されています：

  1. Transaction ID (TxID): トランザクション識別子。
  2. 操作タイプ (Type): Insert, Update, Delete。
  3. スペースID (Space ID) とページID (Page ID): 物理データが存在する場所。
  4. Before-Image (Undo Log): ロールバック (rollback) に使用される変更前のデータ。
  5. After-Image (Redo Log): ロールフォワード (roll-forward) に使用される変更後のデータ。
  6. PrevLSN: 同じトランザクションの前のレコードを指すポインタで、逆方向の連結リストを形成します。

LSNシーケンスのシリアル化により、データベースエンジン内のすべての状態変化 (state mutation) 操作に対して、線形かつ全体的な時間順序 (total global order) が作成されます。数学的に言えば、WALストリームは混沌としたマルチスレッドシステムを厳密に順序付けられた変換状態のシーケンスに変換します。

WAL Bufferレベルでの並行性制御 (Concurrency Control)

ワーカースレッド (worker threads) がトランザクションを実行する際、ユーザ空間メモリ (User-space Memory) 内のWAL Bufferと呼ばれる中間バッファ構造にデータを書き込む権利を競います。WAL Bufferレベルでの並行性制御には、レイテンシを最小限に抑えるための最適なマイクロアーキテクチャ技術が必要です。なぜなら、ここがシステム全体のシステムスループット (throughput) における最も深刻なボトルネックになるからです。

全体的な std::mutex を使用してWAL Bufferを保護する素朴な (naïve) 設計では、システムが毎秒数千のトランザクションのしきい値に達すると、即座にCPU飽和 (lock contention) を引き起こします。これを解決するために、エンジニアはハードウェアの fetch_add 命令 (Compare-And-Swap) を通じたロックフリー (lock-free) なLSN割り当てメカニズムを使用します。

ロードするレコードの予想サイズを $S_{record}$ とします。実行スレッドはアトミックなハードウェア命令を呼び出します：

$$ LSN_{allocated} = text{AtomicFetchAndAdd}(text{Global_LSN}, S_{record}) $$

この命令は、他のスレッドをブロックすることなく、1 CPUクロックサイクルの時間内にそのスレッド専用の $LSN_{allocated}$ 位置を返します。その後、スレッドは対応するオフセットでデータをWAL Buffer配列にコピー (memcpy) するだけです。この構造はLock-free Ring Bufferと呼ばれ、ScyllaDBやInnoDBなどのシステムが数十万TPSのスループットを達成する背後にある秘訣です。

Torn Writeの惨劇とCRC32Cチェックサム

物理的な観点から見ると、RAMから不揮発性ストレージデバイス (NVMe/SSD) へのWALフラッシュ (WAL flushing) プロセスは、Torn WriteまたはSector Tearingと呼ばれる大きな課題に直面しています。

SSDは、セクター単位 (512バイトまたは4096バイト) でアトミックな書き込み (atomic write) 機能を提供します。データベースがハードウェアのアトミックな制限を超えるサイズ (例：16KBのログレコード) のログ書き込みコマンドを発行し、ハードウェアがこのデータブロックを書き込んでいる途中で突然の停電が発生した場合、WALレコードの一部 (最初の4KBなど) は保存されますが、残りの部分には古いガベージデータが含まれることになります。

このLSNチェーン構造の破損を検出するために、各WALレコードの先頭には常に、ペイロード全体をカバーするハードウェアアクセラレーション暗号化ハッシュ関数、代表的には CRC32C (Castagnoli) による整合性チェックコードが追加されます。

リカバリフェーズでログを読み取る際、厳格なCRCチェックが行われます。もし：

$$ text{CRC32C}(text{ReadPayload}) ne text{StoredCRC} $$

であれば、システムはTorn Writeイベントが発生したと結論付け、その不完全なログレコードの処理を直ちに拒否します。CRCエラーが発生したポイントが、リカバリデータストリームの有効な論理的終了点となります。

#include <atomic>
#include <cstdint>
#include <cstring>
#include <vector>
#include <nmmintrin.h> // For hardware CRC32 instruction (_mm_crc32_u64)

struct LogRecordHeader {
    uint64_t lsn;
    uint32_t txn_id;
    uint32_t payload_size;
    uint32_t crc32; // Mathematical integrity check
};

class WALManager {
private:
    std::atomic<uint64_t> current_lsn_{0};
    std::atomic<uint64_t> flushed_lsn_{0};
    uint8_t ring_buffer_[1024 * 1024 * 16]; // 16MB lock-free circular buffer
    
public:
    uint64_t AppendRecord(uint32_t txn_id, const std::vector<uint8_t>& data) {
        // Lock-free allocation of buffer space via hardware atomics
        uint32_t total_size = sizeof(LogRecordHeader) + data.size();
        uint64_t allocated_lsn = current_lsn_.fetch_add(total_size, std::memory_order_relaxed);
        
        LogRecordHeader header;
        header.lsn = allocated_lsn;
        header.txn_id = txn_id;
        header.payload_size = data.size();
        header.crc32 = CalculateHardwareCRC32(data.data(), data.size());
        
        // Copy header and payload into the ring_buffer_ using modulo arithmetic
        size_t offset = allocated_lsn % sizeof(ring_buffer_);
        // (Implementation handles wrapping around the edge of the ring buffer)
        
        return allocated_lsn;
    }

    void GroupCommit(uint64_t target_lsn) {
        // Wait-free check if the data has already been flushed by a concurrent thread
        if (flushed_lsn_.load(std::memory_order_acquire) >= target_lsn) {
            return; 
        }
        
        // Acquire internal mutex strictly for disk I/O coordination
        // Issue O_DIRECT write from ring_buffer_ down to physical NVMe
        // Update the global visibility of flushed data
        flushed_lsn_.store(target_lsn, std::memory_order_release);
    }

    uint32_t CalculateHardwareCRC32(const uint8_t* data, size_t length) {
        uint32_t crc = 0xFFFFFFFF;
        const uint64_t* ptr64 = reinterpret_cast<const uint64_t*>(data);
        size_t i = 0;
        
        // Exploit x86-64 SSE 4.2 instruction for massive throughput
        for (; i + 8 <= length; i += 8) {
            crc = _mm_crc32_u64(crc, *ptr64++);
        }
        // ... handle remaining bytes
        return crc ^ 0xFFFFFFFF;
    }
};

待ち行列理論とGroup Commitの最適化

I/Oシステムのパフォーマンスは、ソースコードだけでなく、シリアライゼーションとディスクフラッシュ (I/O flushing) 操作のコスト関数に強く左右されます。

個々のトランザクションを個別に物理ディスクに書き込むこと (Sync-on-commit) は、Write Amplificationのペナルティをもたらし、PCIe帯域幅を浪費します。コストのかかるI/Oレイテンシに対抗するため、アーキテクチャエンジンは Group Commit (グループコミット) 戦略を採用しています。この技術は意図的にマイクロ遅延ループ ($T_{wait}$) を設定し、トランザクションのセットが一緒にWALへの追加操作を完了するのを待ちます。数がしきい値に達すると、システムはハードディスクへの単一のI/Oストリームを発生させます。

解析学および待ち行列理論 (Queueing Theory) の観点から見ると、システムのスループット (Throughput) はIOPS操作数に反比例します。Group Commitを適用すると、待ち行列モデルは $M/M/1$ からバッチ処理 $M/G/1$ (Bulk Service) に移行します。

1つのトランザクションのログ追加時間を $t_{append_i}$、ディスクハードウェアへのフラッシュ遅延時間を $t_{flush_i}$ とします。$N$ 個の個別のトランザクションの総時間コストは次のようになります：

$$ Cost_{naive} = sum_{i=1}^{N} (t_{append_i} + t_{flush_i}) $$

ブロック全体の $N$ 個のトランザクションにGroup Commitを適用すると：

$$ Cost_{group} = sum_{i=1}^{N} (t_{append_i}) + t_{flush_group} $$

個々のトランザクションの遅延 (Latency) と全体のスループット (Throughput) の間のトレードオフ (trade-off) は、凸最適化 (convex optimization) 問題です。$T_{wait}$ が大きすぎると、トランザクションの遅延が急増し、アプリケーションがハングアップ (timeout) します。$T_{wait}$ が小さすぎると、I/O数が急増し、ハードウェアシステムがクラッシュします。PostgreSQLのような最新のシステムは、リアルタイムで測定されたディスク帯域幅に基づいてGroup Commitのしきい値を自動調整 (auto-tuning) します。

ARIESリカバリアルゴリズム：数学的分析と収束性

データ復旧システムのアーキテクチャ分野において、1992年にIBMのC. Mohan博士によって発明された ARIES (Algorithms for Recovery and Isolation Exploiting Semantics) は、ゴールドスタンダード (gold standard) として広く認識されています。ARIESの完璧な設計は、「歴史の繰り返し (Repeating History)」モデルとデータ操作の論理的セマンティクスの活用を中心に展開しています。

ARIESは以下のシステム原則から発展しました：

  • No-Force: トランザクションのコミット時にダーティページ (dirty pages) のディスクへのフラッシュを強制しない (ランダムI/Oを最小限に抑える)。
  • Steal: システムが 未 コミットのトランザクションのデータページの一部をディスクにプッシュすることを許可する (必要に応じてRAMバッファを解放するため)。

この柔軟性こそが、ディスク上のデータ破損のリスクを生み出します。これを解決するために、ARIESは厳格な3つのフェーズのプロセスを使用します：Analysis (分析), Redo (再実行), Undo (元に戻す)。ARIESの心臓部は、複雑なLSNチェーン設計と、収束 (convergence) を保証する役割を果たす Compensation Log Record (CLR) 構造にあります。

フェーズ1：分析フェーズ (Analysis Phase)

最も近い有効なチェックポイント (Checkpoint) を読み取ることから始まります。このフェーズの数学的な目標は、クラッシュ直前 (Crash point) のメモリ空間の状態を正確に再構築することです。2つの重要なデータ構造を再作成します：

  1. トランザクションテーブル (TT): 進行中のトランザクションとそれらの最後のLSNを保存します。
  2. ダーティページテーブル (DPT): RAM上で変更されたがまだディスクにフラッシュされていないページのリストを追跡します。DPTの各エントリには、ページをダーティにした最初の変更のLSNである変数 $recLSN$ が含まれます。

Checkpointからログの最後まで順方向スキャン (forward scan) することで、システムは新しいトランザクションをTTに、新しいページをDPTに追加します。このスキャンプロセスにより、システムは最も重要なパラメータである $RedoLSN$ を計算できます。

$$ RedoLSN = min({recLSN mid text{page} in DPT}) $$

$RedoLSN$ の値は、時空間の絶対的な下限 (lower bound) を設定します。LSNが $RedoLSN$ 未満のすべてのレコードは確実に物理的にディスクにフラッシュされているため、その古いWAL部分を再度読み取る必要はありません。グラフ理論的に言えば、このスキャンはシステム状態を表すノード (ページ) の集合の構造を再構築することです。

struct DirtyPageEntry {
    page_id: u32,
    rec_lsn: u64, // The LSN of the first update that dirtied the page
}

struct TransactionEntry {
    txn_id: u32,
    last_lsn: u64,
    status: TransactionStatus, // Active, Committing, Aborted
}

fn analysis_phase(wal_iterator: &mut WalIterator, checkpoint_lsn: u64) -> (HashMap<u32, TransactionEntry>, HashMap<u32, DirtyPageEntry>, u64) {
    let mut transaction_table: HashMap<u32, TransactionEntry> = HashMap::new();
    let mut dirty_page_table: HashMap<u32, DirtyPageEntry> = HashMap::new();
    
    wal_iterator.seek(checkpoint_lsn);
    
    while let Some(record) = wal_iterator.next() {
        match record.record_type {
            RecordType::Update(page_id) => {
                transaction_table.entry(record.txn_id).or_insert(TransactionEntry {
                    txn_id: record.txn_id,
                    last_lsn: record.lsn,
                    status: TransactionStatus::Active,
                }).last_lsn = record.lsn;
                
                // Track the very first LSN that dirtied this page
                dirty_page_table.entry(page_id).or_insert(DirtyPageEntry {
                    page_id,
                    rec_lsn: record.lsn,
                });
            },
            RecordType::Commit => {
                transaction_table.get_mut(&record.txn_id).unwrap().status = TransactionStatus::Committing;
            },
            // ... other record types handling
        }
    }
    
    let redo_lsn = dirty_page_table.values().map(|entry| entry.rec_lsn).min().unwrap_or(u64::MAX);
    (transaction_table, dirty_page_table, redo_lsn)
}

フェーズ2：再実行フェーズ (Redo Phase) とべき等性の保証

ARIESの哲学は「盲目的な歴史の繰り返し (blind history repeating)」に表れています。トランザクションがコミットされたかアボートされたかに関係なく、システムはデータの各状態を正確に再作成します。

システムは開始点 $RedoLSN$ からWALをスキャンします。Updateレコードに遭遇すると、ARIESは厳密な論理テストのセット (ブール代数) を実行し、物理ディスクデバイス上でこの操作を再実行する必要があるかどうかを判断します：

  1. DPTの確認: WALレコードに記載されているページ $P_x$ は、DPT内に存在している必要があります。そうでない場合、このページはクラッシュ前に安全にディスクにフラッシュされています。
  2. recLSNの確認: DPTに保存されているページ $P_x$ の $recLSN$ 値は、$LSN_{record}$ の値以下でなければなりません。
  3. 物理LSN (Physical PageLSN) の確認: 上記2つの条件が満たされた場合、システムはディスクからページ $P_x$ をロードする必要があります。最終的な検証ポイントは、ロードされたばかりのページのメタデータに適用されます：

$$ PageLSN le LSN_{record} $$

$PageLSN$ (そのページのディスクに物理的に刻印されたLSN) が $LSN_{record}$ より小さい場合、この変更はまだディスクに書き込まれていないことを意味します。システムはページにAfter-Imageを適用 (apply) し、直ちに $PageLSN = LSN_{record}$ を更新します。

$PageLSN$ のマーキングは、本質的に論理クロックベクトル (logical clock vector) のように機能し、重複する再実行の問題を徹底的に防ぎます。この特性は Idempotence (べき等性) と呼ばれます。連続的な電源障害のためにRedoフェーズを何回適用しても、データベースの最終状態が破損することはありません。

フェーズ3：元に戻すフェーズ (Undo Phase) とCLRの謎

TT内でコミット済みとしてマークされていないトランザクション (Active Transactions) のすべての残骸をロールバック (rollback) するためのレビューを実行します。前の2つのフェーズとは異なり、Undoは時間を遡ってスキャン (backward traversal) します。

システムはTTから最も高いLSNを選択し、$PrevLSN$ ポインタチェーンをたどって後退します。ARIES設計における数学的な優位性は、Undoプロセスで CLR (Compensation Log Record) レコードが生成されるときに明らかになります。

Before-Imageを適用して操作を元に戻すとき、ARIESは新しい $CLR_i$ レコードをWALに書き込みます。$CLR_i$ には特別なポインタが与えられます：

$$ UndoNextLSN = PrevLSN(U_i) $$

この論理的不変構造 (logical invariant) は、有限収束 (Finite Convergence) を保証します：Undoを試みている間にシステムが2回目、3回目とクラッシュし続けた場合、次の復旧時にシステムは保存されたCLRに遭遇します。すでにUndoされたレコードを再度Undoしようとすることはありません (データ破損の原因となります)。代わりに、$UndoNextLSN$ ポインタを見て過去へと時間を跳躍し、取り消されたアクションをスキップします。

リカバリ状態のサイクルは不可逆な非巡回有向グラフ (irreversible cycle-free graph) となり、無限の元に戻すループに行き詰まることなく、システムが常に完全な状態に向かうことを保証します。

バッファアーキテクチャ、OSとの相互作用、そしてNUMAの悪夢

システムが数百万トランザクションの速度で動作する場合、WALを維持するプロセスは、OSカーネルとの深い相互作用およびストレージパス設計との戦いに移行します。

write() などのPOSIX API呼び出しは、デフォルトでOS Page Cacheにデータをダンプします。カーネルパニックはWALデータをすべて蒸発させてしまいます。fsync() または fdatasync() コマンドは、カーネルにダーティデータバイトをディスクにプッシュさせるために作られました。しかし、fsync() はi-nodeをロックしてメタデータを同期しなければならないため、非常に莫大なコストがかかります。

根本的な解決策は、ODIRECT フラグを使用することです。このフラグはOS Page Cacheシステムを完全に排除し、WAL BufferがBlock IOおよびNVMeサブシステムと直接対話できるようにするDMA (Direct Memory Access) トンネルを開きます。ODIRECTを使用することで、データベースはCPUジッター (CPU jittering) やKernel Flusher Threadsによるランダムなロックを排除します。

ただし、O_DIRECTでは、ブロックサイズ (例：$4KB$) に応じたユーザ空間メモリのメモリ境界アライメント (Memory Boundary Alignment) が必要であり、malloc() の代わりに posix_memalign() を使用する必要があります。

NUMAの壁とPartitioned WAL

さらに、Non-Uniform Memory Access (NUMA) のトポロジは厄介な問題です。マルチソケットCPUアーキテクチャがある場合、隣接するNUMAノードに属するRAMへのアクセスは非常に遅くなります。WAL Bufferを単一の集中型Ring Bufferとして設計すると、Cache Line Bouncing 現象が発生します。散在するプロセッサコアがTailポインタ変数を変更する権利を競い合い、システムにQPI/UPIバス経由でMESIプロトコルを解決させ、スループットを低下させます。

アーキテクチャは、各スレッドグループが独立したログラックを所有する Thread-Local WAL Buffers または Partitioned WAL Buffers をデプロイすることでこの問題を解決します。これらのトラックがしきい値に達すると、中央のFlusherスレッドが、物理I/OをSSDにプッシュする前に、ソート (Sorting) とLSNのマージ (Merge) を行います。断片化されたシステムにおける全体的なLSNの厳密なグローバル性を保証するという一般的な問題は、数学的に激しいトレードオフをもたらします。

ZNS NVMe：ハードウェア/ソフトウェア協調設計の進化

WALの集中的な環境 (小さなクラスタごとにランダムに追記) におけるNANDフラッシュの寿命は、Write Amplification Factor (WAF) 現象のために常に懸念事項です。SSDコントローラーは、切り捨てられた (truncate/trim) 古いWALスペースをクリーンアップするために、コストのかかるガベージコレクションプロセスを実行する必要があります。

Zoned Namespaces (ZNS) NVMeのような新世代のハードウェアプロトコルを使用すると、データベースはストレージデバイスにSequential-Write-Onlyルールの下でゾーン (Zones) を管理するように直接コマンドを出すことができ、ディスクファームウェアのガベージコレクターを完全に排除できます。ZNS上のWALファイルが削除されると (チェックポイントが安全であるため)、システムは単にReset Zoneコマンドを発行します。書き込みサイクル (Write Cycles) の寿命を消耗することなく、理想的な帯域幅を即座に回復します。

1990年代のARIES理論から21世紀のZNS NVMe規格へのシームレスな進化は、最も形式的な定理がシリコンウェハーに直接刻まれるハードウェアとソフトウェアの協調設計 (Co-design) 哲学の最も輝かしい証拠です。

教訓 (Lessons Learned)

極めて信頼性の高いストレージシステムを構築するエンジニアへ：

  1. Lock-free設計は必須: マルチスレッドシステムでは、LSNの割り当てはハードウェアのAtomic Compare-And-Swapオペレーターを介して実行する必要があります。WAL BufferにMutexを使用すると、CPUのスケーラビリティが完全に破壊されます。
  2. チェックサムによる二重確認: ハードウェアを盲目的に信用しないでください。Torn Writesは常に発生します。SSE/AVXハードウェアマイクロ命令を使用したCRC32Cチェックコードの統合は、物理的な破損に対する唯一の盾です。
  3. べき等性は復旧の王様: すべての復旧操作 (Redo/Undo) は、PageLSNおよびCLRメカニズムを介して無限に繰り返すことができる (Idempotent) ように設計する必要があります。復旧中のクラッシュがデータベースを破壊することは許されません。
  4. カーネルのバイパス: サブミリ秒のレイテンシと絶対的な決定性 (determinism) を達成するには、OS Page Cacheを完全にバイパスし、ODIRECTとiouringを中心にI/Oアーキテクチャを設計します。
  5. I/Oのグループ化 (Group Commit): 個々のトランザクションに対してディスクフラッシュ (sync) を絶対に実行しないでください。PCIe帯域幅を最大限に活用し、IOPSコストを最小限に抑えるために、スマートキューを使用して複数の書き込み操作をバッチ処理 (batching) します。
