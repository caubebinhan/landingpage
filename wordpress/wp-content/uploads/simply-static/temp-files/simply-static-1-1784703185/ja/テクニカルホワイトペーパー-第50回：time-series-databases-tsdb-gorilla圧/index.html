テクニカルホワイトペーパー 第50回：Time-Series Databases (TSDB) – Gorilla圧縮アルゴリズムとChunking Memoryアーキテクチャの解剖

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー (Executive Summary)

このホワイトペーパーは、世界をリードする時系列データベース (Time-Series Database – TSDB、例: Prometheus, InfluxDB, Gorilla) がIoTシステムやマイクロサービスから生成される膨大なデータ量をどのように処理するかについて、深く包括的な視点を提供します。Gorilla圧縮アルゴリズムの数学的構造 (Delta-of-DeltaおよびIEEE-754 XOR) と、OSカーネルレベルでのChunkingメモリ管理戦略の分析を通じて、読者は数CPUクロックサイクルでデータを64:1の比率に圧縮する原理を理解し、リアルタイム分散システムに対する貴重な設計の教訓を得ることができます。

はじめに：時系列データの不変の特性

クラウドコンピューティングとモノのインターネット (IoT) の時代において、機械は常に会話をしています。数十億のセンサー、サーバー、コンテナが継続的に状態信号 (telemetry data) を発信しています。これらのデータは時系列データ (Time-Series Data) と呼ばれます。

一般的なリレーショナルデータベース管理システム (RDBMS) やNoSQLのデータテーブルとは異なり、TSDBは非常に厳格で特異な物理的特性を持っています。

  1. 極端に多い書き込みトラフィック (Write-Heavy): 毎秒数百万回の書き込み操作 (millions of ops/sec) に達する可能性があります。
  2. 時間的な単調性 (Time-Monotonic): データポイントは常に増加するタイムスタンプと共に到着します。過去を更新 (UPDATE) することはなく、ほとんどが追記 (APPEND) のみです。
  3. シンプルなデータ構造: 1つのデータポイントは通常、タイムスタンプ (Timestamp – 64-bit Int) と測定値 (Metric – 64-bit Float)、それにいくつかの識別子文字列 (Tags) のみで構成されます。

この単調さこそが、ストレージにおける巨大な課題であると同時に、システムエンジニアがマイクロチップレベルで最適化の魔法を生み出すための絶好の機会でもあります。

コアとなる問題：IoTデータの津波とストレージのボトルネック

1 帯域幅、ディスクI/O、そしてRDBMSの破綻

100,000台のサーバーを持つデータセンターの監視システムを想像してみてください。各サーバーは毎秒100個のパラメータ (CPU, RAM, Disk I/Oなど) を発信します。 合計: $100,000 times 100 = 10,000,000$ データポイント/秒。 各データポイント (8 bytes Timestamp + 8 bytes Float Valueの16 bytes)。 生の取り込みレート (Raw Ingestion Rate): $160 text{ MB/秒} approx 13.8 text{ TB/日}$。

もしこのデータ量を INSERT コマンドを通じてPostgreSQLやMySQLに押し込んだ場合、B-Treeインデックスシステムはロックの競合 (lock contention) により即座に崩壊します。Cassandraのようなシステムを使用したとしても、1年間のストレージライフサイクル (約 $sim 5$ Petabytesに相当) のために毎日13.8 TBを保存するNVMeハードドライブの購入費用は、財政的な大惨事になります。

2 辞書圧縮アルゴリズムの無力さ

その圧力に直面したとき、エンジニアの自然な反応はデータ圧縮を使用することです。しかし、LZ77、Snappy、Gzip、Zstandardなどの業界標準の圧縮アルゴリズムは、「辞書検索」 (Dictionary-based Compression) の原理に基づいています。それらは繰り返されるバイト列を見つけようとします。しかし不幸なことに、タイムスタンプは連続して増加する数値であり (決して繰り返されない)、浮動小数点数 (Float) は末尾のビットが継続的に変化します。SnappyやZstdは全く役に立たず、圧縮率が低いにもかかわらず大量のCPUを消費します。

提起される問題：辞書を使用せず、分岐 (branching) なしでCPUマイクロチップ上で直接動作し、整数や浮動小数点数のシーケンスに徹底的に最適化された圧縮アルゴリズムが必要です。 2015年にMeta/Facebookによって導入されたGorillaアルゴリズムの誕生こそが、最も完璧な解決策でした。

アルゴリズムによる解決策：Gorilla圧縮アルゴリズム (数学とビット論理)

Gorillaは問題をTimestampsの圧縮とMetricsの圧縮の2つのストリームに分割します。鍵は情報理論にあります：絶対値は継続的に変化しますが、それらの間の差分 (Delta) は極めて安定しています。

1 Delta-of-Deltaによるタイムスタンプの圧縮

データ収集システムは通常、固定周期 (例: 10秒ごと) で動作します。各タイムスタンプに64ビットを保存する代わりに：

$T = [1000, 1010, 1020, 1030]$

1次誤差 (Delta – $D_n = T_n – T_{n-1}$) を計算します：

$D = [10, 10, 10]$

ネットワークの遅延 (jitter) のため、実際のDeltaは次のようになる可能性があります：

$D = [10, 12, 9, 11]$

アルゴリズムはさらに2次誤差 (Delta-of-Delta – $D^{(2)}_n = D_n – D_{n-1}$) を計算します：

$D^{(2)} = [2, -3, 2]$

Delta-of-Delta (DoD) の背後にある数学：安定したシステムの96%のケースにおいて、DoD = 0です。Gorillaは可変長でエンコードします (ハフマン符号化に似ています)：

  • DoD = 0 の場合: 正確に 1ビット 0 を記録します (64ビットから1ビットへの圧縮！)。
  • DoD $in [-63, 64]$ の場合: 10 + 7ビットのデータを記録 (合計9ビット)。
  • DoD $in [-255, 256]$ の場合: 110 + 9ビットのデータを記録 (合計12ビット)。
  • より広い範囲についても同様に続きます。

2 IEEE 754 XOR論理による測定値の圧縮

64ビット浮動小数点数 (IEEE 754) は、圧縮アルゴリズムにとっての悪夢です。それは1ビットの符号、11ビットの指数部、52ビットの仮数部で構成されています。小さな変化 (例えば2.0から2.01) でも、ビット構造が完全に反転します。 しかし、Gorillaは気付きました：センサーからの温度値 (例：摂氏37.5度) が急上昇することはめったにありません。ビットレベルでは、隣接する2つの数値は多くの場合、符号、指数部、および上位の大量の仮数部ビットを共有しています。

Gorillaは隣接する2つの値を取得し、減算を行うのではなく、64ビット上で直接 XOR ($oplus$) 論理演算を使用します：

$$X_n = V_n oplus V_{n-1}$$

結果 $X_n$ は、先頭に多くの 0 (Leading Zeros – LZ) と末尾に 0 (Trailing Zeros – TZ) を含む64ビットのシーケンスになります。中央にあるのは「異なるビットのブロック」 (Meaningful Bits) です。

  1. $X_n = 0$ (値が変化しない) の場合：1ビット 0 のみを記録します。
  2. $X_n neq 0$ の場合、システムはLZとTZを前の値と比較します。Meaningful Bitsブロックが前のブロックの領域に収まる場合、システムは制御ビット 10 を記録し、それらの異なるビットのみを記録して、LZ/TZ構造を再利用します。
  3. 構造が変化する場合、11 + 新しいLZの長さを含む5ビット + 異なるブロックの長さを含む6ビット + 異なるブロック自体を記録します。

3 ハードウェアレベルのマイクロアーキテクチャ

Leading Zerosのカウントには、高価な while ループは必要ありません。C++/Rust/Goコンパイラのおかげで、このアルゴリズムはx86_64 CPUのハードウェア命令セットである LZCNT および TZCNT を直接トリガーします。0を数える操作は1マシンサイクル (1 clock cycle) で完了します。分岐 (branching) が最小限に抑えられ、CPUのハードウェア分岐予測器 (Hardware Branch Predictor) がスムーズに機能します。

システムマイクロアーキテクチャ：メモリページング管理とChunking

圧縮データは連続したビット列 (bit-stream) を生成します。しかし、「昨日」のデータに対してクエリを実行する必要がある場合、システムは年初から順番に解凍することはできません。ここで Chunking Architecture の出番です。

1 Memory Chunkingとは？

無限の時系列データは「塊」 (Chunks) に切り分けられます。Chunkは通常、期間 (例：2時間) またはバイト量 (例：4 KB) を表します。 どの時点においても、各Time-Seriesは1つの Active Chunk のみを持ちます (RAM上にあり、Append-onlyスタイルで継続的にデータを受信します)。

2 CPUコアのトレードオフ問題 (False Sharing vs TLB Misses)

Chunkサイズが小さすぎる (< 4KB) 設計の場合： RAMの物理メモリページ (Memory Page 4KB) には、複数の異なるSeriesの多くのChunksが含まれます。異なるCPUコアが同じページ上で隣接する異なるSeriesを更新すると、「False Sharing」 (偽共有) 現象が引き起こされます。CPUのMESIプロトコルは継続的に互いのL1/L2キャッシュを無効化するメッセージを発行し、速度が急落します。

Chunkが大きすぎる (例: 20MB) 設計の場合： RAMが断片化します。仮想ページマネージャー (Translation Lookaside Buffer – TLB) は、物理アドレスを追跡しすぎるために使い果たされ、継続的なTLB Missesが発生します。 一般的な解決策は、jemallocなどのRAM割り当てライブラリによって管理されるHuge Pages (2MB) を使用してChunkを「大きなメモリ領域」 (Memory Arenas) にプールし、キャッシュヒット率を常に最大に保つことです。

オペレーティングシステムとの収束：OS Page CacheとMmap()

Active Chunkが封印 (Sealing) され、Immutable Chunk (不変) になると、TSDBはLinuxオペレーティングシステムのカーネルに対する神聖な「委任」トリックを開始します。

1 RAMとSSDの境界を曖昧にする

TSDBはImmutable ChunkをBlock StorageとしてNVMe SSDにプッシュします。ファイルの読み取りを自己管理する代わりに (read() コマンドとKernel Space / User Space間のコンテキストスイッチにコストがかかる)、TSDBは mmap() システムコールを使用します。 mmap() はSSD上のブロックをプロセスの仮想アドレス空間 (Virtual Address Space) にフラットにマッピングします。

  • ユーザーが先月のデータチャートをクエリすると、TSDBはメモリポインタ (pointer) で直接読み取り操作を行うだけです。
  • ハードウェア (Memory Management Unit) はRAM上にデータがないことを検出し、「Page Fault」エラーを発生させます。
  • Linuxカーネルは即座にNVMeに行き、その4KBブロックをRAMに持ち込みます。さらに、後続のブロックもL3キャッシュに自動的に予測 (Read-ahead heuristics) して引き込みます。

2 Zero-copy Deserialization

ディスク上のChunkのバイト構造は、RAM上にあったときのGorillaビットストリームの構造と100%同一であるため、TSDBは「構造のデコード」 (Deserialization) に労力を費やすことはありません。CPUはRAMから圧縮ビットを直接読み取り (mmapによってディスクからプルされたばかり)、XOR/LZCNT関数に渡して、グラフを即座に表示します。

分散システム設計の哲学に関する教訓 (Lessons Learned)

GorillaとChunkingの魔法から、私たちは核となる教訓を引き出します。

  1. データドメインの深い理解 (Domain-specific Optimization): 万能鍵となるアルゴリズムはありません。Zstandardはテキスト圧縮において無敵かもしれませんが、数学的な時系列データには惨敗します。Gorillaは証明しています：データの物理的性質、確率、および分布 (時間の単調性、センサーの相関関係) を正確に理解すれば、従来の情報理論を打ち破るシンプルなアルゴリズム (Delta-of-Delta, XOR) を設計できるということです。
  2. 不変性 (Immutability) は拡張の鍵: 不変のデータブロック (Immutable Chunks) を持つシステム設計は、ロックメカニズム (Locks) を排除し、デッドロックを回避し、システムがOS Page Cacheを最大限に活用できるようにするための基盤です。Chunkがロックされると、キャッシュがダーティ (Dirty Pages) になることはなくなり、メモリが不足したときにカーネルがいつでもRAM上のデータを破棄できるようになります。
  3. ハードウェアとソフトウェアは共進化しなければならない: 現代のデータベースの設計は、単にロジックコードを書くことではありません。アーキテクトはマイクロアーキテクチャの言語 (Cache Lines (64 bytes), L1/L2 Cache Hit/Miss, TLB, SIMD, Hardware Instructions (LZCNT)) でコミュニケーションをとる必要があります。良いコードを書くだけでは十分ではなく、シリコンの物理法則に適合するコードを書くことこそが究極の目標です。

結論

超高性能なTime-Series Databaseの構築は、ネットワーク帯域幅、限られたCPUリソース、および物理的ストレージ制限の間の巧妙な調整プロセスです。Gorilla圧縮アルゴリズムとChunkingアーキテクチャは、最小限のブール数学、CPUマイクロアーキテクチャの理解、およびOSページングメカニズムを組み合わせて、システム監視データベース (Observability) の標準を形作りました。この理解は、Web 3.0時代のデータインフラストラクチャやグローバルなIoTデバイスネットワークを設計し、マスターしたいバックエンドエンジニア (Backend Engineers) にとっての出発点となります。
