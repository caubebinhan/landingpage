スプリットブレイン(Split-Brain)問題 – Fencing Tokens、Quorums、そして一貫性を守る戦い (詳細な技術ホワイトペーパー)

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー (Executive Summary)

分散システム運用エンジニアリングにおいて、「スプリットブレイン (Split-Brain)」ほど悪夢のような現象はありません。この現象は、ネットワークが分断され、サーバークラスターが独立した島に分割されたときに発生します。帯域外 (out-of-band) の通信チャネルがない場合、各パーティションは自分が唯一の生存者であると誤認し、自動的に新しいリーダー (Leader) を選出します。その結果、2つの「脳」が並行して機能し、交差する分散ロック (Distributed Locks) を発行してデータの無秩序な上書きを引き起こし、コアストレージシステムに回復不可能な損害を与えます。

本技術ホワイトペーパーでは、Quorum (集合論) と CAP定理に照らしてスプリットブレイン現象を徹底的に解剖します。さらに重要な点として、オペレーティングシステムのマイクロアーキテクチャを深く掘り下げ、クロックスキュー (Clock Skew) やメモリのガベージコレクションの一時停止 (GC Pauses) といった物理的な落とし穴が、最良のロック管理サービス (ZooKeeper、Redis Redlock など) であってもどのように欺き得るかを分析します。

最後に、究極の解決策である Fencing Tokens (状態フェンシングトークン) を提示します。これは、ストレージエンジン (Storage Engine) のソースコードに深く埋め込まれた侵入不可能なデジタル防御メカニズムであり、一貫性の保護を脆弱なネットワークの取り組みではなく、数学的定理に変えます。

問題提起 (Problem Statement): 現代のシステムアーキテクチャでは、オペレーティングシステムとコンピュータネットワークが信頼できる「リアルタイム」標準を提供できると誤って想定されることがよくあります。分散ロック (Lease/TTL) を使用する際、有効期限が切れればロックは安全であると信じています。しかし、クライアントの実行スレッドが (OS Page Fault や GC Pause によって) 凍結された場合、物理的な時間の契約は破棄されます。「目を覚ました」クライアント (Zombie Client) は、期限切れのロックを保持したまま、まだ自分に権限があると思い込み、他人の有効なデータを上書きしてしまいます。ストレージシステムは、このような内部からの攻撃からどのように身を守ることができるのでしょうか？

学んだ教訓と知識 (Lessons Learned):

  1. 物理時計の崩壊: 合意 (Consensus) を得るための唯一の真実の情報源として、物理的な Clock/TTL を絶対に使用しないでください。論理時計 (Logical Clocks) を使用する必要があります。
  2. Zombie Clients は見えない暗殺者: Fencing Tokens のないシステムは、鍵はかかっているが中に警備員がいない家のようなものです。ZooKeeper のロックはネットワークアクセス権を決定するだけであり、Storage Engine がアクセス許可トークンをチェックする最後の関所である必要があります。
  3. Quorum は数学的交換シールド: Paxos/Raft アルゴリズムは、分散セット上で2つの Leader が同時に書き込み操作を実行できないことを代数幾何学的に証明するために、過半数 ($R + W > N$) を使用します。

理論的背景: ネットワーク分断と CAP / PACELC 定理

ネットワーク分断 (Network Partition) は、接続されたネットワークグラフ $G = (V, E)$ が非接続の部分集合 $V_1, V_2, dots, V_k$ に分断され、それら間の通信帯域幅が $to 0$ に近づき、遅延 $RTT to infty$ になることとして形式的に定義されます。

この分断は、海底光ファイバーケーブルの切断によるものだけでなく、多くの場合、スイッチの ASIC 内の Buffer Bloat に起因し、大規模なパケットドロップを引き起こし、TCP 輻輳制御 (Congestion Control) を麻痺させ、Heartbeat パケットを失敗させます。CAP 定理によれば、システムは窮地に追い込まれます：一貫性 (Consistency) を犠牲にするか、可用性 (Availability) を犠牲にするかのトレードオフです。

Availability を選択した場合、各パーティション $V_k$ はクライアントへのサービス提供を継続し、並行した履歴の分岐を作成します。これが Split-Brain の根本的な原因です。財務データや B-Tree インデックス構造は破壊され、二度と自動的にマージすることはできなくなります。

1 Split-Brain の天敵: Quorum 合意アルゴリズム

2つの脳を防ぐために、Apache ZooKeeper、HashiCorp Consul は Paxos / Raft を使用します。 それらの数学的バックボーンは Quorum (定足数) システムです。 Quorum 集合 $S = {Q_1, Q_2, dots, Q_m}$ は、交差点が常に存在することを要求します： $forall Q_i, Q_j in S, Q_i cap Q_j neq emptyset$.

  • 総ノード数 ($N$) に対する書き込み集合 ($Q_w$) と読み取り集合 ($Q_r$) のサイズ: $Q_r + Q_w > N$
  • Split-Brain を防ぐために: $Q_w > frac{N}{2}$

例: クラスターに 5 つのノードがあるとします。3 ノードと 2 ノードに分断されます。 3 ノードのクラスターのみが Leader を選出し続けるのに十分な投票数 ($> 2.5$) を集めることができます。少数の 2 ノードクラスターは継続的にタイムアウトし、Follower に降格します。

マイクロアーキテクチャの謎: 分散ロックと時間の幻想

複数のクライアントが同じファイルを同時に編集するのを防ぐために、例えば 10 秒という期限 (Lease TTL) を設定した分散ロック (Distributed Lock) を付与します。クライアントに障害が発生した場合、10 秒後にロックは自動的に解放されます。 しかし、災害のシナリオは常にオペレーティングシステム層に潜んでいます。

1 “Stop-The-World” の悪夢

ランタイム環境 (Runtime Environment) の途方もない複雑さは、時間の連続性を完全に破壊する可能性があります。

  • Garbage Collection (GC): 仮想マシン (JVM、Go Runtime) は “Stop-The-World” サイクルをトリガーする可能性があります。Heap メモリのガベージをクリーンアップするために、アプリケーションの実行スレッド全体が数十秒間凍結されます。
  • OS Page Fault: RAM からのデータ読み取りエラー。MMU (Memory Management Unit) はハードウェア割り込みを発生させ、遅いディスクからデータを取得するために CPU にアプリケーションを停止させます。
  • Hypervisor CPU Steal Time: クラウド環境では、仮想サーバーは Hypervisor によって物理 CPU を突然奪われる可能性があります。

Client A が Userspace で「無意識に凍結」されている間、Lock Service の時間は経過し続けます。10 秒の契約が期限切れになると、Lock Service はロックを解放し、それを Client B に付与します。 数秒後、Client A は目を覚まします！Kernel は時間のブラックホールに関するアラート (Interrupt) を送信しないため、Client A は内部の Boolean フラグをチェックし、自分がまだロックを保持していると判断します。

絶対防御アーキテクチャ: Fencing Tokens (フェンシングトークン)

時間の幻想の崩壊に直面して数学的な正確さを証明するために、分散ストレージ業界は Fencing Tokens を発明しました。

  • 非線形の物理時計を放棄します。Logical Clocks を使用します：単調増加する整数のシーケンス ($T_i > T_{i-1}$)。
  • Lock Service は、ロックを付与するたびに自動的に新しい Token を生成し ($33, 34, 35dots$)、それをクライアントに添付します。
  • Storage Engine の究極の役割: Storage Layer はプロアクティブな検閲バリアになります。これまでに遭遇した最大のトークンであるグローバルレジスタ $T_{max}$ を維持します。
  • Atomic Validation ルール: クライアントがパケット + Token を持って書き込みに来たとき、Storage は Boolean 代数をチェックします：
  • $T_{req} ge T_{max}$ の場合: 書き込みを受け入れ、$T_{max} = T_{req}$ を更新します。
  • $T_{req} < T_{max}$ の場合: 即座に拒否します (409 Conflict エラーを返す)。なぜなら、この Token は古すぎるからです。

1 マイクロチップレベルでの “Zombie Client” の排除

上記のタイムブラックホールのシナリオを Fencing Token を使って繰り返してみましょう：

  1. Client A がロックを取得し、Token 33 を発行されます。JVM GC Pause がトリガーされます。
  2. ロックの有効期限が切れ、Client B がロックを取得し、Token 34 を発行されます。
  3. Client B は Token 34 を Storage に送信します。Storage は $34 ge 0$ を確認し、書き込みを許可します。$T_{max} = 34$ に更新されます。
  4. Client A が目を覚まし、Zombie Client となり、上書きしようと古い Token 33 を Storage に持ち込みます。
  5. Storage ALU はチェックします: $33 < 34$。条件は False です。ハードウェア割り込み命令は即座にパケットを破棄し、Storage への扉を閉じます。

Split-Brain は、磁気ハードディスクがビット状態を変更する暇もなく、入り口で直ちに殺されます。

Rust と Compare-And-Swap (CAS) による低レベルの最適化

この Storage Fencer の障壁は、数千万の IOPS に耐えなければなりません。メモリをロックするために Mutex を使用することはできません (CPU がパンクします)。エンジニアは、アセンブリ言語の Lock-free Atomic Operations、具体的には CAS (Compare-And-Swap) 命令を使用する必要があります。

以下は、オーバーヘッドを引き起こすことなく $T_{max}$ を保護するために、インターキャッシュプロトコル MESI と Hardware Bus Locking を最大限に活用した、Rust 言語による Kernel レベルでの Storage Fencer の説明例です。

use std::sync::atomic::{AtomicU64, Ordering};

pub struct HighThroughputStorageFencer {
    // AtomicU64 は 64-bit マイクロアーキテクチャ上で絶対的なメモリ安全性を保証します
    current_max_fencing_token: AtomicU64,
}

pub enum FencingViolationError {
    StaleZombieToken { provided_token: u64, current_system_max: u64 },
}

impl HighThroughputStorageFencer {
    pub fn new() -> Self {
        Self { current_max_fencing_token: AtomicU64::new(0) }
    }

    pub fn validate_and_atomically_update(&self, incoming_fencing_token: u64) -> Result<(), FencingViolationError> {
        // Acquire 命令は CPU Reordering を防ぎます
        let mut local_current_view = self.current_max_fencing_token.load(Ordering::Acquire);
        
        loop {
            // ZOMBIE CLIENT の迎撃
            if incoming_fencing_token < local_current_view {
                return Err(FencingViolationError::StaleZombieToken {
                    provided_token: incoming_fencing_token,
                    current_system_max: local_current_view,
                });
            }
            
            // ロックフリー CAS を試行。別のコアが介入した場合、直ちにリトライします
            match self.current_max_fencing_token.compare_exchange_weak(
                local_current_view, 
                incoming_fencing_token, 
                Ordering::Release, 
                Ordering::Relaxed
            ) {
                Ok(_) => return Ok(()),
                Err(new_updated_val_from_ram) => local_current_view = new_updated_val_from_ram,
            }
        }
    }
}

低レベルの compare_exchange_weak ループを使用することで、Fencer Engine はゼロオーバーヘッドガベージの $O(1)$ に漸近する運用コストをほぼ達成します。

まとめ

分散システムアーキテクチャは、緩い API ソフトウェアを結合する技術ではありません。それは、数学哲学に直結した厳密な科学です。多数を閉じ込めるために Quorum 理論を定量化し、時空を超える Zombie Clients の触手を断ち切るためにアトミック化された Fencing Token を適用することで、最も堅牢な Storage Engine を設計することができます。半導体マイクロチップの分子粒子レベルで発生する熱力学的カオスの状態を完全に制御するのです。
