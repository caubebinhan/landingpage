Adaptive Radix Trees (ART): In-Memory DBのための究極のデータ構造

July 22, 2026 · admin · 4 分で読めます

Executive Summary (概要 / Overview)

ビッグデータ時代におけるインメモリデータベースシステム (in-memory database systems – IMDBs) の爆発的な普及は、インデックスデータ構造 (indexing data structures) のコア設計に対して非常に厳しい要求を突きつけています。IMDBシステムにおいて、ボトルネックは従来のハードドライブのシーク時間 (seek time) や回転遅延 (rotational latency) ではなくなり、メモリ帯域幅 (memory bandwidth) とCPUキャッシュアクセス遅延 (CPU cache latency) に完全に移行しています。

このテクニカルホワイトペーパー (Whitepaper) では、現代のハードウェアのパワーを最大限に活用するために特別に設計された、非常に学術的かつ実践的な画期的なデータ構造であるAdaptive Radix Tree (ART)に関する詳細な分析を提示します。B-Treeの範囲クエリ (range queries) の特性、Hash Tableの漸近的定数時間計算量、およびCPUマイクロアーキテクチャに応じた微調整機能 (cache-line alignment, SIMD instructions) を組み合わせることで、ARTは実行時間パフォーマンスとメモリ領域の最適化の完璧なバランスを実現します。ARTは、動的ノードスケーリング (dynamic node scaling) メカニズムによって標準のRadix Treeの容量の無駄を徹底的に解決するだけでなく、キャッシュミス (cache miss) やTLBミス (TLB miss) の数を最小限に抑え、超高速トランザクションおよび分析処理システム (OLTP/OLAP) にとって完璧な「心臓部」となります。

Core Problem Statement (中核となる課題)

インメモリ時代における古いパラダイムの崩壊

  1. B+-Treeとそのバリアントの限界:

    2. ディスクベースのデータベース (disk-based databases) の時代に生まれたB+-Treeは、ディスクブロックアクセス (disk block access) に最適化されています。メインメモリにおいて、大きなサイズのB+-Treeノード (例: 4KB – 16KB) は数十のキャッシュライン (cache lines – x86-64プロセッサでは通常64 bytes) を占有します。内部B+-Treeノードのトラバーサルには、$mathcal{O}(log_2(mathcal{B}))$の計算量を持つ二分探索 (binary search) アルゴリズムが必要となり、無数の条件分岐命令 (branching) が発生します。これらの命令はしばしばCPUの分岐予測パイプライン (branch prediction penalty) を破壊し、誤予測ごとに数十クロックサイクルの遅延を生み出します。

  1. 範囲クエリに対するHash Tableの無力さ:

    2. Hash Tableは$mathcal{O}(1)$のアクセス時間計算量を提供しますが、データをメモリ空間全体にランダムに分散させます。この特性により、分析やレポート処理のバックボーンである範囲スキャンクエリ (range scan queries) を実行する機能が完全に破壊されます。さらに、ロードファクター (load factor) が高い場合、ハッシュ衝突 (hash collisions) によってパフォーマンスが著しく低下します。

  1. Radix Tree (Trie) の空間と時間のトレードオフのパラドックス:

    2. 標準のRadix TreeはHash Tableの範囲クエリの弱点を克服します。ツリーの最大深さは数式 $mathcal{D} = lceil frac{mathcal{K}}{s} rceil$ で定義されます。ここで、$mathcal{K}$はビット単位のキー長、 $s$ (span) は各ステップの幅です。しかし、Radix Treeは過酷なトレードオフに直面します:

  • $s$が小さい場合 (例 $s=1$ bit): グラフツリーは非常に深くなります。64ビット長のキーをトラバースするには、最大64回のポインタジャンプ (pointer chasing) が必要です。各ジャンプには、致命的なキャッシュミスやTLBミスのリスクが伴います。
  • $s$が大きい場合 (例 $s=8$ bits): グラフは大幅に浅くなりますが (64ビットキーでわずか8レベル)、メモリ容量は恐ろしいほど膨れ上がります。各ノードは静的に $2^s = 256$ 個のポインタの配列を割り当てる必要があります。スパースなインデックス空間 (sparse data) では、この配列の99%以上が空になり、物理メモリが極端に無駄になります。

このマイクロアーキテクチャのパラドックスを根本的に解決するために、Adaptive Radix Tree (ART) 構造が誕生しました。ARTは、多次元インデックス空間におけるキーの分布密度が決して均一ではないことを認識しています。静的割り当てのノードサイズを固定することはリソースの無駄です。ARTは、実行時 (runtime) の実際のデータ密度に基づいて自動調整されるハイブリッドグラフ構造を形成します。

Deep Technical Knowledge / Internals (詳細な技術知識 / 内部構造)

形態学的エコシステム (Morphological Ecosystem) のマイクロアーキテクチャ

ARTは単一の均一なノードの超構造を維持する代わりに、非線形の容量を持つ4つのノードタイプ (node types): Node4、Node16、Node48、および Node256 を含むエコシステムを定義します。Trieネットワークにより、ノードは実際の小枝の数に基づいて自動的に透過的な昇格 (promotion) または降格 (demotion) を行うことができます。

各ノードタイプの構造の内訳:

  • Node4 (40 bytes): 最もスパースな交点を表します。最大4つのサブキー (各1バイト) と4つのメモリポインタ (8 bytes/ポインタ) を並列に格納し、さらにメタデータ (metadata) を追加します。合計40 bytesはキャッシュライン (64 bytes cache line) の境界内に完全に収まります。プロセッサはNode4にアクセスする際、ストールメモリアクセスに直面することはありません。
  • Node16 (144 bytes): 5番目の要素がNode4に挿入されると、Node16に昇格します。連続した16個のサブキーと16個のリンクポインタを含みます。Node16の画期的な点は、並列実行 (SIMD) のためにCPUのXMM/YMMレジスタに直接マッピングできる機能です。
  • Node48 (600 bytes): 16ブランチを超えると、並列配列のシフトはコストが高くなります。Node48は間接モデル (indirection) を使用します。マッピング (index map) として機能する256 bytesの配列を所有します。この配列のインデックスはバイト値 (0-255) に対応し、配列に格納された値は、48個のポインタの静的配列における実際のポインタの位置を指します。これにより、高コストなポインタ配列検索操作が排除されます。
  • Node256 (2048 bytes): 物理的限界の頂点です。256個の静的ポインタの配列です。この配列へのアクセスは無条件のオフセット計算 $mathcal{O}(1)$ です。

プロセッサ命令セットの最適化 (CPU SIMD Instructions & Branchless Logic)

分岐予測ペナルティ (branch prediction penalty) を排除するために、ARTは標準のSSE2/AVX2 SIMD (Single Instruction, Multiple Data) 命令セットによるブランチレスプログラミング (Branchless Programming) 技術を適用します。特にNode16で有効です。

// ARTアルゴリズムのマイクロアーキテクチャのコア構造 (標準C++20表現)
#include <immintrin.h>
#include <cstdint>

struct alignas(64) ARTNode {
    uint8_t type_descriptor;
    uint32_t active_children_count;
    uint32_t compression_prefix_length;
    uint8_t compressed_prefix[8]; // 小さなプレフィックスの制限
};

struct alignas(64) Node16 : public ARTNode {
    uint8_t vector_keys[16];      // キーの連続バイト配列
    ARTNode* memory_pointers[16]; // 並列メモリポインタ配列
};

// ... Node4, Node48, Node256 の定義 ...

ARTNode* ART_Microkernel_Lookup(ARTNode* current_node, const uint8_t* query_key, uint32_t max_key_len, uint32_t current_depth) {
    while (current_node != nullptr) {
        // [1] パス圧縮 (Path Compression) の解決
        if (current_node->compression_prefix_length > 0) {
            uint32_t prefix_matched_bytes = Check_Optimized_Prefix(current_node, query_key, current_depth);
            if (prefix_matched_bytes != current_node->compression_prefix_length) return nullptr;
            current_depth += current_node->compression_prefix_length;
        }
        if (current_depth == max_key_len) return current_node; 
        
        uint8_t active_byte = query_key[current_depth];
        
        // Type Descriptor に基づく分岐ロジック
        switch (current_node->type_descriptor) {
            case NODE16_TYPE: {
                Node16* specialized_node = static_cast<Node16*>(current_node);
                
                // 検索対象のバイトをXMMレジスタの全16バイトにロード (broadcast)
                __m128i target_byte_vector = _mm_set1_epi8(active_byte);
                
                // メモリから16キーの配列をレジスタにロード
                __m128i stored_keys_vector = _mm_loadu_si128(reinterpret_cast<const __m128i*>(specialized_node->vector_keys));
                
                // 1クロックサイクルで同時に等価比較を実行 (SIMD Eq)
                __m128i comparison_result = _mm_cmpeq_epi8(target_byte_vector, stored_keys_vector);
                
                // 比較結果を16ビットのビットマスクとして抽出
                unsigned matched_bitmask = _mm_movemask_epi8(comparison_result) & ((1 << specialized_node->active_children_count) - 1);
                
                if (matched_bitmask) {
                    // __builtin_ctz は末尾のゼロの数を数え、子ノードのインデックスを瞬時に見つけ出します
                    current_node = specialized_node->memory_pointers[__builtin_ctz(matched_bitmask)];
                } else {
                    return nullptr; // ミス、見つからない
                }
                break;
            }
            // Node4, Node48, Node256サブクラスのマイクロ分岐ロジック (省略)
        }
        current_depth++;
    }
    return nullptr;
}

_mm_cmpeq_epi8 と __builtin_ctz を組み合わせることで、16個のブランチ内での子ブランチの検索操作は、ループや if ステートメントを使用せずに実行されます。これにより、プロセッサのIPC (Instructions Per Cycle) が常に上限にとどまることが保証されます。

パス圧縮アルゴリズム (Molecular Path Compression) と遅延展開 (Polymorphic Lazy Expansion)

ツリーの深さを圧迫する「単一子リンクチェーン」 (single-child link chains) 現象を克服するために、ARTは2つの調整された技術を使用します:

  1. Lazy Expansion (遅延展開): 単一の値のみを含むリーフ (leaf nodes) は、ブランチノードに完全に展開されません。データへの直接ポインタとして保存されます。現在のリーフノードで衝突 (hash collision anomaly) が発生した場合にのみ、グラフは分岐を進行します。
  2. Path Compression (パス圧縮): 連続した構造サブクラスが単一の子ポインタのみを持つ場合、それらは次のノードに格納される「プレフィックス」 (prefix) に「圧縮」されます。「A-B-C-D-E」シーケンスを検証するためだけに5つのノードをジャンプする代わりに、ARTは「ABCDE」を下位ノードの compressed_prefix 配列に圧縮し、ブロックメモリマッチコマンド (例: 超高速の memcmp) を使用して文字列を比較します。

OSレベルの対話、NUMA、およびメモリ管理 (OS-Level Allocations)

POSIXオペレーティングシステムの標準的な malloc ライブラリに依存すると、仮想メモリの断片化 (Virtual Memory Fragmentation) が発生します。これは、ハードウェアレベルのプリフェッチャ (Hardware Prefetcher) の効率を低下させるだけでなく、TLBスラッシング (TLB Thrashing) も引き起こします。

ARTには、カスタムの Hierarchical Slab Allocator が急務です:

  • 全仮想アドレス空間 (Virtual Address Space) は、巨大なモノリシックメモリチャンク (Monolithic memory chunks) に断片化されます。
  • これらの領域は特化されています: 領域Aは Node4 の割り当て専用、領域Bは Node16 専用など。
  • NUMA-Awareness: マルチソケットサーバーシステムにおいて、アロケータは幾何学的空間の物理的認識を持ちます。新しく作成されたノードのメモリアドレスを操作中のプロセッサ (ソケット) のシリコン層にバインドし、クロスドメインバス (QPI/UPI cross-domain bus overhead) のデータ転送遅延を排除します。

マルチコア並行性制御 (Concurrency Control: OLC & ROWEX)

超並列処理 (Massively Parallel Processing) システムでは、アクセススレッドを保護するロックメカニズム (heavyweight latches / read-write locks) が、システムバスを過負荷にする「キャッシュラインバウンシング」 (Cache Line Bouncing) の大惨事を引き起こします。

ARTは、Optimistic Lock Coupling (OLC) 哲学と ROWEX (Read-Optimized Write EXclusive) プロトコルを適用します:

  • 各ノード構造は、アトミックなバージョンカウンター (Version Counter / Lock Word) を維持します。
  • Read Transactions (読み取りトランザクション): Acquire Lock を決して行いません。読み取りスレッドはバージョン値 $V_{pre}$ を記録し、データを読み取り、コンパイラ/CPUのリオーダリングを防ぐためにメモリバリア命令 mfence を実行し、その後 $V_{post}$ を再確認します。$V_{pre} == V_{post}$ であり、ロックビットがアクティブでない場合、読み取りセッションは完全に安全に成功します。不一致がある場合、読み取りプロセスは自己中止して再試行 (Retry / Spin-loop) します。
  • Write Transactions (書き込みトランザクション): Out-of-place Atomic Replace 技術を使用します。進行中の読み取りスレッドに損傷を与える可能性のある、所定の位置でのメモリの直接変更の代わりに、エンジンはノードを新しいメモリ領域にコピー/昇格させます。コピー先での更新が完了すると、書き込みスレッドは超高速のアトミック命令 lock cmpxchg (Compare-And-Swap) を実行するだけで親ノードからのポインタを交換します。この書き込み操作は、古いバージョンを読み取っているスレッドからは完全に見えません。

Practical Applications & Case Studies (実用的なアプリケーションとケーススタディ)

In-Memory データベース管理システム (Hyper-Scale OLTP/OLAP)

HyPer Database (TUM/Tableau) や SAP HANA などのトップデータベースエンジンのコア構造は、ARTに似た基盤となる基数木 (radix tree) をプライマリインデックス (primary index) およびディクショナリエンコーディング (Dictionary Encoding) のストレージとして適用しています。HyPerでは、ARTはコアスレッド (core thread) ごとに毎秒5000万以上のポイントクエリ (point queries) の処理速度を実証し、B+-Treeのバリアント (Masstree や Bw-Treeなど) を20%から150%の卓越した差で完全に打ち負かしました。

ネットワークIPルーティング (Network IP Routing / Prefix Matching)

通信インフラ分野では、IPアドレスの検索は Longest Prefix Match (LPM) モデルに従います。巨大なIPv6サイズにより、ARTはノードのスケーリング機能によってハイエンドハードウェアスイッチのラインレート (Line-Rate) レベルでのBGP (Border Gateway Protocol) ルーティング問題を解決し、ルーティングテーブル (Forwarding Information Base – FIB) のサイズを数メガバイト (L3キャッシュに収まるサイズ) に最適化します。

構成チューニング (Configuration Tuning)

ARTを完全に動作させるには、メタデータ圧縮サイズ (Max Prefix Length) を微調整する必要があります。長すぎると ARTNode のベースサイズが膨れ上がり、短すぎるとPath Compressionのパフォーマンスが低下します。実際のアプリケーションでは、compressed_prefix の8〜16バイトの値が、メタデータ構造を常に16〜32バイトのしきい値内に収めるための黄金比 (Golden Ratio) と見なされています。

Lessons Learned (得られた教訓)

  1. ハードウェアの理解がアルゴリズムの生存を決定する:

    2. 次の10年では、キャッシュミス、TLBスラッシング、およびIPCメトリクスを無視した場合、従来の数学的な $mathcal{O}$ の計算量は実際のパフォーマンスを反映しなくなります。最良のアルゴリズムは、シリコンと「調和」するアルゴリズムです。

  1. 分岐命令の排除 (Branchless over Branching):

    2. 分岐予測ミス (branch misprediction) を処理するコストは、冗長なデータをロードし、ブラインド並列計算 (SIMD blind execution) を処理するよりもはるかに高くなります。ARTは、Node16の設計において優れた模範を示しています。

  1. 動的データ構造 (Morphological Flexibility):

    2. 1つのサイズ (one-size-fits-all) が、高度に偏った実データ (skewed data) に適合することは決してありません。データ密度に基づいて動作と形態を変更することで、時間的にも物理的空間的にも圧倒的な優位性がもたらされます。

  1. Persistent Memory と CXL による未来:

    2. NVDIMM (Intel Optaneなど) とCXL (Compute Express Link) プロトコルの時代に直面して、物理的な書き込み帯域幅は非常に高価になります。FPTree (Fingerprinting Persistent Tree) や REC-ART などのバリアントは、ARTのコア構造から学び、$mathcal{CLFLUSHOPT}$ ストレージバリア命令と組み合わせて、ダイレクトパーシステントストレージの将来の限界を再定義しています。
