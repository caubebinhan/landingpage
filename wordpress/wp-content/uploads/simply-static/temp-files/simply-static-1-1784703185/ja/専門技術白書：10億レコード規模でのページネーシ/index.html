専門技術白書：10億レコード規模でのページネーションクエリ – Offset/Limitに対するKeyset Paginationの台頭

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー (Executive Summary)

ページネーション (Pagination) は、あらゆるアプリケーションにおいて最も基本的な機能の一つです。しかし、データセットが10億レコード (billion-record scale) の閾値を超えると、初期の素朴なSQLページネーションクエリは致命的な欠陥を露呈し、パフォーマンスの崩壊を引き起こします。

この技術白書は、データベースの世界で最も一般的な2つのページネーション戦略であるOffset/LimitとKeyset Pagination (Seek Pagination)の包括的な解剖を提示します。単なるSQL構文にとどまらず、このドキュメントはこれらのステートメントがB+Tree構造、オペレーティングシステムのページキャッシュ (Page Cache)、およびCPUマイクロアーキテクチャ (L1/L2 Cache, Branch Prediction) とどのように相互作用するかに深く掘り下げます。数学的およびマイクロアーキテクチャの分析を通じて、分散システムにおいてOffset/Limitがなぜ目に見えない大惨事であるのか、そしてKeyset Paginationが最大規模でリアルタイムのレイテンシ ($O(1)$) を維持するための鍵である理由を証明します。

このドキュメントを読み終えると、システムエンジニア、DBA、およびソフトウェアアーキテクトはページネーションの内部動作原理を習得し、「Deep Pagination Penalty」(深いページネーションのペナルティ) 現象の根本原因を理解し、自身のシステムに対する非常に価値のある設計の教訓を引き出すことができます。

はじめにと中核となる問題 (The Core Problem)

問題とは何か？ 数十億のトランザクションを持つEコマースシステムを運用していると想像してください。ユーザー (またはバッチジョブプロセス) がトランザクションリストの「10,000ページ目」の表示を要求します。バックエンドのソースコードは次のような古典的なSQLステートメントを発行します： SELECT * FROM transactions ORDER BY created_at DESC OFFSET 100000 LIMIT 50;

最初はデータが少ないため、クエリは10ミリ秒で返されます。しかしシステムが成長すると、同じクエリに10秒かかり、データベースサーバーのCPUが100%で実行され、他の一連のクエリがハングアップします。これはDeep Pagination Penalty (深いページネーションのペナルティ) 現象と呼ばれます。

なぜわずか50行を取得するクエリがデータベースクラスター全体をクラッシュさせることができるのでしょうか？その答えはOffset/Limitの「カウントして破棄する」原理にあり、これはハードウェアの共生 (Mechanical Sympathy) の原則に完全に反する設計モデルです。

Offset/Limitメカニズムの解剖：巨大な無駄

数学的およびアルゴリズムの複雑さ

RDBMSが OFFSET O LIMIT K を受け取ったとき、直接位置 $O$ に「ジャンプ」することはできません。開始位置から正確に $O + K$ 個のレコードをスキャンし、最初の $O$ 個のレコードを捨てる痛みを押し殺して、最後の $K$ 個のレコードのみを返すことを余儀なくされます。 合計実行時間 $T_{offset}$ は一次関数です：

$$ T_{offset}(O, K) = C_{seek} cdot log_b(N) + C_{scan} cdot sum_{i=1}^{O+K} c_i $$

コスト $c_i$ は決して小さくありません。これには、各レコードに対するディスクからの読み取り、解凍 (decompression)、および条件の確認 (filtering) が含まれます。

バッファプールスラッシング (Buffer Pool Thrashing)

$O + K$ 個のレコードを読み取るために、データベースはそれらを含むすべてのデータページ (Data Pages) をSSDからRAMに引き込む必要があります。OFFSET 1000000 の場合、捨てるためだけに数ギガバイトのガベージデータをRAMにロードさせている可能性があります。

この膨大なガベージデータは、データベースのLRU (Least Recently Used) クリーンアップアルゴリズムをトリガーし、他の重要なクエリを提供している「ホット」なデータページ (hot pages) をフラッシュ (flush) します。この現象はキャッシュ汚染 (Cache Pollution) と呼ばれ、システム全体のキャッシュヒット率 (Cache Hit Ratio – CHR) を著しく低下させ、データベースにハードドライブからの読み取りを強制し、死の螺旋 (Death Spiral) を引き起こします。

分散システムにおける大惨事 (Distributed Systems)

シャーディングアーキテクチャ (Elasticsearch、Cassandraなど) では、Offset/Limitは悪夢に変わります。10個のシャードがあり、OFFSET 100000 LIMIT 50 を呼び出す場合、コーディネーターノード (Coordinator Node) は各シャードに $100,050$ レコードを返すよう要求する必要があります。その後、Coordinatorは $100,050 times 10 = 1,000,500$ レコードをRAMに保持し、Merge-Sortを実行して、$1,000,000$ 行を破棄しなければなりません。この操作は簡単にメモリ不足 (Out-Of-Memory – OOM) を引き起こしたり、ガベージコレクターを停止 (GC Stop-The-World) させたりします。

Keyset Pagination (Seek Pagination) アーキテクチャ：大規模での救済策

Keyset Paginationは「絶対位置」 (Offset) の概念を放棄します。代わりに、前のページの最後のレコードの状態 (例：last_id, last_timestamp) を保存し、それを WHERE 条件として次のクエリに渡します。

SELECT * FROM transactions WHERE (created_at, id) < (last_timestamp, last_id) ORDER BY created_at DESC, id DESC LIMIT 50;

検索木アルゴリズムの優位性

Keyset PaginationはB+Treeインデックス構造を最大限に活用します。線形カウントの代わりに、データベースはIndex Seek (木での二分探索) 操作を実行して、 (last_timestamp, last_id) と等しい値を持つレコードに正確に「着地」します。

複雑さから $O$ 要素が完全に排除されます：

$$ T_{keyset}(K) = C_{seek} cdot log_b(N) + C_{scan} cdot K $$

$N = 10$億の場合、B+Treeの高さはわずか3〜4レベルです。最上位レベル (Root/Internal Nodes) は常にL3キャッシュに存在します。したがって、このSeek操作にかかる時間は数マイクロ秒にすぎません。着地後、データベースはリーフノード (Leaf Nodes) 上の正確に次の $K$ (50) レコードをスライドするだけです。

分散ボトルネックの徹底的な解決

分散システムでKeyset Paginationを使用する場合、Coordinatorはキーペア (last_timestamp, last_id) を直接シャードに渡します。各シャードは超高速なIndex Seekを実行し、正確に50レコードのみを返します。Coordinatorは $50 times 10 = 500$ レコードを受け取り、Merge-Sortを実行して返すだけです。消費されるRAMとネットワーク帯域幅は、Offset/Limitと比較して数千倍減少します。

マイクロアーキテクチャ (Micro-architecture) の最適化とハードウェアの相互作用

これら2つのモデル間の違いは、CPUマイクロチップとオペレーティングシステムのレンズを通して観察すると、実に恐ろしいものです。

分岐予測 (Branch Prediction) と命令パイプライン (Instruction Pipeline)

Offset/Limitモデルでは、while ループがカウント変数 skipped < offset であるかどうかを継続的にチェックします。数十万のガベージレコードに対して、CPUの分岐予測器 (Branch Predictor) は混乱します。予測が外れると、CPUは命令パイプライン全体をフラッシュ (Instruction Pipeline Flush) し、ミスごとに15〜20クロックサイクル (clock cycles) を無駄にしなければなりません。

$$ T_{cpu_cycles} = N_{instructions} cdot CPI_{ideal} + N_{misses} cdot Penalty_{cache_miss} + N_{mispredicts} cdot Penalty_{pipeline_flush} $$

Keyset Paginationでは、リーフノードをスライドするループには破棄条件の分岐が一切含まれません。これは直線的な実行プログラミング設計 (Straight-line execution) に準拠しており、CPUパイプラインを常に満杯にし、100% IPC (Instructions Per Cycle) の閾値で動作させます。

TLBスラッシング現象とL1/L2キャッシュ

OffsetがCPUに過剰なガベージデータをロードさせると、L1/L2キャッシュから有用な命令とデータが追い出されます。さらに悪いことに、数十万の仮想ページ (Virtual Pages) を物理ページ (Physical Pages) にマッピングしなければならない場合、変換ルックアサイドバッファ (TLB – Translation Lookaside Buffer) も過負荷になり、TLBスラッシング (TLB Thrashing) が発生します。結果として、CPUはページテーブルを再スキャンするためにOSカーネルを呼び出す (Page Walk) 必要があり、システム全体が遅くなります。小さな $K$ レコードを持つKeyset Paginationは、TLBヒット率を99.9%近くに保ちます。

SSD NVMeキューレイテンシとハードウェアプリフェッチャ (Hardware Prefetcher)

NVMeディスククラスターでは、Offset/Limitからの数千のガベージDMAリクエストのプッシュがディスク制御キュー (Queue Depth) を詰まらせます。対照的に、Keyset Paginationは使いやすい線形読み取りモデルです。これにより、CPUとオペレーティングシステムのハードウェアプリフェッチャ (Hardware Prefetcher) は、次に使用されるキャッシュライン (Cache Line) を極めて正確に予測し、アルゴリズムが要求する前にプロアクティブに内部SRAMにデータを引き込み、I/Oメモリストール (I/O Memory Stall) を完全に排除できます。

学んだ教訓と実践的な経験 (Lessons Learned)

詳細な分析の後、データベース設計における重要な教訓を引き出します：

  1. 本番環境での巨大なOFFSETの絶対的な禁止： OFFSET > 10,000 を持つクエリは、システムの安定性を脅かす時限爆弾です。ユーザーインターフェイスが100,000ページへのジャンプ (例：Jump to Page) を要求する場合、UX/UI設計を再考するか (Keysetを使用したInfinite Scrollingへの切り替えなど)、正確なカウントが本当に必要な場合はサードパーティのインデックスシステムの使用を強制します。
  2. Keyset用のカバリングインデックス (Covering Index) の設計： Keyset Paginationは、ORDER BY基準とページネーション条件に一致する複合インデックス (Composite Index) がある場合にのみ、最大の力を発揮します。例：CREATE INDEX idx_created_id ON transactions(created_at DESC, id DESC);。
  3. ページネーションキーの一意性 (Uniqueness) の確保： Keysetを使用する際の一般的な間違いは、重複する値を持つ列 (例：created_at) のみを使用することです。同じ created_at を持つレコードが100個ある場合、アルゴリズムはデータ欠落エラーになります。アンカー (Tie-breaker) として、常に一意の列 (例：id または UUID) を追加してください。
  4. 異常 (ファントムデータ) への耐性： Offsetを使用する場合、前のページに新しいデータがINSERTされると、後続のすべてのページが「シフト」 (Shift) され、ユーザーに重複したデータが表示されます。Keyset Paginationはインデックス空間の特定の物理ノードを直接指すため、このデータシフトエラーの影響を完全に受けません。

結論

ページネーション戦略の選択は、単に数行のSQLを入力するだけではありません。それは、基盤となるハードウェアマイクロチップ層とソフトウェアがどのように通信するかについてのエンジニアの理解を反映しています。Offset/Limitの時代遅れの線形カウントの考え方を放棄し、Keyset Paginationのエレガントな数学的原理を受け入れることで、システムを崩壊から守るだけでなく、現代のサーバーアーキテクチャの無限の計算能力を解き放ち、何億人ものユーザーにスムーズにサービスを提供できるアプリケーションプラットフォームへの道を開くことができます。
