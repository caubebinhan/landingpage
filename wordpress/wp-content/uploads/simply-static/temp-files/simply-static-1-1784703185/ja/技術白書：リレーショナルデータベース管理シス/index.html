技術白書：リレーショナルデータベース管理システム（RDBMS）における Cost-Based Optimizer の内部アーキテクチャとアルゴリズム

July 22, 2026 · admin · 4 分で読めます

エグゼクティブサマリー (Executive Summary)

データが指数関数的に爆発的に増加している状況下において、データの迅速な検索と処理は、すべての企業にとって中核となる競争上の優位性となっています。リレーショナルデータベース管理システム（RDBMS）におけるこのプロセスの中心となるのがクエリオプティマイザ（クエリ最適化）です。この記事では、SQL ステートメントの実行方法を決定する「頭脳」として機能する、RDBMS のコアコンポーネントである Cost-Based Optimizer（CBO）について深く分析します。

マイクロアーキテクチャ、コスト評価のための数学的モデル、動的計画法アルゴリズム、データの選択性予測メカニズム（Cardinality Estimation）、そして CBO の決定に対するハードウェアやオペレーティングシステムの影響について詳しく掘り下げます。この白書を読み終えれば、CBO の内部動作原理や、複雑なクエリの最適化に伴う痛烈な問題点を深く理解し、データベースの設計や保守に適用できる実践的な教訓を引き出すことができるでしょう。

はじめにと中核となる問題の定義

リレーショナルデータベース管理システム（RDBMS）におけるクエリの最適化は、コンピュータサイエンスの分野で最も複雑な計算問題の 1 つであり、グラフ理論、確率・統計学、およびコンピュータシステムアーキテクチャの密接な統合が求められます。

問題とは何か？ ユーザーが論理クエリ（通常は SQL）を送信すると、RDBMS システムはその論理クエリを、ハードディスクや RAM とやり取りするための物理的な実行計画（Physical Execution Plan）に変換する必要があります。複数の複雑な結合（Join）を含むクエリの場合、考えられる実行計画の数は数百万に及ぶこともあります。実行計画の選択を誤る（例：インデックスを使用せずにテーブル全体をスキャンしたり、結合順序を間違えたりする）と、数ミリ秒の応答時間が数時間に変わったり、システムがクラッシュ（Out of Memory）したりする可能性があります。

この問題を解決する心臓部が Cost-Based Optimizer (CBO) です。硬直したルール（インデックスがあれば常に使用するなど）にのみ依存していた初期の Heuristic-Based Optimizer や Rule-Based Optimizer とは対照的に、Cost-Based Optimizer は、最適な経路を選択する前に、何万もの潜在的な実行計画の実行時間とリソースのコスト（Cost）をシミュレートする包括的な数学的モデルを使用します。

CBO の意思決定プロセスは、以下に強く依存します。

  • クエリ自体の構造。
  • データの統計的特性（メタデータ）。
  • ハードウェア構成（CPU、I/O 帯域幅）。
  • オペレーティングシステムのバッファ管理メカニズム。

優れたオプティマイザ（PostgreSQL や Oracle など）と貧弱なデータベースの根本的な違いは、探索空間モデルを分解するアルゴリズムの洗練度にあります。この問題の計算量（Computational Complexity）は NP 困難（NP-Hard）クラスに属するため、システムは確率的推定（Stochastic Estimation）を用いた継続的な近似戦略を使用する必要があります。

Cost-Based Optimizer のアーキテクチャの解剖

実行計画は、本質的には物理演算子のツリー（Physical Execution Tree）です。

  • リーフノード (Leaf nodes): 基本的なデータアクセス方法（Sequential Scan、B+ Tree Index Scan、Bitmap Scan）を表します。
  • 中間ノード (Intermediate nodes): Join（Hash、Merge、Nested Loop）、Aggregation、Sort などの関係代数演算を表します。

CBO は、これらの各ノードのコストを定量化し、パイプライン（pipeline）を通じたデータフロー（data flow）を計算し、すべてのリソース消費を合算して単一のコスト数値を導き出す必要があります。

最適化サイクルは、いくつかのフェーズに分かれています。

  1. Parser/Analyzer: SQL を抽象構文木（AST：Abstract Syntax Tree）にコンパイルし、次に論理計画（Logical Plan）に変換します。
  2. Logical Optimization (Rule-based): 式を簡素化し、フィルタリング条件を押し下げます（Predicate Pushdown）。
  3. Physical Optimization (Cost-based): 探索空間（Search Space）を巡回し、コストモデル（Cost Model）を使用してスコアリングを行い、最適な物理計画を選択します。

コスト評価の数学的モデルとハードウェアリソースの推定関数

CBO における定量化プロセスの核心は、実際のハードウェア操作に対して忠実度の高いコストモデル（cost model）を構築することです。総コストモデル $C_{total}$ は、ハードディスクの I/O、CPU の計算サイクル、メインメモリの割り当て、およびネットワーク遅延の組み合わせ関数です。

一般的なコスト関数は次のように表すことができます。

$$C_{total} = W_{IO} cdot C_{IO} + W_{CPU} cdot C_{CPU} + W_{MEM} cdot C_{MEM} + W_{NET} cdot C_{NET}$$

ここで、重み（$W$）はハードウェア（HDD 対 SSD など）を正確に反映するように設定によって微調整されます。

I/O コストの分解 ($C_{IO}$)

コンポーネント $C_{IO}$ は、データの読み取り/書き出しコストを推定し、シーケンシャルリード（sequential reads）とランダムリード（random reads）に細分化されます。HDD の場合、ランダムリードのコストはシーケンシャルリードよりもはるかに大きくなります。SSD ではこの差は小さくなりますが、仮想ブロックの制限によりランダムリードの方が依然として高価です。

$$C_{IO} = N_{seq} cdot C_{seq} + N_{rand} cdot C_{rand} + N_{dirty_flush} cdot C_{write_barrier}$$

CPU コストの分解 ($C_{CPU}$)

CPU コストは、各物理演算子を通過するレコード（tuples）の数、述語評価（predicate evaluation）、およびハッシュ計算（hashing）に基づいてモデル化されます。

$$C_{CPU} = N_{tuples} cdot C_{tuple_eval} + N_{index_probes} cdot C_{index_lookup} + N_{hash_collisions} cdot C_{resolution_penalty}$$

CPU を正確にシミュレートするには、CBO がデータ行のサイズ（Cardinality）に関する詳細なカウンターを維持する必要があります。これを実現するために、CBO は Estimator を使用します。

Cardinality Estimation: CBO の心臓部と確率的データ構造

条件 $P$ の選択性（Selectivity）は、その条件を満たすレコードの割合です。

$$Sel(P) = frac{|sigma_P(R)|}{|R|}$$

従来、CBO は複数の条件が結合されている場合（AND、OR）、統計的独立性（Statistical Independence）を仮定します。

$$Sel(P_1 land P_2) = Sel(P_1) cdot Sel(P_2)$$

よくある間違い: 実際には、属性間には強い相関関係があることがよくあります（例：国 = 'ベトナム' と 市外局番 = '+84'）。独立性を仮定すると、極端な過小評価（under-estimation）が発生します。CBO は、返される結果が 100万行ではなくわずか 1 行であると考え、誤って Nested Loop Join を選択し、システムのパフォーマンスを破壊してしまうことになります。

これを克服するために、最新の CBO は以下を使用します。

  1. 多次元統計 (Multi-dimensional statistics): 列間の相互相関を評価します。
  2. V-Optimal Histograms: 平均二乗誤差（Mean Squared Error）を最小化するデータパーティショニングアルゴリズム。不均一なデータ分布（data skew）の予測に役立ちます。
  3. HyperLogLog (HLL) と Count-Min Sketch: 非常に大きな個別要素のカウント（Count Distinct）問題に対して、HLL は RAM の消費をキロバイト程度に抑えながら、わずか数パーセントの誤差で確率的近似推定を可能にします。

$$SE approx frac{1.04}{sqrt{m}}$$

ここで、$m$ はビットレジスタの数であり、このアルゴリズムはプロセッサの L1/L2 キャッシュの帯域幅を破壊することなく極限の最適化を達成します。

動的計画法アルゴリズムと探索空間

実行計画の探索空間は巨大な組み合わせの壁です。結合する $N$ 個のテーブルの場合：

  • Left-Deep Trees（パイプラインメカニズムを優先する線形ツリー）のみを考慮した場合：順列の数は $N!$ です。
  • Bushy Trees（並列結合をサポートする分岐ツリー）も考慮した場合：その数はカタラン数の変種まで膨れ上がります。

$$text{総 Bushy Trees 数} = frac{(2N-2)!}{(N-1)!}$$

System R のアルゴリズム（IBM Research によって生み出された）は、動的計画法（Dynamic Programming）を使用して、データテーブルの各サブセットに最適な結合構成をメモ化（memoize）します。ベルマンの最適性の原理（Bellman’s Principle of Optimality）を使用することで、システムは最悪の構成を枝刈りできます。

$$OptPlan(S) = min_{S_1, S_2 subset S, S_1 cap S_2 = emptyset} { Cost(OptPlan(S_1) bowtie OptPlan(S_2)) }$$

しかし、このボトムアップ（Bottom-Up）アーキテクチャは、データのソートを優先するなど、トップダウンの要求駆動型（Top-Down Demand-Driven）のインタラクティブなフィードバックが必要な場合に困難に直面します。

Cascades Framework: オプティマイザ設計の革命

Microsoft SQL Server、CockroachDB、Apache Calcite で使用されている Cascades アーキテクチャは、System R の問題を完全に解決します。

Cascades は、トップダウン（Top-Down）トラバーサルと超グラフ構造 Memo（Memoization Graph）を組み合わせたものです。Memo は、論理的な結果は同じですが物理計画が異なる式のグループである、等価クラス（Equivalence Classes）を格納します。

中核となる優位性は、物理プロパティの要求（Physical Properties Demand）にあります。親演算子（例：GROUP BY X）が X でソートされたデータを要求する場合、子ツリーにその要件を満たす物理計画を検索させます。これにより、Hash Join の方がコストが低くなる可能性があっても、Hash Join（ソート構造を破壊する）ではなく Merge Join（ソートを維持する）が優先されます。

次の Rust コードは、Memo Graph 内での分枝限定法（Branch and Bound）によるコスト囲い込みプロセスを示しています。

use std::collections::HashMap;
use std::sync::{Arc, RwLock};

#[derive(Clone, Hash, PartialEq, Eq)]
struct LogicalExpressionId(u64);

#[derive(Clone)]
struct PhysicalPlan {
    cost: f64,
    operator_type: String,
}

struct MemoTable {
    best_plans: RwLock<HashMap<LogicalExpressionId, PhysicalPlan>>,
}

impl MemoTable {
    fn new() -> Self {
        MemoTable { best_plans: RwLock::new(HashMap::new()) }
    }

    fn optimize_group(&self, group_id: &LogicalExpressionId, current_upper_bound: f64) -> Option<PhysicalPlan> {
        {
            let read_guard = self.best_plans.read().unwrap();
            if let Some(cached_plan) = read_guard.get(group_id) {
                if cached_plan.cost <= current_upper_bound {
                    return Some(cached_plan.clone()); // しきい値を超えた場合の枝刈り (Pruning)
                }
            }
        }

        // ルールエンジンによるバリアント生成
        let candidates = vec![
            PhysicalPlan { cost: 1500.0, operator_type: "GraceHashJoin".to_string() },
            PhysicalPlan { cost: 800.0, operator_type: "ParallelMergeJoin".to_string() },
        ];

        let mut local_best: Option<PhysicalPlan> = None;
        let mut min_cost = current_upper_bound;

        for candidate in candidates {
            if candidate.cost >= min_cost { continue; }
            min_cost = candidate.cost;
            local_best = Some(candidate);
        }

        if let Some(ref best) = local_best {
            let mut write_guard = self.best_plans.write().unwrap();
            write_guard.insert(group_id.clone(), best.clone());
        }
        local_best
    }
}

ハードウェアアーキテクチャと OS メモリの影響 (Hardware-Aware CBO)

コストモデルは、多くの場合、物理的な障壁、すなわちメモリの断片化（Memory Fragmentation）、ページング（Paging）、およびキャッシュ階層（Cache Hierarchy）に直面したときに悲惨な失敗を招きます。

Grace Hash Join – メモリオーバーフローの悪夢

Hash Join を実行する場合、ハッシュテーブル（Hash Table）が L3 キャッシュにぴったり収まれば、プローブ（Probe）速度は非常に高速（数ナノ秒）になります。RAM に溢れると、TLB ミスによって遅延が増加します。しかし、ハッシュテーブルが規定の RAM を超えて肥大化（Out of Memory）すると、システムは Grace Hash Join をトリガーし、データを半分にスライスして磁気ディスクに書き出さなければなりません。

このときの I/O コストは次の式に従って爆発的に増加します。

$$C_{hash_join} = 3 cdot (|R| + |S|) cdot C_{IO_seq} + C_{cpu_partitioning}$$

（データを RAM から読み込み -> ディスクに書き出し -> プローブするために読み込み直す必要があります）。このとき、オプティマイザがディスクのスピルイベントを早期に認識していれば、Hash Join ではなく Sort Merge Join を選択していた可能性があります。

NUMA アーキテクチャとハードウェアプリフェッチ

マルチ CPU サーバー（NUMA）では、ソケット間のクロスメモリアクセスは非常に高コストです。最新の CBO（NUMA-aware）は、データに局所性（Data Locality）がない場合、コストにペナルティ（Penalty）を適用します。同時に、CBO はプロセッサの Hardware Prefetcher メカニズム（CPU が実際に要求する前に自動的にデータをキャッシュにロードする）のおかげで、シーケンシャル I/O コストを微調整して削減します。

得られた教訓と実践的な経験 (Lessons Learned)

CBO の内部構造を深く研究することで、ユーザー、データエンジニア (Data Engineer)、およびシステム管理者 (DBA) は、次のような貴重な教訓を引き出すことができます。

  1. システムを責めるのではなく、本質を理解する:

    2. クエリが遅いとき、すぐにデータベースプラットフォームを変更しようとしないでください。根本的な問題は、多くの場合、CBO に情報が不足しているか、データの分布に関する情報が間違っている（Cardinality Misestimation）ことにあります。

  2. 統計情報の維持（Statistics Maintenance）を常に行う:

    3. システムで ANALYZE (PostgreSQL) や GATHER_STATS (Oracle) などのコマンドを定期的に実行するようにしてください。古い統計情報は CBO を盲目にし、何十億行ものデータに対して Hash Join ではなく Nested Loop Join を選択させてしまいます。

  3. データ相関 (Data Correlation) の危険性:

    4. RDBMS が Cross-Column Statistics をサポートしていない場合は、密接に関連する列（市や郵便番号など）で AND/OR 条件を使いすぎないようにしてください。多次元統計を宣言して、この状況を克服します。

  4. ハードウェアリソースの制限の認識 (Memory Config):

    5. PostgreSQL の work_mem のような RAM 割り当てパラメータに注意してください。割り当てが小さすぎると、Sort 操作や Hash 操作がディスクに溢れ（Spill to Disk / Grace Hash Join）、パフォーマンスが何十倍も低下します。割り当てが大きすぎると、サーバー全体の RAM がクラッシュする可能性があります。

  5. 物理的制約 (Physical Constraints) についての理解:

    6. ディスク上のデータの順序付け（Clustered Index）は、CBO に無限の利点をもたらします。システムは「すでにソートされている（Interesting Orders）」という特性を受け継ぐため、安価な Merge Join アルゴリズムを活用できるからです。

結論

Cost-Based Optimizer は、ソフトウェアエンジニアリング、高度なデータ構造、および最適化理論の交差点の頂点に立つものです。ヒストグラム、Cascades の Memoization メカニズムから、L3 キャッシュや NUMA アーキテクチャの影響まで、この機械の最も小さな歯車を理解することで、オプティマイザが「好む」クエリを記述し、最大限のパフォーマンスで動作し、何百万人ものユーザーにスムーズにサービスを提供できるアプリケーションを作成することができます。
