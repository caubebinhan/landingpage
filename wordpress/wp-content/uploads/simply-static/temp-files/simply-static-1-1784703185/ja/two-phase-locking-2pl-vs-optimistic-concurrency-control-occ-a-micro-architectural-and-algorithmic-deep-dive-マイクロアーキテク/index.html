Two-Phase Locking (2PL) vs. Optimistic Concurrency Control (OCC): A Micro-Architectural and Algorithmic Deep-Dive (マイクロアーキテクチャおよびアルゴリズムの徹底解説)

July 22, 2026 · admin · 3 分で読めます

エグゼクティブ・サマリー (What You Are Reading)

この高度な技術ホワイトペーパーは、データベースのconcurrency control (並行処理制御) の2つの基盤であるTwo-Phase Locking (2PL)とOptimistic Concurrency Control (OCC)の厳密なアーキテクチャ比較を提示します。基本的な教科書の定義にとどまらず、このドキュメントではこれらのアルゴリズムが純粋なシリコン物理学、オペレーティングシステムのメモリアロケータ、およびCPUキャッシュコヒーレンシプロトコルとどのように相互作用するかを解剖します。

読者のための主要なポイント:

  • アルゴリズムの二面性: 悲観的ブロッキング (2PL) と楽観的アボート＆リトライ検証 (OCC) の間の基本的なトレードオフを理解する。
  • マイクロアーキテクチャのボトルネック: 2PLがどのようにロックに起因するキャッシュの無効化 (False Sharing, MESIプロトコル) に苦しむのか、またOCCがどのようにthread-local storageやメモリバリアの限界を押し上げるのかを探求する。
  • 競合下の漸近的振る舞い: OCCが極度のZipfian skew (偏り) の下でなぜ「optimistic thrashing (楽観的スラッシング)」へと崩壊するのか、一方2PLがなぜスリープキューへと緩やかに性能低下するのかを学ぶ。
  • 実践的なエンジニアリングの教訓: いつどのプロトコルを使用すべきか、そして現代の分散データベースエンジンがMVCC + 2PLや適応型並行制御などのハイブリッドアプローチを普遍的に採用する理由を発見する。

根本的な問題 (The Core Problem)

マルチスレッドのデータベース環境において、Serializability—並行実行がある順次実行と同等であるという数学的保証—を確保することは究極の課題です。

データベースアーキテクトは残酷なジレンマに直面しています:

  • 競合が頻繁に発生すると想定する場合、Pessimistic Locking (2PL)を使用します。しかし、ロックはスレッドを待機させ、重大なCPUの利用率低下、コンテキストスイッチのオーバーヘッド、そして複雑なdeadlock (デッドロック) を引き起こします。
  • 競合が稀であると想定する場合、Optimistic Concurrency Control (OCC)を使用します。しかし、競合が実際に頻繁に発生した場合 (例: eコマースシステムでのバイラル商品の販売開始など)、スレッドは絶えず検証、失敗、および再試行を繰り返します。これにより、CPU利用率が100%に張り付く一方でシステムのスループットがゼロに落ち込むという致命的な正のフィードバックループが生じます (Optimistic Thrashing)。

根本的な問題は、アトミックなCPU命令を最小限に抑え、バスの飽和を避け、そして壊滅的なスループットの崩壊を引き起こすことなく、ワークロードの偏りの動的な変化を優雅に処理する同期メカニズムを設計することです。

深い技術的分析 (Deep Technical Analysis)

トランザクション並行処理制御の理論的基盤

serializabilityの数学的基盤は、conflict graph (競合グラフ) $G = (V, E)$ に依存しています。ここで頂点 $V$ はコミットされたトランザクションを表し、エッジ $E$ は競合 (read-write, write-read, または write-write) を表します。スケジュールは、$G$が厳密に非巡回 (strictly acyclic) である場合にのみconflict-serializableとなります。

Two-Phase Locking (2PL)

2PLは、厳格なプロトコルを実施することにより、非巡回性を保証します:

  • Growing Phase: トランザクションはロックを取得できますが、ロックを解放することはできません。
  • Shrinking Phase: トランザクションはロックを解放できますが、新しいロックを取得することはできません。

カスケーディングアボート ($T_j$が$T_i$から未コミットのデータを読み取り、$T_i$がアボートする状況) を防ぐため、システムはコミットまたはアボートまで全ての排他ロックを保持するStrict Two-Phase Locking (S2PL)を使用します。2PLは、プログラムによるデッドロック検出 (タージャンの強連結成分アルゴリズムを $mathcal{O}(V+E)$ 時間で実行する) または防止スキーム (Wait-Die / Wound-Wait) を必要とします。

Optimistic Concurrency Control (OCC)

OCCはトランザクションを3つのフェーズに分割します:

  1. Read Phase: 全ての操作はデータベースのローカルコピーで実行されます。Read set $RS(T_i)$ と Write set $WS(T_i)$ が記録されます。
  2. Validation Phase: システムは、$T_i$のセットと同時にコミットされたトランザクションのセット間の交差をチェックします。
  3. Write Phase: 検証が成功した場合、ローカルの変更がグローバル状態にフラッシュされます。

検証が失敗した場合、ワークスペースは破棄されます。高いデータ競合下では、検証の失敗確率 $P_{abort}$ は指数関数的に増大します。

マイクロアーキテクチャへの影響とハードウェアレベルの同期

2PLとOCCの理論的な分岐は、QPI/UPIインターコネクトを介したCPUキャッシュコヒーレンシプロトコル (MOESI)に特に関連して、シリコンレベルで根本的に現れます。

ハードウェアにおける2PLのコスト

ロックマネージャーは、ミューテックス/スピンロックの巨大なハッシュテーブルです。スレッドがロックを取得する際、アトミック命令 (x86_64でのLOCK CMPXCHG) を実行します。これらの命令は、プロセッサのストアバッファをバイパスし、完全なメモリバリアとして機能し、命令実行をシリアル化し、アウトオブオーダー実行を防ぐため、非常に高コストです。

さらに、関連のない2つのロックが同じ64バイトのキャッシュライン上に存在する場合、False Sharingが発生します。MESIプロトコルは、1つのスレッドがロックを取得する際に、すべてのCPUコアにわたってキャッシュラインを強制的に無効化し、途方もないメモリバストラフィックを引き起こします。高度なロックマネージャーは、NUMAアーキテクチャで生き残るために、alignas(64) を使用して構造体を積極的にパディングしなければなりません。

$$
T_{throughput_2PL} = frac{N_{cores}}{T_{exec} + N_{locks} times left(T_{atomic} + P_{contention} times T_{wait}right) + T_{deadlock_detection}}
$$

ハードウェアにおけるOCCのコスト

OCCは、リードフェーズ中のアトミック操作を回避します。トランザクションはThread-Local Storage (TLS) を変更し、L1/L2キャッシュをコヒーレンシ無効化から保護します。

しかし、Validation Phaseは究極のアムダールのボトルネックになります。検証には、グローバルシーケンスロック (seqlock) によって保護されることが多いクリティカルセクションに入ることが必要です。さらに、巨大な一時ワークスペースの割り当てとガベージコレクションは、OSのメモリアロケータ (例: jemalloc) にストレスを与えます。割り当て率がカーネルの仮想メモリサブシステムを飽和させると、コア全体でTLBシュートダウンを引き起こします。

$$
T_{throughput_OCC} = frac{N_{cores} times (1 – P_{abort})}{T_{read_phase} + T_{validation_phase} + T_{write_phase} + P_{abort} times T_{retry_penalty}}
$$

アルゴリズムの漸近線とZipfianワークロード

パフォーマンスを評価するには、これらのプロトコルが高度に偏ったデータ分布 (Zipfian分布によってモデル化される、$alpha > 0.9$) に対してどのように反応するかを理解する必要があります。

OCCでは、高度に偏ったワークロードによって $P_{abort}$ が急増します。アボートされたトランザクションはすぐに再起動し、到着率 $lambda$ を人為的に膨らませます。これにより、Optimistic Thrashing Threshold (楽観的スラッシングの閾値) が形成されます。リトライ率がコミット率を上回ると、システムは崩壊します。CPU利用率は100%に達しますが、有効なスループットはゼロになります。検証アルゴリズムはセットを交差させる必要があり、ブルームフィルターやロックフリーハッシュセットで最適化されない限り、$mathcal{O}(K times R_{size} times W_{size})$ の時間を要します。

// Simplified Rust OCC Validation Logic
pub fn validate_and_commit(&self, mut txn: Transaction) -> Result<(), &'static str> {
    let commit_timestamp = self.global_timestamp.fetch_add(1, Ordering::SeqCst);
    let history = self.committed_transactions.read().unwrap();
    
    // Critical Validation Phase: Check for overlapping read/write sets
    for past_txn in history.iter() {
        if past_txn.start_timestamp > txn.start_timestamp {
            // Validation fails if past transaction modified memory we read
            if !txn.read_set.is_disjoint(&past_txn.write_set) {
                return Err("Validation Failed: Read-Write Conflict");
            }
        }
    }
    // Proceed to Write Phase...
    Ok(())
}

2PLでは、ロックキューの長さ $L$ は $mathcal{O}(L)$ に成長します。しかし、2PLは極端な偏りをより洗練されたアルゴリズムで処理します。明示的なロックメカニズムは、自己調整スロットルとして機能します。OSのスレッドスケジューラはブロックされたスレッドをスリープ状態にし、CPUリソースを節約し、スラッシングフィードバックループを防ぎます。スループットは頭打ちになりますが、崩壊することはありません。

bool acquire_lock(uint64_t txn_id, uint64_t data_id, LockMode mode) {

    if (mode == LockMode::EXCLUSIVE && state->shared_count == 0 && !state->exclusive_held) {
        state->exclusive_held = true;
        return true;
    } else {

Hardware Transactional Memory (HTM) の革命

高度なプロセッサ (Intel TSXなど) は、OCCをシリコン論理ゲートに押し込もうとしています。Hardware Transactional Memory (HTM) は、L1キャッシュを投機的バッファとして使用し、キャッシュラインのメタデータビットを介してセットを追跡します。メモリバス上で競合がスヌープされた場合、ハードウェアはクロックサイクルのほんの一瞬でトランザクションをアボートさせます。HTMはソフトウェアの検証オーバーヘッドを排除しますが、L1キャッシュ容量に厳密に制限されます。トランザクションのデータセットがL1を超えた場合、キャパシティアボートを引き起こし、システムを悲観的な2PLのソフトウェアフォールバックに強制的に戻します。

教訓とベストプラクティス (Lessons Learned & Best Practices)

  1. ワークロードの偏りを知る: 激しい競合があるシステム (チケット予約システム、在庫減少など) にピュアなOCCをデプロイしてはなりません。楽観的スラッシングは壊滅的なダウンタイムを引き起こします。OCCは、読み取り中心のアナリティクスや分割されたマイクロサービスでその真価を発揮します。
  2. パディングは必須: 2PLロックマネージャーを実装する場合は、ロックバケットに alignas(64) または alignas(128) を指定する必要があります。そうしないと、False Sharingが発生し、64コアのモンスターマシンがデュアルコアのラップトップよりも遅いマシンに変わってしまいます。
  3. 検証のボトルネックに注意: OCCは魔法の弾丸ではありません。検証フェーズにはクリティカルセクションが必要です。高速なセット交差 (ブルームフィルター) とエポックベースのメモリ回収 (Epoch-Based memory reclamation) で最適化されていない場合、それが究極のボトルネックになります。
  4. ハイブリッドが標準: 現代のエンジンが純粋な2PLや純粋なOCCを使用することはほとんどありません。Multi-Version Concurrency Control (MVCC) と書き込み用のStrict 2PLを組み合わせるか、機械学習ヒューリスティクスを使用してキューの深さに基づいてプロトコルを動的に切り替える適応型システムを使用します。

結論

Two-Phase LockingとOptimistic Concurrency Controlの間の議論は単なる学術的な演習ではありません。それは、データベースエンジンが純粋なシリコン物理学とどのように相互作用するかを決定する、根本的なアーキテクチャ上の決定です。2PLは、高競合下での予測可能な動作のためにCPUパイプライン効率を犠牲にしますが、OCCは低競合下で最大のロックフリースループットを得るためにメモリ管理のオーバーヘッドと検証の複雑さを犠牲にします。MESIキャッシュコヒーレンシからOSスレッドスケジューリングまで、これら両方のマイクロアーキテクチャへの影響をマスターすることは、エリート・データベース・システム・アーキテクトの決定的な特徴です。
