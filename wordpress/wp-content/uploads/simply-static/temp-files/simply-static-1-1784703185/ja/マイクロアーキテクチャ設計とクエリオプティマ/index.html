マイクロアーキテクチャ設計とクエリオプティマイゼーション：リレーショナルデータベースにおけるCovering Indexes、Index Condition Pushdown (ICP)、およびIndex Merge

July 22, 2026 · admin · 6 分で読めます

Executive Summary (概要 / Overview)

現代のリレーショナルデータベース管理システム（RDBMS）の複雑さは、データの永続的な保存やACID特性の保証だけでなく、非常に洗練されたクエリオプティマイゼーション（Query Optimization）アルゴリズムに深く依存しています。オプティマイザ（Optimizer）の究極の目標は、メモリアクセスサイクル（memory access cycles）の数を最小限に抑え、オペレーティングシステムのコンテキストスイッチ（context switches）を可能な限り制限し、最も重要なこととして、従来のボトルネックであったI/O操作（I/O operations）を削減することです。

RDBMSアーキテクチャにおいて、CPU（ナノ秒レベルでの処理）とストレージドライブ（マイクロ秒またはミリ秒レベルでの処理）のパフォーマンスのギャップは、巨大なメモリの壁（memory wall）を作り出します。この壁を乗り越えるため、システムエンジニアは、マイクロアーキテクチャレベルに直接介入する3つのコア最適化メカニズムを開発しました。Covering Indexes（カバリングインデックス）、Index Condition Pushdown – ICP（インデックス条件プッシュダウン）、およびIndex Merge（インデックスマージ）です。

この技術レポート/ホワイトペーパー（Whitepaper）では、数学的分析、データ構造、ハードウェア（CPU Cache、SIMD、Virtual Memory）との相互作用を深く掘り下げるとともに、大規模システムにおいて超低遅延（ultra-low latency）のクエリパフォーマンスを提供するために、これらの技術がどのように組み合わされるかの全体像を描き出すための実践的な例を提供します。

Core Problem Statement (核心的な問題の定義)

解決策に踏み込む前に、大規模データ（テラバイト/ペタバイト）上で動作する際に、単純なクエリ（naive query）が直面するパフォーマンスの問題を明確に定量化する必要があります。

標準的なRDBMSシステムモデルは、主に2つの層（layers）に分かれています：

  1. SQL Execution Engine (実行層): 構文解析（parsing）、リレーショナル代数計画（relational algebra planning）、および論理述語（logical predicates）の評価を担当します。
  2. Storage Engine (ストレージ層): B+ Tree構造、Buffer Pool、ディスクI/O、および行レベルのロック（locking）を管理します。MySQLのInnoDBやMongoDBのWiredTigerがその代表例です。

課題1：ランダムI/Oのコスト (Random I/O Cost) クエリがセカンダリインデックス（secondary index）を使用してデータを検索する場合、セカンダリインデックスのB+ Treeは通常、インデックスキーと、元のテーブル（Clustered Index/Heap Table）を逆参照するポインタ（通常は主キー – Primary Key）のみを格納します。クエリがセカンダリインデックスに含まれない列を要求する場合、システムはBookmark Lookup（ブックマークルックアップ）操作を実行する必要があります。この操作は、ランダムI/Oアクセスモデル（Random I/O）を生成します。回転ディスク（HDD）ではシークタイムのコストが非常に高く、SSDではIOPSが高いものの、広範なランダムI/Oは依然としてWrite Amplificationを引き起こし、バス帯域幅（PCIe）を消費します。

課題2：キャッシュの汚染 (Cache Pollution) とページフォールト (Page Faults) Bookmark Lookupがトリガーされた際、データページ（data page、通常16KB）がBuffer Pool（RAM）にない場合、Major Page Faultが発生します。オペレーティングシステムは実行中のスレッドを一時停止（suspend）し、ディスクからRAMへページをロードしなければなりません。Buffer Poolが満杯の場合、LRU（Least Recently Used）アルゴリズムが他のページを退去（evict）させる必要があり、ホットデータ（hot data）が失われ、キャッシュの汚染を引き起こす可能性があります。

課題3：層間の通信オーバーヘッド (Inter-layer Communication Overhead) 従来のアーキテクチャでは、Storage Engineはインデックスキーに一致するデータを見つけ、タプル全体をSQL Engineに返すことしか知りません。クエリに追加のフィルタ条件 WHERE col_A = 1 AND col_B LIKE '%x%' がある場合、Storage Engineは col_A のみを使用してツリーをトラバースし、すべての行をロードしてデシリアライズ（deserialize）し、API境界（ABI）を越えてSQL Engineに押し上げ、col_B を評価します。これにより、無駄なメモリ割り当て、冗長なデータコピー（data copying overhead）が発生し、CPUの命令パイプライン（instruction pipeline）が遅延します。

標準的なクエリコスト関数：

$$C_{naive} = C_{traverse_idx} + N_{matches} cdot (C_{page_fault} + C_{deserialize} + C_{eval_api})$$

ここで、$N_{matches}$ はインデックスのルーティング条件に一致するレコードの数です。最適化技術の目標は、括弧内のコスト定数を排除することです。

Deep Technical Knowledge / Internals (詳細な技術知識 / 内部構造)

1 B+ Tree構造とCovering Indexesによるメモリシステムの最適化

Covering Indexは、CREATE COVERING INDEXコマンドで作成する物理的なオブジェクトではありません。これは、クエリが必要とするすべてのデータ（列）（SELECT、WHERE、ORDER BY、GROUP BY）が、セカンダリインデックスのリーフノード（leaf nodes）内にすでに存在する場合に発生するクエリ現象（query phenomenon）です。

数学的基礎: リレーションを $T$ とします。クエリ $Q$ は属性セット（projection list & predicates） $A_{Q} = {a_1, a_2, dots, a_n}$ を要求します。インデックス $I$ はキー $K_{I} = {k_1, k_2, dots, k_m}$ から構成されます。InnoDBのようなエンジンでは、主キー $K_{PK}$ は常にセカンダリインデックスの各エントリの末尾に暗黙的に追加されます。 したがって、$I$ が $Q$ のCovering Indexとなるのは次の場合に限定されます：

$$A_{Q} subseteq (K_{I} cup K_{PK})$$

マイクロアーキテクチャへの影響 (Micro-architectural Impact): カバー条件が満たされると、システムはBookmark Lookupのステップを完全にスキップします。

  1. シーケンシャルアクセス (Sequential Access): クラスタ化インデックスのページ間を飛び回る代わりに、CPUはセカンダリインデックスのリーフノードを接続する双方向リンクリスト（doubly-linked list）に沿ってスライドするだけです。
  2. CPU L1/L2/L3キャッシュの最適化: セカンダリインデックス構造は非常にコンパクト（数個の列のみを含む）であるため、16KBのページに配置されるタプルの数は非常に大きくなります（高い空間局所性 – high spatial locality）。CPUにロードされる1つのキャッシュライン（64 bytes）には有用なタプルが多数含まれており、CPU Prefetcherが最大限の効率で動作するのに役立ちます。すべての計算命令で99%以上のキャッシュヒット率（Cache Hit）を達成します。

Storage Engineでのスキャンプロセスを説明するC++の疑似コード：

// Covering Indexがある場合の最適化パス (Fast Path)
void ScanLeafNode(const BTreeNode* node, const QueryContext& ctx, ResultSet& result) {
    if (ctx.is_covering) {
        // Spatial Locality Optimization
        // スキーマが固定長(fixed-length)の場合、コンパイラはループをアンロールし、SIMDを使用可能
        for (int i = 0; i < node->num_records; ++i) {
            if (EvaluatePredicates(node->records[i], ctx.predicates)) {
                result.PushBack(Project(node->records[i], ctx.projection));
            }
        }
    } else {
        // Slow Path: Bookmark lookup
        for (int i = 0; i < node->num_records; ++i) {
            RowId rid = node->records[i].GetRowId();
            // FetchFromBufferPool関数はIO Waitに遭遇した場合スレッドをブロックする可能性がある
            Tuple full_tuple = buffer_pool_manager->FetchFromClusteredIndex(rid);
            if (EvaluatePredicates(full_tuple, ctx.predicates)) {
                result.PushBack(Project(full_tuple, ctx.projection));
            }
        }
    }
}

2 述語分割アルゴリズムとIndex Condition Pushdown (ICP)

Covering Indexが不可能な場合（クエリが多すぎる列を要求する場合）、システムは $N_{matches} cdot C_{page_fault}$ のコストに直面します。Index Condition Pushdown (ICP) は、データフィルタをStorage Engineのすぐ内側に挿入することでこの問題を解決します。

述語分割 (Predicate Splitting): 条件セット $P$ は、字句解析および最適化によって3つの部分に分割されます：

  • $P_{seek}$: B+ Treeのルーティング（traversing）に使用されます（例：col1 = 'A'）。
  • $P_{icp}$: ルーティングはできないが、列がセカンダリインデックスに存在している条件（例：col2 LIKE '%xyz%'）。
  • $P_{server}$: インデックスにない列に関する条件（例：col3 > 100）。

ICP以前は、$P_{icp}$ は $P_{server}$ に統合されていました。Storage Engineは $P_{seek}$ を満たすすべてのレコードを返し、その後SQL Engineが $P_{icp}$ をチェックしていました。 ICPを使用すると、$P_{icp}$ はAPIの境界を越えてStorage Engineへ「プッシュダウン（push down）」されます。

ICPのマイクロアーキテクチャ設計: これにより通信オーバーヘッド（Context Switch / Function call）が排除されます。最新のRDBMSではLLVMを使用して述語セット $P_{icp}$ をネイティブマシンコード（native code）にJIT Compileし、CPUがタプルをデシリアライズすることなく、インデックスレコードの生のバイト配列（raw byte array）上で条件を直接評価できるようにし、プロセッサのスーパースカラー（superscalar）アーキテクチャを活用します。

3 Bitmapデータ構造とIndex Mergeにおける論理マージ

一次元のB+ Tree構造は、複数の独立した列に関する多くのORまたはAND条件（例：WHERE status = 'ACTIVE' OR category_id = 5）を含むクエリに対しては非常に脆弱です。単一のインデックスでは解決できず、すべての順列に対して複合インデックス（composite index）を作成することはストレージの観点から大惨事になります。

Index Mergeは、異なるB+ tree上で複数のスキャンを同時に開始する動的なメカニズムです。アルゴリズムは以下のフェーズから構成されます：

  1. スキャンと抽出 (Scan & Extract): 各インデックスはスキャンを実行し、RowID（またはPrimary Key）のリストを収集します。
  2. Bitmapデータ構造 (Bitmap Data Structure): RAMを消費する配列（array）やハッシュテーブル（hash table）の代わりに、識別子はビット配列にマッピングされます。疎な（sparse）識別子領域でRAMを節約するために、Roaring Bitmapsのような圧縮アルゴリズムが使用されます。
  3. ビット単位の論理演算 (Bitwise Logic):
  • ANDフィルタ (Index Merge Intersection): Bitwise AND（$land$）操作を使用して2つのRowIDセットを積集合します。
  • ORフィルタ (Index Merge Union): Bitwise OR（$lor$）操作を使用して2つのRowIDセットを和集合します。
  1. テーブル取得 (Table Fetch): 最終的なビットマップに基づいて実際のデータを取得します。

SIMD（Single Instruction, Multiple Data）によるCPUの最適化: ビットマップに対するビット単位の操作は、SIMD（x86のAVX2、AVX-512、またはARMのNEON）の完璧な候補です。各ビットを個別に処理する代わりに、CPUは1クロックサイクル（clock cycle）で512ビット（512レコードに相当）をAND/OR処理できます！

// SIMDによるI/OおよびCPU最適化を示すRustの疑似コード
// 2つのインデックスのBitmapに対するIntersectionアルゴリズムの統合。
#[cfg(target_arch = "x86_64")]
use std::arch::x86_64::{__m512i, _mm512_and_si512, _mm512_loadu_si512, _mm512_storeu_si512};

#[target_feature(enable = "avx512f")]
pub unsafe fn avx512_bitmap_intersect(bitmap_idx1: &[u64], bitmap_idx2: &[u64], result: &mut [u64]) {
    let len = bitmap_idx1.len();
    // 各512-bitベクタは8つのu64チャンクを含む
    let chunks = len / 8;
    
    for i in 0..chunks {
        // L1 Cacheから512 bitsを同時にロード
        let ptr1 = bitmap_idx1.as_ptr().add(i * 8) as *const __m512i;
        let ptr2 = bitmap_idx2.as_ptr().add(i * 8) as *const __m512i;
        let res_ptr = result.as_mut_ptr().add(i * 8) as *mut __m512i;

        let vec1 = _mm512_loadu_si512(ptr1);
        let vec2 = _mm512_loadu_si512(ptr2);

        // わずか1 CPU Cycleで512レコードを交差 (Intersection)！
        // if-elseループを完全に排除し、ブランチ予測の失敗(branch misprediction)を防ぐ
        let vec_res = _mm512_and_si512(vec1, vec2);
        
        _mm512_storeu_si512(res_ptr, vec_res);
    }
}

構成の注意事項: オプティマイザは通常、コストを計算します（Cost-Based Optimizer – CBO）。セットされたビットの総数がテーブルサイズの約20%を超える場合、CBOはIndex Mergeをキャンセルし、テーブル全体のシーケンシャルI/OコストがRowIDを介したランダムI/O読み取りコストよりも安くなるため、Full Table Scanを選択します。

Practical Applications & Case Studies (実践的なアプリケーションとケーススタディ)

1 Case Study 1: Eコマースの商品フィルタリングシステム (High Dimensionality)

Eコマースプラットフォームでは、ユーザーは brand_id、color、price_range でフィルタリングできます。

  • 問題: (Brand, Color)、(Color, Price)、(Brand, Price)のすべての組み合わせに対してインデックスを作成することは不可能です。
  • 解決策: brand_id、color に単一列インデックス（single-column index）を作成します。
  • 影響: MySQLは Index Merge Intersection を使用します。idx_brand と idx_color をトラバースし、SIMDビットマップを使用してRAM上でRowIDの積集合を瞬時に見つけ、その後ディスクにアクセスして商品情報を取得します。

2 Case Study 2: ログ＆時系列分析システム

クエリ要求：特定のユーザーの500エラーリクエストの数をカウントする。 SELECT COUNT(*) FROM access_logs WHERE user_id = 123 AND status_code = 500;

  • 問題: ログテーブルには数十億の行があり、各行は数KBです。テーブルスキャンは不可能です。
  • 解決策: インデックス idx_user_status (user_id, status_code) を作成します。
  • 影響: これは完全な Covering Index です。COUNT(*) 関数は行データを読み取る必要がありません。RDBMSは idx_user_status のB+ツリー上のリーフノードの数をカウントするだけです。クエリは数ミリ秒（ms）で完了し、Clustered Index（Heap）構造には全く触れられず、Buffer Poolはクリーンなまま保たれます。

3 Case Study 3: 空間クエリとマルチテナント (Multi-tenant SaaS)

クエリ：説明に「refund」というキーワードを含む tenant_id = 5 のトランザクションリストを取得する。 SELECT * FROM transactions WHERE tenant_id = 5 AND description LIKE '%refund%';

  • 問題: LIKE '%...' はB+ Treeを使用した二分探索には使用できません。
  • 解決策: インデックス idx_tenant_desc (tenant_id, description) を用いた Index Condition Pushdown (ICP) に依存します。
  • 影響: Storage Engineは tenant_id を使用してリーフノード（Pseek）を見つけます。リーフノードでは、JIT正規表現を適用して description（Picp）をチェックします。テナントには10,000のトランザクションがありますが、「refund」という単語が含まれているのは50のみです。ICPは、Storage Engineがリーフレベルで直接9,950行を拒否するのを助け、ディスクへのランダムI/Oを9,950回節約します！ EXPLAIN 句には Using index condition と表示されます。

Lessons Learned (得られた教訓)

  1. ハードウェアアーキテクチャがソフトウェア設計を決定する: SQLの抽象的な概念は、最終的にはL1キャッシュ、磁気ディスクの回転速度、PCIeアーキテクチャに直面しなければなりません。Covering Indexは、インデックスに追加の1〜2列を詰め込む（クエリをカバーするため）ことで、ディスクスペースをCPU速度と交換することが非常に価値のあるトレードオフであることを証明しています。
  2. 実行計画の監視と分析: 常に EXPLAIN FORMAT=JSON (MySQL) または EXPLAIN ANALYZE (PostgreSQL) を使用して、システムが Using index (Covering)、Using index condition (ICP)、または Using intersect/union (Index Merge) を使用しているかを確認します。
  3. 書き込みペナルティ (Write Penalty) を考慮する: CoveringやIndex Mergeをサポートするために多くのインデックスを作成することによる読み取りの最適化（Read optimization）は、INSERT/UPDATE/DELETE コマンドのパフォーマンスを低下させます。書き込みが激しいOLTP環境では、インデックスの数を厳密に制御する必要があります。
  4. CBO (Cost-Based Optimizer) の限界: オプティマイザは完璧ではありません。古くなったデータ統計（Table Statistics）により、CBOがIndex MergeとFull Table Scanの間で誤った選択をする可能性があります。これらのマイクロアーキテクチャ機能が機能するためには、ヒストグラムを適切に構成し、定期的なデータ分析を行うことが不可欠です。

Covering Indexes、ICP、およびIndex Mergeの統合は、巨大な処理パイプラインを形成します。優れたデータベースエンジニアは、正確なSQLを記述するだけでなく、プラッタ（platter）からシステムバスを通過し、L1キャッシュに入り、CPUコア内のSIMD命令セットによってデータがどのように形成されるかを想像できなければなりません。
