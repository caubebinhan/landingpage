Logical vs. Physical Replication – カーネルレベルのデータストリームの解剖 (詳細な技術レポート)

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー (Executive Summary)

分散データベース管理システム（RDBMS）と大規模なデータストレージアーキテクチャにおいて、高い一貫性と可用性を維持するには、ノード間で洗練されたデータ同期メカニズム（Replication）が必要です。業界では、物理レプリケーション (Physical Replication) と 論理レプリケーション (Logical Replication) という2つのアーキテクチャ派閥が絶対的な支配力を持っています。

この技術レポートはアプリケーションの表面にとどまらず、システムのスタックの最も低い階層（カーネルメモリ、NVMeディスク構造、CPUマイクロアーキテクチャ、TCP/IPネットワークプロトコル）に深く入り込み、プライマリノード (Primary) からターゲットノード (Replicas) へのデータ転送メカニズムを解読します。

問題の核心 (Problem Statement): ネットワークシステムをクラッシュさせたり、CPUリソースを完全に消費したりすることなく、毎秒数ギガバイトのデータをサーバー間で同期するにはどうすればよいでしょうか？ 物理レプリケーションを選択すると、極限の速度（Zero-copy DMA）を達成できますが、特定のオペレーティングシステムのバイナリ構造や特定のデータベースバージョンにロックインされます。論理レプリケーションを選択すると、最大の柔軟性（データフィルタリング、アーキテクチャの違いの許容）を得られますが、デコード (decoding) のための莫大な「計算税」を支払う必要があり、RAMのオーバーフロー（Out-of-Memory）のリスクが生じます。どのアーキテクチャを選択するかは、高負荷（High-load）環境におけるシステムの生死を決定します。

教訓と得られる知識 (Lessons Learned):

  1. Physical Replicationはパフォーマンスの王様です: CPUを完全にバイパス（Kernel Bypass）し、sendfile() または splice() を使用することで、物理レプリケーションはRAMからネットワークカードへバイトを直接$O(1)$のコストで転送します。
  2. Logical Replicationの計算税: データ構造をデコードするために、CPUは生のWALストリームを構文解析 (parsing) しなければなりません。このプロセスは、L1/L2キャッシュレベルでInstruction Cache Missesの爆発を引き起こし、システムに巨大なメモリ領域の割り当て（Reorder Buffer）を要求します。
  3. レプリケーションにおける待ち行列定理: 論理レプリケーションにおけるReplication lag（同期遅延）は、物理的に $M/M/1$ モデル（Single-threaded apply worker）に制限されており、マスターの書き込みが速すぎる場合、ボトルネックの悲劇を招きやすくなります。

Physical Replication: バイナリブロック同期の芸術 (Block-Level Synchronization)

物理レプリケーションは、ブロックレベルの同期の原則に基づいて動作します。メモリページ（8KBのmemory pages）または先行書き込みログ（Write-Ahead Logging – WAL）ファイルの正確なバイナリコンテンツが、ソースノードからターゲットノードにそのままコピーされます。

1 LSN (Log Sequence Number) の概念

物理アーキテクチャでは、システムは単調増加の識別子である LSN を使用します。これは、ログファイル内のレコードの絶対的な物理座標です。数学的には次のように表されます。

$$LSN_{i+1} = LSN_{i} + Delta_{size}(Record_{i}) + sigma(alignment)$$

ここで、$sigma(alignment)$ は、CPUのデータバス (Data bus) を最適化するための8バイトまたは16バイト境界でのバイト丸め係数です。LSN構造は、破ることのできない「時間の矢」（happens-before relationship）を作成します。レプリカ (Replica) ノードは単に、そのハードディスク上の正しいオフセット座標にこれらのバイトを適用 (apply) します。

2 Zero-copy Memory Transfer とカーネル通信

物理レプリケーションの絶対的な力は、User-Spaceを完全にバイパスする能力にあります。 従来のI/Oモデルでは、データは4回のコピー（Disk $rightarrow$ Kernel Cache $rightarrow$ User Buffer $rightarrow$ Socket Buffer）を経る必要があります。

Physical Replicationシステムは、Linux上のシステムコール sendfile() または splice() を介した Zero-copy (Direct Memory Access – DMA) 技術によってこれを最適化します。

  • WAL (Write-Ahead Log) ストリームは、DMAによってNVMe SSDからPage Cache (Kernel Space) に直接ロードされます。
  • sendfile() はネットワークカード (NIC) に命令を出し、Page Cacheからデータを直接吸い上げて光ファイバーに打ち出します。
  • CPUコアは完全にアイドリング状態です！ そのバイトの内容を読み取るために1クロックサイクル (clock cycle) も消費する必要はありません。

同期コミット遅延 (Synchronous Commit Latency) を計算する方程式は次のように要約されます。

$$T_{sync_commit} = T_{local_flush} + T_{network_RTT} + T_{remote_flush} + T_{ack}$$

（ハードディスクの遅延とネットワークのRTTにのみ依存し、CPUには依存しません）。

3 帯域幅制限と TCP プロトコル (BBR/CUBIC)

書き込み速度 (Write-throughput) が1000 MB/sを超えると、ボトルネックはSSDからTCP/IPスタックに移行します。パケットロス (Packet Loss) が発生した場合、Head-of-Line Blocking（キューの先頭でのブロック）という現象が現れます。

システムは、確認されていないWALセグメントを保持するために、リングバッファ (Ring Buffer – 例えば PostgreSQLにおける wal_keep_size) を割り当てる必要があります。安全な最小サイズは、帯域幅とネットワーク遅延の積 (BDP) によって制限されます。

$$BDP = C times RTT$$

レプリカがロードに間に合う前にRing Bufferがオーバーフローした場合、物理リンクは永久に切断され、レプリカは完全な再構築 (Full Re-sync) を余儀なくされます。

Logical Replication: デコードプロセスの力学 (Logical Decoding Pipeline)

Physical Replicationがバイトを盲目的に転送するのに対し、Logical Replicationは言語学者のようなものです。物理バイトブロックを遡り、解剖し、純粋なSQLステートメント (INSERT, UPDATE, DELETE) に逆翻訳します。

この機能は、PostgreSQLからMySQLに、x86_64サーバーからARMサーバーにレプリケーションしたい場合、または logs テーブルを無視して users テーブルのみをレプリケーションしたい場合に非常に重要です。

1 マイクロレベルでの計算税 (Computational Tax)

この柔軟性は、プロセッサのマイクロアーキテクチャレベルでの巨大な計算税を伴います。デコードアルゴリズム (Decoding) は数ギガバイトのWALストリームを処理する必要があるため、タプル構造を理解するためにMVCC (Multi-Version Concurrency Control) のメタデータにアクセスする必要があります。

CPUは数十億回の構文解析 (Parsing) の計算を実行しなければなりません。このデータは絶えず変化するため、L1/L2キャッシュ内のデータは継続的に押し出され（Cache Eviction）、Instruction & Data Cache Misses の爆発を引き起こします。CPUのパフォーマンス指標は急降下します。

2 トランザクション再構築の構造 (Reorder Buffer)

論理デコードにおける最も残酷なアルゴリズムの課題は Reorder Buffer です。 WALファイルでは、何千ものトランザクションのデータが混ざり合っています (Interleaved)。 例: Tx1_Start, Tx2_Start, Tx1_Insert, Tx2_Update, Tx1_Commit。

しかし、Logical Replicationは原子性の完全性 (ACID) を保証しなければなりません。 Commit コマンドを受け取る前に Tx1 の一部を送信してはなりません。 したがって、システムは Tx1 と Tx2 の完全なデータをRAMメモリ内に保持するための ステートレスハッシュマップ (in-memory hash map) を作成し、Commit を捉えるまで待機しなければなりません。

トランザクションの総量 (In-flight Transactions) が許容RAM制限を超えた場合（例：1000万件のレコード更新トランザクション）、メカニズムは強制的に Spill-to-Disk（ディスクへのスピル） アルゴリズムをトリガーします。

template <typename DataType>
class HighlyConcurrentReorderBuffer {
private:
    std::unordered_map<TransactionId, std::vector<LogicalTupleChange>> active_inflight_txns;
    std::atomic<size_t> current_memory_footprint{0};
    const size_t HARD_MEMORY_LIMIT = 1024 * 1024 * 512; // 512 MB Threshold

    // システムをOut-Of-Memoryの災害からSpill-to-Diskで救う
    void evict_to_disk_spill(TransactionId victim_xid) {
        int fd = create_anonymous_temp_file(victim_xid);
        size_t stream_size = active_inflight_txns[victim_xid].size() * sizeof(LogicalTupleChange);
        
        // Zero-copy Mapped I/Oを割り当てる
        void* virtual_mapped_mem = mmap(nullptr, stream_size, PROT_WRITE, MAP_SHARED, fd, 0);
        memcpy(virtual_mapped_mem, active_inflight_txns[victim_xid].data(), stream_size);
        
        // ディスクへのフラッシュを強制する
        msync(virtual_mapped_mem, stream_size, MS_ASYNC);
        active_inflight_txns[victim_xid].clear();
        munmap(virtual_mapped_mem, stream_size);
        current_memory_footprint.fetch_sub(stream_size, std::memory_order_release);
    }
};

真の災害は COMMIT コマンドの時に発生します。 システムはRAM上に残っているデータと磁気ディスク上に散らかっているガベージチャンクの間で External Merge Sort を実行しなければなりません。 ハードドライブは非常に高いランダム読み書きI/Oサイクル (Random I/O) によって酷使されます。

Replication Lagの待ち行列理論 (Queuing Theory)

論理レプリケーションのターゲットノード (Replica) における最大の問題は、シングルスレッドの障壁 (Single-threaded Bottleneck) です。 Primaryノードは64個のCPUコアで同時にデータを書き込みます (Write)。しかし、Replicaノードは外部キー (Foreign Keys) の完全性制約を保護するために、論理的なSQLコマンドを再実行するために たった1つのCPUコア (Single-threaded Apply Worker) しか使用しません。

待ち行列理論 (M/M/1) の ポラツェク＝ヒンチン (Pollaczek–Khinchine) 定理によると、キューの長さ $L_q$（つまりReplication Lag）は次のようになります。

$$L_q = frac{rho^2 + rho^2 C_s^2}{2(1 – rho)} quad text{ここで負荷変数} quad rho = frac{lambda_{total}}{mu_{apply}}$$

$lambda_{total}$ (Primaryのデータ送信速度) が $mu_{apply}$ (ReplicaのSQL実行速度) に近づくと、$rho to 1$ になります。 その場合、遅延 $L_q$ は無限大に近づきます。 これは越えられない物理的障壁であり、Primaryに書き込みを停止させるバックプレッシャー (Backpressure) 信号を設定しない限り、データが何時間もドリフト (Drift) することになります。

一方、物理レプリケーションはこの問題を非常に見事に解決します。 SQLや外部キーに関心がないため、物理WALプロセッサはMemory Pageの変更コマンドを何十もの異なるWorker Threadに分散させます。 各ThreadはRAM上の8KB Blockを独立して変更します。 適切なメモリバリア (Memory Barriers) を使用し（NUMA False Sharingを回避する限り）、物理レプリケーションはターゲットノードで完璧に水平方向の拡張 (Scale-out) が可能です。

結論

論理レプリケーションと物理レプリケーションは、コンピュータエンジニアリングにおける2つの偉大な哲学を表しています。

  • Logical Replication は無限の自由を提供します。 データを硬直したハードウェアから分離し、フォーマットを解放し、ハイブリッドクラウド (Hybrid Cloud)、CDC (Change Data Capture)、およびStreamingの時代を切り開きます。 しかしその代償として、システムのエントロピー、CPUサイクルの消費、そしてシングルスレッドキューモデルの遅延の災害が発生します。
  • Physical Replication は物理的なブロックボリュームに盲目的に忠実です。 その代わり、原始的な信頼性、比類のないZero-copy転送I/Oパフォーマンス、およびハードディスクの帯域幅に近づく速度での災害復旧能力を提供します。

これら2つのモデルの力学を徹底的に理解することは、毎秒数百万のトランザクションを処理するデータベースクラスターを運用するための前提条件です。
