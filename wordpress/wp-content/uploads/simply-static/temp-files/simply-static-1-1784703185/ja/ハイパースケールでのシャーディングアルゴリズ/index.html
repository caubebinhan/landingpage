ハイパースケール時代のシャーディングアルゴリズム — 数学、データ構造、マイクロアーキテクチャ

July 22, 2026 · admin · 3 分で読めます

なぜシャーディングが避けられなくなるのか

データ量がペタバイトの領域に達し、トラフィックが秒間数百万トランザクションに近づくと、単一ノードのデータベースはどうやっても行き詰まる。インデックスを工夫しても、ディスクを大きくしても解決しない。1台のディスクと1個のCPUという物理的な制約そのものが上限を決めているからだ。ここに至って初めて、シャーディング（データを複数の独立したノードに分割すること） は「最適化の一手段」ではなく「唯一の選択肢」になる。

シャーディングを「テーブルを機械的に分割するだけ」と捉えるのは危険な誤解だ。実際にうまく機能させようとすると、離散数学、グラフ理論、そしてかなり低レベルなハードウェア工学にまで踏み込むことになる。この記事では、実運用でくり返し登場する3つのシャーディングモデル——ハッシュベースシャーディング、レンジベースシャーディング、ディレクトリベースシャーディング——を、単なる抽象アルゴリズムとしてではなく、CPUキャッシュの挙動、メモリアライメント、SIMD命令、OSレベルのI/Oの現実まで踏まえて見ていく。

核心にある問題は単純に見える。データを保持する物理サーバーへクエリを正しくルーティングすること、それだけだ。しかし方式ごとに弱点は明確に異なる。ハッシュベースはノードが1台落ちた瞬間にリバランシングの嵐が起き、ネットワーク帯域を食い尽くす。レンジベースは範囲スキャンには強いが、書き込みが「今」を担当する1つのシャードに集中し、そのディスクだけが焼け付き、隣のシャードは暇を持て余す。ディレクトリベースは間接層を挟むことでこの両方を回避するが、その間接層自体が新たなボトルネックとレイテンシ問題を抱え込む。本稿が答えようとしている問いは、CPUレベルでのルーティングコストを数十ナノ秒に保ったまま、この分散問題をどう解くかということだ。

本稿で扱う内容:

  1. コンシステントハッシュはデータ移動量を概ね $O(1/N)$ に抑えるが、それだけでは負荷分布が偏る。仮想ノードは選択肢ではなく必須要件だ。
  2. ルーティング速度を左右するのはキャッシュラインの挙動である。連結リストで実装されたシャードルーターはL1/L2キャッシュミスで性能を失う。連続かつ64バイトアライメントされた配列以外に現実的な選択肢はない。
  3. 負荷がかかった状態でのレンジシャード分割は分散システム設計の難所の一つで、カーネルバイパス、ゼロコピーI/O、分離されたスナップショットが要求される。これを怠るとサービスが分割の途中で落ちる。
  4. ディレクトリベースシャーディングは中央ボトルネックをクライアント側キャッシュで解決するが、そのキャッシュにはリース機構が必要であり、ストレージノード側もリース切れのクライアントを遮断（フェンシング）する仕組みを持たねばならない。

ハッシュベースシャーディング: コンシステントハッシュリングとCPUのマイクロダイナミクス

ハッシュベースシャーディングは、レコードのキー（$k$）をハッシュ関数にかけ、その結果をサーバーにマッピングする。単純な実装である $S(k) = h(k) pmod N$ は書くのは簡単だが、本番環境では厄介な代物だ。サーバーを1台追加して $N$ を $N+1$ に変えるだけで、キーの約99%が以前とは違うノードに割り当てられてしまう。これがリバランシングストームを引き起こし、東西方向のネットワーク帯域を数時間にわたって圧迫することもある。

1 コンシステントハッシュと仮想ノード

David Kargerのコンシステントハッシュは、このマッピングの考え方そのものを作り直すことで問題を解決した。剰余演算の代わりに、識別子空間をリング状に丸める——たとえば $0$ から $2^{160}-1$ まで。

  • 各サーバーノードはハッシュ化され、リング上の座標を持つ。
  • 各キーも同様にハッシュ化され、時計回りに進んで最初に出会うノードに割り当てられる。

ノードが1台故障しても、そのノードだけが所有していたデータのみが隣のノードへ移動すればよい。データ移動量は理論上の最小値 $approx frac{1}{N}$ に近づく。

問題は、ハッシュの配置がランダムであるため、たまたま1台のサーバーがリングの半分近くを担ってしまうことがある点だ——これが実際の負荷不均衡につながる。解決策は仮想ノードである。物理ノードを1回だけハッシュ化するのではなく、異なるサフィックスを付けて何千回もハッシュ化する: $V(node_i) = {h(node_i parallel j) mid j in [1, v]}$。十分な数の仮想レプリカをリング上に散らせば、分布はほぼ均一になる。

2 リングを高速化する: 連続配列とSIMD

ここで理論とハードウェアの現実がぶつかる。リングを（自然な発想である）循環連結リストで実装すると、キャッシュミスの代償を払うことになる。C++やRustでは、ヒープ上に散らばったポインタを辿るたびにCPUの命令パイプラインが止まってしまう。

解決策は、リングを静的でソート済みの連続配列に平坦化することだ。

  • 各要素のサイズとアライメントを、CPUのL1キャッシュライン（64バイト）に正確に一致させる。
  • 各エントリは16バイト（ハッシュ8バイト、ノード識別子へのポインタ8バイト）。
  • これにより、RAMから読み込まれる1つのキャッシュラインに4つの仮想ノードが同時に収まる。

この配列に対する二分探索は $O(log(N times v))$ で済む。これに、MurmurHash3やCityHashのような高速な非暗号ハッシュ関数と、コンパイラが可能な範囲で活用するSIMD（AVX-512）を組み合わせれば、大量のキーストリームのルーティングも1キーあたり数ナノ秒で処理できる。

// メモリを圧縮した静的ハッシュリング構造を示す低レベルのRustソースコード
use std::hash::{Hash, Hasher};
use fasthash::murmur3::Murmur3Hasher_x64_128;

pub struct ConsistentHashRing {
    // 連続して割り当てられたベクター配列 (Contiguous Memory Allocation)
    ring: Vec<(u64, String)>,
    virtual_nodes: usize,
}

impl ConsistentHashRing {
    pub fn new(virtual_nodes: usize) -> Self {
        ConsistentHashRing {
            ring: Vec::with_capacity(10_000), // ヒープの断片化を防ぐ
            virtual_nodes,
        }
    }

    pub fn add_node(&mut self, node_id: &str) {
        for i in 0..self.virtual_nodes {
            let virtual_key = format!("{}#{}", node_id, i);
            let mut hasher = Murmur3Hasher_x64_128::default();
            virtual_key.hash(&mut hasher);
            self.ring.push((hasher.finish(), node_id.to_string()));
        }
        // ミクロレベルの二分探索に備えて静的にソート
        self.ring.sort_unstable_by(|a, b| a.0.cmp(&b.0));
    }

    pub fn get_node(&self, key: &str) -> Option<String> {
        let mut hasher = Murmur3Hasher_x64_128::default();
        key.hash(&mut hasher);
        let hash_val = hasher.finish();

        // Hardware Branch Predictor はこの二分探索ループを極めて正確に予測する
        match self.ring.binary_search_by(|probe| probe.0.cmp(&hash_val)) {
            Ok(idx) => Some(self.ring[idx].1.clone()),
            Err(idx) => {
                let wrapped_idx = if idx == self.ring.len() { 0 } else { idx };
                Some(self.ring[wrapped_idx].1.clone())
            }
        }
    }
}

レンジベースシャーディング: 動的リバランスとI/Oホットスポット

ハッシュシャーディングの最大の弱点は、データの局所性を完全に失うことだ。SELECT * WHERE id BETWEEN 10 AND 50 のようなクエリは、キーの近さと物理配置の間に何の関係もないため、全シャードへのスキャッター・ギャザーを強いられる。

レンジベースシャーディングはこれとは逆のアプローチを取る。キー空間 $mathcal{K}$ を、隣接し重複しない範囲 $R_i = [K_{i, min}, K_{i, max})$ に分割する。範囲クエリはクラスタ全体ではなく、隣接する1つか2つのシャードだけにルーティングされ、ネットワークコストの面で大きな利点となる。

1 データホットスポット問題

その範囲スキャンの効率性には代償が伴う——ホットスポットだ。シャードキーがタイムスタンプの場合、システム内のすべての新規INSERTは「現在」の時間帯を担当する単一のシャードに集中する。そのディスクは書き込み負荷を100%背負う一方、昨日のデータを持つシャードはほとんど遊んでいる状態になる。これに対する本質的な解決策は、シャードがボトルネックになる前に、実行時に動的に分割することしかない。

2 シャード分割の実際の仕組み

シャード $R_0 = [K_{min}, K_{max})$ がサイズの上限を超えて成長すると、ある分割点 $K_{split}$ で二つに分けられる。

  • 新しいシャード $R_{new} = [K_{split}, K_{max})$ が、新しいサーバー上に作られる。
  • 古いシャードは $R_{old} = [K_{min}, K_{split})$ に縮小する。

難しいのは、これを書き込みをブロックせずに行うことだ——数百GBのコピーがダウンタイムの原因になってはならない。実務では、MVCCとファイルシステム自体のCopy-on-Write機構を組み合わせて対応する。

  • カーネルレベルでは、LinuxがCopy-on-Write（ZFSやBtrfsがネイティブに対応）によって、瞬時に、物理バイトを一切コピーせずにSSTableファイルのスナップショットを取得する。
  • バックグラウンドのコンパクションプロセスが、フォアグラウンドのI/Oを飢えさせない程度のペースで、そのデータをネットワーク経由で新しいサーバーへ転送する。
  • その間、コピーがまだ進行中の $R_{new}$ のキー範囲に対する書き込みは、キャッチアップログバッファに記録される。
  • 一括コピーが完了すると、ルーターはおよそ1マイクロ秒でトラフィックを切り替え、キャッチアップログを新しいサーバーに適用し、PaxosやRaftによってクラスタ全体で分割成功を確定させる。
// C++ による超高度な範囲ルーティング構造 
struct alignas(64) RangeShard {
    std::string min_key;
    std::string max_key;
    std::string node_endpoint;

    // CPU 向けの Branchless ロジック技術
    inline bool contains(const std::string& key) const noexcept {
        return key >= min_key && key < max_key;
    }
};

class ZeroCopyRangeRouter {
    std::vector<RangeShard> shards;
public:
    std::string route_point_query(const std::string& key) const {
        // SIMD 最適化された lower_bound が事前にキャッシュにロードされる
        auto it = std::lower_bound(shards.begin(), shards.end(), key, 
            [](const RangeShard& s, const std::string& k) { return s.max_key <= k; });

        if (it != shards.end() && it->contains(key)) [[likely]] {
            return it->node_endpoint;
        }
        throw std::runtime_error("Consistency check failed.");
    }
};

ディレクトリベースシャーディング: 中央ルックアップとレイテンシのコスト

ディレクトリベースシャーディングは、抽象化を論理的な極限まで押し進めたモデルだ。キーからサーバーへのマッピングを表す数式は存在せず、ただのルックアップテーブルがある: $S(k) = DirectoryLookup(k)$。

  • 利点は実質的な柔軟性だ。どのテナントのどのデータでも、固定のハッシュ式や範囲式に縛られることなく、いつでもどのノードにも移動できる。Google SpannerやFoundationDBは、いずれもこのモデルの変種を採用している。
  • 一方で弱点も同じくらい実質的だ。このルックアップテーブルが単一の競合ポイントになる。システム内のすべてのクエリが、ZooKeeperやetcdのようなディレクトリサービスにルーティング先を尋ねる必要がある。

1 一貫性を壊さずにディレクトリをキャッシュする

ディレクトリサービスへのRPC往復には数十ミリ秒かかる。たまにであれば問題ないが、クエリのたびに行うのでは現実的ではない。素直な解決策は、ルーティングテーブルをクライアント自身のメモリにキャッシュすることだ。

だがこれは本質的に厄介な一貫性問題を生む。シャードが移動した際にクライアントのキャッシュがまだそれを知らなければ、誤ったサーバーへ書き込みを送ってしまい、それが既に移動したデータを静かに壊す恐れがある。

解決策はリース付きキャッシュだ:

  1. ディレクトリサーバーは、クライアントにルーティングテーブルとリーストークンを渡す。有効期間は $Delta t$（典型的には5秒程度）に区切られている。
  2. クライアントはリースが有効な間、シャードサーバーと直接自由にやり取りしてよい。条件は $T_{current} + epsilon < T_{issued} + Delta t$ であり、$epsilon$ はマシン間のNTPクロックスキューを吸収する項だ。
  3. 本当の安全装置はストレージノード側にある。ストレージノード自身もディレクトリセンターからリースを受け取っており、そのリースが切れた場合（ケーブル断線、ネットワーク分断など）、ストレージノードは自らをフェンシングする。NVMeのI/O提供を止め、すべての受信リクエストを拒否し、409 Conflict を返す。
  4. この自己フェンシングの挙動こそが、実際にデータ整合性を守っている部分だ。古いルーティング情報を持ったクライアント（クロックが同期していないゾンビのような存在）がいくら書き込みを送っても、ストレージノードは自らのリースが切れた時点でそれを拒否する。これによって、誤ルーティングされた「汚れた」書き込みがあってもACID保証が保たれる。

3つのモデルをどう選ぶか

すべての場面で勝つ単一のシャーディングアルゴリズムは存在しない。

  • ハッシュベースシャーディングはCPUとメモリの性能を最大限に引き出し、うまくチューニングされたハッシュリングは負荷不均衡にほぼ強い。ただし範囲スキャン性能は完全に犠牲になる。
  • レンジベースシャーディングはデータを物理的に順序立てて保持するため、大規模な分析スキャンには有利だが、ストレージ層で継続的な——時に骨の折れる——シャード分割作業を引き受けることになる。
  • ディレクトリベースシャーディングは、どんな理由であれデータをどこへでも動かせる自由を与えてくれるが、その代償としてクライアント側キャッシュ、ディレクトリリース、各ストレージノードでのフェンシングロジックという、本質的に複雑な支援体制が必要になる。

これらのアルゴリズムをキャッシュラインやレジスタのレベルまで理解することは、決して学術的な自己満足ではない。深夜3時にノードが1台落ちた瞬間に破綻するシャーディング層と、負荷の下で緩やかに劣化していくシャーディング層とを分けるのは、まさにこの理解の深さである。
