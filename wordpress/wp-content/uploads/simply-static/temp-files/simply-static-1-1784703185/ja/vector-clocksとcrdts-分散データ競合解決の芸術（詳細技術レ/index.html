Vector ClocksとCRDTs – 分散データ競合解決の芸術（詳細技術レポート）

July 22, 2026 · admin · 4 分で読めます

エグゼクティブサマリー (Executive Summary)

グローバル規模のフォールトトレラントなクラウドストレージシステムを構築する競争において、Amazon DynamoDB（オリジナルアーキテクチャ）やBasho Riakなどの先駆的なプラットフォームは、絶対的な可用性（Availability、99.999%のアップタイム基準を達成）を得るために、強い一貫性（Strong Consistency）を自発的に放棄しました。この構造的な決定により、システムは分散サーバーが独立かつ非同期に書き込みデータを受け取ることを許容する、結果整合性（Eventual Consistency）の領域へと導かれます。

しかし、この自由は致命的なリスクを生み出します：データ状態の競合（Data Conflicts）。もし2人のユーザーが、海底光ケーブルが切断されている間に2つの異なるデータセンター（例：米国と欧州）で同じドキュメントを編集した場合、ネットワークが再接続されたとき、システムはどちらのバージョンが正しいかをどのようにして知るのでしょうか？

この技術レポートは、この問題を解決するための2つの究極の数学的ツールであるVector Clocks（ベクタークロック）とCRDTs（競合なしレプリケーションデータ型）について深く分析します。Leslie Lamportの「Happens-before」理論からハードウェアのマイクロアーキテクチャまでを探求することで、このドキュメントはデータベースがどのように遅延と分岐（Divergence）を処理するかについての包括的なレンズを提供します。

問題提起 (Problem Statement): 中央集権的なロックシステム（distributed locks）に依存することなく、分散ネットワーク上でデータの完全性と因果律（causality）の順序をどのように維持するか？ロックシステムはパフォーマンスのボトルネックを引き起こします。データが安全に分岐し、その後自動的かつ決定論的に、そして情報の損失なくマージ（merge）されることを可能にする分散アルゴリズムが必要です。

教訓と得られた知識 (Lessons Learned):

  1. 時間は相対的である: 物理的な時計（wall-clock）に頼って、どのイベントが先に発生したかを判断することはできません。Vector Clocksは完璧な「論理時間（Logical Time）」の参照フレームを提供します。
  2. アプリケーションへの委任 vs. 自動化: Vector Clocksは競合を検出するだけで、解決の責任をアプリケーション層（Application Layer）に押し付けます。対照的に、CRDTsは、交換則、結合則、冪等則という数学的性質のおかげで、すべての競合を自動的にマージする巨大な前進です。
  3. ガベージの圧力（Garbage）と状態空間のリーク: 分散アルゴリズムには高いメモリコストがかかります。CRDTsのTombstones（墓石）マーカーやVector Clocksのリストは、洗練された剪定（Pruning/Garbage Collection）メカニズムがないと無限に膨張する可能性があります。

因果関係とVector Clocksの基礎

競合を管理するためには、システムはイベントの順序を理解しなければなりません。1978年、Leslie Lamportは$rightarrow$と表記される「先行発生」（Happens-Before）関係を定義しました。

  • イベント$a$が$b$の前に発生する場合（$a rightarrow b$）、$a$から$b$への情報の流れ（または因果関係）が存在します。
  • $a rightarrow b$でも$b rightarrow a$でもない場合、2つのイベントは並行（Concurrent）と呼ばれ、$a parallel b$と表記されます。まさにこの時に競合が発生します！

1 Vector Clocksの動作メカニズム

$N$個のノードを持つシステムにおけるVector Clockは、$N$個の整数からなるベクトル$V$です。 数学的な更新ルール：

  1. 初期状態では、すべての$V_i[j] = 0$。
  2. イベント（データの書き込み）を実行する前に、ノード$i$はローカル変数をインクリメントします：$V_i[i] leftarrow V_i[i] + 1$。
  3. メッセージを送信する際、ノードは$V_i$を添付します。
  4. メッセージを受信すると、ノード$j$は自身のベクトルと受信したベクトルをMax演算によってマージします：$forall k, V_j[k] leftarrow max(V_j[k], V_m[k])$。その後、$V_j[j]$をインクリメントします。

競合の検出方法: （ベクトル$V_A$と$V_B$を持つ）2つのデータバージョン$A$と$B$を比較するには：

  • もし$forall k, V_A[k] le V_B[k]$：バージョン$A$は$B$の祖先です。システムは安全に$A$を$B$で上書きします。
  • もし$V_A notle V_B$ かつ $V_B notle V_A$：2つのバージョンは並行（分岐）しています。システム（Dynamoなど）は両方のバージョン（Siblingsと呼ばれる）を保存し、クライアントに返します。クライアントは、それらをどのようにマージするかを自身で決定しなければなりません（例：Amazonのショッピングカートのマージ）。

2 状態空間の枯渇 (State Space Explosion)

Vector Clocksの最大の欠点は、構造の膨張です。数千台のサーバーからなるクラスターでは、Vector Clockは数千の要素の長さになります。この巨大なメタデータをすべての読み取り/書き込み操作に付与すると、ネットワーク帯域幅が破壊され、CPUキャッシュが粉砕されます。 これを克服するために、AmazonはPruning Algorithm（剪定）を適用します：Vector Clockは(NodeID, Counter, Timestamp)のリスト形式に圧縮されます。許容数を超えると、最も古いNodeIDが切り捨てられます。しかし、これによりFalse Positives（偽陽性）が発生します – 因果履歴が失われるため、システムは2つの順次レコードを競合と誤認します。

CRDTs: 自動数学的マージの芸術 (Basho Riak)

Vector Clockが競合解決の責任をアプリケーション層に「丸投げ」するのに対し、CRDT (Conflict-free Replicated Data Type)はそれをデータベース層で根本的に解決します。これはBasho Riakにおいてコア技術として適用されています。

分散代数の観点から、CvRDT（状態ベース）タイプのCRDTは、マージ関数$m(x,y)$または$x sqcup y$が結び半束（Join-semilattice）を形成することを要求します。この関数は、3つの厳しい数学的法則に従わなければなりません：

  1. 交換則 (Commutativity): $x sqcup y = y sqcup x$。ネットワークパケットがどの順序で到着しても重要ではなく、結果は同じです。
  2. 結合則 (Associativity): $(x sqcup y) sqcup z = x sqcup (y sqcup z)$。データがどのルーティングの分岐点を通過しても、一貫して収束します。
  3. 冪等則 (Idempotency): $x sqcup x = x$。ネットワークがループし（TCP再送）、同じパケットを100回送信しても、システムはデータ破損を起こしません。

1 OR-Set (Observed-Remove Set) 構造の解剖

異なる大陸で複数の人が要素$E$を追加（Add）および削除（Remove）しても競合が発生しないSet（集合）構造をどのように設計するのでしょうか？

CRDTはこれをOR-Setで解決します。要素$E$を追加するたびに、システムは単に$E$を保存するだけでなく、それに一意のUUID (Tag)を付与します。状態は(E, Tag)のペアの集合です。

  • 米国がEを追加: (E, tag_US)
  • 欧州がEを追加: (E, tag_EU)

    • 同期時、システムは次のように収束します：{ (E, tag_US), (E, tag_EU) }。要素$E$は存在します。

もし欧州が$E$を削除しても、メモリから物理的に削除するわけではありません。(E, tag_EU)を墓石の集合（Tombstone Set）に押し込みます。 米国と再度同期すると、システムは(E, tag_EU)が削除マークされていることを確認しますが、(E, tag_US)はまだ生き残っています！結果：$E$は引き続き画面に表示されます。「ファントム（幻影）」現象は、識別子の厳格な論理によって完全に排除されます。

#include <iostream>
#include <set>
#include <string>
#include <algorithm>

// 低レベルOR-Set CRDTアーキテクチャのシミュレーション
struct Element {
    std::string value;
    std::string tag;
    bool operator<(const Element& other) const {
        if (value != other.value) return value < other.value;
        return tag < other.tag;
    }
};

class ORSet {
private:
    std::set<Element> add_set;
    std::set<Element> tombstone_set; // 墓石の集合

    std::string generate_tag() {
        static int counter = 0;
        return "uuid_" + std::to_string(++counter);
    }

public:
    void add(const std::string& val) {
        add_set.insert({val, generate_tag()});
    }

    void remove(const std::string& val) {
        // 観察されているすべてのタグをtombstoneに移動
        for (const auto& elem : add_set) {
            if (elem.value == val) tombstone_set.insert(elem);
        }
    }

    bool contains(const std::string& val) const {
        for (const auto& elem : add_set) {
            // add_set内に存在し、かつtombstone_setに入っていない場合に存在する
            if (elem.value == val && tombstone_set.find(elem) == tombstone_set.end()) {
                return true;
            }
        }
        return false;
    }

    // 結び半束 (Join Semilattice) 演算: 冪等則、交換則、結合則
    void merge(const ORSet& other) {
        std::set<Element> new_add;
        std::set_union(add_set.begin(), add_set.end(),
                       other.add_set.begin(), other.add_set.end(),
                       std::inserter(new_add, new_add.begin()));
        add_set = new_add;

        std::set<Element> new_tomb;
        std::set_union(tombstone_set.begin(), tombstone_set.end(),
                       other.tombstone_set.begin(), other.tombstone_set.end(),
                       std::inserter(new_tomb, new_tomb.begin()));
        tombstone_set = new_tomb;
    }
};

マイクロアーキテクチャ、メモリ管理、LSM-Treeの最適化

CRDTの理論は完璧ですが、物理ハードウェア上で実行すると、キャッシュ（Cache）やガベージコレクション（Garbage Collection）に関する悪夢を引き起こします。

1 NUMAファイアウォールとFalse Sharing

マルチコアNUMA環境では、数百のスレッドが毎秒数千のトランザクションを処理します。スレッドがCRDTにデータを書き込む際、CRDTの巨大なサイズ（多数のTombstoneとVector Clockメタデータを含む）はしばしば64バイト（標準的なL1 Cache Lineサイズ）を超えます。 1つの書き込み操作が他のすべてのCPUのキャッシュラインを無効化（invalidate）し、最悪のFalse SharingとCache Line Contentionを引き起こします。

解決策は、Thread-Local State Shards（スレッドローカル状態シャード）を使用することです。各CPUスレッドは、自身のローカルRAM上に独立したCRDTを保持します。スレッドはメモリを共有せず、Mutex Lockも使用しません。サイクル終了時（例：5msごと）に、バックグラウンドスレッドがすべてのローカルシャードをスキャンし、極めて高速なアトミック命令（Compare-and-Swap – CAS）を使用してCRDTのマージコマンド$m(x,y)$を呼び出します。

2 LSM-TreeのBlock I/O層へのCRDTマージ関数の統合

OR-Set CRDTの最大の欠点は、Tombstone（墓石の集合）タグが削除されないことです。放置すれば、データベースはRAM/Diskのオーバーフローでクラッシュします。 Riakのアーキテクチャでは、驚異的な解決策が設計されました：CRDTのマージ関数をLog-Structured Merge-tree (LSM-tree)の物理ストレージ層の深部に押し込むのです。

LSM-tree（LevelDB、RocksDBなど）は、定期的にCompaction（ハードドライブ上のSSTableファイルのマージと圧縮）を実行します。コンパクター（Compactor）が2つのSSTableファイルをスキャンする際、単に新しいバージョンで古いバージョンを上書きするのではなく、CRDTの$sqcup$ (Join) 関数が埋め込まれます。 このI/O Block層の深いスキャン中に、Compactorは全体像を観察する能力を持ちます。Tombstoneがクラスター内のすべての物理ノードに安全に同期されたことを確認した場合（Epoch-based Reclamationを使用）、NVMeなどの磁気ディスクからそのTombstoneを永久に蒸発させます。このメカニズムは、トランザクションを処理するCPUスレッドを中断させることなく、ストレージ容量を透過的に回復させます（Zero-overhead Garbage Collection）。

結論

Vector ClocksとCRDTsは、純粋な抽象的代数定理が強力なハードウェアマシンに組み込まれる、システム設計の芸術に対する偉大な証拠です。

これらはCAP定理の限界を打ち破り、ノイズの多いネットワーク環境で真実の断片が無数の方向に投げられても、最終的には常に無傷で完璧な全体像に収束するようなスーパーデータベースを人類が運用することを可能にします。このメカニズムを理解することは、最新のデッドスポットのないCloud-Nativeシステムを運用するための鍵です。
