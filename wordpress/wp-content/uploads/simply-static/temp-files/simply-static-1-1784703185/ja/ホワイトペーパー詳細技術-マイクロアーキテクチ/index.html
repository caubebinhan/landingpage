ホワイトペーパー詳細技術: マイクロアーキテクチャレベルからのJoinアルゴリズムの解剖（Nested Loop、Hash Join、およびSort-Merge Join）

July 22, 2026 · admin · 4 分で読めます

エグゼクティブサマリー (Executive Summary)

リレーショナルデータベース理論において、結合演算子（join operator）は最もコストのかかる操作の1つであり、クエリ実行プロセスにおいて中心的な役割を果たします。Joinアルゴリズムの設計と選択は、単なる時間計算量（$O(N)$）の問題ではなく、CPUクロックサイクル、キャッシュメモリ（Cache）の各バイト、およびストレージ帯域幅をめぐる戦いでもあります。

このホワイトペーパーでは、結合アルゴリズムの3つの柱である Nested Loop Join、Hash Join、Sort-Merge Join について詳細な解剖を行います。基本的な教科書の概念を超えて、Radix Partitioning、SIMD Vectorization命令セット（AVX-512）、NUMAアーキテクチャの影響、オペレーティングシステムのメモリ管理メカニズム（Huge Pages）、Bloom Filterを使用したストレージへの処理オフロード技術（Storage Offloading）など、マイクロアーキテクチャ（Micro-architecture）レベルの最新の最適化手法に深く踏み込みます。

このホワイトペーパーを読み終えると、ハードウェアレベルでのjoinアルゴリズムの動作原理を習得し、クエリが突然何百倍も遅くなる理由を明確に理解し、非常に価値のあるシステム設計の教訓を得ることができるでしょう。

はじめに：Joinの本質と問題 (The Core Problem)

問題とは何か？ 2つの巨大なデータテーブル、Orders（10億行）とCustomers（1000万行）があると想像してください。「ハノイ（Hanoi）」の顧客に属するすべての注文を見つける必要があります。数学的に、これはフィルタと組み合わせたデカルト積（Cartesian product）です。ナイーブな方法で行うと、$1,000,000,000 times 10,000,000 = 10^{16}$回の比較を実行する必要があります。効率的な結合アルゴリズムがなければ、このワークロードはどんなスーパーコンピュータをもクラッシュさせるのに十分です。

joinアルゴリズムのパフォーマンスのボトルネックは時間とともに移行してきました：

  • 磁気ディスク（HDD）時代: I/Oコスト（Seek time）が絶対的なボトルネックでした。最適なアルゴリズムは、ディスクの読み取り回数を最小限に抑えるアルゴリズムです。
  • インメモリDB & SSD NVMe時代: データが完全にRAMまたは超高速SSD（数百万IOPS）上にある場合、CPUとメモリ帯域幅（Memory Bandwidth）がボトルネックになります。キャッシュミス（Cache Miss）とTLBミス（TLB Miss）こそがパフォーマンスを破壊する真の要因です。

したがって、現代のデータベース管理システム（RDBMSおよびOLAPエンジン）は、ハードウェアの物理構造に「適応」するために、Joinアルゴリズムを進化させ続ける必要があります。

理論的基礎と漸近計算量 (Theoretical Foundations)

2つのリレーション$R$（outer/build relation）と$S$（inner/probe relation）を考え、それぞれのタプル数を$|R|$および$|S|$とします。

Nested Loop Join (NLJ) とそのバリアント

最も基本的な形式は、単一のレコードを反復処理する2つのネストされたループを設定することです。

  • Tuple-centric Nested Loop Join: 計算量 $mathcal{O}(|R| times |S|)$。数学的にも実践的にも最悪です。
  • Block Nested Loop Join (BNLJ): $R$と$S$のブロック（blocks/pages）をRAMにロードすることで、I/Oアクセスモデルを改善します。利用可能なRAMページ数を$M$とした場合、I/Oコストは $mathcal{C}_{I/O} = P_R + lceil frac{P_R}{M-2} rceil times P_S$ に減少します。BNLJはボトルネックをディスクの回転遅延からメモリ帯域幅に変更します。
  • Index Nested Loop Join (INLJ): $S$のB+ Treeインデックスを活用し、$S$のフルスキャンを排除します。計算量は $mathcal{O}(|R| log_b |S|)$ に減少します。これは$R$が事前に小さくフィルタリングされるOLTPシステムのゴールドスタンダードです。

Hash Join の時代

結合述語が等価（equi-join）である場合、Hash Join が期待時間計算量 $mathcal{O}(|R| + |S|)$ で台頭します。アルゴリズムは2つのフェーズに分かれます：

  1. Build Phase: $R$をスキャンし、ハッシュ関数$h(k)$を使用してレコードをRAM上のハッシュテーブル（Hash Table）に配置します。
  2. Probe Phase: $S$をスキャンし、ハッシュ関数$h(k)$を使用してハッシュテーブル内の一致を検索します。

古典的な Hash Join の問題: ハッシュテーブルがRAMに収まる場合は完全に機能します。RAMから溢れると、オペレーティングシステムはディスクへのスワップ（ページング）を行い、「スラッシング（Thrashing）」（ディスク上のランダムアクセス）を引き起こし、パフォーマンスが劇的に低下します。

これを解決するために、Grace Hash Join が誕生しました。これは、$R$と$S$の両方をハッシュ関数$h_1(k)$でディスク上の$k$個のパーティションに分割し、各パーティション$R_i$がRAMにぴったり収まるようにします。その後、ペア$(R_i, S_i)$をロードして結合します。I/Oコスト: $mathcal{C}_{I/O} = 3(P_R + P_S)$（パーティション分割のための読み取り、ディスクへの書き込み、結合のための再読み取り）。

Sort-Merge Join (SMJ)

SMJはハッシュに依存せず、ソートの特性を活用します。データが結合キーで事前にインデックス付けされている場合に非常に効率的です。

  1. Sort Phase: データが大きい場合、External Merge Sortを使用します。計算量: $mathcal{O}(|R| log_M |R| + |S| log_M |S|)$。
  2. Merge Phase: $R$と$S$に沿ってスキャンする2つのストリームポインタを配置します。線形計算量: $mathcal{O}(|R| + |S|)$。

SMJの大きな利点はシーケンシャルモデル（Sequential I/O）です。ランダムアクセス（Random Access）が100%排除されるため、HDDやRAMが制限された構成に最適です。

マイクロアーキテクチャの最適化とハードウェアの認識 (Hardware-Aware Implementations)

インメモリデータベース（SAP HANA、MemSQLなど）への移行は、ゲームを完全に変えます。ボトルネックはCPUのマイクロアーキテクチャ（L1/L2/L3キャッシュ、分岐予測器（branch predictor）、TLB）に移ります。

Cache Miss と TLB Thrashing の悪夢

Hash Tableの作成とプローブのプロセスにはランダムアクセスが必要です。Hash Tableが大きい場合（例: 10GB）、L3キャッシュのミス（Cache Miss）は避けられません。DRAMの呼び出しには100〜300のCPUサイクルがかかり、CPUのALUが完全にフリーズ（Memory Stall）します。

これに対抗するため、Radix Hash Joinは極端な「分割統治」戦略を適用します。ハッシュ値のビットシフト操作（bit shift）を使用して、L1キャッシュ（32KB）またはL2（256KB）にぴったり収まる極小のブロックにデータを分割します。 一度に多くのパーティションに分割しすぎると、ページマッピングテーブル（Page Table）がTLB（Translation Lookaside Buffer）から溢れ、TLB Thrashingを引き起こします。したがって、アルゴリズムは複数ステップで分割する必要があります（Multi-pass radix partitioning）。ここでのトレードオフは、シーケンシャルメモリ帯域幅を消費する（複数回繰り返す）方が、過酷なランダムメモリアクセス遅延に耐えるよりも常に有益であるということです。

Vector化 (SIMD) の力

Volcanoモデルでは、if/else命令（ハッシュ衝突条件のチェックなど）が分岐予測器（Branch Predictor）を継続的に誤らせ、数十サイクルのコストがかかるPipeline Flush（CPUが事前計算された命令を破棄する）につながります。

Vectorizationは、AVX-512（16個の32-bit整数を同時に処理）によってこの問題を解決します。インライン組み込み（intrinsic）C++擬似コードは、Software Prefetchingと組み合わせた分岐のないSIMDの哲学を示しています：

#include <immintrin.h>

// インメモリHash Joinのためのベクトル化されたプロービングロジック
inline void simd_probe_hash_table(
    const int32_t* probe_keys, 
    const int32_t* hash_table, 
    uint32_t num_keys, 
    uint32_t* output_buffer) 
{
    uint32_t out_idx = 0;
    // 16個のキーを並列処理
    for(uint32_t i = 0; i < num_keys; i += 16) {
        __m512i v_probe = _mm512_loadu_si512((__m512i*)&probe_keys[i]);
        
        // Software Prefetching: 64ループ前にCPUにデータをキャッシュにロードさせる
        _mm_prefetch((const char*)&probe_keys[i + 1024], _MM_HINT_T0);

        // XOR / ビットシフト演算による並列ハッシュ計算
        __m512i v_hashes = _mm512_xor_si512(v_probe, _mm512_srli_epi32(v_probe, 15));
        
        // Gather: Hash tableからバケットを並列にランダム取得
        __m512i v_ht_entries = _mm512_i32gather_epi32(v_hashes, hash_table, 4);
        
        // IF命令を使用しないSIMD比較
        __mmask16 match_mask = _mm512_cmpeq_epi32_mask(v_probe, v_ht_entries);
        
        // 圧縮と連続的な結果の書き込み
        __m512i v_matched = _mm512_maskz_compress_epi32(match_mask, v_probe);
        _mm512_storeu_si512((__m512i*)&output_buffer[out_idx], v_matched);
        out_idx += _mm_popcnt_u32(match_mask);
    }
}

NUMA構造の影響

マルチCPUサーバー（Multi-socket）では、RAMはローカルに分割されます。Intel QPIを介したソケット間RAMアクセス（Remote access）は超高遅延のペナルティを受けます。現代のDBは、特定のCPUコアで構築されたHash TableがそのソケットのRAMにのみ存在することを保証するために、Data Pinning（データの固定）とThread Affinity（スレッドの固定）を実行し、サーバーを縮小版の「Share-nothing」アーキテクチャに変換する必要があります。

オペレーティングシステムのメモリ管理と非同期I/O

Huge Pages (ヒュージページ)

デフォルトのページサイズ（4KB）を使用すると、OSが巨大なHash Table（数十GB）を動的に割り当てる際に、システムが何百万ものページエントリを維持する必要があるため、TLB Missesという惨事が発生します。Huge Pages（2MBまたは1GB）を有効にすることで、ページエントリの数が指数関数的に減少します。TLBヒット率は99.9%に達し、インメモリクエリが数倍高速化されます。同時に、DBは標準の割り当てライブラリを避け、特化したArena Allocatorを独自に構築します。

Asynchronous I/O と Direct I/O

同期アクセス（Blocking I/O）はSSD NVMeの可能性を台無しにします。現代のDBはio_uring（Linux上）またはAIOを使用して、数万の非同期データロード要求をプッシュします。これにより、ストレージデバイスがDMAチャネルを介してデータを送り込んでいる間にCPUは自由にハッシュ計算を行うことができ、I/Oと計算の重複率が100%に達します。

さらに、Direct I/Oを使用することで、DBはOS Page Cacheを「回避」します。joinアルゴリズムには本質的に「1回だけ読み取る（scan-once）」というセマンティクスがあり、Page Cacheを経由すると他の有用なデータが汚染されてRAMから追い出され、二重のデータコピー帯域幅の無駄が発生します。

Storage Offloading と Bloom Filter Pushdown

限界をさらに押し広げる、Computational Storage（計算ストレージ）の概念とBloom Filterの組み合わせは、古典的なシナジーを生み出します。マザーボード（PCIeバス）を介してSSDからRAMに数TBのデータを送り込む代わりに、CPUはBuildリレーション$R$から小さなBloom Filter構造を合成し、それをSSDのマイクロコントローラ（SmartNIC/FPGA）に直接送信します。

$$p = left( 1 – e^{-frac{kn}{m}} right)^k$$

この構造に基づいて、SSDは自律的に述語を評価し、一致しない$S$のデータ行（false）を物理ハードウェアレベルで直ちに破棄し、システム帯域幅を完全に節約します。

学んだ教訓 (Lessons Learned & Best Practices)

Data Engineers と System Architects へ：

  1. データ特性を理解してアルゴリズムを制御する: 盲目的にシステムにHash Joinを強制しないでください。最終結果を結合キーでORDER BYする必要がある場合、システムのヒント（hinting）を使用してSort-Merge Joinを指定する方が、最後にコストのかかるソートプロセスを排除できるため、はるかに安価になります。
  2. メモリ管理はサバイバルアート: Hash Join時のメモリ不足（Out-of-memory）はデータベースをクラッシュさせませんが、クエリをGrace Hash Join（ディスクへのスピル）に追いやります。5秒のクエリが5時間に延長される可能性があります。バッファサイズパラメータ（例：Postgresのwork_mem）を調整する際は注意してください。
  3. OSの設定は遊びではない: 超大規模データベースシステムにとって、Huge Pagesの設定を無視することはCPUリソースの恐ろしい無駄遣いです。同様に、NUMAアーキテクチャを理解して適切に無効化または微調整することで、ソケット間のパフォーマンスの罠を回避できます。
  4. Hardware and Software Co-design: 現代のデータベースは単なるソフトウェアではありません。真の速度は、ソフトウェアとハードウェアの間の機械的共生（Mechanical Sympathy）から生まれます – SIMD、キャッシュ構造、インテリジェントなNVMeディスク帯域幅の活用です。

結論

Joinアルゴリズムのエコシステム – 初歩的なネストされたループから、ベクトル化されたマルチパスRadix Hash Joinや並列ネットワークを組み合わせたSort-Mergeまで – は、ソフトウェアエンジニアリングと計算物理学を調和させる技術の記念碑として立っています。これらを習得することは、AI時代の巨大なデータウェアハウスを運用するための究極の鍵となります。
