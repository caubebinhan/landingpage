データベース・オブザーバビリティにおけるBPFとeBPF: カーネルレベルでのデータベースの超音波検査

July 22, 2026 · admin · 3 分で読めます

概要と中核的な問題 (Core Problem Statement)

分散マイクロサービス (Distributed Microservices) と高負荷のトランザクション処理システムの時代において、データベースは最大のボトルネック (bottleneck) として機能することがよくあります。しかし、データベースの障害診断とパフォーマンス監視 (Observability) は非常に困難な課題です。

中核的な問題: 従来のデータベースは「ブラックボックス」 (Black Box)として動作します。 SQLクエリがなぜ遅いのかを知るために、エンジニアはしばしばスロークエリログ (Slow Query Log) を有効にしたり、インバンドプロファイリング (In-band profiling) のためのエージェントをインストールしたり、straceやtcpdumpのような古典的なOSツールを使用したりする必要があります。

  • ログの有効化やインバンドエージェントは、最大30〜50%のパフォーマンスのオーバーヘッド (Overhead) を発生させ、エラーを見つける前にシステムをクラッシュさせます。
  • strace (ptrace メカニズムに基づく) を使用すると、システムコール (System Call) のたびにOSがデータベースプロセスを停止する必要があるため、遅延が何百倍にも増加します。
  • tcpdump を使用してネットワークパケットをキャプチャすると、膨大なジャンクデータが生成され、カーネル (Kernel) からユーザースペース (User Space) にデータをコピーするためにCPUを消費しすぎます。

eBPF (Extended Berkeley Packet Filter) の出現は、オブザーバビリティにおいてアーキテクチャの革命をもたらしました。eBPFにより、エンジニアはOSのカーネル空間 (Kernel Space) の内部で直接、超軽量の「超音波」コードを実行できます。eBPFは、ネットワークカードを通過するネットワークパケット、PostgreSQL内のすべての関数呼び出し、SSDに書き込まれるデータブロックなどを、ほぼゼロのオーバーヘッド (< 1%) で監視できます。

この詳細な技術レポートでは、eBPF仮想マシンのマイクロアーキテクチャ、再起動せずに実行中のデータベースのソースコードを「ハッキング」するためのUprobes/Kprobesのセットアップ方法、およびデータセンター規模でeBPF監視インフラストラクチャを展開する際の苦い教訓について解剖します。

eBPF仮想マシンのマイクロアーキテクチャ (eBPF Virtual Machine)

元々はネットワークパケットをフィルタリングするために設計された (tcpdumpツールのcBPF) eBPFですが、現在ではLinuxカーネル (Linux Kernel) の心臓部で実行される汎用仮想マシン (VM) へと進化しています。

ハードウェアへの直接マッピングを行う命令セットアーキテクチャ (ISA)

最新のeBPFアーキテクチャは、x86_64やARM64などの64ビットプロセッサのハードウェア構造に直接かつ最適にマッピング (one-to-one mapping) されるように設計されています。 eBPFプロセッサは、11個の仮想64ビットレジスタ ($R0$ から $R10$) のモデルを使用します。

  • $R0$: 関数の戻り値を格納します。
  • $R1 – R5$: 関数呼び出し時の入力引数を保持します。
  • $R6 – R9$: 関数呼び出し間で保持されるレジスタ (Callee-saved)。
  • $R10$: 読み取り専用のフレームポインタ (Frame Pointer) であり、スタック (Stack) へのアクセスに使用されます。

実行時コンパイル (Just-In-Time – JIT compilation) プロセスは、eBPFバイトコード (bytecode) をCPUのネイティブマシンコード (native machine code) に変換します。eBPFのレジスタアーキテクチャはx86_64のレジスタアーキテクチャをほぼそのままコピーしているため、このJITプロセスは超高速で行われ、生成されたマシンコードは直接コンパイルされたCコードと同じくらい高速に実行されます。

門番: eBPF Verifier

OSカーネルは不可侵な領域です。もしeBPFコードにゼロ除算、NULLポインタ、あるいは無限ループが含まれていれば、OSを完全にクラッシュ (Kernel Panic) させる可能性があります。 絶対的な安全性を確保するために、eBPFコードはVerifierというスキャナーを通過しなければなりません。 Verifierは静的な制御フローグラフ (Control Flow Graph – CFG) 解析技術を使用して、以下を保証します：

  1. 無限ループがないこと (eBPFコードは終了が保証されていなければなりません – 実践的な意味でTuring Incomplete)。
  2. 配列外メモリへのアクセス (Out-of-bound access) がないこと。
  3. 禁止されたカーネルメモリ領域へのアクセスがないこと。
  4. 最大スタックサイズが512バイトを超えないこと。

Verifierの計算量は $mathcal{O}(N times E)$ (ここで $N$ は状態数、$E$ はフローグラフの辺の数) に漸近します。Verifierが「安全」のスタンプを押した場合にのみ、eBPFコードはJITにロードされることを許可されます。

eBPF Maps: 超高速データ共有 (Zero-Copy)

カーネル内で実行されるeBPFプログラムがクエリの遅延を収集した場合、コピーのコストをかけずにそのデータをユーザースペース (例：PrometheusやDatadog Agent) に送信するにはどうすればよいでしょうか？ その解決策が eBPF Maps です。

eBPF Mapsは、カーネルのページング不可能な物理RAM (Non-pageable memory) 上に静的に割り当てられるKey-Valueデータ構造です。eBPFプログラムとユーザースペースアプリケーションの両方が、ファイルディスクリプタを介してこのMapに読み書きできるため、データをコピーするためのシステムコールが完全に不要になります。

eBPF Hash Map (RCU-Protected)

状態の保存に使用されます。例：リクエストパケットが入ってきたとき、eBPFは TCP_Tuple (IP, Port) をKeyとして Time_Start をHash Mapに書き込みます。レスポンスパケットが出ていくとき、eBPFはHash Mapを読み取って Time_Start を取得し、Latency を計算して保存します。 内部のハッシュアルゴリズムは Read-Copy-Update (RCU) メカニズムを使用しており、完全にロックのない同時読み取り (Lock-free concurrent reads) を保証し、CPUのボトルネックなしに数千万IOPSのアクセス速度を達成するのに役立ちます。

eBPF Ring Buffer

カーネルからユーザースペースにイベントストリーミング (Event Streaming) をプッシュするため (例：遅いクエリのリスト) に、eBPFはRing Bufferを使用します。 これはロックフリーのMPSC (Multi-Producer Single-Consumer) 循環キュー構造です。イベントを生成するCPUコアは、Ring Bufferに直接書き込みます。ユーザースペースエージェントは、ポーリングメカニズム (epoll) を使用してデータを継続的に収穫します。 この分離により、ユーザースペースが過負荷になってもカーネルがブロック (block) されないことが保証されます。

監視料金所の設置方法 (Probing)

eBPFは、データベースが何をしているかをどのようにして知るのでしょうか？ eBPFは、OSの血管に取り付けるための「釣り針」 (hooks) を提供します。

ネットワークフック (Network Hook – TC / XDP / Socket Filter)

データベースプロセスに干渉したくない (またはできない) 場合は、ネットワークレベルで監視できます。 eBPFは、トラフィック制御 (Traffic Control – TC) レイヤー、またはネットワークカード (NIC) ドライバーの最下位レベルであるeXpress Data Path (XDP) にアタッチできます。 PostgreSQL/MySQLのWire Protocolに従ってTCPパケットを解析 (parse) することで、eBPFプログラムはデータベース自体が全く気付かないうちに、クエリ時間、クエリ数、ネットワークエラー率 (Retransmission) を正確に測定できます。この方法はオーバーヘッドが非常に低いですが、複数のパケットにまたがる断片化されたTCP (TCP Fragmentation) クエリを解析するのは困難です。

User Probes (uprobes & uretprobes)

これが究極の武器です。Uprobesにより、eBPFはデータベースのソースコード (ユーザースペース) 内のC/C++関数に直接「穴を開ける」ことができます。 たとえばPostgreSQLでは、exec_simple_query()関数の先頭にUprobeを付け、その関数のリターンポイントにUretprobeを付けることができます。

ハードウェア割り込みの原理 (int3): eBPFが exec_simple_query() 関数にUprobeをアタッチすると、この関数の最初のマシンコード (opcode) を密かに int3 命令 (Breakpoint Trap) に置き換えます。 CPUがこの関数を実行すると、int3 命令がハードウェア例外 (Hardware Exception) をスローします。OSが制御を奪い、カーネルにジャンプして、私たちのeBPFハンドラ関数を呼び出します。eBPFがログの書き込みを終えた後、CPUは古いopcodeを復元し、データベースの実行を再開します。

Uprobesの致命的な注意点: Uprobeを1回ジャンプするコスト (Exception + Context Switch の生成) は、最新のCPUで約 1.5〜3.0 マイクロ秒 かかります。

  • もし exec_simple_query 関数 (1秒間に10,000回しか実行されない) にUprobeを付けた場合、CPUは30msしか消費しません (約3%のオーバーヘッド) -> 非常に安全です。
  • もし btr_search_leaf 関数 (B-Treeの葉をトラバースする、1秒間に1,000,000回実行される) にUprobeを付けた場合、「割り込みストーム」 (Interrupt Storm) を引き起こします。CPUは実際の1秒に対して3秒を消費します -> データベースが完全にハングアップします。

    • Uprobeを挿入する前に、関数の呼び出し頻度 (Invocation Frequency) を常に慎重に測定してください！

Kernel Probes (kprobes)

データベースとハードドライブおよびRAMの相互作用を監視するために使用されます。 kprobeを vfs_read() および vfs_write() 関数にアタッチし、識別モデルと組み合わせることで、eBPFはページキャッシュのミス率 (Page Cache Miss Ratio) を追跡できます。エンジニアは、MySQLの SELECT ステートメントが実際にはNVMeディスクに何回読み込みに行かなければならなかったか、そしてカーネルのブロックレイヤーで何ナノ秒かかったかを見透かすことができます。

システムマイクロアーキテクチャの最適化 (NUMA & Cache)

数十億のトランザクションを処理する128コアサーバークラスター上にeBPFでオブザーバビリティシステムを展開するためには、eBPF Mapsの設計はプロセッサの物理法則に厳密に従う必要があります。

NUMA問題とPer-CPU Maps

NUMA (Non-Uniform Memory Access) アーキテクチャでは、あるCPUが別のCPUのRAMにアクセスしようとすると、非常に大きな遅延が発生します。もしコア0とコア64のeBPFプログラムが同時に単一のeBPF Hash Map (Global Map) に書き込もうとすると、メモリの競合 (Memory Contention) が発生し、相互接続バス (UPI/Infinity Fabric) が飽和状態になります。

解決策: Per-CPU Maps 構造を使用します。 OSは、ハードウェアCPUコアごとにMapの個別のスペース (Local Copy) を割り当てます。どのコアで実行されているeBPFプログラムも、そのコアのRAM部分にのみローカルな静的データを書き込みます。これにより、すべての書き込み操作がロックフリーで競合なし (Lockless & Contention-Free) になり、スループット (Throughput) はL1キャッシュの帯域幅の物理的な限界に達します。

インストラクションキャッシュの保護 (I-Cache Thrashing)

プロセッサのL1命令キャッシュ (L1i) は非常に小さいです (わずか32KB)。データベースは、複雑な論理命令 (Query Planner, Execution Engine) を格納するためにこのスペースを必要とします。 もしeBPFプログラムがコンパイルされてマシンコードの量が多すぎる場合、(kprobe/uprobeを介して) 呼び出されるたびにL1iキャッシュに溢れ込み、データベースのソースコードを追い出します (Eviction)。データベースの実行が再開されると、キャッシュミスが発生し、L2/L3から命令をリロードしなければならず、パフォーマンスの深刻な低下 (Stall Cycles) を引き起こします。

BPFエンジニアは、非常に無駄のないコード (Lean Code) を書かなければなりません。チェックを統合し、unroll loops を慎重に使用し、L1iキャッシュ内でデータベースと「平和に共存」するためにBPFプログラム全体を4KB未満に保つように努めてください。

システムエンジニアのための教訓とベストプラクティス

  1. kprobesとネットワークから始め、uprobesは慎重に: 本番環境 (Production) で初めてuprobeをテストしないでください。遅延収集を展開する前に、常にステージング環境と bcc または bpftrace ツールを使用して、(count() 関数により) 1秒あたりの関数呼び出し回数を測定してください。
  2. フラットなデータ構造 (Flattening FSM): eBPFにはヒープがありません (mallocがありません)。複数のステップを経るSQLトランザクションを追跡する必要がある場合は、ユーザースペースに有限状態機械 (FSM) を構築してください。eBPFはRing Bufferを介して個々のイベント (Event) シグナルをプッシュする役割のみを果たすべきです。カーネル内で複雑な結合ロジックを詰め込もうとしないでください。
  3. Mapの容量管理 (Bounded Capacity): すべてのeBPF Mapには max_entries があります。ハッシュテーブルがいっぱいになると、新しいデータベーストランザクションは監視されなくなります。エンジニアは、新しいデータのためのスペースを空けるために、Mapから古いキーを「クリーンアップ」 (delete) するのに十分な速度のユーザースペースエージェントを書く必要があります。
  4. 権限とセキュリティ (Security Lockdown): eBPFはカーネルのRAM全体を読み取る機能を持っています (転送中のパスワードやシークレットキーを含む)。本番環境では、CAP_BPF または CAP_SYS_ADMIN グループに属するユーザーのみが操作を許可されるべきです。コード署名システムを使用して、出所不明のすべてのeBPFプログラムをブロックします。
  5. 可視化ツール (Observability Stack): BPFエージェントをCで一から書かないでください。Cilium、Tetragon、Pixie などのエコシステムを使用するか、Golangと cilium/ebpf ライブラリを使用して書いてください。Grafanaのパーセンタイルチャート (P99, P99.9) を活用するために、生データをPrometheus Histogramにプッシュしてください。
