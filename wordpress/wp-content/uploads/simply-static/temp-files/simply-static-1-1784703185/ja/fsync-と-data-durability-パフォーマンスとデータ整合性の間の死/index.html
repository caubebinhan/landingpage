`fsync()` と Data Durability: パフォーマンスとデータ整合性の間の死闘の解剖

July 22, 2026 · admin · 4 分で読めます

要約と中核となる問題 (Core Problem Statement)

分散システム (distributed systems)、リレーショナルデータベース (RDBMS)、および極度のデータ整合性を必要とするアプリケーション（コアバンキングトランザクションシステムなど）の世界において、データの耐久性 (Data Durability – ACIDの’D’) の概念は、妥協できない基盤としての役割を果たします。

中核となる問題: ユーザーが「保存」をクリックしたばかりのデータの行が、誰かがサーバーの電源プラグを即座に乱暴に引き抜いたとしても、決して消えないことをソフトウェアはどうやって100％確信できるでしょうか？

この学術的な議論と物理的な限界の中心は、POSIX標準のシステムコール (system call) である fsync() と、その最適化されたバリアントである fdatasync() を中心に展開しています。このシステムコールは、揮発性メモリ (Volatile Memory – RAM) 状態と不揮発性物理ストレージデバイス (Non-volatile storage – HDD/SSD) の間の過酷なアーキテクチャ上の境界を表しています。

fsync() の実行には、オペレーティングシステムカーネル (OS Kernel) がページキャッシュ (Page Cache) からストレージデバイスへすべてのダーティページ (dirty pages) をフラッシュ (flush) すると同時に、デバイスコントローラー (device controller) にハードウェアキャッシュ (hardware cache) をストレージメディアへフラッシュするよう要求するコマンドを発行する必要があります。

  • 速度の錯覚: fsync() を使用しないことは、極端に高いスループット (throughput) の錯覚をもたらします。RAMへのデータの書き込みには1マイクロ秒かかります。しかし、電源が落ちると、データは永遠に蒸発します。データベースは「破損 (Corrupted)」します（構造的な破損）。
  • パフォーマンスの死刑宣告: 継続的に fsync() を乱用すると、壊滅的なパフォーマンスの低下につながります。それは、電子が NAND フラッシュチップのフローティングゲート (floating gates) 内に実際に正常にトラップされるまで、CPUスレッドをブロック (block) し、数ミリ秒（RAMの数千倍遅い）待機させることを強制します。これにより、巨大なI/Oボトルネックが生じ、SSDの寿命が縮まります (Write Amplification)。

この技術ホワイトペーパー (whitepaper) は、ユーザースペース (user space) からLinuxカーネル (VFS, Block Layer)、物理トランジスタレベルに至るまで、ストレージスタック (storage stack) アーキテクチャ全体の詳細な解剖を行い、それにより fsync() のメカニズムと世界をリードするデータベース管理システムがそれに直面してどのように生き残るかを明らかにします。

ストレージシステムの階層アーキテクチャとキャッシュメカニズム

現代のコンピュータアーキテクチャは、マイクロプロセッサ（光の速さ）と二次記憶装置（亀の歩み）の間の巨大なレイテンシの差を調整するためのメモリ階層の原則に基づいて構築されています。

非同期 write() コマンドのライフサイクル

アプリケーション（NodeJSやPythonなど）が write() コマンドを実行するとき、データが直接ハードドライブに書き込まれることは決してありません。代わりに、それは Page Cache に入ります。 Page Cacheは、Linuxの仮想ファイルシステム (Virtual File System – VFS) によって管理されます。Page Cacheは、書き込み操作を吸収するスポンジのように機能します。新しいデータを含むメモリページは「ダーティ (dirty)」としてマークされます。

アプリケーションの write() スレッドは即座に Success ステータスを返します。アプリケーションはデータが安全であると考えますが、実際にはRAM上に浮かんでいます。カーネルのバックグラウンドワーカースレッド（bdi_writeback など）は、vm.dirty_ratio または vm.dirty_expire_centisecs パラメータに基づいて、これらのダーティページをゆっくりと集め、後でディスクにフラッシュします。

レイテンシ分解の解剖 (Latency Decomposition)

非同期書き込み操作のレイテンシ ($L_{async}$) には以下のみが含まれます。

$$L_{async} = L_{syscall} + L_{copy_to_kernel} + L_{page_cache_update} + L_{lock_contention}$$

この期間にはわずか1〜5マイクロ秒しかかかりません。

しかし、fsync() 呼び出しがトリガーされると、スレッドは投獄されます。同期レイテンシ ($L_{sync}$) は、何十もの層を通過する脱獄の旅です。

$$L_{sync} = L_{syscall} + L_{vfs_flush} + L_{fs_journal} + L_{block_queue} + L_{pcie_tlp} + L_{nvme_ctrl} + L_{ftl_mapping} + L_{nand_prog}$$

$L_{nand_prog}$ コンポーネント（フラッシュチップの物理プログラミング時間）には、200マイクロ秒（SLCチップ）から1500マイクロ秒以上（QLCチップ）かかります。つまり、RAMへの書き込みの 300倍から1500倍 遅いです。さらに、PCIeのトランザクションレイヤーパケット (Transaction Layer Packets – TLP) の構築とブロック I/O スケジューラの競合もレイテンシに寄与します。

エンタープライズの武器: PLP Capacitor と FUA Flag

SSDのコントローラー (Controller) の内部には、バッファ (Disk Write Cache) として機能する独自のRAMスティックがあります。OSが fsync() を呼び出すと、PCIeコマンドに特別なフラグ、つまり FLUSH コマンドフラグまたは FUA (Force Unit Access) ビットを挿入する必要があります。

FUA フラグはSSDに命令します。「データを内部RAMに置いて私を騙そうとするな。今すぐNANDチップに焼き付けろ！」。この強制的な直接焼き付けにより、SSDは恐ろしく遅くなります。

エンタープライズ向けのソリューションは？ PLP (Power Loss Protection) キャパシタ。エンタープライズSSDには大容量のキャパシタ（コンデンサ）が搭載されています。このキャパシタのおかげで、SSDはFUAフラグを「無視する」権利を持ちます。データを内部RAMに保存し（超高速）、すぐにOSに成功を報告します。サーバーの電源が落ちた場合、キャパシタに蓄えられたエネルギーがSSDに約50〜100ミリ秒の追加時間を提供し、内部RAMからの残りのデータをNANDチップにゆっくりとフラッシュさせます。これが、書き込みが重い (Write-Heavy) テストにおいて、エンタープライズSSD上で実行されるデータベースがSamsung EVOドライブ（コンシューマーSSD）上で実行されるよりも50倍速い根本的な理由です。

双子のコマンド間のマイクロアーキテクチャ分析: fsync() vs fdatasync()

ファイルシステム構造の奥深くでは、fsync() の呼び出しは単にデータを書き込むだけではありません。ファイルサイズ、権限 (permissions)、変更時間 (mtime)、アクセス時間 (atime) などのメタデータ (Metadata) も書き込む必要があります。

ログファイルの末尾にさらに10バイト書き込むと想像してください。

  • fsync() を呼び出すと、オペレーティングシステムは2つの物理I/O操作を実行する必要があります。1つは10バイトのデータを書き込むため、もう1つはファイルの inode 構造の mtime タイムスタンプの更新を書き込むためです。
  • mtime を継続的に更新するという厳格さは、無意味なI/Oタスクの増幅をもたらします。

fdatasync() 命令は妥協のために生まれました。それは、ファイル拡張子 (file extension) のようにファイルサイズを増加させるような 破壊的な変更 (breaking changes) の性質を持つ場合のみのメタデータと、ファイルの内容を直接構成するデータ部分の整合性のみを保証します。mtime を変更するだけの場合、fdatasync() は「怠惰」になり、メタデータをディスクにフラッシュしません。

物理コマンドのI/O要求数 ($N_{io}$) の数学的相関関係:

$$N_{io}(text{fdatasync}) le N_{io}(text{fsync})$$

ほとんどの現代のデータベース管理システム（MySQLのInnoDBやPostgreSQLのWALなど）は、スループットを保護するために fdatasync() を非常に好みます。トランザクションのコミットごとに不要な1回のメタデータI/O操作を節約することで、全体のパフォーマンスを2倍に高めることができます。

Write Amplification Factor (WAF) の亡霊 – Flash NANDの破壊者

fsync() の乱用は、システムを遅くするだけでなく、SSDを文字通り「燃やし」ます。

NANDフラッシュチップは、磁気ディスク (HDD) のようなインプレース更新 (in-place update) 上書きメカニズムをサポートしていません。新しいデータを更新するたびに、SSDは完全に空の物理ページ (Physical Page – 通常16KB) に書き込む必要があります。古いページは「ゴミ (garbage)」としてマークされます。ドライブの空き容量がなくなると、FTLマイクロコントローラーはガベージコレクション (Garbage Collection – GC) アルゴリズムを開始します。

アーキテクチャの悲劇: 書き込み操作はページ (Page – 16KB) レベルで行われますが、ゴミの消去操作はブロック (Block – 通常4MBから16MB) レベルで行われなければなりません。GCはブロック全体をSSD RAMに取り込み、まだ使用可能なページを拾い上げて別の場所に移動し、高電圧をかけてその巨大なブロック全体を消去する必要があります。

素朴なアプリケーションをシミュレートしてみましょう。100バイトのログを書き込むたびに、fsync() を1回呼び出します。 ハードウェアレベルでは、SSDは100バイトを書き込むことはできません。FUAバリアが壊れないようにするために、その100バイトを格納するために16KBの物理ページ全体を燃やし尽くす（割り当てる）ことを余儀なくされます。

書き込み増幅係数 (WAF) の方程式:

$$WAF = frac{text{Flash NANDにフラッシュされる実際のデータの合計バイト数}}{text{Host OSが書き込みを要求したデータ構造の合計バイト数}} approx frac{S_{page}}{S_{payload}} + WAF_{GC_overhead}$$

アプリケーションが $S_{payload} = 100text{ bytes}$ をページ $S_{page} = 16384text{ bytes}$ に書き込む場合、基本WAFはすでに 163.8倍 です。つまり、1GBのデータの書き込みを要求すると、SSDは実際に163GBを書き込まなければなりません。ハードドライブの物理的耐久性 (TBW – Terabytes Written) の摩耗率は急上昇します。その結果、GCアルゴリズムが狂ったように実行され、コントローラーをロックし、テールI/Oレイテンシピーク (tail latency peaks) を生成し、p99レイテンシが数百ミリ秒に急上昇します。

データベースエンジンからの「ハードル越え」最適化戦略

整合性 (Durability) と速度 (Throughput) の間の不可能なパラドックスに直面して、ソフトウェアアーキテクトは古典的なメカニズムを発明しました。

Group Commit（グループコミット）アルゴリズム

これはデータベース業界全体を救った数学の魔法です。各トランザクションスレッドが独立して fsync() を呼び出す代わりに、DBエンジンは次のようにします。

  1. トランザクションスレッドはログをMemory Ring Bufferキューにプッシュし、スリープ状態になります。
  2. リーダースレッド (Leader Flusher) が起動し、そのバッファで待機している何百ものトランザクションをすべて収集します。
  3. リーダーは、すべてのために単一の巨大な fdatasync() コマンドを呼び出します。
  4. 物理I/Oが完了すると、リーダーは数百のクライアントスレッドすべてを起こし、同時に成功を報告します。

素朴なスループットの限界:

$$ lambda_{naive} approx frac{1}{L_{fsync}} $$

fsync() に1ミリ秒かかる場合、CPUの数に関係なく、システムは最大でも1000 TPSにしか達しません。

グループコミットでのスループットの限界:

$$ lambda_{group_commit} = minleft( lambda_{max_hardware_io_bandwidth}, frac{bar{N}_{batch}}{L_{fsync}} right) $$

グループ化 ($bar{N}_{batch}$) することにより、システムのスループットはディスクのシークレイテンシによって制限されなくなり、PCIeバスの超高速帯域幅に近づきます。システムが混雑するほど、待機しているスレッドが多くなり、$N_{batch}$ が大きくなり、システムは自動的により効率的になります。

io_uring 革命

最近のLinuxカーネルは、従来のブロッキング (blocking) fsync() モデルを完全に置き換えるために io_uring を導入しました。カーネルとユーザースペース間で共有される2つのmmapキュー構造 (Submission Queue – SQ および Completion Queue – CQ) により、コンテキストスイッチングが排除されます。 現代のデータベース（ScyllaDBなど）は、CPUスレッドを待たせることなく、IORING_OP_FSYNC フラグ付きの数万のI/OコマンドをSQに詰め込みます。すべてがイベント駆動型 (Event-Driven) になり、ハードドライブがまだデータを焼き付けている間、CPUスレッドは完全に自由に他のタスクを実行できます。

Direct I/O (O_DIRECT) vs Buffer I/O + fsync()

別のアーキテクチャの岐路は、Page Cacheを使用してから fsync() を呼び出すべきか、それとも O_DIRECT を使用してPage Cacheをバイパスし、手動でメモリを管理する（MySQL InnoDB Buffer Poolのように）べきかということです。

  • O_DIRECT の使用: アプリケーションは独自の Buffer Pool メカニズムを構築し、ダーティデータをディスクにフラッシュする責任を負わなければなりません。利点は、I/Oライフサイクルを完全に制御できることです（OSカーネルに奪われない）。O_DIRECT を fsync() と組み合わせると（ファイルシステムによりますが）、大規模データベースにおいて最も予測可能な最高のパフォーマンスを提供します。
  • Buffer I/O の使用: プログラミングが容易で、読み取りにOSキャッシュを活用できます。ただし、fsync() を呼び出すと、アプリケーションスレッドはカーネルの bdi_writeback アルゴリズムによって引き起こされるI/O爆発の結果を負担しなければなりません。

未来のアーキテクチャ: Storage-Class Memory (SCM) と NVDIMM

半導体材料が進化するにつれ、SCM技術（Intel OptaneやNVDIMMなど）はRAMとSSDの境界を完全に曖昧にしています。SCMはRAMスロット（PCIeバスの代わりにDDRバス）に直接接続され、不揮発性ストレージ機能を備えていますが、ナノ秒単位のアクセス速度を持っています。

中核となる武器は DAX (Direct Access kernel interface) インターフェースです。DAXファイルをmmapすることにより、ファイルシステムはPage CacheとBlock Layerの両方を完全にバイパスします。かつて有名だった fsync() 操作は完全に排除されます。代わりに、CPUは SFENCE バリアを伴うプリミティブなアセンブリマイクロコマンド CLWB (Cache Line Write Back) を呼び出すだけで済みます。データはレジスタ (CPU L1 Cache) から電源安全なSCMモジュールへわずか数十ナノ秒で直接送られます。NVDIMMをログキャッシュ層として使用するデータベースは、Durabilityのボトルネックを徹底的に解決し、WAFや fsync() に対するすべての懸念を過去のものにします。

システムエンジニアのための教訓とベストプラクティス

  1. 基盤となるハードウェアを理解する: コンシューマーSSD (Client SSD) に書き込みの重いデータベースシステムを展開しないでください。エンタープライズSSD上のPLPキャパシタは、fsync() がシステムのパフォーマンスを殺さないようにする唯一の「ハードウェアの魔法」です。
  2. fsync() よりも fdatasync() を優先する: カスタムログ / WALを書き込むケースの99%では、コンテンツデータのみを気にし、mtime タイムスタンプは気にしません。fdatasync() を使用すると、大量の無駄なI/Oを削減できます。
  3. 常にバッチ処理 (Batching) を使用する: グループコミットの原則と同様に、永続的なログストレージを必要とするアプリケーションを作成する場合は、fsync を使用してディスクにフラッシュする前に、RAM上でデータを大きなチャンク（16KBや64KBなど）にグループ化します。数バイトのログ行ごとにfsyncしないでください。
  4. カーネルのダーティ比率を制御する: アプリケーションが大量のダーティページを生成する場合（まだfsyncされていない場合）、Linuxカーネルが静かにゆっくりとクリーンアップするように、vm.dirty_background_ratio を低く（5%など）設定します。20%まで蓄積してからクリーンアップさせないでください。システムが完全にフリーズする「I/O stall」が発生します。
  5. データベースエンジンを自作しようとしない: 物理I/O上でACIDを調整する複雑さは計り知れません。InnoDB、RocksDB、PostgreSQLなどの巨人の肩の上に立ちましょう。
