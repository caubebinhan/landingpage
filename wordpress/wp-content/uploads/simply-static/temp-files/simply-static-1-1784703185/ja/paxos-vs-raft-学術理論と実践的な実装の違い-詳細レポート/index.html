Paxos vs. Raft – 学術理論と実践的な実装の違い (詳細レポート)

July 22, 2026 · admin · 3 分で読めます

エグゼクティブサマリー (Executive Summary)

コンピュータサイエンスと耐障害性分散システム (fault-tolerant distributed systems) の発展の歴史において、コンセンサス (consensus) 問題は最大の課題の1つです。この問題を解決することを目的とした最も顕著な2つのアルゴリズムは、PaxosとRaftです。どちらも信頼性の低いネットワーク環境で複製されたステートマシン (Replicated State Machine) の状態の一貫性を維持することを目指していますが、それらの設計思想は2つの対極にあります。

この技術レポートでは、Paxos (柔軟性、分散化、学術的な数学に焦点を当てている) とRaft (理解しやすさ、中央集権的な独裁性、実践的な実装可能性を重視している) の根本的な違いを徹底的に解剖します。

問題提起 (Problem Statement): 独立したコンピュータ (nodes) は、ネットワークが中断されたり、メッセージが失われたり、一部のノードがクラッシュしたりした場合でも、単一の値またはコマンドのシーケンスについてどのように合意できるでしょうか？FLP定理 (Fischer, Lynch, Paterson) によると、1つのノードに障害が発生した場合、完全に非同期なネットワークにおいて安全性 (Safety) と活性 (Liveness) の両方を保証できる決定論的なコンセンサスアルゴリズムはありません。システムは、ランダム性 (randomness) と部分同期 (partial synchrony) ネットワークの仮定を使用することで、この制限を打破する方法を見つける必要があります。この問題へのアプローチの違いが、PaxosとRaftを生み出しました。

教訓と得られた知識 (Lessons Learned):

  1. 理論 vs. 実践: Paxosは数学的には完璧ですが、実装するのは「悪夢」です。Raftは、理解しやすさ (understandability) を優先することで、より安定した実践的なシステムをもたらすことができることを証明しています。
  2. 構造上のトレードオフ (Structural Trade-offs): Paxosは順不同のコミット (out-of-order commits) を許可し、ヘッドオブラインブロッキング (head-of-line blocking) を軽減しますが、障害回復の複雑さが増します。Raftは厳格なシーケンシャルな書き込みを強制し、絶対的な明確さと引き換えにパフォーマンスの一部を犠牲にします。
  3. スプリットブレイン (Split-Brain) とクーデター: どちらのアルゴリズムもクォーラム交差 (Quorum Intersection) の原則に依存しています。ただし、Raftはネットワーク分断時の「任期のインフレ」 (term inflation) 問題に直面しており、Pre-Voteなどの補完的なメカニズムによる完成が求められます。

分散コンセンサスの理論的基礎

1 FLP定理とクォーラム交差 (Quorum Intersection)

コンセンサスアルゴリズムの信頼性を形成する基礎となる数学的根拠は、クォーラム交差プロパティ (quorum intersection properties) に線形に依存します。

クォーラム (定足数) $mathbb{Q}$ は通常、$N = 2F + 1$ 個のノードからなる最小過半数グループとして定義されます。ここで、変数 $F$ は、システムが回復力を維持できる障害ノードの最大数を表します。

集合の交差特性は、次の公理によって形式化されます：

$$ forall Q_1, Q_2 in mathbb{Q}, Q_1 cap Q_2 neq emptyset $$

この非対称な交差公理は、任意の2つの過半数グループが確実に少なくとも1つのメンバーを共有するという認知的アンカーを提供します。この中間ノードは「メモリキーパー」 (memory keeper) として機能し、システムが同じ時間サイクルで2つの矛盾する決定を承認するのを防ぎます。これは、PaxosとRaftの両方が依存する究極の安全障壁 (Safety barrier) です。

Paxosアルゴリズムの解剖: 分散化の芸術

優れたコンピュータサイエンティストであるLeslie Lamportによって架空のSynod評議会のメタファーを通じて提示されたPaxosアルゴリズムは、単一のリーダーに依存しません。代わりに、システムを3つの役割に分解します：

  • Proposers (提案者): 評議会が投票するための値を提案します。
  • Acceptors (承認者): 提案を記録し、システムの「メモリ」として機能します。
  • Learners (学習者): コンセンサスが得られたときに最終結果を更新します。

1 Basic Paxosの2フェーズコミット (Two-Phase Commit)

Basic Paxosモデルでコンセンサスを達成するプロセスは、2つのフェーズで実行されます：

Phase 1 (Prepare): 約束の収集

  1. Proposerは識別番号 $n$ (この番号は一意であり、これまでに使用したどの番号よりも大きくなければなりません) を生成し、$Prepare(n)$ メッセージをAcceptorsの過半数に送信します。
  2. Acceptorが $Prepare(n)$ を受信し、$n$ がこれまでに見たどの提案番号よりも大きい場合、$Promise(n, v_a, n_a)$ で応答します。この約束は「今後 $n$ より小さい番号の提案は絶対に受け入れない」と断言するものです。存在する場合、(以前に番号 $n_a$ で承認された) 値 $v_a$ が添付されます。

Phase 2 (Accept): 承認要求の送信

  1. Proposerがクォーラムから十分な約束を収集した場合、$Accept(n, v)$ を送信します。
  2. 値 $v$ は任意に選ぶことはできません！受け取った $Promise$ の中で最大の $n_a$ を持つ値でなければなりません。すべての $v_a$ が空の場合、Proposerは独自の新値を提案する権利を得ます。
  3. Acceptorは、$n$ より大きい提案番号をすでに約束していない場合にのみ $Accept(n, v)$ を承認します。

2 数学的帰納法による証明

ある値 $v$ が元の提案番号 $n$ でAcceptorsの評議会ネットワークによってすでに証明されていると仮定します。証明命題は次を要求します：以降の提案 $n’ > n$ はすべて、その値 $v$ を持たなければなりません。

$Q_c$ を提案 $n$ を保護したAcceptorsの集合とします。$Q_p$ を提案 $n’$ に約束したAcceptorsの集合とします。 集合の原則によると：$Q_c cap Q_p neq emptyset$ です。 この交点は番号 $n$ とともに $v$ をすでに「記憶」しています。$Prepare(n’)$ に応答するとき、それは $v$ を返します。新しいProposerは強制的に $v$ を確認し、その正確な値 $v$ を再伝播しなければなりません。安全性 (Safety) は永久に保証されます！

3 欠陥: ライブロック (Livelock) とProposer Dueling

分散化は致命的な弱点をもたらします：Proposer Dueling (Proposer同士の衝突)。 Proposer Aが $Prepare(10)$ を送信し、Acceptorsが同意したとします。Aが $Accept(10)$ を送信する前に、Proposer Bが $Prepare(11)$ を送信します。Acceptorsは今度はBに約束します (Aを見捨てます)。Aが送信した $Accept(10)$ は拒否されます。怒ったAは $Prepare(12)$ を送信します…このループはLivelockを引き起こし、システムは実行され続けますが決定に至ることはありません (Livenessの喪失)。 これを解決するために、Paxosは指数バックオフ (Exponential Backoff) アルゴリズムを使用するか、Multi-Paxos (静的リーダーを持つ) に進化しなければなりません。

Raftアルゴリズムの解剖: 中央集権的独裁の崇拝

対極にあるRaft (Diego OngaroとJohn Ousterhoutによる設計) は、強力な権力の中央集権化を崇拝する設計哲学 (strong leader paradigm) を持って登場しました。

Raftは、リーダー選出 (Leader Election) 問題とトランザクションログ複製 (Log Replication) プロセスを巧みに統合しています。任意の任期 (Term) $T$ において、Leaderはただ1つしか存在できません。

1 ランダム化された選挙とLivelockの回避

Raftは、ランダム化されたハートビートタイムアウト (randomized heartbeat timeouts) を適用することで、Paxosの「Proposer Dueling」を完全に解決します。 複数のFollowerが同時に目覚めて競合したときの衝突確率 (collision probability) の背後にある数学的構造は、次のように推定できます：

$$ P(X) = 1 – prod_{i=1}^{k} (1 – frac{i-1}{W}) $$

係数 $W$ (ランダムウィンドウ) をネットワーク遅延 (Network RTT) を超えて拡大することにより、複数のノードが同時に立候補して票が割れる (split vote) 可能性は漸近的に0に近づきます。

2 リーダーの完全性 (Leader Completeness)

Paxos (すべてのノードがいつでもデータを伝播できる) とは異なり、Raftは純粋で不可逆的な情報フローを強制します：データはLeaderからFollowerへとしか流れません。Leaderが自身のログを上書きしたり削除したりすることは決してありません。

Raftは、コミット済みのすべてのログエントリ (committed) が含まれていない場合、CandidateがLeaderになることを禁止します。 「最新かどうか」 (up-to-date) の比較は、座標ペア $(Term, Index)$ 上の辞書式代数を使用して計算されます：

$$ (T_{last}^A > T_{last}^B) lor (T_{last}^A = T_{last}^B land Index_{last}^A > Index_{last}^B) $$

この冷酷な制約のおかげで、Raftの新しいLeaderは確実に古いLeaderのすべてのデータを含んでいます。したがって、選出後すぐに、RaftのLeaderはPaxosのように過去を掘り返す時間を無駄にすることなく、Read/Writeクエリを即座に処理できます。

マイクロアーキテクチャ、OS I/O、そしてテールレイテンシ (Tail Latency)

理論は理論ですが、RaftやPaxosを物理ハードウェア上で実行することは、メモリ帯域幅やハードウェア割り込み (Hardware Interrupts) といった物理法則との戦いです。

1 OSと異なるメモリの脅威 (fsync)

最も残酷な法則の1つは次のようなものです：システムがネットワークに応答 (ACK) する前に、すべての状態はNVMeドライブ (Non-volatile storage) 上の先行書き込みログ (WAL) にしっかりと固定されなければなりません。

Linux (POSIX) で fsync() 関数を呼び出すと、マイクロチップの嵐が巻き起こります：カーネルはRAM (Page Cache) からDMA経由で物理SSDドライブにバイトを強制的に押し出します。この鼓動のたびに15〜50マイクロ秒かかります。すべてのパケットに対して継続的に fsync を呼び出すと、システムのパフォーマンスが悲惨なレベル (数千 IOPS) に低下します。

解決策: RaftとMulti-Paxosはどちらも、Group Commit BatchingとカーネルバイパスI/Oテクノロジー (Kernel-bypass, io_uring, SPDK) を使用する必要があります。 以下の最適化されたRust擬似コードは、Zero-copyアーキテクチャのブレークスルーを表しています：

/// 内部のRaft状態: 共有されたアトミック変数はキャッシュラインを強制的に分離されます。
#[repr(align(64))]
pub struct AtomicRaftState {
    pub current_term: AtomicU64,
    pub commit_index: AtomicU64,
}

pub async fn handle_append_entries_optimized(
    &mut self, 
    request: ZeroCopyAppendEntriesReq<'_>
) -> AppendEntriesResp {
    let current_term = self.state.current_term.load(Ordering::Acquire);
    
    // 特性安全1: 時代遅れのLeaderを拒否する
    if request.term < current_term {
        return AppendEntriesResp { term: current_term, success: false };
    }
    
    // ロックフリーな任期の進化アルゴリズム (Lock-free Term Evolution)
    if request.term > current_term {
        self.state.current_term.store(request.term, Ordering::Release);
        self.voted_for.store(0, Ordering::Relaxed);
        // 非同期WALの保存
        self.wal.persist_metadata(request.term, None).await; 
    }
    
    // キャッシュラインバリアとI/Oバッチ処理
    let batch_bytes = request.extract_payload_zero_copy();
    self.wal.submit_sqe_write(batch_bytes);
    self.wal.await_cqe_fsync().await; // ディスクを待っている間、他のプロセスにスレッドを譲る

    AppendEntriesResp { term: request.term, success: true }
}

2 ガベージコレクタ (GC) のSTW (Stop-The-World) の脆弱性

Java (JVM) やGolangなど、ガベージコレクタを持つ言語によって編まれた分散システムにとって、Stop-The-World現象はサイレントキラーです。

GCの一時停止が200ミリ秒続く瞬間が、もろい election_timeout の境界を越える可能性があります。Followerノードは、現職のLeaderが突然死んだと誤って診断します。これにより不要な選挙の嵐 (Election Storm) が引き起こされ、処理能力が崩壊します。 ハイパースケール管理システム (TiKV、Redpanda) がC++やRustを選択する理由はこれです。手動のメモリ管理モデルにより、GCの不確実な遅延が完全に排除されます。

容赦なき戦い: いつRaftを選び、いつPaxosを選ぶべきか？

1 Out-of-Order Commits (Multi-Paxos) vs. Strict Linearity (Raft)

Multi-Paxos (例: Google Spanner) Multi-Paxosの究極の特権は、「順不同」のコミット能力 (out-of-order commits) です。ログ内の各スロット (インデックス座標) は独立した評議会であるため、パケット12が失われた場合でも、システムはスロット番号 $10, 11, 13$ を承認できます。システムは「ヘッドオブラインブロッキング」 (Head-of-Line Blocking) に悩まされることはありません。この機能は、TrueTime原子時計と組み合わせることで、Multi-Paxosをマルチリージョン地理分散 (Multi-Region Geo-Distributed) ストレージ用の無敵のマシンにします。

Raft (例: TiKV, CockroachDB, etcd) Raftは厳密な直線性を強制的に義務付けます (strict linearity)。インデックス $N$ をコミットするには、$1$ から $N-1$ までのすべてのインデックスが完全でなければなりません。パケット番号12が失われた場合、パケット番号12が正常に再送信されるまで、パイプライン全体が停止します。 この物理的なボトルネックは、分かりやすさ (Monolithic Transparency) を得るためにピークパフォーマンスを犠牲にします。これにより巨大な技術的負担が解放され、これが今日のクラウドネイティブ (Cloud-Native) のオープンソースの世界をRaftが完全に支配している理由です。

2 障害からの回復: Paxosの傷跡とRaftのPre-Vote

Leaderが崩壊した時：

  • Raft: 選出された新しいLeaderは、完全なデータを即座に持っています。定数コスト $O(1)$ で復旧します。
  • Multi-Paxos: 新しいLeaderは、空所を「埋める」ために、孤児となった (完全にコミットされていない) すべてのスロットをスキャンする巨大なPrepareフェーズを実行する必要があります。回復コストは $O(L)$ であり、非常に遅く、高価です。

しかし、ローカルネットワークが分断された場合、Raftには残酷な欠点があります。ネットワークから隔離されたFollowerノードは、継続的にLeaderを見つけることができません。選挙を試みるために、自身の任期 (Term) を数千、数万倍と自己増加させます。ネットワークが回復すると、この巨大な任期の数値をネットワークにもたらし、正当な現職Leaderに即時の退位を強要し、システムを破壊します。

これを克服するために、Raftのアーキテクトは Pre-Vote (事前投票) フェーズを追加する必要がありました。孤立したCandidateは「モック信任投票」を行わなければなりません。他のノードに「Leaderが見えますか？見えない場合のみ、任期を増やして立候補することが許可されます」と尋ねます。このメカニズムは完璧な毒素フィルターとして機能し、架空の任期インフレーション現象を完全に無効化します。

結論

RaftはPaxosと比較して数学的に後退しているわけではありません。それは実践的で天才的な技術的妥協の産物です。Paxosは分散コンセンサスの深淵なる限界を人類に教えましたが、Raftはすべてのソフトウェアエンジニアに、真の永遠のクラウドを創造するためのツールを与えました。Paxosのランダムな流れとRaftの直線的な単調さの違いを理解することは、プログラマーとコアインフラストラクチャアーキテクトを区別する境界線となります。
