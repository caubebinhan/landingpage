技術ホワイトペーパー: 現代データベースにおけるJust-In-Time(JIT)コンパイル — アーキテクチャ徹底講座

July 22, 2026 · admin · 4 分で読めます

エグゼクティブサマリー

リレーショナルデータベース管理システムの進化は、歴史的にクエリ実行エンジン(query execution engine)に付随するオーバーヘッドによって制約を受けてきた。数十年にわたり、データベースはディスクI/Oが主要なボトルネックであった時代向けに設計された、解釈実行(interpreted execution)モデルに依存してきた。しかし、データがインメモリ構造へと移行し、プロセッサが深いパイプラインを持つスーパースカラー(superscalar)の怪物へと進化するにつれ、この解釈オーバーヘッドは致命的なボトルネックと化した。

本ホワイトペーパーでは、現代データベースにおけるJust-In-Time(JIT)コンパイルを解剖する——これは、クエリを解釈された実行計画から、実行時(runtime)にその場で生成される、専用の高度に最適化されたマシンコードへと変換するパラダイムシフトである。我々は、クエリコストの数学的基盤、Volcanoイテレータモデルから LLVMベースのIR(中間表現)生成への移行、そして深遠なマイクロアーキテクチャ上の利点(L1i命令キャッシュの局所性、分岐予測、レジスタ割り当て)、さらに動的に生成されたコードを実行する際のオペレーティングシステム上の制約について探究する。この徹底講座を理解することで、エンジニアは、PostgreSQL(LLVMを用いた)やApache Spark Tungstenといった現代のエンジンが桁違いの性能向上をどのように達成しているかについて深い技術的洞察を得るとともに、そうしたシステムを実装する上での重要な教訓を学ぶことができる。

はじめに:核心となる問題提起

問題とは: なぜ従来型のデータベースエンジンは、分析クエリを実行する際に現代のCPUの能力を完全に活用できないのか?

従来型のデータベースシステムは主にVolcanoイテレータモデル(Volcano iterator model)に依存しており、クエリプランは演算子(operators)のツリーとして表現される。各演算子は標準インターフェースを実装し、next()メソッドを公開する。このメソッドは、タプルを生成するために子演算子から呼び出される。この設計は高い拡張性とクリーンな抽象化を提供する一方で、現代のスーパースカラープロセッサアーキテクチャ上では深刻な性能ボトルネックを引き起こす。

非効率性の主な原因は、タプル1件ごとの処理に必要な仮想関数呼び出し(動的ディスパッチ、dynamic dispatch)を広範に使用することにある。数十億行を処理する典型的なデータウェアハウジングのシナリオでは、これらの間接分岐命令(indirect branch instructions)によるオーバーヘッドが、頻繁なパイプラインフラッシュと命令キャッシュ(L1i)ミスを引き起こし、1サイクルあたりの命令実行数(IPC)を大幅に低下させる。

実行コストを形式化するため、$C_{eval}(t_i)$をタプル$t_i$上の述語を評価するのに必要なCPUサイクル数、$C_{dispatch}$を仮想関数のディスパッチオーバーヘッドとする。純粋に解釈実行されるエンジンにおける総実行時間$T_{interp}$は次のとおりである。

$$ T_{interp} = sum_{i=1}^{N} left( C_{dispatch} + C_{eval}(t_i) + C_{materialize}(t_i) right) $$

$N$が数十億にまで増加すると、$C_{dispatch}$がCPU時間を完全に支配するようになる。CPUは実質的に、データを実際に評価するよりも、次にどのコードを実行すべきかを判断することに多くの時間を費やしていることになる。

解決策:コード生成とPushベースのパイプライン

この本質的な限界を回避するため、現代のデータベース管理システムはJust-In-Time(JIT)コンパイルを採用する。JITコンパイルは、解釈された実行計画を、実行時に動的に生成される、高度に最適化されタイトに結合されたマシンコードへと変換する。

PullベースからPushベースの実行へ

クエリコンパイルの根本的な前提は、演算子が結合されたパイプラインへと融合される、データ中心(data-centric)のコードを生成することによって、解釈のオーバーヘッドを排除することである。仮想関数呼び出しを通じて演算子ツリーからタプルを引き出す(pull、Pullベース)代わりに、クエリコンパイルはPushベースの実行モデルを採用する。

このパラダイムでは、データはストレージ層から読み込まれ、集約ハッシュテーブル、ソートバッファ、あるいは最終的な結果セットといったマテリアライゼーションポイント(materialization point)に到達するまで、演算子パイプラインを通じて上方へ押し上げられる(push)。このPushベースのモデルは、現代のメモリ階層の特性と完璧に合致し、データ局所性を最大化し、キャッシュやメインメモリへ溢れる前にできる限り長くデータをCPUレジスタ内に留めておく。

特定のクエリ構造に合わせて設計された専用ループを合成することで、JITコンパイラは$C_{dispatch}$を事実上完全に排除する。

性能向上の数学

WHERE句における述語の数学的評価を考えてみよう。解釈実行モデルでは、複雑な式ツリーをタプルごとに走査しなければならない。式ツリーが$k$個のノードから成るとする。コスト$C_{eval}(t_i)$は$O(k)$回のポインタ参照解決(dereference)の関数となる。

対照的に、JITコンパイルは式ツリーを一連のスカラー算術演算へと畳み込み、$C_{eval}(t_i)$を$O(1)$回のネイティブCPU命令にまで低減する。性能向上係数$Gamma$は次のように表される。

$$ Gamma = frac{sum_{i=1}^{N} (C_{dispatch} + alpha cdot k)}{sum_{i=1}^{N} (beta) + T_{compile}} $$

$N to infty$のとき、JITコンパイルされたクエリの漸近的性能は厳格に支配的となり、その上限はメモリ帯域幅のみによって規定される。

JIT対ベクトル化(Vectorization)

ベクトル化(タプルを配列にバッチ化する)もまた、$C_{dispatch}$をベクトルサイズ$V$で割ることで償却する。しかし、ベクトル化は演算子間でのマテリアライゼーションオーバーヘッド(materialization overhead)を被る——データはキャッシュへ書き込まれ、再び読み戻されなければならない。JITコンパイルは、パイプライン全体を通じてタプルデータを厳密にCPUレジスタ内に留め続けることで、これを超越する。これにより、複雑で計算負荷の高いクエリに対しては、JITが数学的に優れていることになる。

アーキテクチャ統合:LLVM IRの生成

この変換を達成するために、データベースエンジンは通常、LLVMのような堅牢なコンパイラフレームワークを活用する。データベースのコードジェネレータは物理プランを走査し、Static Single Assignment(SSA)形式に基づく、強く型付けされたアーキテクチャ非依存のアセンブリ言語である、LLVM中間表現(LLVM Intermediate Representation、IR)を出力する。

コードジェネレータは、生成されたIR内でSQLのデータ型、null許容性のセマンティクス、およびMVCC可視性チェックをネイティブに、かつ細心の注意を払って扱わなければならない。SQLは3値論理(True、False、Unknown)を要求するため、生成されるコードには条件チェックが組み込まれる。これを最適化するため、コンパイラは投機的実行(speculative execution)とループアンスイッチング(loop unswitching)を用いて、カタログメタデータが非null制約を保証している場合には、null チェックを内側のループの外へ引き上げる。

// Hash Join Probeパイプライン向けのIR生成を示すRust風の疑似コード
fn generate_hash_join_probe_pipeline(
    builder: &mut IRBuilder, module: &mut IRModule, plan: &PhysicalJoinPlan
) -> Function {
    let pipeline_func = builder.create_function("HashJoinProbePipeline");
    let loop_block = builder.create_basic_block("loop", pipeline_func);
    let probe_block = builder.create_basic_block("probe_hash_table", pipeline_func);
    let emit_block = builder.create_basic_block("emit_joined_tuple", pipeline_func);
    
    // セットアップしてループへジャンプ
    builder.build_br(loop_block);
    
    // LOOPブロック:取得とハッシュ計算
    builder.set_insert_point(loop_block);
    let probe_tuple = emit_storage_layer_fetch(builder);
    let join_key = emit_expression_evaluation(builder, probe_tuple, plan.probe_key_expr);
    let hash_value = emit_murmur_hash3(builder, join_key);
    builder.build_br(probe_block);
    
    // PROBEブロック:ハッシュテーブルを確認
    builder.set_insert_point(probe_block);
    let bucket_ptr = emit_hash_table_lookup(builder, hash_table_ptr, hash_value);
    let is_match = emit_key_comparison(builder, bucket_ptr, join_key);
    builder.build_cond_br(is_match, emit_block, loop_block);
    
    // EMITブロック:一致を出力し継続
    builder.set_insert_point(emit_block);
    let combined_tuple = emit_tuple_concatenation(builder, probe_tuple, bucket_ptr);
    emit_pipeline_continuation(builder, combined_tuple, plan.parent_operator);
    builder.build_br(loop_block); 
    
    return pipeline_func;
}

OSレベルのメモリ管理とセキュリティ上の制約

JITコンパイルされた関数のメモリ管理は、OSレベルで深刻な課題をもたらす。LLVMがIRをマシンコードへと落とし込む際、生成されたバイナリは実行可能権限がマッピングされた物理メモリページへとフラッシュされなければならない。

$W oplus X$(Write XOR Execute)セキュリティ原則

現代のオペレーティングシステムは、バッファオーバーフロー攻撃を軽減するために$W oplus X$を強制する。メモリは書き込み可能かつ実行可能の両方を同時に満たすことはできない。データベースはシステムコール(POSIXではmprotect、WindowsではVirtualProtect)を編成しなければならない。

1. ページを読み書き可能(`PROT_READ PROT_WRITE`)として確保する。
  1. マシンコードを生成しリンクする。
3. ページを読み取り/実行可能(`PROT_READ PROT_EXEC`)へ遷移させる。
  1. CPUがメインメモリから新しい命令を確実にフェッチするよう、命令キャッシュをフラッシュする(__builtin___clear_cache)。

TLBスラッシングとHuge Pages

巨大なデータセットに対する実行にはTLB効率が求められる。JIT実行は処理速度を最大化する一方、ボトルネックをメモリアドレス変換(memory translation)へと移す。データベースは、Huge Pages(2MBまたは1GB)でマッピングされた専用のアリーナアロケータ(arena allocator)を管理しなければならない。これにより、単一のTLBエントリが広大な連続メモリ領域をカバーすることが保証され、実行中のTLBミスが事実上ほぼ排除される。

マイクロアーキテクチャの力学:CPUの解放

JITの真の力は、LLVMがアーキテクチャを意識した最適化パス(optimization pass)を適用したときに解き放たれる。

レジスタ割り当て(グラフ彩色)

変数はグラフ彩色(graph coloring)を用いて物理CPUレジスタへとマッピングされる。演算子を融合することで、中間変数(タプル属性)は極めて短い生存期間(live range)を持つようになる。干渉グラフ(interference graph)は疎なままとなり、すべてのデータが厳密にCPUレジスタファイル内に留まり続け、L1キャッシュへの読み書きを完全に迂回する確率を最大化する。

ループアンローリングとソフトウェアパイプライニング

ループアンローリングは、$N$回反復するループを$frac{N}{U}$回の反復へと変換する。より大きな基本ブロック(basic block)は、CPUのアウトオブオーダー実行エンジン(リオーダーバッファ、ROB)に対して、独立した命令からなる広大な見通し(horizon)を提供し、命令レベル並列性(Instruction-Level Parallelism、ILP)を最大化する。 ソフトウェアパイプライニングはループの反復を重ね合わせ、メモリレイテンシを隠蔽する。反復$i$がALUで評価されている間に、反復$i+1$はL1キャッシュからデータをフェッチしている。

L1iキャッシュと分岐予測

インタプリタは命令キャッシュ(L1i)を破壊する。なぜなら、タイトな内側ループには巨大なswitch文や間接ポインタが含まれ、CPUのスラッシングを引き起こすからである。JITコンパイルはロジックを連続した逐次命令の配列へと畳み込み、決定論的な命令プリフェッチを保証する。さらに、制御フロー分岐をデータフロー演算(cmov)へと変換することで、分岐予測ミスによるペナルティを排除する。

損益分岐点と適応的実行(Adaptive Execution)

コンパイルは無料ではない。LLVMの最適化パスを実行することは、相当量のCPUサイクルを消費する。では、いつJITコンパイルが数学的に正当化されるのか?

$T_{compile}$をコンパイルのレイテンシとする。$c_{interp}$と$c_{jit}$をそれぞれ、解釈実行パスとコンパイル実行パスにおけるタプルあたりの時間とする。$N$個のタプルに対し、JITが有益となるのは次の場合である。

$$ T_{compile} + N cdot c_{jit} < N cdot c_{interp} $$

あるいは$N$について解くと:

$$ N > frac{T_{compile}}{c_{interp} – c_{jit}} $$

小規模なOLTPクエリでは$N$が小さすぎ、コンパイルのオーバーヘッドが性能を損なう。一方、大規模なOLAPクエリでは$N$が膨大となり、JITは桁違いの性能改善をもたらす。

適応的実行(Adaptive Execution)

$T_{compile}$を隠蔽するため、高度なシステムは適応的実行(Adaptive Execution)を用いる。クエリは軽量なベクトル化インタプリタを用いて即座に開始される。同時に、バックグラウンドスレッドがJITコンパイルを開始する。コンパイルが完了すると、エンジンは実行途中で関数ポインタをホットスワップし、処理を中断することなく最大スループットへとシームレスに移行する。

学びの要点とベストプラクティス

  1. コンパイルオーバーヘッドは現実のものである: すべてのクエリに盲目的にJITを適用すると性能の退行(regression)を引き起こす。堅牢なコストベースのヒューリスティクスと、パラメータ化クエリに対するコンパイル済みバイナリを再利用するためのクエリプランキャッシュを実装せよ。
  2. JITに対してはPushベースがPullベースに勝る: VolcanoのPullベースモデルをそのままJITコンパイルしようとしても得られる効果はごくわずかである。レジスタ局所性を最大化するには、アーキテクチャをPushベースの、パイプライン融合型モデルへ移行させなければならない。
  3. OSとのやり取りは重要である: JITコードに対する頻繁なmprotect呼び出しや断片化したメモリ割り当てはレイテンシを引き起こす。専用のアリーナアロケータとHuge Pagesを使用せよ。
  4. C-ABIへのフォールバック: すべてをJITコンパイルしようとしてはならない。複雑なロジック(正規表現の解析、ロックの取得、バッファプールのスピル)は、JITコードが標準のC-ABI経由で呼び出す、静的にコンパイルされたC/C++関数として実装すべきである。

結論

JITコンパイルの統合は、データベースを単なるアプリケーションソフトウェア層として扱うことから、専用の動的コンパイラとして扱うことへの、根本的なパラダイムシフトを意味する。実行時にカスタムマシンコードをその場で生成することにより、現代のデータベースは、高水準の宣言的クエリ言語(SQL)と、現代のハードウェアプラットフォームが持つ低水準で容赦のないマイクロアーキテクチャの現実との間にある、歴史的な意味論的ギャップを架橋し、シリコンの真の潜在能力を解き放つ。
