`io_uring` vs `epoll`: The New Era of Asynchronous I/O in Database Architecture

July 22, 2026 · admin · 11 min read

Executive Summary & Core Problem Statement

For more than two decades, the Linux server ecosystem has leaned on the epoll mechanism (introduced in kernel 2.5) to overcome the C10K problem (handling 10,000 concurrent connections). However, as storage hardware evolved from mechanical spinning disks (HDDs — millisecond-scale latency) to PCIe NVMe Gen 4/5 solid-state drives (microsecond-scale latency), epoll and the traditional I/O model exposed critical, fatal weaknesses at the micro-architectural level.

The core problem: Modern database engines (such as ScyllaDB, Aerospike, and PostgreSQL) are being “strangled” in performance not by the physical limits of the storage medium, but by the software overhead of the Linux operating system itself. To perform asynchronous I/O operations using epoll or the old POSIX AIO standard, the CPU must continuously cross the boundary between User Space and Kernel Space. Every system call (syscall) such as read(), write(), or epoll_wait() demands thousands of CPU clock cycles for context switching, page table protection (KPTI), and data copying. In a high-speed NVMe environment, the CPU ends up spending more time “handling administrative paperwork” with the OS than actually moving data.

The arrival of io_uring (from Linux kernel 5.1) represents a revolution that tears down and completely rebuilds Linux’s I/O architecture. By establishing two memory-mapped (mmap) Ring Buffers shared directly between User Space and Kernel Space, io_uring allows an application to submit millions of I/O requests and receive their results without performing a single System Call.

This in-depth technical report dissects the quantum-level principles of io_uring‘s micro-architecture, contrasts it with the physical limitations of epoll and linux-aio, and analyzes how database management systems (DBMS) are redesigning their entire core engines to exploit this technological marvel.

The Crisis of epoll and the Origins of Asynchronous I/O

The Nature of epoll (Event-Driven Readiness)

epoll was born for Network I/O (network sockets). Its execution model is bound by the nature of readiness notification rather than completion notification.

The workflow:

  1. You call epoll_wait() (blocking the thread, waiting).
  2. When the network card receives a packet, the Kernel wakes you up: “Socket 5 has data, go read it.”
  3. The application must then invoke a further read(5) syscall to actually copy the data from the Kernel buffer into the User buffer.

The total latency $T_{epoll}$ can be modeled mathematically as:

$$T_{epoll} = t_{syscall(epoll_wait)} + t_{ctx_switch} + t_{syscall(read)} + t_{vfs_lookup} + t_{hardware_io} + t_{interrupt}$$

The epoll Disaster for Disk Storage I/O

epoll works beautifully for networking, but fails utterly and catastrophically with local disk storage (File System I/O). On Linux, regular storage files (ext4, xfs) ALWAYS report a “ready” status to epoll. When you call epoll_wait() for a file on disk, it returns immediately. But when you call read(), if the data is not already resident in RAM (a Page Cache Miss), the Linux kernel will silently block your application thread while it waits for the data to arrive from disk.

This “silent blocking” behavior completely violates the design principle of the Event Loop. A single stalled read() call can freeze thousands of other client connections running on that same NodeJS or NGINX thread.

The Collapse of Linux AIO (io_submit)

To handle asynchronous Disk I/O, Linux once offered the linux-aio interface (via io_submit and io_getevents). However, Linus Torvalds himself publicly criticized this architecture. It carried a host of flaws:

  1. Only supports O_DIRECT: You must manage your own buffering, bypassing the Page Cache entirely. It does not work with Buffered I/O.
  2. Still gets blocked by metadata operations: Even with O_DIRECT, io_submit can still block if the file system needs to perform block allocation or acquire an inode lock.
  3. Syscall overhead: Submitting a single batch of requests still costs at least one io_submit syscall.

To dodge these poor designs, databases such as PostgreSQL, MySQL, and MongoDB were forced to create massive Thread Pools. They dump blocking read/write calls onto background threads so as not to affect the main thread. The constant context-switching among hundreds of these I/O threads creates a thrashing effect on the CPU’s L1/L2 caches.

The Micro-architecture of io_uring: The Shared Memory Marvel

io_uring was designed by Jens Axboe (the maintainer of the Linux Kernel’s Block Layer) to eliminate every weakness of the previous generations. It shifts from the “Readiness” model to a “Completion” model. You hand the Kernel a description of the work, the Kernel handles it from A to Z, and once done, it writes the result back into memory for you.

Lock-Free Ring Buffer Data Structures

io_uring establishes two one-directional circular ring buffers:

  1. Submission Queue (SQ): The queue for submitting tasks (from User -> Kernel). It contains SQE (Submission Queue Entry) slots. Each SQE describes a command such as: “Read 4KB from File X into Buffer Y.”
  2. Completion Queue (CQ): The queue for receiving results (from Kernel -> User). It contains CQE (Completion Queue Entry) slots. Each CQE reports: “The previous command finished reading, returning error code 0 (Success).”

These two array structures are memory-mapped (mmap) directly into the application process’s virtual memory. This means the application and the Kernel are literally looking at the same span of RAM.

Synchronization via Memory Barriers

There is not a single lock (Mutex/Spinlock) used here! Synchronization relies entirely on atomic Head and Tail pointers combined with hardware ISA-level Memory Barriers.

When the Database Engine wants to submit 100 write tasks to disk, it simply writes the data into 100 SQE slots in RAM, then increments the SQ_Tail pointer. Next, it makes a single io_uring_enter() syscall to “ring the bell” and wake the Kernel up to process the work. Instead of 100 write() syscalls, the system now costs just 1.

The Ultimate Frontier: Eliminating Syscalls Entirely with SQPOLL

To reach the pinnacle of ultra-low latency, io_uring offers the IORING_SETUP_SQPOLL configuration flag. When this feature is enabled, the Kernel spawns a dedicated kernel thread pinned to a CPU core. This thread continuously runs a polling loop watching the application’s SQ_Tail pointer.

When the application pushes a request into the SQ, the Kernel thread sees it immediately through the shared memory region and automatically picks it up for processing. At this point, the number of System Calls is exactly zero. The application submits I/O and receives results without triggering a single context switch. The I/O latency at this point ($T_{iouring_sqpoll}$) is reduced to just the time it takes for electrical signals to travel across the PCIe bus:

$$T_{iouring_sqpoll} = t_{mem_barrier} + t_{pcie_dma_transfer} + t_{nvme_flash_prog}$$

Advanced Weapons for Database Engines

io_uring doesn’t just solve basic I/O — it also arms database engines with an entire arsenal of rocket launchers.

Registered Fixed Buffers

Normally, when calling read() or write(), the Linux kernel must set up an iovec structure, map the User’s RAM pages into the IOMMU (Input-Output Memory Management Unit) so the hardware controller can DMA directly, and then unmap them again afterward. This procedure is extremely expensive. With io_uring, a database can pre-register a massive block of RAM ahead of time. The Kernel pins that RAM region physically and pre-configures the IOMMU mapping. Afterward, every I/O operation (IORING_OP_WRITE_FIXED) flies straight from the disk controller into that exact RAM region without any address-translation procedure whatsoever.

Chained Requests (Linked SQEs)

In a database, I/O ordering is a matter of life and death. For example: you must write data to disk (Command 1), ensure it is flushed (Command 2 — fsync), and only then update the metadata (Command 3). In Linux AIO, you had to do this sequentially: wait for Command 1 to finish -> submit Command 2 -> wait for Command 2 to finish -> submit Command 3. In io_uring, you use the IOSQE_IO_LINK flag. You submit all 3 commands into the queue at once. The Kernel guarantees that Command 2 only runs if Command 1 succeeded. This drastically minimizes the number of interactions between User and Kernel.

Unifying Network and Disk I/O

The greatest breakthrough of io_uring is that it supports EVERY TYPE of operation (Network, File, Timeout, fsync, fallocate, and so on). A database no longer needs to maintain 2 separate architectures — one epoll for Network Sockets, and one Thread Pool for Disk I/O. An architect only needs to set up a Single Event Loop for a single CPU core. That one core can accept a TCP connection (IORING_OP_ACCEPT), read an HTTP request (IORING_OP_RECV), and write data to disk (IORING_OP_WRITEV) — all funneled into one single io_uring ring.

A Non-Blocking C++ Implementation

Below is a simplified pseudo-code anatomy in C++ of a modern storage engine built on liburing. Notice how the application attaches a C++ class object (context) to user_data in order to match up commands when their results come back.

#include <liburing.h>
#include <memory>
#include <cstdint>
#include <iostream>
#include <stdexcept>

// Custom request structure carrying internal application context
struct IOTransactionContext {
    int file_descriptor;
    uint64_t disk_offset;
    std::unique_ptr<char[]> memory_buffer;
    size_t length;
    uint32_t transaction_id;
};

class UltraFastStorageEngine {
private:
    struct io_uring ring;
    const unsigned int RING_DEPTH = 4096;

public:
    UltraFastStorageEngine() {
        struct io_uring_params params = {};
        // Maximum optimization: use SQPOLL to eliminate syscalls entirely
        params.flags |= IORING_SETUP_SQPOLL;
        params.sq_thread_idle = 2000; // Kernel thread sleeps after 2ms of no work
        
        if (io_uring_queue_init_params(RING_DEPTH, &ring, &params) < 0) {
            throw std::runtime_error("Kernel does not support this or ulimit restricts it!");
        }
    }

    void submit_async_write(IOTransactionContext* ctx) {
        // Fetch an empty SQE slot from the shared Ring Buffer
        struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
        if (!sqe) {
            // SQ is full, proactively submit so the Kernel can consume some entries
            io_uring_submit(&ring);
            sqe = io_uring_get_sqe(&ring);
        }
        
        // Configure the low-level asynchronous write opcode
        io_uring_prep_write(sqe, ctx->file_descriptor, 
                            ctx->memory_buffer.get(), ctx->length, ctx->disk_offset);
                           
        // CRITICAL: attach the C++ object pointer to the SQE's metadata (64-bit integer)
        io_uring_sqe_set_data(sqe, ctx);
    }

    void reap_completions_lockfree() {
        struct io_uring_cqe *cqe;
        unsigned head;
        unsigned count = 0;

        // Scan the CQ Ring Buffer entirely in local RAM (No Syscalls)
        io_uring_for_each_cqe(&ring, head, cqe) {
            // Cast the user_data pointer back to restore the context
            IOTransactionContext* ctx = static_cast<IOTransactionContext*>(io_uring_cqe_get_data(cqe));
            
            if (cqe->res < 0) {
                std::cerr << "I/O Error on Transaction " << ctx->transaction_id 
                          << " (Error code: " << cqe->res << ")n";
            } else {
                // Handle successful business logic
                finalize_transaction(ctx, cqe->res);
            }
            count++;
        }
        
        if (count > 0) {
            // Update the atomic CQ Head to tell the Kernel we've harvested these entries
            io_uring_cq_advance(&ring, count);
        }
    }

private:
    void finalize_transaction(IOTransactionContext* ctx, int bytes_written) {
        // Send an ACK back to the client over the network, or mark the WAL
        delete ctx; // Clean up memory
    }
    
    ~UltraFastStorageEngine() {
        io_uring_queue_exit(&ring);
    }
};

Lessons Learned & Best Practices for Systems Architects

The generational shift from epoll to io_uring is well underway (Redis 7.0, PostgreSQL 15, and NodeJS 20+ have begun experimenting with it). However, wielding this sword requires deep understanding.

  1. Understand the Limits of the OS Cache: If your application calls io_uring to read a file whose data already sits in the OS Page Cache (RAM), the Kernel’s background io_wq worker thread has to step in, causing slowdowns. io_uring reaches its absolute peak performance (leaving every other mechanism far behind) when combined with the O_DIRECT (Direct I/O) flag. The database must build its own Buffer Pool and bypass the operating system entirely.
  2. Beware of Security Vulnerabilities: Directly mapping Kernel structures into User Space via mmap has historically been associated with a number of privilege-escalation risks (CVEs). Container systems (Docker, Kubernetes) or SELinux often disable io_uring by default (via seccomp filters). DevOps engineers need to explicitly unlock it (whitelist the io_uring syscalls) before the database can even start.
  3. Manage the Memory Lifecycle Carefully: When programming with io_uring in C/C++/Rust, the buffer you supply to the Submission Queue (SQE) must remain valid until you receive the Completion notification (CQE). If a careless thread frees (delete/free) that buffer before the disk finishes writing, the Kernel will DMA data into a garbage RAM region, causing Memory Corruption that can destroy the entire system.
  4. Use Polling I/O (IOPOLL) for Ultra-Fast NVMe: If you’re using Intel Optane or PCIe Gen 5 drives with response latencies under 10 microseconds, the hardware interrupt (IRQ) mechanism used to signal completion becomes a bottleneck itself (a context switch costs 3-4 us). Enable the IORING_SETUP_IOPOLL flag so the Kernel busy-waits, polling the SSD’s status instead of sleeping. This will burn 100% of a core’s CPU, but it delivers latency approaching the physical quantum limit.
  5. Don’t Rush to Tear Down epoll for Web Servers: Although io_uring can handle networking (network sockets) as well, benchmarks show that for typical web servers (short-lived HTTP/TCP connections), io_uring‘s performance advantage over epoll is not large enough to justify the complexity of a full architectural rewrite. io_uring is the absolute king in the Storage domain, but in the Networking domain, epoll remains a battle-hardened veteran.
