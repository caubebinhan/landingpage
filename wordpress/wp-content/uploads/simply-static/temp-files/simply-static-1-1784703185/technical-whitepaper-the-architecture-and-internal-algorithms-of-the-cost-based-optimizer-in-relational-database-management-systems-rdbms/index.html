Technical Whitepaper: The Architecture and Internal Algorithms of the Cost-Based Optimizer in Relational Database Management Systems (RDBMS)

July 22, 2026 · admin · 10 min read

Executive Summary

As data volumes explode exponentially, the ability to retrieve and process data quickly has become a core competitive advantage for every business. At the heart of this process in relational database management systems (RDBMS) lies query optimization. This article provides an in-depth analysis of the Cost-Based Optimizer (CBO) — a core RDBMS component that acts as the “brain” deciding how SQL statements get executed.

We will dive into the micro-architecture, the mathematical cost-evaluation models, dynamic-programming algorithms, cardinality-estimation mechanisms, and the impact of hardware and operating-system behavior on the CBO’s decisions. By the end of this whitepaper, you will have a firm grasp of how the CBO works internally, the thorniest problems in optimizing complex queries, and practical lessons you can apply to database design and maintenance.

Preface and Defining the Core Problem

Query optimization in relational database management systems (RDBMS) is one of the most computationally complex problems in computer science, requiring a tight fusion of graph theory, probability and statistics, and computer systems architecture.

What is the problem? When a user submits a logical query (typically SQL), the RDBMS must translate that logical query into a physical execution plan capable of interacting with disk and RAM. For a query involving multiple complex joins, the number of feasible execution plans can reach into the millions. Choosing the wrong execution plan (for example, a full table scan instead of using an index, or the wrong join order) can turn a response time of a few milliseconds into hours — or even crash the system outright (Out of Memory).

The heart of the solution to this problem is the Cost-Based Optimizer (CBO). Unlike primitive Heuristic-Based or Rule-Based Optimizers that rely solely on rigid rules (such as “always use an index if one exists”), the Cost-Based Optimizer uses a comprehensive mathematical model to simulate the time and resource cost of tens of thousands of potential execution plans before deciding on the optimal route.

The CBO’s decision-making process depends heavily on:

  • The structure of the query itself.
  • The statistical characteristics of the data (metadata).
  • The hardware configuration (CPU, I/O bandwidth).
  • The operating system’s buffer/cache management mechanisms.

The core difference between an excellent optimizer (such as PostgreSQL’s or Oracle’s) and a mediocre database lies in the sophistication of the algorithm used to decompose the search-space model. Because the computational complexity of this problem falls into the NP-Hard class, systems must rely on continuous approximation strategies, using stochastic statistical estimation.

Anatomy of the Cost-Based Optimizer’s Architecture

An execution plan is, in essence, a tree of physical operators (Physical Execution Tree).

  • Leaf nodes: represent base data-access methods (Sequential Scan, B+ Tree Index Scan, Bitmap Scan).
  • Intermediate nodes: represent relational algebra operations such as Join (Hash, Merge, Nested Loop), Aggregation, and Sort.

The CBO must quantify the cost of each node, calculate the data flow through the pipelines, and sum up the total resource consumption to arrive at a single overall cost figure.

The optimization cycle is divided into phases:

  1. Parser/Analyzer: compiles SQL into an Abstract Syntax Tree (AST) and then a Logical Plan.
  2. Logical Optimization (Rule-based): simplifies expressions, pushes filter conditions downward (Predicate Pushdown).
  3. Physical Optimization (Cost-based): traverses the search space, using the Cost Model to score candidates and select the optimal physical plan.

The Mathematical Cost Model and Hardware Resource Estimation Functions

At the core of the CBO’s quantification process is building a cost model that faithfully mirrors real hardware operation. The total cost model $C_{total}$ is a composite function of disk I/O, CPU compute cycles, main memory allocation, and network latency.

The general cost function can be expressed as:

$$C_{total} = W_{IO} cdot C_{IO} + W_{CPU} cdot C_{CPU} + W_{MEM} cdot C_{MEM} + W_{NET} cdot C_{NET}$$

Here, the weights ($W$) are tuned through configuration to accurately reflect the hardware (e.g., SSD versus HDD).

Decomposing I/O cost ($C_{IO}$)

The $C_{IO}$ component estimates the cost of reading/writing data, broken down into sequential reads and random reads. On HDD, random-read cost is far greater than sequential-read cost. On SSD, this gap narrows, but random reads remain more expensive due to virtual block-size constraints.

$$C_{IO} = N_{seq} cdot C_{seq} + N_{rand} cdot C_{rand} + N_{dirty_flush} cdot C_{write_barrier}$$

Decomposing CPU cost ($C_{CPU}$)

CPU cost is modeled based on the number of tuples passing through each physical operator, predicate evaluation, and hashing computation.

$$C_{CPU} = N_{tuples} cdot C_{tuple_eval} + N_{index_probes} cdot C_{index_lookup} + N_{hash_collisions} cdot C_{resolution_penalty}$$

Accurately simulating CPU cost requires the CBO to maintain a detailed count of row-set size (Cardinality). To do this, the CBO relies on the Estimator.

Cardinality Estimation: The Heart of the CBO and Its Probabilistic Data Structures

The selectivity of predicate $P$ is the fraction of records satisfying that predicate.

$$Sel(P) = frac{|sigma_P(R)|}{|R|}$$

Traditionally, the CBO assumes statistical independence when multiple conditions are combined (AND, OR).

$$Sel(P_1 land P_2) = Sel(P_1) cdot Sel(P_2)$$

A common pitfall: in practice, attributes are often strongly correlated (for example, country = 'Vietnam' and area_code = '+84'). The independence assumption leads to severe under-estimation. The CBO may believe the result set contains only 1 row instead of 1 million, and mistakenly choose a Nested Loop Join, destroying system performance.

To address this, modern CBOs employ:

  1. Multi-dimensional statistics: evaluating cross-correlation between columns.
  2. V-Optimal Histograms: a data-partitioning algorithm that minimizes Mean Squared Error, helping predict uneven data distributions (data skew).
  3. HyperLogLog (HLL) and Count-Min Sketch: for counting distinct elements (Count Distinct) at massive scale, HLL allows approximate, randomized estimation with only a few percent of error, while consuming only kilobytes of RAM.

$$SE approx frac{1.04}{sqrt{m}}$$

Where $m$ is the number of bit registers, this algorithm achieves extreme optimality without disrupting the processor’s L1/L2 cache bandwidth.

Dynamic Programming Algorithms and the Search Space

The search space of execution plans is an enormous combinatorial barrier. For $N$ tables to be joined:

  • Considering only Left-Deep Trees (linear trees favoring pipeline data flow): the number of permutations is $N!$.
  • Considering Bushy Trees as well (branching trees supporting parallel joins): the count balloons to a variant of the Catalan number sequence.

$$text{Total number of Bushy Trees} = frac{(2N-2)!}{(N-1)!}$$

The System R algorithm (originated by IBM Research) uses Dynamic Programming to memoize the best join configurations for every subset of tables. By applying Bellman’s Principle of Optimality, the system can prune away the worst-performing configurations.

$$OptPlan(S) = min_{S_1, S_2 subset S, S_1 cap S_2 = emptyset} { Cost(OptPlan(S_1) bowtie OptPlan(S_2)) }$$

However, this bottom-up traversal architecture struggles when interactive, top-down demand-driven feedback is needed — for instance, when it is preferable to return data that is already sorted.

The Cascades Framework: A Revolution in Optimizer Design

Used by Microsoft SQL Server, CockroachDB, and Apache Calcite, the Cascades architecture thoroughly solves System R’s problem.

Cascades combines top-down traversal with a hypergraph structure called the Memo (Memoization Graph). The Memo stores equivalence classes — expressions that share the same logical result but different physical plans.

Its core advantage lies in the Physical Properties Demand. When a parent operator (for example, GROUP BY X) requires data to be sorted by X, it forces child subtrees to search for physical plans satisfying that requirement, thereby favoring a Merge Join (which preserves sort order) over a Hash Join (which destroys sort order) — even if the Hash Join would be cheaper.

The following Rust code illustrates the Branch and Bound cost-pruning process inside the Memo Graph:

use std::collections::HashMap;
use std::sync::{Arc, RwLock};

#[derive(Clone, Hash, PartialEq, Eq)]
struct LogicalExpressionId(u64);

#[derive(Clone)]
struct PhysicalPlan {
    cost: f64,
    operator_type: String,
}

struct MemoTable {
    best_plans: RwLock<HashMap<LogicalExpressionId, PhysicalPlan>>,
}

impl MemoTable {
    fn new() -> Self {
        MemoTable { best_plans: RwLock::new(HashMap::new()) }
    }

    fn optimize_group(&self, group_id: &LogicalExpressionId, current_upper_bound: f64) -> Option<PhysicalPlan> {
        {
            let read_guard = self.best_plans.read().unwrap();
            if let Some(cached_plan) = read_guard.get(group_id) {
                if cached_plan.cost <= current_upper_bound {
                    return Some(cached_plan.clone()); // Prune if it exceeds the bound
                }
            }
        }

        // Rule engine generates candidate variants
        let candidates = vec![
            PhysicalPlan { cost: 1500.0, operator_type: "GraceHashJoin".to_string() },
            PhysicalPlan { cost: 800.0, operator_type: "ParallelMergeJoin".to_string() },
        ];

        let mut local_best: Option<PhysicalPlan> = None;
        let mut min_cost = current_upper_bound;

        for candidate in candidates {
            if candidate.cost >= min_cost { continue; }
            min_cost = candidate.cost;
            local_best = Some(candidate);
        }

        if let Some(ref best) = local_best {
            let mut write_guard = self.best_plans.write().unwrap();
            write_guard.insert(group_id.clone(), best.clone());
        }
        local_best
    }
}

The Impact of Hardware Structure and Operating-System Memory (Hardware-Aware CBO)

The cost model often fails catastrophically when it collides with physical barriers: memory fragmentation, paging, and the cache hierarchy.

Grace Hash Join — the nightmare of memory overflow

When running a Hash Join, if the hash table fits neatly within L3 cache, probe speed is extremely fast (a few nanoseconds). If it spills into RAM, latency rises due to TLB misses. But if the hash table balloons beyond the allotted RAM (Out of Memory), the system must trigger a Grace Hash Join — splitting the data in half and writing it out to disk.

At that point, I/O cost explodes according to the formula:

$$C_{hash_join} = 3 cdot (|R| + |S|) cdot C_{IO_seq} + C_{cpu_partitioning}$$

(Data must be read from RAM, written to disk, then read back for probing.) At this stage, if the optimizer had recognized this disk-spill event early, it might have chosen a Sort Merge Join instead of a Hash Join.

NUMA Architecture and Hardware Prefetching

On multi-CPU servers (NUMA), accessing memory across sockets is extremely expensive. A modern NUMA-aware CBO applies a penalty to the cost when data lacks locality. At the same time, the CBO also fine-tunes sequential I/O cost downward, benefiting from the processor’s Hardware Prefetcher (which automatically loads data into cache before the CPU actually requests it).

Lessons Learned and Practical Experience

After a thorough study of the CBO’s internals, users, data engineers, and database administrators (DBAs) can draw the following valuable lessons:

  1. Understand the root cause instead of blaming the platform:

    2. When a query runs slowly, don’t rush to switch database platforms. The core issue often lies in the CBO lacking, or having incorrect, information about data distribution (Cardinality Misestimation).

  2. Always maintain statistics:

    3. Make sure the system regularly runs commands such as ANALYZE (PostgreSQL) or GATHER_STATS (Oracle). Stale statistics turn the CBO blind, causing it to choose a Nested Loop Join for billions of rows instead of a Hash Join.

  3. The danger of data correlation:

    4. Avoid stacking too many AND/OR conditions on tightly related columns (such as city and postal code) unless the RDBMS supports Cross-Column Statistics. Declare multi-dimensional statistics to remedy this situation.

  4. Be aware of hardware resource limits (memory configuration):

    5. Pay attention to RAM-allocation parameters such as work_mem in PostgreSQL. If it’s set too small, Sort and Hash operations will spill to disk (Grace Hash Join), degrading performance by tens of times. If set too large, it can crash the entire server’s RAM.

  5. Understand physical constraints:

    6. Ordering data physically on disk (Clustered Index) grants the CBO an enormous, ongoing advantage, because the system inherits the “already sorted” property (Interesting Orders), enabling it to exploit the cheap Merge Join algorithm.

Conclusion

The Cost-Based Optimizer represents the pinnacle of the intersection between software engineering, advanced data structures, and optimization theory. By understanding the smallest gears in this machine — from histograms, to the Memoization mechanism in Cascades, to the impact of L3 cache and NUMA architecture — you can write queries that the optimizer “loves,” building applications that run at peak performance and serve millions of users smoothly.
