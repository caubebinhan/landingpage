Whitepaper: The Micro-architecture and Optimization Engineering of Skip Lists in Redis ZSET

July 22, 2026 · admin · 11 min read

Executive Summary (Overview)

Redis (Remote Dictionary Server) has established the gold standard in the world of in-memory database management systems by delivering outstanding performance together with a rich collection of data structures. Among these, the Sorted Set (ZSET) stands out as one of the most sophisticated, distinctive, and widely used data structures Redis offers. ZSET maintains a set of unique elements, where each element is bound to a floating-point score. These elements are automatically ordered by score, enabling real-time positional lookups, score-range queries, and rank queries with optimal performance.

This whitepaper serves as an in-depth technical deep-dive into how Redis implements ZSET. At the core of this architecture lies a clever combination of a Hash Table and an enhanced variant of the Skip List. The document dissects, in detail, why Redis rejected traditional Balanced Binary Search Trees, the mathematics underlying the Skip List algorithm, the micro-level memory allocation mechanisms built around jemalloc, the interplay with CPU cache architecture and the operating system, and finally real-world industry applications. The goal is to give systems engineers a thorough, multi-dimensional view of the historical trade-off decisions made by Redis’s core team.

Core Problem Statement

In computer science, the problem of maintaining a dynamic, ordered dataset while supporting search, insert, delete, and range-query operations with $mathcal{O}(log N)$ asymptotic complexity is a classic challenge.

When the Redis engineers faced the task of building ZSET, they had to weigh the standard data structures available:

  1. AVL Trees / Red-Black Trees: These guarantee a worst-case time complexity of $mathcal{O}(log N)$ for all operations. However, maintaining the tree’s balanced shape is extremely costly. Every insertion or deletion in a Red-Black Tree requires rotations and recoloring. Although Redis itself is single-threaded and therefore unaffected by lock contention, the constant rotations consume a large number of CPU cycles, increasing local latency jitter. Furthermore, executing range queries (ZRANGE) requires either an expensive in-order traversal or maintaining additional parent pointers, which bloats the size of each object.
  2. B-Tree / B+ Tree: These are the undisputed kings of disk-based storage due to their excellent optimization for cache lines and OS page sizes. However, in Redis’s in-memory environment, splitting and merging B-Tree nodes incurs heavy memory-copying overhead, and the implementation is far too complex — running counter to Redis’s philosophy of “keeping the source code simple.”

Faced with the rigid limitations of deterministic balancing models, the Skip List data structure — introduced by William Pugh in 1990 — emerged as a savior. The Skip List abandons strict structural balance in favor of probabilistic balancing achieved through a randomized algorithm. Its simplicity of implementation, snappy operation speed, natural support for range queries via the bottom-level linked list, and flexible tunability were the deciding factors behind Redis’s choice of the Skip List as the core of ZSET.

Deep Technical Knowledge / Internals

Dual Composite Structure: Dict + Skip List

To satisfy the demanding requirements of its API functions, a Redis ZSET is not merely a plain Skip List. It is a synchronized, polymorphic data structure composed of:

  • dict (Hash Table): Provides the ability to look up the score of any element in $mathcal{O}(1)$ time.
  • zskiplist: Preserves the linear ordering of the data, enabling rank queries and range retrieval in $mathcal{O}(log N)$ time.

This combination is a textbook example of the Space-Time Tradeoff principle. Redis accepts the extra memory cost of storing a hash table in order to supercharge the response time of the ZSCORE command.

Micro-structure: zskiplistNode and the span Metadata

William Pugh’s original Skip List design never solved the problem of rank counting. How do you determine that element $X$ sits at a given position among millions of elements in $mathcal{O}(log N)$ time? Redis solved this with a brilliant innovation: adding a span variable to the zskiplistLevel structure.

The standard C source for the Redis Skip List node structure:

/* Structure defining a Skip List node */
typedef struct zskiplistNode {
    sds ele;                              // Dynamic SDS string holding the element's content
    double score;                         // Sorting score (IEEE 754 double precision)
    struct zskiplistNode *backward;       // Backward pointer (level 0) supporting ZREVRANGE
    struct zskiplistLevel {
        struct zskiplistNode *forward;    // Forward pointer (points to the next node at the same level)
        unsigned long span;               // Distance (number of physical nodes skipped)
    } level[];                            // Flexible Array Member holding the node's levels
} zskiplistNode;

/* List control block */
typedef struct zskiplist {
    struct zskiplistNode *header, *tail;
    unsigned long length;                 // Total number of elements
    int level;                            // Current highest level of the list
} zskiplist;

How does the span variable work? The span variable precisely measures the logical distance between the current node and the target node that the forward pointer points to. During a search, the ZRANK command simply accumulates the span value across every edge of the graph the algorithm traverses on its path from the header node to the target node. This turns what would be an $mathcal{O}(N)$ sequential counting problem into a lightning-fast $mathcal{O}(log N)$ cumulative arithmetic operation.

The Mathematics Behind Randomized Level Generation

Whenever a new element is inserted into a ZSET, the algorithm determines how many levels that node will have. Redis simulates this using a Bernoulli process.

  • Every node is guaranteed to have at least 1 level (Level 1).
  • The probability of climbing an additional level is $p = 0.25$.
  • The coin-flip process (a micro dice roll) repeats until it fails or reaches the ZSKIPLIST_MAXLEVEL cap (currently 32).

The probability that a node has exactly $k$ levels follows a Geometric Distribution:

$$ P(L=k) = p^{k-1}(1-p) $$

The expected number of levels per node, $mathbb{E}[L]$, is:

$$ mathbb{E}[L] = sum_{k=1}^{32} k cdot p^{k-1}(1-p) approx frac{1}{1-p} $$

Substituting $p = 0.25$, we get $mathbb{E}[L] = 1.33$. In other words, on average, each node has only 1.33 levels. Instead of bearing the physical burden of 2 pointers as in a binary tree, the Skip List’s asymmetric design shrinks the pointer metadata overhead down to just $1.33 times text{sizeof(pointer)}$ for the forward structure. This is a spectacular structural compression ratio!

Memory Management (Jemalloc) and the Flexible Array Member

Redis delegates heap allocation to jemalloc (or tcmalloc), a custom allocator optimized for highly fragmented workloads. Looking at the zskiplistNode structure, Redis exploits the C99 Flexible Array Member (FAM) trick (level[]) placed at the end of the struct. This lets Redis allocate a single node with just one system call:

$$ text{Size} = text{sizeof}(zskiplistNode) + (k) times text{sizeof}(zskiplistLevel) $$

The returned memory block is fully contiguous for each node’s metadata. This design completely eliminates internal fragmentation within each object.

However, the flip side of dynamic per-node allocation is an increased risk of external fragmentation as a ZSET is repeatedly resized. jemalloc absorbs this risk fairly well, but SysAdmin engineers must still keep a constant eye on the mem_fragmentation_ratio metric via the INFO MEMORY command.

Friction with the CPU Cache and Branch Predictor

Despite its algorithmic elegance, the Skip List reveals its biggest blind spot when running on modern hardware. Contemporary CPUs rely heavily on spatial locality and hardware cache prefetchers.

Because zskiplistNode objects are scattered randomly across the heap, traversing a Skip List is a worst-case exercise in pointer chasing. When the CPU loads a node, most of the data in the 64-byte cache line containing that node goes to waste, and the address of the next node is entirely unpredictable. This triggers a storm of cache misses (L1, L2, L3) and TLB misses. The CPU pipeline is forced to stall for hundreds of clock cycles just waiting for the memory controller to fetch data from DRAM.

This is precisely the problem Redis tried to paper over with an intermediate structure called listpack.

The Listpack Alternative Encoding (Linear Super-Compression Optimization)

To fix the Skip List’s fatal CPU-cache weakness for small collections, Redis employs an adaptive alternative encoding mechanism. Below certain configured thresholds:

  • zset-max-listpack-entries (default 128)
  • zset-max-listpack-value (default 64 bytes)

Redis does not create a dict hash table or a zskiplist at all. Instead, the entire ZSET’s data is packed and stamped into a single contiguous physical byte stream: the listpack (inheriting the philosophy of the earlier ziplist).

In a listpack, element/score pairs sit sequentially adjacent to one another: [Header] [Element_1] [Score_1] [Element_2] [Score_2] ... [End]

Despite the algorithm degenerating from $mathcal{O}(log N)$ into a flat-array $mathcal{O}(N)$ search, the actual execution speed with a listpack is FASTER by tens of times. Why? A listpack chunk of a few hundred bytes fits neatly inside just a few L1/L2 cache lines. Scanning the array perfectly wakes up the hardware prefetcher, turning memory reads into speeds approaching register access. Reallocation operations are performed via realloc and memmove, fully exploiting the CPU’s Vector SIMD instruction sets. Only when the dataset grows past the threshold does the listpack irreversibly decompose and restructure itself into a Skip List.

Friction with the Operating System: COW (Copy-on-Write) and System Forks

In persistence strategies such as RDB Snapshotting or AOF Rewrite, the parent Redis process invokes the OS fork() call to spawn a background shadow process. Linux uses Copy-on-Write (COW) at the virtual memory paging layer.

Everything is ideal until write operations start hitting a ZSET backed by a Skip List. A single insertion into a Skip List changes the address of dozens of span and forward variables across numerous scattered nodes. Because these nodes are chaotically dispersed, one tiny insertion can trigger OS page faults, forcing the kernel to physically re-copy dozens of scattered 4KB memory pages. This phenomenon produces enormous memory-consumption spikes, putting server nodes at risk of being mercilessly executed by the Linux OOM Killer (Out Of Memory Killer). For this reason, the Linux overcommit_memory = 1 configuration parameter and a reserved RAM headroom ratio are mandatory.

Practical Applications & Case Studies

Thanks to the flexible $mathcal{O}(log N)$ power of the Skip List and its refined micro-level optimizations, Redis’s ZSET serves as a “silver bullet” for a great many modern system architecture workloads.

Real-time Streaming Leaderboards

In the eSports, gaming, or online e-commerce ecosystem, maintaining a constantly fluctuating leaderboard for tens of millions of users requires extremely high I/O throughput. Commands such as ZINCRBY (score update), ZREVRANGE (get Top K), and ZRANK (get a user’s current position) run smoothly with sub-millisecond latency. In particular, ZREVRANGE benefits enormously from the span metadata, which fully solves the problem of computing rank at lightning speed.

Sliding Window Rate Limiters

One of the most effective L7-layer anti-DDoS (distributed denial-of-service) systems is the Sliding Window. By storing a timestamp (Unix timestamp in microseconds) as the score and a UUID as the member value, the system can use the ZREMRANGEBYSCORE function to automatically purge old requests outside the time window (e.g., wiping out traffic older than 60 seconds), then use ZCARD to count the currently valid request frequency in an instant.

Time-Series Indexing

In IoT (Internet of Things) infrastructure, continuous telemetry data streams in from millions of sensors. Using ZSET, Redis manages events with a timestamp as the weight. Extracting a device’s data from time $T_1$ to $T_2$ becomes a cheap ZRANGEBYSCORE query with $mathcal{O}(log N + M)$ complexity, where the Skip List pointer immediately cuts through the entirety of the chaotic data straight to the $T_1$ starting position before sequentially scanning the base-level array in $mathcal{O}(M)$ to sweep up the measurement events.

Delayed Task Queues

Using the score as the scheduled execution timestamp (a future timestamp). Worker nodes continuously poll the system using ZRANGEBYSCORE -inf <current_timestamp> LIMIT 0 1. The Skip List guarantees that the latency for locating the next task always stays within $Theta(log N)$, keeping the worker node system’s lifecycle running smoothly.

Lessons Learned

From dissecting the ZSET data structure’s source code and surveying its low-level mechanisms, we can draw valuable engineering lessons broadly applicable across software engineering:

  1. Rejecting perfect complexity in favor of probabilistic performance: By accepting a random geometric distribution instead of a strict mathematical balancing rule, Redis built a system that is less error-prone, easier to maintain, has smoother response times, and completely eliminates bottlenecks caused by spin-locks.
  2. Locality effects rule the processor: No perfectly asymptotic algorithm can ever defeat the physical laws of the cache. The evolution toward the Listpack encoding is ironclad proof of the principle: when the dataset is small enough, a mindless sequential $mathcal{O}(N)$ scan over contiguous memory always crushes clever $mathcal{O}(log N)$ binary-search operations scattered across the heap.
  3. The hidden cost of a tangled pointer architecture: No matter how powerful an in-memory algorithm is, OS memory paging and COW (Copy-on-Write) remain the ultimate limiting factor. Understanding this friction helps SysAdmin engineers master the risks of OOM crashes when provisioning resources for massive distributed data clusters.
  4. Combining structures is a high art: Trading a small amount of memory to duplicate a referential graph structure — as with how dict combines with zskiplist — can shatter the ceiling of algorithmic complexity, pushing the entire database’s I/O limits into a new era of performance.
