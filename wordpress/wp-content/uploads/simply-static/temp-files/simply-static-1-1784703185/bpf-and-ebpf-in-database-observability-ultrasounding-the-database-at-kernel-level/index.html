BPF and eBPF in Database Observability: Ultrasounding the Database at Kernel Level

July 22, 2026 · admin · 8 min read

Summary & Core Problem Statement

In the era of Distributed Microservices and high-intensity transaction processing systems, the Database is often the largest bottleneck. Yet diagnosing failures and observing the performance (Observability) of a Database remains a formidable challenge.

Core problem: Traditional databases behave like “Black Boxes”. To find out why a SQL query is slow, engineers typically have to turn on the Slow Query Log, install in-band profiling agents, or fall back on classic operating-system tools such as strace or tcpdump.

  • Turning on logging or an in-band agent introduces overhead of 30-50%, crashing the system before the root cause is even found.
  • Using strace (built on the ptrace mechanism) requires the operating system to stop the Database process on every single System Call, inflating latency by hundreds of times.
  • Using tcpdump to capture network packets generates an enormous volume of junk data and burns excessive CPU copying data from Kernel Space up to User Space.

The emergence of eBPF (Extended Berkeley Packet Filter) triggered an architectural revolution in Observability. eBPF lets engineers run ultra-lightweight “ultrasound” code directly inside the operating system’s Kernel Space. eBPF can track every network packet crossing the NIC, every function call inside PostgreSQL, every block of data written down to an SSD… with overhead that is virtually ZERO (< 1%).

This in-depth technical report dissects the microarchitecture of the eBPF virtual machine, how to set up Uprobes/Kprobes to “hack” into a running Database’s source code without a restart, and the hard-won lessons from deploying eBPF observability infrastructure at data-center scale.

The Microarchitecture of the eBPF Virtual Machine

Originally designed to filter network packets (cBPF in the tcpdump tool), eBPF has today evolved into a general-purpose Virtual Machine (VM) running right at the heart of the Linux Kernel.

An Instruction Set Architecture (ISA) Mapped Directly to Hardware

The modern eBPF architecture is designed to map directly and optimally (one-to-one mapping) onto the hardware structure of 64-bit processors such as x86_64 and ARM64. The eBPF processor uses a model of 11 virtual 64-bit registers ($R0$ through $R10$):

  • $R0$: Holds the function’s return value.
  • $R1 – R5$: Hold the input arguments when calling a function.
  • $R6 – R9$: Callee-saved registers preserved across function calls.
  • $R10$: A read-only Frame Pointer used to access the Stack.

Just-In-Time (JIT) compilation converts eBPF bytecode into the CPU’s native machine code. Because the eBPF register architecture almost exactly mirrors the x86_64 register architecture, this JIT process happens at blazing speed, and the resulting machine code runs as fast as directly-compiled C code.

The Gatekeeper: The eBPF Verifier

The kernel is sacrosanct territory. If eBPF code contains a divide-by-zero bug, a NULL pointer, or an infinite loop, it could crash the entire operating system (Kernel Panic). To guarantee absolute safety, eBPF code must pass through the Verifier scanner. The Verifier uses static Control Flow Graph (CFG) analysis to guarantee:

  1. No infinite loops (eBPF code must be guaranteed to terminate — practically Turing-incomplete).
  2. No out-of-bound memory access.
  3. No access to forbidden Kernel memory regions.
  4. Maximum Stack size does not exceed 512 Bytes.

The Verifier’s complexity approaches $mathcal{O}(N times E)$ (where $N$ is the number of states and $E$ is the number of edges in the flow graph). Only once the Verifier stamps “SAFE” is the eBPF code allowed to be loaded into the JIT.

eBPF Maps: Ultra-High-Speed Data Sharing (Zero-Copy)

An eBPF program runs in the Kernel and collects a Query’s latency — but how does that data get sent up to User Space (e.g., a Prometheus or Datadog agent) without the cost of copying? The answer is eBPF Maps.

eBPF Maps are Key-Value data structures statically allocated in the Kernel’s non-pageable physical RAM. Both the eBPF program and the User Space application can Read/Write to this Map via a File Descriptor, entirely eliminating the data-copying System Call.

eBPF Hash Map (RCU-Protected)

Used to store state. Example: when an incoming Request packet arrives, eBPF writes Time_Start into the Hash Map keyed by the TCP_Tuple (IP, Port). When the Response packet leaves, eBPF reads the Hash Map to retrieve Time_Start, computes the Latency, and stores the result. The internal hashing mechanism uses Read-Copy-Update (RCU), guaranteeing fully lock-free concurrent reads and delivering access speeds of tens of millions of IOPS without CPU contention.

eBPF Ring Buffer

To stream events (Event Streaming) from the Kernel up to User Space (e.g., a list of slow queries), eBPF uses a Ring Buffer. This is a lock-free MPSC (Multi-Producer Single-Consumer) circular queue structure. CPU cores producing events write directly into the Ring Buffer. The User Space Agent uses polling (epoll) to continuously harvest the data. This design guarantees the Kernel is never blocked, even if User Space becomes overloaded.

Methods for Placing Monitoring “Toll Booths” (Probing)

How does eBPF know what the Database is doing? eBPF provides “hooks” that attach to the operating system’s bloodstream.

Network Hook (TC / XDP / Socket Filter)

If you don’t want to (or cannot) touch the Database process itself, you can monitor at the Network level. eBPF can attach to the Traffic Control (TC) layer or the eXpress Data Path (XDP) — the lowest level of the NIC driver. By parsing TCP packets according to the PostgreSQL/MySQL Wire Protocol, an eBPF program can precisely measure query latency, query counts, and network error rates (Retransmission) without the Database itself ever knowing. This method has extremely low overhead, but it is difficult to reassemble queries that are fragmented (TCP Fragmentation) across multiple packets.

User Probes (uprobes & uretprobes)

This is the ultimate weapon. Uprobes let eBPF “drill a hole” directly into C/C++ functions inside the Database’s source code (user space). For example, in PostgreSQL, we can attach a Uprobe at the entry of exec_simple_query() and a Uretprobe at that function’s return point.

The hardware-interrupt principle (int3): When eBPF attaches a Uprobe to exec_simple_query(), it silently replaces the function’s first machine opcode with an int3 (Breakpoint Trap) instruction. When the CPU executes into this function, the int3 instruction throws a Hardware Exception. The OS seizes control, jumps into the Kernel, and calls our eBPF handler. Once eBPF finishes logging, the CPU restores the original opcode and lets the Database continue running.

The Achilles’ heel of Uprobes: The cost of a single jump through a Uprobe (raising an Exception + Context Switch) is roughly 1.5 to 3.0 microseconds on a modern CPU.

  • If you attach a Uprobe to exec_simple_query (which only runs 10,000 times/second), the CPU only spends 30ms (equivalent to 3% overhead) — very safe.
  • If you attach a Uprobe to btr_search_leaf (traversing B-Tree leaf pages, running 1,000,000 times/second), you’ll create an “Interrupt Storm.” The CPU spends 3 seconds of work for every 1 real second elapsed — the Database hangs completely.

    • Always carefully measure a function’s Invocation Frequency before attaching a Uprobe!

Kernel Probes (kprobes)

Used to monitor the Database’s interaction with Disk and RAM. By attaching kprobes to the vfs_read() and vfs_write() functions, combined with a pattern-recognition model, eBPF can track the Page Cache Miss Ratio. Engineers can see straight through to how many times a MySQL SELECT statement actually had to dig down to an NVMe disk, and how many nanoseconds it spent in the Kernel’s Block Layer.

System Microarchitecture Optimization (NUMA & Cache)

To deploy an eBPF-based Observability system on a 128-core server cluster handling billions of transactions, the design of eBPF Maps must strictly obey the physical laws of the processor.

The NUMA Problem and Per-CPU Maps

In a NUMA (Non-Uniform Memory Access) architecture, a CPU trying to access another CPU’s RAM incurs enormous latency. If an eBPF program on CPU Core 0 and one on Core 64 both try to write into a single, global eBPF Hash Map, they will cause Memory Contention and saturate the interconnect bus (UPI/Infinity Fabric).

Solution: Use the Per-CPU Maps structure. The operating system allocates a separate space (a Local Copy) of the Map for each physical CPU core. eBPF programs running on a given core only write static data locally into that core’s slice of RAM. This turns every write into a Lockless and Contention-Free operation, with throughput reaching the physical limit of L1 Cache bandwidth.

Protecting the Instruction Cache (I-Cache Thrashing)

The processor’s L1 instruction cache (L1i) is very small (only 32KB). The database needs this space to hold its own complex logic (Query Planner, Execution Engine). If your eBPF program compiles down to too much machine code, every time it is invoked (via kprobe/uprobe) it will overflow into the L1i Cache, evicting the Database’s own instructions. When the Database resumes execution, it suffers a Cache Miss and must reload instructions from L2/L3, causing severe performance degradation (Stall Cycles).

BPF engineers must write extremely lean code: merge checks where possible, use unroll loops carefully, and keep the entire BPF program under 4KB so it can “coexist peacefully” with the Database inside the L1i Cache.

Lessons Learned & Best Practices for Systems Engineers

  1. Start with kprobes and Network hooks; be cautious with uprobes: Never test a uprobe for the first time in Production. Always use a Staging environment and tools such as bcc or bpftrace to measure function-call frequency (using the count() function) before rolling out latency collection.
  2. Flatten Data Structures (Flattening the FSM): eBPF has no Heap (no malloc). If you need to track a SQL transaction that spans multiple steps, build the Finite State Machine (FSM) in User Space. eBPF’s role is only to push each event signal up through the Ring Buffer. Don’t try to cram complex correlation logic into the Kernel.
  3. Manage Map Capacity (Bounded Capacity): Every eBPF Map has a max_entries limit. If the Hash table fills up, new Database transactions simply won’t be observed. Engineers need to write a User-space Agent fast enough to “clean up” (delete) stale Keys from the Map to make room for new data.
  4. Permissions and Security Lockdown: eBPF is capable of reading the Kernel’s entire RAM (including passwords or secret keys in flight). In Production, only users in the CAP_BPF or CAP_SYS_ADMIN group should be allowed to operate it. Use a Code Signing system to block any eBPF program of unknown origin.
  5. Visualization Tooling (Observability Stack): Don’t write a BPF Agent from scratch in C. Use an ecosystem such as Cilium, Tetragon, Pixie, or write in Golang with the cilium/ebpf library. Push raw data out to a Prometheus Histogram to take advantage of percentile charts (P99, P99.9) in Grafana.
