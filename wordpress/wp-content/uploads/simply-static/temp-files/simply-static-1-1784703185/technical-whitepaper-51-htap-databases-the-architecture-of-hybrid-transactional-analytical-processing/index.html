Technical Whitepaper #51: HTAP Databases – The Architecture of Hybrid Transactional/Analytical Processing

July 22, 2026 · admin · 8 min read

Executive Summary

This technical whitepaper provides an exhaustive, micro-architectural analysis of Hybrid Transactional/Analytical Processing (HTAP) database systems. Historically, the database industry accepted a hard bifurcation between OLTP (Online Transaction Processing) and OLAP (Online Analytical Processing). HTAP breaks this compromise. By exploring the mathematics of dual-format memory topologies (Delta-Main architecture), the intricacies of Multi-Version Concurrency Control (MVCC) across divergent data layouts, and the sheer computational power of SIMD vectorization and LLVM JIT compilation, this document equips systems engineers, database architects, and researchers with the deep technical knowledge required to understand and build next-generation unified data engines.

Introduction: The Historic Divergence of Data Processing

For over four decades, the database ecosystem has been characterized by a fundamental dichotomy: systems are either designed to write fast, or read fast.

  • OLTP (Online Transaction Processing): Systems like PostgreSQL or MySQL are optimized for high-frequency, low-latency, point queries and updates. They utilize row-oriented (N-ary Storage Model) layouts, ensuring that an entire record (e.g., a user profile) can be retrieved or mutated in a single disk I/O or cache line fetch.
  • OLAP (Online Analytical Processing): Systems like Snowflake, ClickHouse, or BigQuery are optimized for complex, read-heavy, long-running queries that aggregate massive volumes of data. They rely on column-oriented (Decomposition Storage Model) layouts, which maximize sequential memory bandwidth and allow for extreme data compression.

The problem? Businesses today require real-time analytics on live transactional data. The traditional approach to bridge this gap involves extracting data from the OLTP system, transforming it, and loading it (ETL) into the OLAP data warehouse. This creates an unacceptable latency window, turning “real-time analytics” into “yesterday’s analytics.”

HTAP is the architectural paradigm shift designed to unify these workloads into a single engine, eliminating the ETL pipeline while strictly maintaining performance isolation. But how does a single engine serve two fundamentally opposing masters?

The Core Problem: The Impedance Mismatch

1 The Analytical Penalty of Row Stores

When an OLAP query (e.g., SELECT SUM(salary) FROM employees) runs on a row-store, the CPU must load entire rows into its L1/L2 caches just to read a single salary attribute. This results in massive cache pollution (loading unneeded attributes like address or biography) and squanders memory bandwidth. The CPU becomes starved for data, bottlenecking the entire system.

2 The Transactional Inflexibility of Column Stores

Conversely, executing an INSERT or UPDATE on a column-store is an operational nightmare. A single logical row mutation requires seeking and writing to dozens or hundreds of disparate columnar files scattered across the disk. This scattered random I/O completely destroys transactional throughput, turning a microsecond operation into a millisecond operation.

3 The Cost of Data Movement (ETL)

Maintaining two separate databases requires an ETL pipeline. ETL is notoriously fragile, computationally expensive, and introduces a staleness bound $Delta T$ that often ranges from minutes to hours. In modern algorithmic trading, fraud detection, or dynamic pricing, an hour of staleness translates directly to millions of dollars in lost revenue.

The Unified Engine Solution: Delta-Main Architecture

To resolve the impedance mismatch, HTAP databases (such as TiDB, SingleStore, or SAP HANA) employ a multi-format, dual-storage architecture. Data logically resides in a unified schema but is physically instantiated in both row and columnar formats within the same system.

1 The Delta-Main Memory Topology

Because synchronously maintaining two physical representations for every microsecond mutation is prohibitively expensive, HTAP engines utilize an in-memory delta-main architecture.

  • Delta Store (Row-Oriented): All incoming write operations (INSERT, UPDATE, DELETE) are ingested into a highly concurrent, lock-free, row-oriented buffer in RAM. This absorbs high-velocity transactional mutations, ensuring sub-millisecond commits.
  • Main Store (Column-Oriented): The bulk of the historical data resides in a heavily compressed, read-optimized columnar format.

2 Asynchronous Compaction and Tuple Movers

To prevent the row-oriented Delta Store from consuming all available RAM and slowing down analytics, an asynchronous background pipeline (often called the Tuple Mover or Compactor) continuously runs. It takes a snapshot of the immutable rows in the Delta store, transposes them into a columnar format, applies heavy compression (Run-Length Encoding, Dictionary Encoding, Bit-packing), and flushes them into the Main Store.

The mathematical cost of an analytical query $C_{olap}$ across attribute $A$ involves scanning both:

$$C_{olap} = C_{column_scan}(N – Delta) + C_{row_scan}(Delta)$$

Because $C_{row_scan}$ is significantly more expensive per tuple than $C_{column_scan}$, the Tuple Mover’s algorithm must operate aggressively enough to keep $Delta$ small, but cautiously enough to avoid starving OLTP threads of CPU cycles and memory bandwidth.

3 Hardware Implications: NUMA, TLB, and Huge Pages

Building an in-memory HTAP system requires intimate awareness of OS and hardware limitations. Scanning terabytes of columnar data rapidly triggers Translation Lookaside Buffer (TLB) misses if standard 4KB OS memory pages are used. HTAP engines invariably configure the Linux kernel to allocate Huge Pages (2MB or 1GB), keeping the TLB resident and the page walker idle.

Furthermore, Non-Uniform Memory Access (NUMA) topologies dictate that if a thread on CPU Socket 0 attempts to scan columnar data residing in the RAM of CPU Socket 1, the data must traverse the QPI/UPI interconnect, adding parasitic latency and bottlenecking throughput. HTAP allocators pin specific column chunks to the NUMA node of the executing analytical threads, maximizing local memory bandwidth.

Multi-Version Concurrency Control (MVCC) in HTAP

Serving high-throughput writes and long-running reads concurrently on the same engine requires bulletproof isolation. Traditional Two-Phase Locking (2PL) is catastrophic here: a 5-minute OLAP read query would acquire shared locks on millions of rows, completely blocking OLTP writes for 5 minutes.

1 Snapshot Isolation via MVCC

HTAP uniformly leverages Multi-Version Concurrency Control (MVCC). Writers do not overwrite data in place; they append a new version of the tuple. Readers do not acquire locks; they read from an immutable, temporally consistent snapshot of the database corresponding to their specific logical read timestamp ($T_{read}$).

2 Visibility Resolution Across Dual Formats

The complexity arises when resolving visibility across the dual formats.

  1. The analytical query scans the heavily compressed Columnar Main Store. However, some of these tuples might have been updated or deleted in the Delta Store. The engine uses a Roaring Bitmap or an invalidation list to exclude these stale tuples.
  2. The query simultaneously scans the Delta Store. For every row, it evaluates the visibility predicate:

$$Visible(V) = (BeginTS le T_{read}) land (EndTS > T_{read})$$

Only the specific version of a tuple that satisfies this condition is merged into the final result set.

3 Garbage Collection and Watermarks

If an HTAP engine processes 10,000 updates per second, it generates 10,000 obsolete versions per second. The system maintains a global $T_{watermark}$ representing the oldest active transaction. A background Vacuum process continuously scans the Delta Store, physically deleting any tuple version where $EndTS < T_{watermark}$, reclaiming memory and preserving cache density.

Hardware-Accelerated Query Execution

To compensate for the overhead of MVCC and dual-format reconciliation, HTAP analytical engines bypass traditional tuple-at-a-time (Volcano) processing, utilizing modern CPU microarchitectures to their absolute limits.

1 Vectorized Execution and SIMD

Vectorization processes data in batches (chunks of 1024 or 4096 values). This amortizes virtual function dispatch overhead and keeps tight loops perfectly resident in the L1 Instruction Cache. Crucially, vectorization unlocks Single Instruction, Multiple Data (SIMD) capabilities (e.g., Intel AVX-512). The CPU can load 16 x 32-bit integers into a single 512-bit register and execute an arithmetic operation or predicate check (salary > 50000) on all 16 values in a single clock cycle. This yields a theoretical $16times$ speedup over scalar execution.

2 Just-In-Time (JIT) Compilation via LLVM

An alternative or complementary approach is LLVM-based JIT Compilation. Instead of executing pre-compiled generic C++ functions, the database dynamically writes and compiles bespoke machine code tailored specifically to the user’s SQL query at runtime. JIT compilation fuses multiple operators together, keeping data entirely within CPU registers and eliminating memory round-trips to the L1 cache altogether.

3 The Freshness vs. Performance Tradeoff

In HTAP, data freshness is a controllable spectrum. If an analyst can tolerate data that is 5 minutes stale ($Delta T = 5 text{ min}$), the Query Optimizer can completely bypass the row-oriented Delta Store and scan only the immutable Columnar Main Store. By avoiding the CPU-heavy visibility predicate checks and dual-format merging, the query executes orders of magnitude faster. This exposes a beautiful mathematical tradeoff: sacrificing microsecond-level freshness yields gigabytes-per-second increases in scan throughput.

Lessons Learned & Design Philosophy

The evolution of HTAP systems imparts several profound lessons for software and systems engineering:

  1. Unified Systems Are About Smart Tradeoffs: HTAP does not magically eliminate the laws of physics. It solves the OLTP/OLAP impedance mismatch by trading RAM capacity (storing dual formats) and background CPU cycles (Tuple Movers) for operational simplicity (no ETL) and real-time freshness.
  2. Data Freshness is a Spectrum, Not a Boolean: Modern systems should allow applications to negotiate their freshness requirements. Exposing the staleness bound $Delta T$ to the Query Optimizer allows for massive optimizations, proving that strict serializability is often an unnecessary bottleneck for analytical insights.
  3. Hardware-Software Co-Design is Mandatory: You cannot build a billion-row-per-second engine using generic POSIX abstractions. To achieve true HTAP performance, developers must write code that explicitly respects Cache Lines, NUMA boundaries, TLB page sizes, and SIMD registers. The software must bend to the physical realities of the silicon.

Conclusion

HTAP databases represent the pinnacle of modern data engineering, successfully dismantling the decades-old wall between operational and analytical systems. By orchestrating a delicate symphony of in-memory delta-stores, background columnar compression, lock-free MVCC, and SIMD vectorization, HTAP engines empower businesses to run complex machine learning models and aggregations on live data the very millisecond it is generated. As the volume and velocity of data continue to explode, the HTAP architecture—born from the absolute limits of hardware optimization—will become the undeniable standard for future database platforms.
